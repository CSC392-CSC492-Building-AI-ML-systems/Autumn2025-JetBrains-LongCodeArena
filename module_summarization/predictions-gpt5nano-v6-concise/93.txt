# Basics: Indexing Utilities – Generate documentation for this module

Overview
- This module provides a collection of utilities to manipulate and reason about structured NumPy arrays (recarrays). It includes helpers for:
  - Inspecting and listing field names (with support for nested structures)
  - Flattening and reconstructing field descriptions
  - Filling and recursing over structured fields
  - Composing and manipulating field data across arrays
- Historically contributed by John Hunter for Matplotlib; now generalized for convenience and reuse.

Intended audience
- Developers maintaining or extending the indexing utilities
- Contributors adding new recarray helpers or extending existing behavior
- Users integrating structured array workflows into libraries or applications

API surface (public)
The module exports a core set of utilities (as listed in __all__ in the source). The key public functions are:

- append_fields
  - Append new fields to a structured dtype or array.

- apply_along_fields
  - Apply a function along specified fields of a structured array.

- assign_fields_by_name
  - Assign or copy fields by name between structured arrays.

- drop_fields
  - Remove one or more fields from a structured array.

- find_duplicates
  - Find duplicated field/value combinations within a structured array.

- flatten_descr
  - Flatten a structured dtype description into a flat list of (field, dtype) pairs.

- get_fieldstructure
  - Retrieve the field structure/descriptor of a dtype (used for introspection).

- get_names
  - Return the hierarchical field names for a given structured dtype (preserves nesting).

- get_names_flat
  - Return flattened field names, including nested subfields, as a flat tuple.

- join_by
  - Join multiple arrays by a common key/field.

- merge_arrays
  - Merge multiple arrays into a single structured array.

- rec_append_fields
  - Recursively append fields to a structured array.

- rec_drop_fields
  - Recursively drop fields from a structured array.

- rec_join
  - Recursively join two or more structured arrays.

- recursive_fill_fields
  - Fill fields from one array into another, supporting nested structures.

- rename_fields
  - Rename fields within a structured dtype or array.

- repack_fields
  - Repack or reorganize fields within a structured dtype.

- require_fields
  - Ensure that required fields exist in a structured array; raise if missing.

- stack_arrays
  - Stack multiple structured arrays along a new axis.

- structured_to_unstructured
  - Convert a structured array to an unstructured (regular) ndarray.

- unstructured_to_structured
  - Convert an unstructured ndarray to a structured array.

Note: The actual module contains additional helpers and internal utilities (e.g., _recursive_fill_fields_dispatcher, _get_fieldspec, _zip_dtype) that support the public API and nested dtype handling.

Key internal helpers (for maintainers)
- _recursive_fill_fields_dispatcher(input, output)
  - Dispatcher for array_function overrides used by recursive_fill_fields.

- recursive_fill_fields(input, output)
  - Core helper that copies data from input into output by field, honoring nested dtypes.

- _get_fieldspec(dtype)
  - Produce a list of (name, dtype) pairs describing a dtype, including nested subfields.
  - Used to reconstruct dtypes or inspect nested structures.

- get_names(adtype)
  - Return a tuple of field names, preserving nested structure as tuples.

- get_names_flat(adtype)
  - Return a flat tuple of all field names, including nested ones, in depth-first order.

- flatten_descr(ndtype)
  - Flatten a structured dtype description into a sequence of (field, dtype) tuples, expanding subfields.

- _zip_dtype(seqarrays, flatten=False)
  - Internal helper to assemble a new dtype from a sequence of arrays, optionally flattening nested fields.

Examples of typical usage
- Inspecting field names (including nested fields)
  - Get hierarchical names:
    - adtype = np.dtype([('a', int), ('b', [('ba', int), ('bb', int)])])
    - names = get_names(adtype)
    - Result: ('a', ('b', ('ba', 'bb')))

  - Get flattened names:
    - adtype = np.dtype([('a', int), ('b', [('ba', int), ('bb', int)])])
    - flat_names = get_names_flat(adtype)
    - Result: ('a', 'b', 'ba', 'bb')

- Flattening a descriptor
  - ndtype = np.dtype([('a', '<i4'), ('b', [('ba', '<f8'), ('bb', '<i4')])])
  - flattened = flatten_descr(ndtype)
  - Result: (('a', dtype('int32')), ('ba', dtype('float64')), ('bb', dtype('int32')))

- Recursively filling fields
  - a = np.array([(1, 10.), (2, 20.)], dtype=[('A', np.int64), ('B', np.float64)])
  - b = np.zeros((3,), dtype=a.dtype)
  - recursive_fill_fields(a, b)
  - Result: array([(1, 10.), (2, 20.), (0,  0.)], dtype=[('A', '<i8'), ('B', '<f8')])

Generating and maintaining documentation (how to build)
- Tools: Sphinx, NumPy’s docstring conventions, and doctest support
- Typical workflow:
  - Ensure each public function carries a clear NumPy-style docstring (Parameters, Returns, Notes, Examples).
  - Use Sphinx with extensions such as:
    - sphinx.ext.autodoc
    - numpydoc or hymn to parse NumPy-style docstrings
    - sphinx.ext.doctest to validate code examples
  - Typical commands:
    - sphinx-quickstart to scaffold a docs directory
    - Edit conf.py to enable autodoc and numpy doc style:
      - extensions = ['sphinx.ext.autodoc', 'sphinx.ext.doctest', 'numpydoc']
      - add paths to your module so autodoc can import it
    - In your reST file (e.g., basics/indexing.rst):
      - .. automodule:: numpy.lib.recfunctions
         :members:
         :undoc-members:
         :show-internal:
      - Or specifically document groups:
        .. automodule:: numpy.lib.recfunctions as rfn
           :members:
           :undoc-members:
           :inherited-members:
    - Build HTML: make html or sphinx-build -M html . _build/html
    - Run doctests only: make doctest
- Best practices:
  - Keep docstrings up-to-date with public API changes (add/remove functions in __all__ consistently).
  - Include minimal, reproducible examples demonstrating typical use and edge cases (e.g., nested structures, subarrays).
  - Document dtype-related behavior explicitly (how nested dtypes are handled, subarrays, offsets).

Guidance for maintainers and contributors
- Extending the API:
  - When adding new functions to the public API (appearing in __all__), provide a clear docstring with:
    - Signature and parameter descriptions
    - Return value details
    - Examples that cover both simple and nested structured dtypes
- Internal utilities:
  - Keep internal helpers (e.g., _get_fieldspec, _zip_dtype) well-documented with comments describing their purpose and side effects.
  - Avoid leaking private helpers into docs unless intentionally used in public API or heavily relied upon by users.

Common pitfalls
- Nested field handling:
  - Ensure that nested structures are properly traversed and that operations preserve the original nesting while performing the intended manipulation.
- Subarrays and field offsets:
  - When flattening or reconstructing, be mindful of subarray shapes and field offsets; doc any nontrivial behavior in _get_fieldspec and flatten_descr.
- Doctest reliability:
  - Code examples should be self-contained, using explicit imports and consistent NumPy versions to remain portable across environments.
