Conditions in PyTables
=======================

Overview
--------
PyTables supports filtering data by expressing conditions on table columns. Conditions are supplied as strings and can be used with Table.where, Table.read_where, and related APIs. The condition language is designed to be NumPy-friendly and efficient, with special handling for NaN values and large datasets.

Key ideas
- Reference columns by their names in the table schema.
- Build boolean expressions using a small set of operators and parentheses for grouping.
- Use bitwise logical operators (&, |) to combine atomic predicates (instead of Python and/or), with explicit parentheses around each atomic condition.
- NaN-aware comparisons are applied automatically for numeric columns.
- Conditions can leverage the full range of supported data types (numeric, strings, booleans).

Where to use
- Table.where(condition) returns an iterator over rows satisfying the condition.
- Table.read_where(condition) returns a NumPy structured array containing rows that satisfy the condition.
- Condition strings can be reused across calls and benefit from PyTables’ internal condition caching.

Syntax and operators
- Atomic predicates
  - Supported comparison operators: ==, !=, >, >=, <, <=
  - Typical usage: column_name > 10, status == "active"
  - Strings must be quoted with single or double quotes: name == 'Alice'
- Logical combination
  - Use bitwise operators to combine predicates:
    - & means logical AND
    - | means logical OR
    - ~ means logical NOT (unary)
  - Always parenthesize atomic predicates when combining:
    - (age > 21) & (status == "active")
    - (score >= 80) | (grade == 'A')
- Parentheses
  - Group expressions to control evaluation order
  - Example: ((age > 18) & (age < 65)) | (role == 'admin')
- NaN handling
  - Numeric comparisons are NaN-aware; NaN values do not cause crashes and are treated consistently with PyTables’ internal rules.
  - If a column contains NaN values, results for comparisons involving NaN follow the numeric NaN semantics.
- Column references
  - Columns are referenced by their names as defined in the table schema.
  - For complex column names or path-like references, use the table’s standard access patterns to determine the visible column names for conditions.

Examples
--------
Simple condition
- Read rows where age is greater than 30:
.. code-block:: python

   results = table.read_where('age > 30')
   # results is a NumPy structured array with matching rows

Combined conditions (AND)
- Read rows where age > 30 and status is "active":
.. code-block:: python

   results = table.read_where('(age > 30) & (status == "active")')

Using OR
- Read rows where age < 18 or score >= 90:
.. code-block:: python

   results = table.read_where('(age < 18) | (score >= 90)')

Negation
- Exclude rows where flag is True:
.. code-block:: python

   results = table.read_where(~(flag == True))

String matching
- If your data stores categories as strings, you can compare directly:
.. code-block:: python

   results = table.read_where('(city == "New York") | (city == "London")')

NaN handling example
- Find rows where temperature is greater than 100, ignoring NaNs:
.. code-block:: python

   results = table.read_where('(temperature > 100)')

Iterating with a Where object
- The where() API yields Python tuples (row views) for memory-efficient iteration:
.. code-block:: python

   for row in table.where('(age >= 21) & (active == True)'):
       process(row)

Performance considerations
--------------------------
- Reuse of condition objects: Reusing the same condition string across multiple reads may benefit from internal caching mechanisms.
- Avoid unnecessary data copies: Prefer read_where for bulk retrieval and where for streaming/iteration over rows.
- Complex expressions: Break down very complex conditions into smaller parts if possible to aid readability and potential caching.

Common pitfalls
--------------
- Mixing Python and bitwise operators incorrectly (e.g., using and/or instead of &/|) can lead to exceptions or unexpected results.
- Forgetting parentheses around atomic predicates can lead to incorrect evaluation order.
- Referencing non-existent/aliased column names in the condition string results in a runtime error; ensure the condition uses valid table column names.

API notes
---------
- Condition strings are evaluated against the table’s column data. The same syntax applies across read_where and where; the latter yields an iterator while the former returns a NumPy array.
- NaN handling follows NumPy/PyTables’ standard practices for numeric comparisons; booleans and strings are treated normally.

Best practices
------------
- Always validate condition strings with a small, known subset of data before applying to large datasets.
- Use parentheses liberally to make the intended logic explicit.
- When filtering on string columns, keep quotes consistent and avoid embedding quotes within the string literal.
