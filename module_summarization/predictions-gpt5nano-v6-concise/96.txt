Getting Started with Generate
=============================

This guide provides a concise, practical introduction to Generate and walks you through a complete getting-started tutorial. It is intended for developers who want to scaffold projects and render files from templates using Generate.

Overview
--------

Generate is a command-line tool and library for creating project skeletons, code modules, and documents from reusable templates. It emphasizes:

- Fast scaffolding of common project layouts
- Template-driven rendering with variable substitution, conditionals, and loops
- Simple, repeatable workflows suitable for project bootstrapping and documentation generation

Prerequisites
-------------

- Python 3.8 or newer
- A working virtual environment (virtualenv, venv, or conda)
- Basic familiarity with the command line

Installation
------------

Option A: Install from PyPI (recommended for most users)

- Create and activate a virtual environment
  - python -m venv .venv
  - source .venv/bin/activate  (Unix/macOS)
  - .venv\\Scripts\\activate     (Windows)
- Install Generate
  - pip install generate

Option B: Install from source (if you are developing on Generate)

- Clone the repository
  - git clone https://example.org/generate.git
- Create and activate a virtual environment
  - python -m venv .venv
  - source .venv/bin/activate
- Install development dependencies
  - pip install -r requirements-dev.txt
- Install the package in editable mode
  - pip install -e .

Quick Start Tutorial
--------------------

This section walks you through a small, concrete example: bootstrapping a Python package and rendering a README file from a template.

Scenario: Create a Python package skeleton and generate a README

1) Initialize a new project

- Create a directory for your project and initialize Generate
  - mkdir my_package_demo
  - cd my_package_demo
  - generate init --name my_package_demo

  Expected outcome: A simple project skeleton is created under templates/ and an initial configuration is prepared.

2) Scaffold a Python package

- Create a template blueprint for a Python package (templates/python_package.tpl)
  - The template might look like:
    - contents/templates/python_package.tpl
      - {{ project_name }}/
      - ├── __init__.py
      - └── setup.py

- Use the scaffold command to generate the package structure
  - generate scaffold python_package --name my_package_demo

  Expected outcome: A directory structure resembling a typical Python package is generated, including placeholder __init__.py and setup.py files.

3) Render a README from a template

- Create a template for README (templates/README.md.j2)
  - Example content:
    - # {{ project_name }}
    - Version: {{ version }}
    - Description: {{ description }}

- Render the README with supplied variables
  - generate render --template templates/README.md.j2 --vars '{"project_name": "my_package_demo", "version": "0.1.0", "description": "Demo package generated with Generate."}' --out README.md

  Expected outcome: A populated README.md file appears in your project directory.

4) Inspect the results

- ls -la
- cat README.md

Templates and Variables
-----------------------

Generate templates are typically stored in a templates/ directory within your project or in a global templates repository. Templates use a Jinja2-like syntax for variable interpolation and control flow, including:

- Variables: {{ name }}, {{ project_name }}, etc.
- Conditionals: {% if condition %} ... {% endif %}
- Loops: {% for item in items %} ... {% endfor %}

Example template (templates/README.md.j2)

.. code-block:: text

    # {{ project_name }}
    
    Version: {{ version }}
    
    Description: {{ description }}

Variables can be provided in multiple ways:

- Inline JSON via the CLI:
  generate render --template templates/README.md.j2 --vars '{"project_name": "my_package_demo", "version": "0.1.0"}'
- From a YAML/JSON file:
  generate render --template templates/README.md.j2 --vars-file vars.yml

Project Layout Example
----------------------

A typical Generate-generated project might look like:

- my_project/
  - templates/
    - README.md.j2
    - python_package.tpl
  - src/
    - (generated package files)
  - README.md
  - setup.cfg

CLI Reference (Common Commands)
-------------------------------

- init
  - Initialize a new project scaffold with a default templates set.
  - Example: generate init --name my_project

- scaffold
  - Create a new code skeleton from a blueprint/template.
  - Example: generate scaffold python_package --name my_package

- render
  - Render a template file to an output path using provided variables.
  - Example: generate render --template templates/README.md.j2 --vars '{"name": "Alice"}' --out README.md

- list-templates
  - List available templates in the configured templates directory.
  - Example: generate list-templates

- help
  - Show help for a specific command.
  - Example: generate help render

Extending Generate
------------------

- Custom templates: Create additional templates under templates/ and reference them via your CLI.
- Hooks: Add pre/post render hooks to customize output or perform validation.
- Plugins (advanced): Write Python plugins to extend rendering behavior, e.g., custom filters or post-processing steps.

Examples (More Details)
-----------------------

- Example 1: Generate a minimal Python package skeleton
  - Template: templates/python_package.tpl
  - Steps:
    - generate init --name my_package
    - generate scaffold python_package --name my_package

- Example 2: Produce a configuration file from a template
  - Template: templates/config.tpl
  - Values: name, env, and port
  - Command:
    - generate render --template templates/config.tpl --vars '{"name": "serviceA", "env": "prod", "port": 8080}' --out config.yaml

- Example 3: Conditional content in templates
  - Template includes:
    - {% if include_tests %} tests_enabled: true {% else %} tests_enabled: false {% endif %}
  - Variables:
    - include_tests: true

Testing and Validation
----------------------

- Validate templates with local rendering before committing templates.
- Use a small sample data set to ensure substitutions occur as expected.
- Consider adding unit tests for custom template filters or hooks if you extend Generate.

Troubleshooting
---------------

- If render fails due to missing variables, supply the required values either via --vars or a vars file.
- If template syntax errors occur, check for mismatched tags or typos in Jinja-like syntax.
- Ensure your templates directory is correctly configured and accessible from the working directory.
