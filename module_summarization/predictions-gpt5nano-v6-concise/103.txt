# codegen.rst — Generate documentation for this code

This document describes the dioant.utilities.codegen module, which provides a structured, language-agnostic way to describe routines and translate them into target languages (C, Fortran, Octave/Matlab, etc.). It is intended for developers who want to understand, extend, or use the code-generation framework to produce compilable code from Diofant expressions.

## Overview

- The code generator is designed to produce complete, compilable code (not just a string representation of an expression).
- It separates concerns between the mathematical description (expressions, variables) and programming concerns (input/output handling, headers, libraries).
- It supports multiple backends (C, Fortran 77/90, Octave/Matlab) via language-specific CodeGen descendants.
- It provides a friendly API on top of a more rigorous Routine/CodeGen workflow.

## Key concepts

- Routine: A generic description of a computation that should be translated into code. It describes:
  - name: routine name
  - arguments: input and/or in-out arguments
  - results: return values or outputs produced by the routine
  - local_vars: variables defined within the function
  - global_vars: symbols that are not passed as arguments but are used by the routine
- Argument types:
  - InputArgument: an input to the routine (by value)
  - OutputArgument: an output produced by the routine, often defined by an expression
  - InOutArgument: an argument that is both input and can be modified (pass-by-reference-style)
- Result: a descriptor of a return value or an output variable, typically carrying an expression that defines its value.
- CodeGen: the base class responsible for translating Routine instances into target-language code.
- Language backends:
  - CCodeGen
  - FCodeGen
  - OctaveCodeGen
- Helpers:
  - codegen: a friendly API for common use-cases
  - make_routine: a helper to create Routine objects in common patterns
- Data types: DataType and related utilities describe how inputs/outputs map to language-level types (e.g., double, int).

## How the module differs from printing.ccode

- Code generation produces complete, compilable code (with headers and possible multiple files) rather than a single string representation.
- It handles
  - input/output semantics and function signatures
  - possible headers, includes, and library dependencies
  - transport of intermediate results across multiple expressions (e.g., through common subexpressions)
- It aims to evaluate constants at code-generation time when appropriate, rather than deferring all evaluation to runtime (a difference from simple pretty-printing).
- It supports generating code for multiple routines together and sharing intermediate results when beneficial.

## Basic assumptions

- A generic Routine data structure captures all features needed to emit code across supported languages.
- CodeGen descendants translate Routine instances into language-specific code.
- Friendly API helpers provide simpler workflows on top of the full Routine/CodeGen mechanism.

## API reference (high level)

- Routine
  - Generic description of a routine to be translated into code.
  - Validates that all needed symbols are provided by inputs, locals, or globals.
  - Exposes properties:
    - variables: all symbols that may appear in the routine
    - result_variables: list of variables that hold results
- DataType, default_datatypes, get_default_datatype
  - Mechanisms to describe and query language-specific data types for arguments/results
- Argument types
  - InputArgument
  - OutputArgument
  - InOutArgument
  - Result
- CodeGen, CCodeGen, FCodeGen, OctaveCodeGen
  - Translate Routine instances into code for C, Fortran, or Octave/Matlab
  - Provide methods to emit full source code (and possibly headers) for the target language
- Friendly API
  - codegen: ease-of-use entry point for common patterns
  - make_routine: helper to construct Routine objects in typical scenarios

Note: The exact constructor signatures for InputArgument, OutputArgument, InOutArgument, and Result are defined in the module. Use the documented behavior above to construct Routine instances correctly.

## Typical workflow

1. Build expressions using Diofant (or any Symbol/Expr objects you intend to compute).
2. Create Routine arguments:
   - Identify all inputs (InputArgument)
   - Identify outputs (OutputArgument) and their expressions
   - Optionally define InOutArgument where in-place updates are needed
3. Define local and global variables as needed.
4. Instantiate a Routine(name, arguments, results, local_vars, global_vars).
5. Select a target language backend by creating a CodeGen subclass (e.g., CCodeGen).
6. Generate code using the backend, producing:
   - header (.h) and source (.c) files for C
   - or corresponding files for Fortran or Octave backends
7. Integrate the generated code into your build system or runtime environment.

The following is a conceptual example to illustrate usage. Adjust to the exact constructor signatures in your environment.

```python
from diofant.utilities.codegen import Routine, InputArgument, OutputArgument, Result
# Suppose x is an input symbol and y is produced by an expression of x

# Define routine inputs/outputs
in_arg = InputArgument('x')
out_arg = OutputArgument('y', expr=expr_y)
# No local vars or globals in this simple example
routine = Routine(
    name='compute_y',
    arguments=[in_arg],
    results=[Result('y', expr=expr_y)],
    local_vars=[],
    global_vars=[]
)

# Generate C code (conceptual)
from diofant.utilities.codegen import CCodeGen
codegen = CCodeGen(routine)
generated_code = codegen.get_code()  # or appropriate API call to emit files

print(generated_code)
```

## Design notes and considerations

- Backends should encapsulate language-specific details (e.g., pointer semantics in C, array handling, naming conventions).
- The framework aims to separate high-level mathematical description from language-specific concerns (headers, includes, and I/O conventions).
- Future extensions may include:
  - Common Subexpression Elimination (CSE) across multiple routines
  - User-defined comments in generated code
  - Optional extra include lines for libraries
  - Support for additional languages (Python, C++, Fortran 90+)

## Extending and contributing

- To add a new backend, implement a new CodeGen subclass that knows how to emit code in the target language, respecting the Routine’s structure.
- Ensure the new backend handles argument modes (Input/Output/InOut) consistently with existing backends.
- Update DataType mappings to cover the new language’s types.
- Write tests that verify the emitted code compiles and matches expected behavior on representative inputs.
