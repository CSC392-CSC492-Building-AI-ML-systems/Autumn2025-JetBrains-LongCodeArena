# hyperexpand.rst — Generate documentation for this code

This document describes the purpose, structure, and usage of the
hyperexpand.rst module. The module is designed to generate Sphinx-ready
documentation for hypergeometric expansions by dynamically constructing a
documentation string from a collection of known formulas.

---

## Overview

- Purpose: Create a programmatic documentation source by iterating over a
  collection of hypergeometric formulas and turning each formula into a
  LaTeX/math block suitable for Sphinx documentation.
- Approach: Import a collection of hypergeometric formulas, wrap each formula
  in an equation representation, convert it to LaTeX, and accumulate these
  blocks into a single docstring assigned to `__doc__`.
- Result: When the module is imported, `__doc__` contains the generated
  documentation in reStructuredText with embedded math blocks.

---

## How the code works

- The module constructs a FormulaCollection instance:
  - `c = FormulaCollection()`
- It builds an empty string to accumulate documentation:
  - `doc = ''`
- It loops through the formulas in the collection:
  - `for f in c.formulae:`
- For each formula:
  - It creates a target equation using a hypergeometric function wrapper:
    - `obj = Eq(hyper(f.func.ap, f.func.bq, f.z), f.closed_form.rewrite('nonrepsmall'))`
  - It converts the equation to LaTeX and embeds it in a Sphinx math block:
    - `doc += f'.. math::\n  {latex(obj)}\n'`
- The final documentation string is assigned to `__doc__`:
  - `__doc__ = doc`

- Operators and reduction strategies (A_J, B_J, C_J, D_J, and their inverses).
- Order reduction criteria and reachable index quadruples.

## API / Data Structures

- `c` (FormulaCollection): A collection of hypergeometric formulas used to generate documentation blocks.
- `doc` (str): Accumulated reStructuredText containing math blocks for each formula.
- `__doc__` (str): The generated documentation string after processing all formulas.
- `_mod1(x)` (function): Helper used for modular bucket handling. It ensures consistent treatment for numeric versus symbolic inputs.
- `add_formulae(formulae)` (function): Populates the knowledge base of formulas. Exposed for extension and customization.
  - Inner helpers:
    - `add(ap, bq, res)`: registers a single formula.
    - `addb(ap, bq, B, C, M)`: registers a formula with matrix-based payloads.
- Related classes and imports (used by the generator, not defined here):
  - `Eq`, `hyper`, `latex`: utilities to construct and render equations.
  - `HyperRep_*`, `Hyper_Function`, and other hypergeometric helpers.
  - `Formula`, `Matrix`, and various symbolic components.

## How to use

The module is intended to be imported so that it generates its documentation at import time:

- Importing the module executes the generation and assigns the result to `__doc__`.
- You can access the generated documentation after import via the module’s `__doc__`:
  - Example:
    - import hyperexpand.rst as gen
    - print(gen.__doc__)  # prints the generated reStructuredText with math blocks

Because the generator runs at import time, the generated documentation reflects the current set of formulas in `FormulaCollection` and the existing formula definitions.

## Development and customization

- Purpose of extension:
  - The generator centralizes the creation of documentation blocks for hypergeometric expansions, ensuring consistency and re-use of LaTeX rendering via `latex(obj)`.
- How to extend:
  - Edit `add_formulae(formulae)` to register new formulas.
  - Use the existing `add` and `addb` helpers to add simple and matrix-based results, respectively.
- Testing the generator:
  - Import the module and verify that `__doc__` contains the expected `.. math::` blocks.
  - Validate that LaTeX strings render correctly in Sphinx by building the docs.

---

## Design notes

- Decoupled formatting: The code separates the mathematical construction (`Eq`, `hyper`, `f.closed_form`, `rewrite`) from presentation (`latex` + `.. math::` blocks).
- Extensibility: The known formulas are registered via `add_formulae(...)` with an approachable API (`add` and `addb`), enabling developers to grow the knowledge base without altering the core generator logic.
- Documentation strategy: The module doubles as both a generator and a source of documentation content, making it a single point of truth for the set of hypergeometric expansions.
