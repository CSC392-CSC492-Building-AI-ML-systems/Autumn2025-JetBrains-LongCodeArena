# Library Conventions

This document defines the standard conventions used across the library to ensure consistency, readability, and maintainability for developers contributing code, documentation, and tests.

---

## Scope

- Applies to Python modules within the library (packages, subpackages, and top-level modules).
- Covers code style, naming, API design, documentation, testing, and licensing headers.
- Aims to balance readability with practical performance considerations common in control/embedded-system libraries.

---

## General Philosophy

- Favor clarity and explicitness over cleverness.
- Use clear, descriptive names following established Python conventions.
- Document public APIs with reStructuredText (reST) docstrings compatible with Sphinx and cross-references.
- Maintain backward compatibility where feasible; deprecations should be announced and documented.

---

## File and Package Structure

- Packages use a conventional Python layout with __init__.py as the package initializer.
- Public API is explicitly exported via __all__ in modules to control namespace exposure.
- Submodules implement cohesive functionality; avoid large monolithic modules.
- Relative imports are used within packages (e.g., from .module import Class).

Example pattern (from a module like dtime.py):
- Exported symbols:
  __all__ = ['sample_system', 'c2d']
- Public classes and functions documented in docstrings.
- Internal helpers prefixed with a single underscore (_) to indicate non-public usage.

---

## Naming Conventions

- Functions and variables: snake_case (e.g., sample_system, prewarp_frequency).
- Classes: CamelCase (e.g., OptimalControlProblem, StateSpace).
- Modules: lowercase with underscores (e.g., dtime.py, optimal.py).
- Constants: UPPER_SNAKE_CASE (for module-level constants that are truly constant).

---

## Docstrings and API Documentation

Public APIs must be documented with reST-style docstrings suitable for Sphinx.

Key guidance:
- Include a concise summary line.
- Use NumPy/SciPy-inspired sections: Parameters, Returns, Notes, Examples, See Also, References.
- Use cross-references for types and related symbols, e.g., :class:`StateSpace`, :class:`TransferFunction`.
- Include practical examples that demonstrate typical usage and edge cases.

Example docstring structure (inspired by existing code):

- Parameters: list parameter name, type, and constraints.
- Returns: description of the returned object and its type.
- Notes: additional details, limitations, or platform considerations.
- Examples: short, runnable examples with >>> prompts when appropriate.

Code example snippet (stylized):

```python
def sample_system(sysc, Ts, method='zoh', alpha=None, prewarp_frequency=None):
    """
    Convert a continuous time system to discrete time by sampling.

    Parameters
    ----------
    sysc : LTI (:class:`StateSpace` or :class:`TransferFunction`)
        Continuous time system to be converted.
    Ts : float > 0
        Sampling period.
    method : str
        Sampling method to use (e.g., 'bilinear', 'zoh').
    alpha : float, optional
        Generalized bilinear transformation weighting parameter (used with 'gbt').
    prewarp_frequency : float, optional
        Frequency in rad/s for prewarping (used with 'bilinear').

    Returns
    -------
    sysd : LTI
        Discrete-time system of the same class as the input, sampled at Ts.

    Notes
    -----
    See :meth:`StateSpace.sample` or :meth:`TransferFunction.sample` for details.

    Examples
    --------
    >>> sysc = TransferFunction([1], [1, 2, 1])
    >>> sysd = sample_system(sysc, 1, method='bilinear')
    """
```

Public modules should clearly document the behavior and parameters, including any constraints or defaults.

---

## API Design and Behavior

- Functions should validate inputs and raise clear exceptions (e.g., ValueError) for invalid arguments.
- Return types should be explicit in the docstrings; where practical, state the dynamic type (e.g., LTI, StateSpace).
- The code should delegate complex behavior to well-defined methods or subcomponents, minimizing duplication.
- If a function is a thin wrapper around another, document the wrapper behavior and note any parameter passthrough.

Example pattern:
- A function delegates to a class method (e.g., sysc.sample(...)) and documents the expectation that the input is a continuous-time system.

---

## Coding Style

- Follow PEP 8 for Python style: indentation (4 spaces), reasonable line length, and clear imports.
- Use relative imports within packages (e.g., from .statesp import StateSpace).
- Keep modules focused and cohesive; avoid mixing unrelated functionality.
- Use descriptive, zero-ambiguity error messages.

---

## Testing and Quality

- Tests reside in a parallel structure under a tests/ directory or equivalent.
- Test modules should reflect the same import patterns as the source to catch packaging issues.
- Use pytest (or the project’s chosen framework) and include at least one test per public API edge case.
- Include regression tests for previously reported bugs.
- Ensure tests are fast and deterministic.

---

## Licensing and Headers

- Each source file should begin with a standard license header, reflecting the project’s license.
- Headers should be preserved in derivative works.
- Attribution and copyright must be clear.

Example header (as seen in the code sample):
- A BSD-style license with author, date, and standard warranty disclaimer.

---

## Documentation Patterns

- Provide module-level documentation where appropriate to explain the module’s purpose and design choices.
- Use reST roles for types and cross-references (e.g., :class:`StateSpace`).
- Include a concise “Examples” section to showcase typical usage patterns.
- For complex modules, include a short “Algorithm overview” or “Design rationale” subsection as Notes.

Example of a module-level docstring pattern:

- Class and function docstrings should be self-contained and describe how to use them, not just what they are.

---

## Examples

- Demonstrate typical use with minimal runnable code.
- Include edge-case examples (e.g., invalid input handling) as separate snippets.
- Use consistent example style with >>> prompts and expected outputs when helpful.

Example snippet:

```python
# Typical usage pattern
from control import sample_system, c2d
from control.statesp import StateSpace
from control.transfer_function import TransferFunction

sysc = TransferFunction([1], [1, 2, 1])
sysd = sample_system(sysc, Ts=0.5, method='bilinear')
```
