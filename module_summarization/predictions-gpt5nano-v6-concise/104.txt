diophantine.rst

Introduction
------------
This module provides the public interface for solving Diophantine equations within the project.
The primary entry point is the diophantine(eq, param, syms) function, which simplifies a
Diophantine equation by transforming it into a product of terms that must be equal to zero,
solves each term independently, and then merges the solutions to produce a complete solution
set.

Key ideas
---------
- A Diophantine equation is converted to a product-of-terms form. Each term is solved with
  diop_solve, and the solutions are merged in a way that respects the structure of the
  equation (linear, quadratic, sum of squares, etc.).
- If the equation is rational (n/d = 0), the solver enforces n = 0 and d != 0 (to avoid
  division by zero). This is implemented by solving n = 0 and removing any solutions that also
  render d = 0.
- The solver relies on classification of subexpressions into known Diophantine types and on
  factoring strategies. Some expressions may be left un-factorized to avoid hangs (notably
  for certain sum-of-squares or higher-degree forms).
- The function returns a set of tuples, where each tuple contains the solutions for the
  variables, ordered by the alphabetic ordering of the variables (or a user-specified syms
  order).

Public API
----------
diophantine(eq, param=Symbol('t', integer=True), syms=None)

- eq: Relational or Expr
  An equation (or expression) to be solved in integers. If an Eq is provided, it is treated as
  eq.lhs - eq.rhs == 0.
- param: Symbol
  The parameter symbol used in the parametric family of solutions. By default, a fresh integer
  parameter t is used (Symbol('t', integer=True)).
- syms: sequence of Symbol
  Optional. If provided, determines the order of variables in the solutions returned. The
  elements of the solution tuples are arranged according to the alphabetic ordering of the
  variables by default. If syms is given, only those variables present in the equation are used
  and the resulting tuples are reordered to match the order of syms in the input.

Returns
-------
A set of tuples. Each tuple contains the solutions for the variables in the equation, in
the order defined by syms (or the natural alphabetical order if syms is not provided).

Notes
-----
- The function can raise TypeError if the equation is not a polynomial with rational
  coefficients or if the classification logic cannot handle the expression.
- The implementation tries to avoid expensive factoring steps in some common cases to prevent
  hangs (e.g., certain generalized sum-of-squares forms).

Internal helpers (private)
--------------------------
The module defines several small helper utilities used by diophantine():

- _is_int(i)
  Returns True if i can be treated as an integer (via as_int). Otherwise False.

- _sorted_tuple(*i)
  Returns a sorted tuple of the given elements.

- _remove_gcd(*x)
  Divides all entries by their greatest common divisor and returns the reduced tuple.
  If the inputs cannot be gcd-ed (TypeError/ValueError), the originals are returned.

- _rational_pq(a, b)
  Returns a pair (numer, denom) representing a/b with gcd removed and sign consolidated
  in the numerator.

- _nint_or_floor(p, q)
  Returns the nearest integer to p/q; in case of a tie, returns floor(p/q).

- _odd(i), _even(i)
  Convenience predicates to check parity of an integer.

Example usage
-------------
The following examples illustrate typical usage and expected results.

Basic two-variable difference of squares
- Expression: x**2 - y**2
- Result:
  { (t_0, -t_0), (t_0, t_0) }

Code:
- diophantine(x**2 - y**2)

Three-variable linear form
- Expression: x*(2*x + 3*y - z)
- Result:
  { (0, n1, n2), (t_0, t_1, 2*t_0 + 3*t_1) }

Code:
- diophantine(x*(2*x + 3*y - z))

Quadratic with mixed terms and linear component
- Expression: x**2 + 3*x*y + 4*x
- Result:
  { (0, n1), (3*t_0 - 4, -t_0) }

Code:
- diophantine(x**2 + 3*x*y + 4*x)

Where n1, n2, t_0, t_1, etc., denote new integer parameters introduced by the solver.

Implementation notes
--------------------
- diophantine() optionally accepts syms to fix the variable ordering in the return value. If
  syms is provided, only those symbols present in the equation are included, and the resulting
  tuples are ordered to correspond to syms.
- When the equation is a rational expression n/d, the solver computes solutions to n = 0
  that do not make d = 0. This ensures 0 is achieved without division by zero.
- The function relies on:
  - classify_diop to determine the Diophantine type of terms
  - diop_solve to solve base Diophantine terms
  - factor_list or a controlled factoring path to prepare the expression into solvable terms
  - _mexpand to verify the candidate solutions against the original denominator

Related concepts and components
-------------------------------
- diophantine types (diop_known): a collection of known specialized Diophantine forms such as
  'binary_quadratic', 'cubic_thue', 'general_pythagorean', 'general_sum_of_squares',
  'homogeneous_general_quadratic', 'linear', 'univariate', etc.
- diop_solve: The dedicated term-level solver used to find integer solutions for a base
  Diophantine expression. This function is invoked for each term identified by the
  classification logic.
- classify_diop: Internal function used to classify terms into Diophantine types to select
  the appropriate solving strategy.
- _mexpand: Utility to expand and verify solutions when substituting back into the original
  equation or its components.

Examples (additional)
---------------------
- Solve x + y = 0
  diophantine(x + y)

Expected result (symbolic parameterization):
  {(t_0, -t_0)}

- Solve x*(x + y) = 0
  diophantine(x*(x + y))

Notes
-----
- The solver expects polynomial expressions with rational coefficients. Expressions outside this
  scope may raise TypeError.
- For some complex equations, factoring or classification may be expensive or fail;
  in such cases the function raises an exception or returns a partial set of solutions.

Migration and compatibility
---------------------------
- The interface is designed to be stable within the project. It relies on the project’s core
  symbolic machinery (core, ntheory, polys, and solvers submodules). Any changes in those
  components may impact the behavior of diophantine().
