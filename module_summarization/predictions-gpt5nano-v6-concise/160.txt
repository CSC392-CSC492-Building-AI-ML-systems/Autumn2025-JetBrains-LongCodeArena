# Compatibility Policy

This document defines the Compatibility Policy for the compatibility.rst file and for the project components derived from the provided code (npyrandom, numpy.random extensions, and related build artifacts). It explains what compatibility means for this project, how we enforce it, and how developers should treat deprecations, API changes, and platform differences. It includes practical examples to help implementers and contributors.

---

## 1. Overview

- Purpose: Ensure stable and predictable behavior across Python versions, NumPy versions, platforms, and dependent components while allowing evolution through deprecation and well-defined updates.
- Scope: Covers API/ABI compatibility, binary compatibility, Python/NumPy version compatibility, and build-packaging compatibility for the numpy.random ecosystem (including C/C++ extensions, Cython extensions, and static libraries used by the extension modules).

---

## 2. Key Definitions

- API compatibility: Changes to public functions, classes, and interfaces that would affect user code. If we change an API in a non-backward-compatible way, we must provide a deprecation path or a clear migration guide.
- ABI compatibility: Binary interface stability that affects linking against compiled extensions. Changes that require recompilation by downstream users (e.g., symbol changes, struct layout changes) are restricted.
- Platform compatibility: Behavior and build/run stability across supported OSes (Linux, macOS, Windows/Cygwin). Includes handling of platform-specific flags and linker behavior.
- Deprecation: Marking API elements as deprecated with a migration path and a defined removal timeline.
- Compatibility matrix: A matrix describing supported combinations of Python version, NumPy version, and platform.

---

## 3. Compatibility Levels

- Public API compatibility: Preserve public function names, signatures, and behavior. Introduce deprecated aliases when removing or altering functionality.
- Binary/ABI compatibility: Prefer stable ABIs for extension modules where possible. When necessary, communicate required recompilation and provide migration notes.
- Python/NumPy compatibility: Support a defined set of Python versions and NumPy versions. Avoid hard dependencies on features that are not universally available in the supported matrix.
- Build/packaging compatibility: Ensure the build system produces stable outputs that can be consumed by downstream environments without unexpected behavior across platforms.

---

## 4. Policy Details

### 4.1 API Changes and Deprecations

- Deprecate before removal:
  - Mark the old API with a clear deprecation warning (DeprecationWarning or UserWarning in user-visible context).
  - Provide a recommended replacement (new API) and document the change in release notes.
  - Maintain the old API for at least one major/minor version cycle (e.g., if a major version is v1, deprecate in v1.4 and remove in v2.0, subject to project policy).
- Introduce new features via non-breaking changes:
  - Prefer additive changes (new functions, optional parameters with default values, new classes) while keeping existing behavior intact.
- Backward-compatible defaults:
  - Do not switch default behaviors in a way that changes existing user expectations without deprecation warnings and migration notes.

### 4.2 Deprecation Timeline

- Documented deprecation period: Minimum release cycle(s) between deprecation and removal.
- Clear migration path: Provide examples and guidance to update user code.
- Visible warnings: Emit a deprecation warning at runtime for deprecated API usage.

### 4.3 Versioning Alignment

- Version scheme: SemVer-like approach is recommended (MAJOR.MINOR.PATCH) with:
  - MAJOR: breaking changes including deprecations/removals.
  - MINOR: new features and non-breaking enhancements.
  - PATCH: bug fixes and small improvements.
- Compatibility notes: Release notes must include explicit compatibility statements:
  - Supported Python versions.
  - Supported NumPy versions.
  - Any known breaking changes or deprecations.

### 4.4 ABI and C/C++ Interfaces

- Stable exported symbols: Keep public C API stable where feasible; avoid renaming or removing exported symbols unless absolutely necessary.
- If ABI changes are unavoidable:
  - Document the change clearly.
  - Provide a compatibility alias or wrapper if possible.
  - Require downstream users to recompile extensions against the new ABI.
- Build-time consistency:
  - Use static libraries where appropriate to limit external ABI drift.
  - For cygwin/Windows, handle symbol export pragmatically (e.g., explicit export flags) and document any platform-specific requirements.

### 4.5 Python/NumPy Compatibility

- Minimum versions:
  - Define and publish minimum supported Python and NumPy versions for each release.
- NumPy C-API usage:
  - If using deprecated NumPy C-API features, document which APIs are used and plan migration to modern APIs.
  - When possible, set NPY_NO_DEPRECATED_API to an appropriate level (e.g., 0 for compatibility with older NumPy behavior); adjust as the project’s compatibility stance evolves.
- Testing against matrix:
  - Maintain a compatibility matrix in CI that exercises the code against a representative set of Python and NumPy versions.

### 4.6 Build and Packaging Compatibility

- Build system conventions:
  - Document how to build and test across platforms (Linux/macOS/Windows/Cygwin).
  - Provide platform-specific flags and tricks (e.g., Windows export behavior, Cygwin linker handling).
- Dependencies:
  - Pin external dependencies where possible, and document minimum/maximum supported versions.
- Packaging:
  - Ensure the installed artifacts (static libraries, Python extension modules, stubs, tests) are consistently organized to avoid path ambiguity and ensure reliable imports.

---

## 5. Practical Guidelines for Developers

- Before introducing a breaking change:
  - Prepare a deprecation plan with a migration guide.
  - Add a deprecation warning to affected APIs.
  - Provide a compatibility shim if feasible.
- When adding new features:
  - Add tests to cover both new and existing behavior.
  - Ensure the default behavior remains unchanged for existing users unless explicitly requested.
- For C/C++/Cython code:
  - Keep exported symbols stable where possible.
  - Document any changes to the interface publicly (e.g., in release notes).
- For Windows/Cygwin:
  - Use explicit symbol export strategies if needed (e.g., export-all-symbols when required) and document rationale and any platform-specific caveats.

---

## 6. Testing and Validation

- Compatibility tests:
  - Create a dedicated compatibility test suite that validates:
    - API usage remains stable across supported versions.
    - Correct behavior when deprecated APIs are present (and warnings appear).
    - Basic functionality of new features does not break existing usage.
- CI matrix:
  - Run tests across multiple Python versions and NumPy versions as defined in the compatibility matrix.
  - Include at least one Windows/Cygwin job if Windows support is in scope.
- Data and interface contracts:
  - If test data formats are consumed or produced (e.g., test sets for RNGs), ensure the formats are stable and document any version-specific expectations.

---

## 7. Migration and Release Strategy

- Release notes:
  - Clearly document compatibility expectations for each release.
  - Include a deprecation timeline, migration steps, and example code migrations.
- Migration guides:
  - Provide step-by-step upgrade guidance from older major/minor versions to the latest.
  - Include common pitfalls and platform-specific notes.

## 8. Examples

### 8.1 Deprecating an API with a Migration Path

Before:
- A public function named `legacy_distributions()` is removed or replaced.

After:
- Mark `legacy_distributions()` as deprecated and route calls to `modern_distributions()`.

Python example (runtime warning):
```python
import warnings

def legacy_distributions(*args, **kwargs):
    warnings.warn(
        "legacy_distributions is deprecated and will be removed in v2.0. "
        "Use modern_distributions instead.",
        DeprecationWarning,
        stacklevel=2
    )
    return modern_distributions(*args, **kwargs)
```

Migration:
- Update user code to call:
```python
from npyrandom.distributions import modern_distributions

modern_distributions(...)
```

### 8.2 Adding a New API Without Breaking Existing Users

- Add a new function with a non-breaking default and keep existing functions unchanged.
```python
def sample_from_distributions(n=1, *, mode='auto', rng=None):
    # New feature with a safe default
    pass
```

- Document the new parameter and behavior, but keep the old APIs intact.

### 8.3 Handling Platform Differences

- Document and test platform-specific behavior (e.g., Windows/Cygwin symbol export) and provide platform-specific notes in release docs.

Example CI note:
- On Windows, you may need to enable explicit symbol export for dynamic extensions:
  - Use appropriate linker flags (e.g., export-all-symbols) only when required, and document the rationale and impact.

### 8.4 Minimum Version Enforcement (Build/Runtime)

Check at runtime and/or build time that the environment meets minimum requirements.

Python example:
```python
import sys
import numpy as np

MIN_PY_VERSION = (3, 8)
MIN_NUMPY_VERSION = (1, 21)

if sys.version_info < MIN_PY_VERSION:
    raise SystemExit(f"Incompatible Python version: {sys.version}. "
                     f"Requires Python {MIN_PY_VERSION[0]}.{MIN_PY_VERSION[1]}+.")

np_version = tuple(map(int, np.__version__.split('.')[:2]))
if np_version < MIN_NUMPY_VERSION:
    raise SystemExit(f"Incompatible NumPy version: {np.__version__}. "
                     f"Requires NumPy {MIN_NUMPY_VERSION[0]}.{MIN_NUMPY_VERSION[1]}+.")
```
