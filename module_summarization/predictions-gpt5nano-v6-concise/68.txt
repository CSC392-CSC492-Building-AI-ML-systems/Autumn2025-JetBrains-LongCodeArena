# Gaussian Process Kernels (gaussian_process.rst)

This module provides the building blocks for kernel-based Gaussian process (GP) regression and classification. The kernels implemented here are designed to be highly composable and tunable, enabling kernel engineering through simple arithmetic and exponentiation.

Key features:
- Combine kernels with addition, multiplication, and exponentiation (via **).
- Automatic conversion of scalar values into constant kernels when composing expressions.
- Analytic gradient-based hyperparameter optimization for all kernels.
- Flexible hyperparameter specification with bounds or a “fixed” designation.
- Hyperparameters can be scalar or vector-valued (e.g., anisotropic length-scales).
- Hyperparameters are exposed via a compact, memory-efficient representation.

This documentation is aimed at developers extending or using this module to implement custom kernels and integrate them into Gaussian process models.

---

## Core concepts

- Kernel composition
  - Kernels can be combined using:
    - Addition: A + B
    - Multiplication: A * B
    - Exponentiation: A ** scalar
  - If a scalar is used in an expression, it is automatically converted to a constant kernel.
- Hyperparameters
  - Each kernel exposes a set of hyperparameters to control its behavior (e.g., length-scales, variance).
  - Hyperparameters are described by the Hyperparameter namedtuple with fields:
    - name: parameter name (e.g., "length_scale")
    - value_type: type of hyperparameter (currently "numeric")
    - bounds: 2D array of [lower, upper] bounds or the string "fixed"
    - n_elements: number of elements (scalar = 1; anisotropic parameters may be >1)
    - fixed: whether the hyperparameter is fixed (immutable during optimization)
- Optimization
  - All kernels support analytic gradient-based hyperparameter optimization.
  - The search space is rectangular, defined by lower and upper bounds for each hyperparameter.
  - If a hyperparameter is marked fixed, it is excluded from optimization.

- Hyperparameter metadata
  - Hyperparameters are accessible through kernel.hyperparameters.
  - Bounds can be specified as numeric ranges or as "fixed".
  - When n_elements > 1, bounds must be provided as a pair of arrays with compatible shapes.

## Hyperparameter (Hyperparameter)

The Hyperparameter class captures the specification of a kernel hyperparameter. It is a namedtuple with the following fields:
- name (str): The parameter name. For a kernel with a hyperparameter named "x", the kernel should expose attributes self.x and self.x_bounds.
- value_type (str): The type of the hyperparameter. Currently supports "numeric".
- bounds: A pair of floats (or a 2D array for vector-valued parameters) in the form (lower, upper) or "fixed".
- n_elements (int, default=1): Number of elements in the hyperparameter value. 1 for scalar, >1 for vector-valued hyperparameters.
- fixed (bool): Whether the parameter is fixed (not optimized). If None, this is inferred from bounds.

Key implementation notes:
- Bounds handling
  - If bounds is not the string "fixed", bounds are coerced to a 2D array.
  - For vector-valued parameters (n_elements > 1), bounds must be either:
    - A single row repeated for each element, or
    - A separate row per element.
- Equality
  - Hyperparameter implements __eq__ to allow easy testing of hyperparameter equality.

Example usage (conceptual):
- A ConstantKernel with bounds on the constant value:
  - Hyperparameter(name="constant_value", value_type="numeric", bounds=[[0.0, 10.0]], n_elements=1, fixed=False)

Notes:
- The class is designed to be memory-efficient. It uses a namedtuple with __slots__ and a specialized __new__ to enforce bounds shape and fixed semantics.

## Kernel base class

The Kernel base class serves as the foundation for all kernel implementations. It is an abstract base class (ABC) and defines common semantics and utilities used by concrete kernels.

Key methods (as provided by the base class):
- get_params(self, deep=True)
  - Returns a dictionary of parameter names mapped to their values.
  - Implementation strategy:
    - Inspects the signature of the kernel’s __init__ to identify parameters.
    - For each constructor parameter (excluding self and any varargs/kwargs), includes it in the params dictionary.
  - Important behavioral notes:
    - The code enforces that kernels do not use varargs in their __init__; otherwise, a RuntimeError is raised describing the convention.
  - This introspection mechanism allows external tooling (e.g., hyperparameter optimizers or serialization utilities) to discover and operate on kernel parameters in a uniform way.

- _check_length_scale
  - Helper function that validates length-scale shapes passed to anisotropic kernels.
  - Ensures length_scale is 1D and has a length equal to the number of feature dimensions in X.
  - Returns the normalized length_scale as a 1D array of floats.
  - Raises:
    - ValueError if length_scale has dimensionality greater than 1.
    - ValueError if, for anisotropic kernels, length_scale shape does not match X.shape[1].

Architectural notes:
- The base class is designed to be extended by concrete kernel implementations that provide specific hyperparameters and kernel evaluations (covariance matrices) for GP modeling.
- The operator overloading semantics (+, *, **) are expected to be implemented in concrete kernel classes, enabling kernel composition and scalar-to-constant conversion.

## API usage Patterns

- Extending kernels
  - Subclass Kernel and implement:
    - __init__ with kernel-specific hyperparameters and their defaults
    - __call__(X, Y=None) or the covariance evaluation logic
    - Gradient computation for hyperparameters to support analytic optimization
  - Define a property or attribute (commonly named hyperparameters) that lists Hyperparameter instances describing the tunable parameters.

- Kernel composition
  - Combine kernels using:
    - k = k1 + k2
    - k = k1 * k2
    - k = k1 ** alpha (scalar alpha)
  - When mixing scalars in expressions, scalar values are automatically converted to a constant kernel. This enables concise kernel construction like:
    - k = k1 + 2.0  # equivalent to k1 + ConstantKernel(constant_value=2.0)

- Hyperparameter optimization
  - Define bounds and fixed status for each hyperparameter.
  - Use a gradient-based optimizer to search within the defined rectangular hyperparameter space.
  - Fixed hyperparameters are automatically ignored by the optimizer.

- Inspecting kernel parameters
  - Use get_params(deep=True) to retrieve a dictionary of parameters for serialization, cloning, or optimization workflows.

## Examples

Example 1: Define and inspect a hyperparameter for a kernel (conceptual)

- Suppose you have a kernel subclass ConstantKernel with a hyperparameter constant_value.

  - hyperparam = Hyperparameter(name="constant_value",
                                value_type="numeric",
                                bounds=[[0.0, 10.0]],
                                n_elements=1,
                                fixed=False)

  - print(hyperparam)
  - Access bounds via hyperparam.bounds and determine if it is fixed via hyperparam.fixed.

Example 2: Compose kernels and convert scalars to a constant kernel

- From a hypothetical pair of kernels k1 and k2:
  - k = k1 + k2
  - k = k1 * k2
  - k = k1 ** 2.5
  - scalar in composition: k = k1 + 1.0  # 1.0 becomes a ConstantKernel

Example 3: Anisotropic length-scale for a kernel

- If a kernel uses an anisotropic length-scale, you might specify:
  - length_scale = Hyperparameter(name="length_scale",
  -                                 value_type="numeric",
  -                                 bounds=[[0.1, 3.0]],
  -                                 n_elements=X.shape[1],
  -                                 fixed=False)

  - This yields a per-dimension length-scale vector of size X.shape[1].
  - Ensure _check_length_scale is used to validate shape during construction or evaluation.

Example 4: Hyperparameter bounds and fixed parameters

- Define a kernel with two hyperparameters:
  - hours: numeric in [0.0, 1.0]
  - noise_level: numeric fixed at 0.1
  - The hyperparameter specification would be:
    - Hyperparameter("hours", "numeric", bounds=[[0.0, 1.0]], n_elements=1, fixed=False)
    - Hyperparameter("noise_level", "numeric", bounds="fixed", n_elements=1, fixed=True)

## Design notes and rationale

- Memory efficiency
  - Hyperparameters are implemented as a namedtuple with __slots__ to minimize memory footprint.
  - This design is favorable for GP models that require frequent access to hyperparameter metadata.

- Robustness
  - _check_length_scale enforces consistent dimensionality for anisotropic kernels, reducing runtime errors due to shape mismatches.
  - Kernel.get_params uses signature inspection to provide a uniform API for parameter retrieval, supporting serialization and optimization workflows.

- Extensibility
  - The architecture supports easy addition of new kernels that adhere to the Hyperparameter convention.
  - Users can define custom kernels and participate in hyperparameter optimization without modifying the core GP machinery.

---

## Developer notes

- If you add new kernels, ensure:
  - The __init__ signature is consistent with get_params’s introspection.
  - You expose a hyperparameters attribute or property listing Hyperparameter instances.
  - Scalar constants used in composition are automatically converted to the appropriate constant kernel.

- Testing considerations
  - Validate anisotropic length-scales via _check_length_scale.
  - Verify hyperparameter bounds handling, including the "fixed" case.
  - Confirm that get_params correctly reflects the kernel’s initialization arguments.

- Interactions with downstream components
  - Optimizers should consume kernel.hyperparameters to drive gradient-based searches.
  - Cloning utilities rely on get_params to recreate kernel instances with the same configuration.
