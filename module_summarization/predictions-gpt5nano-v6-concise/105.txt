# g-functions.rst — Generate documentation for this code

Overview
- This page documents the dynamic documentation generator that builds the docstring for the module responsible for describing generated Meijer G related functions.
- The generator runs at import time and composes a reStructuredText block (as a Python string) that becomes the module’s __doc__.
- The resulting documentation is intended to be consumed by Sphinx, typically via a standard docstring-based page or an autosummary/autodoc integration.

What this generator does
- Imports core building blocks (Add, Eq, Symbol) and a latex printer, plus a helper (default_sort_key) and a table builder (_create_lookup_table).
- Creates a mapping t: dict[tuple[type, ...], list[typing.Any]] as a defaultdict(list).
- Invokes _create_lookup_table(t) to populate the mapping with information about elementary functions and more complex function combinations.
- Sorts the items in t by default_sort_key to produce deterministic ordering.
- Builds a documentation string (doc) in reStructuredText format containing:
  - A header for elementary functions (when about == ()).
  - Headers for functions involving specific function types (for non-empty about keys).
  - For each entry (formula, gs, cond, hint) in a category:
    - If gs is not a list, uses a Symbol representing a generated placeholder.
    - Otherwise constructs g as the sum Add(*[fac*f for (fac, f) in gs]).
    - Creates a mathematical equation with Eq(formula, g).
    - Formats an optional condition: if cond is True, omit; otherwise append a “if” clause with LaTeX-safe rendering of cond.
    - Emits a Sphinx math directive:
      .. math::
        <latex(Eq(formula, g))><optional condition>
- Exposes the generated content as __doc__ so that Sphinx can render it directly on the documentation page.

Key data structures and flow
- t: A dict-like structure (defaultdict(list)) mapping:
  - Key: a tuple describing the “about” category (e.g., which base functions are involved).
  - Value: a list of entries, each entry a tuple: (formula, gs, cond, hint).
- For each category (about, category):
  - about == (): header = "Elementary functions:"
  - otherwise: header = "Functions involving <list of function names>:" (names derived from the first-given function’s atoms)
- For each entry:
  - gs is either a list of (factor, function) pairs or a non-list value.
  - If gs is a list, g = Add(sum over fac*f for each pair); otherwise g = Symbol('\text{generated}').
  - obj = Eq(formula, g)
  - cond handling:
    - cond is True → no condition text
    - cond is not True → append ",\text{ if } <latex(cond)>"
  - Emit:
    .. math::
      <latex(obj)><optional cond>

Developer-oriented guidance
- Where the data comes from:
  - _create_lookup_table(t) is the integration point that populates t with categories of functions and their relationships.
  - The exact contents depend on the implementation of _create_lookup_table in the meijerint package, but the generator treats them as:
    - formula: the target expression to be equated
    - gs: a list of (factor, function) pairs describing how to assemble the generated function g; or a non-list to indicate a generated placeholder
    - cond: an optional condition that modifies the rendered math block
    - hint: ancillary information (not directly rendered in the example)
- How to customize ordering:
  - default_sort_key() controls the deterministic order of categories. You can override or modify this function to influence the display order without changing the underlying data.
- Extending functionality:
  - To add new categories or function types, extend _create_lookup_table so t is populated with new keys and corresponding entries.
  - Ensure that entries conform to the (formula, gs, cond, hint) tuple structure so the generator can render them properly.
- Performance considerations:
  - The doc is generated at import time; for very large tables, the import may be slower. If this becomes onerous, consider lazy-loading or caching the generated __doc__.

Examples
- What the generated documentation looks like (conceptual structure):

  Elementary functions:

  .. math::
    f(x) = g(x)

  .. math::
    h(x) = \text{generated}

  Functions involving sin, cos, etc.:

  .. math::
    F(x) = G(x) \quad\text{ if } x > 0

  .. math::
    E(x) = \text{generated}

Notes:
- The actual content depends on the populated t and the latex() rendering. The examples above illustrate the structure (header, math directive, optional condition).

Usage and regeneration
- How to generate/update the docs locally:
  - Ensure the module (containing this generator) is importable in your environment.
  - Import the module (or run a script that imports it) so that __doc__ is populated.
  - Build the Sphinx docs as usual (e.g., sphinx-build -b html <source> <build>).
- Accessing the generated documentation programmatically:
  - After import, the module’s __doc__ contains the reStructuredText content that Sphinx will render.

Important caveats
- Import-time side effects: The generator runs when the module is imported, which means any heavy computation in _create_lookup_table or related code executes at import. Be mindful in constrained environments.
- Consistency: The generated documentation relies on the behavior of latex() and the formatting of Eq. If those helpers change, the rendered docs will differ accordingly.
