Array API support
===================

Overview
--------
This project provides experimental support for the Array API standard to enable backend-agnostic array operations alongside NumPy. The goal is to allow developers to write code that can operate on array objects from any compatible backend (e.g., NumPy, CuPy, JAX, etc.) through a common API surface.

Key ideas
- Use a backend-agnostic interface for array creation, manipulation, and basic operations.
- Maintain NumPy-friendly semantics where possible, with explicit conversions when integrating with scikit-learn estimators.
- Keep the public APIs forward-compatible with the Array API standard while preserving existing NumPy-based workflows.

What this means for developers
- You can source arrays from an Array API backend and perform common operations in a uniform way.
- When integrating with existing scikit-learn codepaths, explicit conversions to NumPy arrays are typically required, unless a downstream path explicitly supports the Array API arrays.
- This support is experimental and subject to change as the ecosystem evolves.

Supported scope
- Basic array creation and casting:
  - asarray, zeros, ones, empty, from Python sequences
  - dtype control and shape inspection
- Elementwise operations and broadcasting:
  - add, subtract, multiply, divide, and equivalent operators
  - comparison operations
- Reductions and aggregations:
  - sum, mean, max, min (and axis-aware variants)
- Basic indexing and slicing
- Minimal linear algebra and reductions provided by the backend (full parity with NumPy is not guaranteed)

Important caveats
- Automatic conversion: Many estimator workflows in this project assume NumPy arrays. If you pass Array API arrays directly, you may need to convert to NumPy arrays before calling scikit-learn estimators (e.g., via np.asarray or equivalent backend-specific conversion).
- Backends and dtype compatibility vary. Some backends may implement additional dtypes or behaviors differently from NumPy; validate expectations in your workflow.
- Features under active development: API surface, performance characteristics, and interop details may change as Array API support matures.

How to enable and use
- Install an Array API backend and the array-api shim compatible with your stack (if applicable in your environment).
- Import and create arrays using the Array API module (commonly imported as xp).
  - Example (pseudo-code; adapt to your backend):
    - import array_api as xp
    - a = xp.asarray([1, 2, 3])
    - b = xp.asarray([4, 5, 6])
    - c = xp.add(a, b)  # elementwise addition
- Convert to NumPy for scikit-learn workflows:
  - import numpy as np
  - a_np = np.asarray(a)
  - b_np = np.asarray(b)
  - c_np = np.asarray(c)
- Use NumPy-based estimators as usual:
  - from sklearn.linear_model import LinearRegression
  - model = LinearRegression().fit(a_np.reshape(-1, 1), c_np)

Examples
1) Basic Array API creation and operations
.. code-block:: python

   import array_api as xp
   a = xp.asarray([1, 2, 3], dtype=xp.float64)
   b = xp.asarray([4, 5, 6], dtype=xp.float64)
   c = xp.add(a, b)          # elementwise add
   s = xp.sum(c, axis=0)     # reduction

2) End-to-end with a scikit-learn estimator (convert to NumPy)
.. code-block:: python

   import array_api as xp
   import numpy as np
   from sklearn.linear_model import LinearRegression

   X_api = xp.asarray([[1.0, 2.0], [3.0, 4.0]])
   y_api = xp.asarray([1.0, 2.0])

   # Convert to NumPy for scikit-learn
   X = np.asarray(X_api)
   y = np.asarray(y_api)

   model = LinearRegression().fit(X, y)
   preds = model.predict(X)

Developer guidance
- Integration points:
  - When accepting inputs for estimators, document whether Array API arrays are accepted directly or require conversion to NumPy.
  - If desired, provide a lightweight conversion helper to convert Array API arrays to NumPy arrays in user code paths:
    - def to_numpy(x): return np.asarray(x)
- Testing considerations:
  - Add tests that verify behavior with both NumPy arrays and Array API arrays (after conversion, where appropriate).
  - Ensure round-trip conversions preserve data values and shapes.
- Performance considerations:
  - Cross-backend conversions can incur overhead; prefer operating in a backend-appropriate path and convert only when necessary for downstream code.
  - Be mindful of memory usage when converting large arrays between backends.

Backwards compatibility
- NumPy-centric code paths remain the primary route for scikit-learn estimators.
- Array API support is opt-in in user code paths; automatic, pervasive interop is not guaranteed at this time.
- When in doubt, convert to NumPy arrays before feeding data into standard scikit-learn estimators.

Future directions
- Broaden and standardize cross-backend interop to reduce or eliminate explicit conversions.
- Expand test coverage to cover more Array API backends and dtypes.
- Improve documentation with concrete, supported backend configurations and examples.
