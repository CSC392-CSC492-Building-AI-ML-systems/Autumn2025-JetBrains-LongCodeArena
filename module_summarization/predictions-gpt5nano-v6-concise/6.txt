Working with user security roles
================================

This document describes how to work programmatically with user security roles in ftrack using the Python API (ftrack_api). It covers the concepts, typical workflows, and concrete examples to create, view, modify, and assign security roles to users or groups.

Prerequisites
-------------
- A working ftrack server reachable from your code (Session can connect to the server).
- An authenticated Session (using API key and user, or environment-configured credentials).
- Knowledge of the security model in your server (roles, permissions, users/groups, and role assignments).
- Sufficient permissions to view and modify security roles and user assignments on the server.

Security model overview
-----------------------
- Security Role: A named collection of permissions. Roles define what actions a user (or group) may perform and on which objects. Roles may be built-in (e.g., Administrator, Developer, Viewer) or custom-defined on the server.
- Permissions: Individual rights that may be combined into roles (e.g., read, write, delete, publish, administer). Permissions are evaluated in the context of the resource (e.g., a project, an asset, an user).
- User/Group: A subject to which security roles can be assigned. Users may belong to groups; groups can also be assigned roles.
- Assignment: The association between a subject (User or Group) and a Role. A subject may hold multiple role assignments.

What you can do with security roles
-----------------------------------
- Inspect available roles and their permissions.
- Create (define) new custom roles.
- Update permissions on existing roles.
- Assign one or more roles to a user or group.
- Query which roles are assigned to a given user or group.
- Revoke roles from a user or group.
- Audit role assignments and modifications (via operation logs, if enabled).

Typical API usage patterns
--------------------------
- Obtain a Session connected to the server.
- Query security roles to inspect definitions.
- Resolve User/Group entities to which roles will be assigned.
- Create or update Role definitions (permissions).
- Create or modify role assignments for Users/Groups.
- Query role assignments for auditing and validation.

Note: The exact API surface (class names, query syntax, and assignment methods) can vary between server versions and schema configurations. The examples below illustrate common patterns and should be adapted to your server’s schema and the ftrack_api API surface in use.

Reference data model (conceptual)
---------------------------------
- SecurityRole
  - id: Unique identifier
  - name: Human-friendly name
  - permissions: List of permissions included in the role
  - inherits: Optional linkage to a parent role for inheritance
- User
  - id: Unique identifier
  - username: Login name
  - name/email: Display information
- Group
  - id: Unique identifier
  - name: Group name
- RoleAssignment (subject -> role)
  - subject: User or Group
  - role: SecurityRole

Common workflows
----------------
1) List all roles
   - Retrieve all SecurityRole entities and print key fields (id, name, permissions).
2) Show details for a specific role
   - Retrieve a role by id or name and display its permissions and inheritance.
3) Create a new custom role
   - Define a role with a name and a set of permissions.
   - Save the role to the server.
4) Update a role’s permissions
   - Modify the permissions list of a role and persist the changes.
5) Assign a role to a user
   - Find the target User entity and the target SecurityRole.
   - Create a RoleAssignment linking the user to the role.
6) Revoke a role from a user or group
   - Locate the existing RoleAssignment and remove it.
7) List roles for a user or group
   - Query assignments for the subject and show the linked roles.
8) Audit and troubleshoot
   - Use operation logs or server-side audits to review changes to roles and assignments.

Examples
--------

Note: Adapt the following examples to your server’s actual API surface. The code blocks illustrate common intent and structure; replace placeholder calls with the corresponding API calls available in your environment.

Example 1: List all security roles
- Intent: Retrieve all defined roles and print their core attributes.

Python (illustrative)
- This example assumes you can query SecurityRole and access fields like id, name, and permissions.

from ftrack_api import Session

# Assume 'session' is an active Session connected to the server
roles = session.query('SecurityRole')  # adapt to your API surface
for role in roles:
    print('Role ID:', getattr(role, 'id', None))
    print('Name:', getattr(role, 'name', None))
    perms = getattr(role, 'permissions', [])
    print('Permissions:', ', '.join(perms))
    print('---')

Example 2: Inspect a specific role
- Intent: Show details for a role named "Publisher" (or by id).

Python (illustrative)
role_name = 'Publisher'
role = session.query('SecurityRole where name is "{}"'.format(role_name))
if role:
    r = role[0]
    print('Role ID:', getattr(r, 'id', None))
    print('Name:', getattr(r, 'name', None))
    print('Permissions:', ', '.join(getattr(r, 'permissions', [])))
else:
    print('Role not found:', role_name)

Example 3: Create a custom role
- Intent: Define a new role with a given set of permissions.

Python (illustrative)
new_role_name = 'ContentPublisher'
new_permissions = ['read', 'write_content', 'publish']

# Pseudo: create role via API; adjust to your actual creation call
new_role = session.create('SecurityRole', {
    'name': new_role_name,
    'permissions': new_permissions
})
session.commit()  # or the appropriate commit/save call

print('Created role', new_role_name, 'with id', getattr(new_role, 'id', None))

Example 4: Assign a role to a user
- Intent: Grant a user the new role.

Python (illustrative)
user_identifier = 'jdoe'  # or user id
role_name = 'ContentPublisher'

# Resolve user and role
user = session.query('User where username is "{}"'.format(user_identifier))[0]
role = session.query('SecurityRole where name is "{}"'.format(role_name))[0]

# Pseudo assignment call; adapt to your API
assignment = session.assign_role(user_id=getattr(user, 'id', None),
                               role_id=getattr(role, 'id', None))
print('Assigned role', role_name, 'to user', user_identifier)

Example 5: Revoke a role from a user
- Intent: Remove an existing role assignment.

Python (illustrative)
user = session.query('User where username is "jdoe"')[0]
role = session.query('SecurityRole where name is "ContentPublisher"')[0]

# Pseudo removal call
success = session.remove_role_assignment(user_id=getattr(user, 'id', None),
                                         role_id=getattr(role, 'id', None))
if success:
    print('Revoked role from user')
else:
    print('Role assignment not found or could not be revoked')

Example 6: List roles for a user
- Intent: Show all roles currently assigned to a user.

Python (illustrative)
user = session.query('User where username = "jdoe"')[0]
assignments = session.query('RoleAssignment where subject_id = "{}"'.format(getattr(user, 'id', None)))
roles = [getattr(a, 'role', None) for a in assignments]
for r in roles:
    if r: print('Role:', getattr(r, 'name', None))

Best practices and considerations
-------------------------------
- Principle of least privilege: Create roles that grant only the permissions needed for a task. Avoid broad, catch-all roles for regular users.
- Use domain-based roles: If possible, create roles aligned to project domains or workflows (e.g., "ContentEditor", "ReviewOnly") to simplify assignment management.
- Inheritance: Leverage role inheritance carefully to avoid unintended permission elevation. Document inheritance relationships.
- Auditing: Enable server-side audits or operation logs for role and assignment changes to facilitate traceability.
- Immutable roles for critical permissions: Consider making highly privileged roles more tightly controlled and audited.
- Validation: Validate role changes in a staging/testing environment when possible before applying to production.

Advanced topics
---------------
- Role inheritance: Some roles may inherit permissions from a parent role. Understand the effective permission set by combining inherited and direct permissions.
- Scoped permissions: Some permissions may be scoped to a project, folder, or resource; ensure assignments respect the correct scope.
- Group-based assignments: When users belong to groups, assigning a role to a group can reflect across all its members. Manage both user-level and group-level assignments to avoid conflicts.

Troubleshooting
---------------
- Missing roles or permissions: Verify that your user has sufficient permissions to view roles and assignments. Check server schema and access controls.
- Assignment failures: Ensure the target user/group and role exist, and that you are using the correct identifiers (id vs. name). Check your server logs for errors.
- Inconsistent state after changes: If changes don’t appear to propagate, ensure you commit changes through the correct API method (e.g., session.commit or a server-side API call) and refresh local objects as needed.
