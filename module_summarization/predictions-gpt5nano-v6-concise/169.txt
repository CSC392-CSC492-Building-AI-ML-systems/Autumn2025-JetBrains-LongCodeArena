Contributing to Qiskit Metal
==============================

This guide explains how to contribute to the Qiskit Metal project, with a focus on developer contributions to the core codebase (Python modules such as qiskit_metal.analyses and related utilities). It covers setup, workflow, coding standards, testing, documentation, and a concrete example of adding a new CPW calculation feature.

Table of contents
-----------------
.. contents::
   :local:
   :depth: 2

Overview
--------
Qiskit Metal is an open-source framework for designing quantum devices. Contributions range from small bug fixes to new analyses, physics models, tests, and documentation improvements. All contributions must respect the project’s licensing and quality standards and go through the PR review process.

Prerequisites
-------------
- Python 3.8+ (check the project’s supported versions in the repository)
- Git
- Access to a development environment with the required development dependencies
- Basic familiarity with reStructuredText (RST) and Sphinx (for docs)

Getting started
---------------
1. Fork the repository on GitHub and clone your fork:
   .. code-block:: bash

      git clone https://github.com/your-username/qiskit-metal.git
      cd qiskit-metal
2. Create a virtual environment and install development requirements:
   .. code-block:: bash

      python -m venv env
      source env/bin/activate  # on Unix/macOS
      env\\Scripts\\activate     # on Windows
      pip install -U pip setuptools wheel
      pip install -r requirements-dev.txt
3. Run the test suite to establish a baseline:
   .. code-block:: bash

      pytest -q
4. Build the documentation locally (optional but recommended for API docs):
   .. code-block:: bash

      sphinx-build -b html docs build/docs
5. (Optional) Install pre-commit hooks for code quality:
   .. code-block:: bash

      pip install pre-commit
      pre-commit install

Repository layout (high level)
--------------------------------
- qiskit_metal/
  - analyses/
    - cpw_calculations.py
    - lumped_capacitive.py
  - other core modules
- tests/
  - unit and integration tests
- docs/
  - source documentation (rst for Sphinx)
- CONTRIBUTING.md, contributor-guide.rst (this file)

Contributing workflow
---------------------
1. Find or propose an issue:
   - Look for open issues in the tracker or open a feature request. If the issue is sizable, discuss approach in a comment.
2. Create a feature branch:
   .. code-block:: bash

      git checkout -b feature/my-improvement
3. Implement the change:
   - Write code in the appropriate module (for example, adding a new function to qiskit_metal.analyses.cpw_calculations).
   - Add or update unit tests under tests/.
   - Update or add documentation (docs and docstrings).
4. Run tests and linting:
   .. code-block:: bash

      pytest -q
      pre-commit run --all-files
5. Create a pull request:
   - Provide a concise PR title and a description outlining the change, motivation, and any potential side effects.
   - Reference related issues (e.g., "Closes #123").
6. Respond to code-review feedback and iterate as needed.

Coding standards and API design
------------------------------
- Language: Python 3.x
- Style: PEP 8, consistent with existing Qiskit Metal code
- Documentation:
  - Public APIs must have docstrings (prefer NumPy-style docstrings)
  - Code modules and functions should include clear descriptions of inputs, outputs, units, and assumptions
  - Use Sphinx-friendly docstrings so autosummary and autodoc can generate API docs
- Exported APIs: Public functions and classes in modules (e.g., qiskit_metal.analyses.cpw_calculations) should be importable via package exports. Update __init__.py or module import paths accordingly when adding new items.
- Tests: Coverage for new functionality; tests should be deterministic and documented

Documentation guidelines
------------------------
- Documentation lives in the docs/ directory and uses reStructuredText.
- Public APIs should be documented with autosummary tables where appropriate:
  .. autosummary::
      :toctree:

      qiskit_metal.analyses.cpw_calculations
      qiskit_metal.analyses.lumped_capacitive
- Update or add docstrings for new functions to reflect usage, parameters, and units.
- Add examples to docs if the feature is nontrivial or has non-obvious usage.
- When modifying APIs, accompany changes with an entry in the release notes (as per your project’s release process).

Testing and quality assurance
-----------------------------
- Add unit tests under tests/, using pytest style.
- Ensure tests are fast and deterministic.
- Validate numerical results with known benchmarks or analytic checks when possible.
- Run the full test suite locally before submitting a PR.
- Run linting and formatting checks (pre-commit, flake8, black, isort if configured).

Example contribution: extending CPW calculations
-----------------------------------------------
This section shows a concrete example: adding a new CPW-related calculation to the analyses for CPW parameters and updating tests and docs accordingly.

Context
- The CPW calculation module currently contains:
  - guided_wavelength(freq, line_width, line_gap, substrate_thickness, film_thickness, dielectric_constant=11.45)
  - lumped_cpw(freq, line_width, line_gap, substrate_thickness, film_thickness, dielectric_constant=11.45, loss_tangent=1e-5, london_penetration_depth=30e-9)
- These functions live in qiskit_metal.analyses.cpw_calculations and are part of the public API, with detailed docstrings.

Proposed change
- Add a new function, e.g., capacitance_per_unit_length, to cpw_calculations.py, which computes the CPW capacitance per unit length for given geometry and dielectric parameters.
- Update the module’s __init__ if needed to export the new function at the package level.
- Add unit tests in tests/test_cpw_calculations.py validating expected results for a few representative sets of parameters (including edge cases).
- Update the docs autosummary so the new function is listed in API docs.
- Document the new function’s purpose, inputs, outputs, and units in a NumPy-style docstring.

Contribution checklist (PR-ready)
-------------------------------
- [ ] Code implements the feature with accompanying tests
- [ ] Public API is well-documented with docstrings
- [ ] Documentation autosummary updated
- [ ] Tests pass locally
- [ ] Linting and formatting checks pass
- [ ] PR description summarizes the change, motivation, and any breaking changes

Closing notes
-------------
- All contributions must retain copyright notices where applicable.
- Ensure changes are backward-compatible or properly documented if breaking changes are introduced.
- For significant changes, discuss with the project maintainers via an issue or PR discussion thread.

If you have questions or need guidance on a contribution, please reach out through the project’s issue tracker or the maintainer contact channels.
