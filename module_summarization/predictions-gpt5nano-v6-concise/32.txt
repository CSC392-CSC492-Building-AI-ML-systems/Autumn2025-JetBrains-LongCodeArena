# w_run — Start or Continue a WESTPA Simulation

Overview
- w_run is the entry point for starting or continuing a WESTPA simulation. It assembles the core WESTPA drivers (simulation manager, system, data, propagator, and WE driver), wires them together, and delegates execution to a configured work manager (which handles master/worker processes, parallelism, and resource management).
- The command-line interface is composed of runtime configuration options (via westpa.rc), a one-segment debugging flag, and work-manager arguments exposed by the environment's work-manager module.

Usage
- Basic usage:
  - w_run
- Help:
  - w_run -h

Key concepts
- Runtime configuration (westpa.rc): w_run delegates to westpa.rc to apply runtime options before the run begins.
- Work manager: The execution is coordinated by a work manager obtained via make_work_manager(). Arguments for the work manager are added to the parser by work_managers.environment.add_wm_args(parser) and processed before the run starts.
- One-segment mode: Use --oneseg to propagate only one segment. This is useful for debugging propagators.

Command-line interface
- --oneseg (alias: --only_one_segment)
  - dest: only_one_segment
  - action: store_true
  - help: only propagate one segment (useful for debugging propagators)
- WM arguments
  - Added by: work_managers.environment.add_wm_args(parser)
  - These options are WM-specific and defined by the active work manager environment.

- Global options
  - Provided by: westpa.rc.add_args(parser)
  - These are runtime/configuration options for WESTPA.

What happens when you run w_run
- The script parses CLI arguments, applies runtime config, and configures the work manager with its arguments.
- It builds the core drivers and ties them together:
  - sim_manager
  - system
  - data_manager
  - propagator
  - we_driver
- The drivers are wired as follows:
  - propagator.system = system
  - data_manager.system = system
  - we_driver.system = system
  - sim_manager.data_manager = data_manager
  - sim_manager.system = system
  - sim_manager.propagator = propagator
  - sim_manager.we_driver = we_driver
- Execution happens within a work_manager context:
  - If work_manager.is_master:
    - Install a SIGINT handler
    - sim_manager.load_plugins()
    - sim_manager.prepare_run()
    - sim_manager.run()
    - sim_manager.finalize_run()
    - KeyboardInterrupt is caught and reported (interrupted; shutting down)
    - Any other exception is reported (exception caught; shutting down) with a traceback
  - If not master:
    - work_manager.run() handles worker-side execution

Internal flow (step-by-step)
1. Parse CLI args
2. Apply runtime config (westpa.rc.process_args)
3. Extend CLI with work-manager args (add_wm_args) and process them (process_wm_args)
4. Create and configure the work manager (make_work_manager)
5. Load core drivers (sim_manager, system, data_manager, propagator, we_driver)
6. Wire drivers together
7. Enter the work_manager context
   - Master path: prepare, run, finalize, with error handling
   - Worker path: delegate to work_manager.run()
8. Exit and return control to the caller

Error handling
- KeyboardInterrupt:
  - westpa.rc.pstatus('interrupted; shutting down')
- Other exceptions:
  - westpa.rc.pstatus('exception caught; shutting down')
  - If the exception message is non-empty, log the message
  - Always log the full traceback for debugging

Plugins
- Plugins are loaded only on the master process via sim_manager.load_plugins() before the run begins.

Debugging and logging tips
- Use --oneseg to quickly validate propagators without full ensemble propagation.
- Use -h to inspect available runtime and WM options (the actual WM options depend on the configured work manager).
- Enable/adjust logging through the WESTPA runtime configuration to capture more detailed output.

Examples
- Basic run (default configuration):
  - w_run
- Run in one-segment debugging mode:
  - w_run --oneseg
- Inspect available options (including WM options):
  - w_run -h
- Extend/run with a configured work manager (options vary by WM; see WM docs):
  - w_run [WM options as provided by your environment]
  - Note: The exact syntax for WM arguments is determined by the active work manager environment (via work_managers.environment.add_wm_args).

Notes for developers
- w_run is designed as an integration point that connects WESTPA components (sim_manager, system, data_manager, propagator, we_driver) with a pluggable work-manager backend.
- The code explicitly wires each driver to the others to ensure consistent configuration before the run begins.
- Error handling is centralized in the master path to provide a clean shutdown and diagnostic information if something goes wrong.
- The --oneseg option offers a targeted path for propagator debugging without requiring full-scale simulation.

See also
- WestPA runtime configuration (westpa.rc) for global options.
- WestPA work-manager documentation for details on available WM backends and their CLI arguments.
