Migrating to nextcord
=====================


Overview
--------

nextcord is a maintained fork of the Discord.py project designed to be API-compatible
where feasible while introducing improvements and fixes. This guide helps you
migrate existing codebases to nextcord with a focus on widget-related data models
and common developer workflows.

Prerequisites
-------------

- Python 3.8+ (as supported by nextcord)
- An existing codebase that consumes Discord widgets (Widget, WidgetChannel, WidgetMember)
- Familiarity with the concepts of widgets in Discord (guild widgets, widget channels, widget members)

Installation
------------

- Install nextcord from PyPI:

  .. code-block:: python

     pip install nextcord

- If you are supporting both discord.py and nextcord in the same environment during migration, you can use a compatibility import helper or separate environments to avoid symbol conflicts.

API changes: widget-related classes
-----------------------------------

nextcord exposes widget-related data models that reflect the structure used by Discord's widget API. The primary classes are:

- WidgetChannel
  - Represents a partial view of a channel within a guild widget.
  - Key attributes: id (int), name (str), position (int)
  - Useful operations: display and mention of a channel.

- WidgetMember
  - Represents a partial member visible in a guild widget.
  - Key attributes (examples): id, name, discriminator, bot, status, nick, avatar, activity, deafened, muted, suppress, connected_channel
  - Activity may be a BaseActivity or a Spotify activity depending on data.

- Widget
  - Represents a Guild widget.
  - Key attributes: id (guild ID), name (guild name), channels (list of WidgetChannel), members (list of WidgetMember)

Notes
- The following properties and behaviors are commonly available:
  - Channel and member objects provide convenient representations (e.g., __str__ returning the channel name or a member’s display name).
  - WidgetMember.display_name uses nick when available, otherwise the username.
  - WidgetChannel.mention provides a string suitable for mentioning the channel.
  - created_at on WidgetChannel derives the creation time from the channel's ID using snowflake_time.

Migration guidance
------------------

1) Imports and basic usage

- Replace imports from discord.* to nextcord.* for widget-related classes.
- The structure of widget-related classes is preserved in nextcord, so many attribute accesses and simple iterations remain the same.

  Example before (conceptual; adapt to your actual codebase):

  .. code-block:: python

     from discord import Widget, WidgetChannel, WidgetMember

     # Assume you have a payload or an API response that constructs these objects
     widget = Widget(state, payload)

     for ch in widget.channels:
         print(ch.name)

  Example after (nextcord):

  .. code-block:: python

     from nextcord import Widget, WidgetChannel, WidgetMember

     widget = Widget(state, payload)

     for ch in widget.channels:
         print(ch.name)

2) Working with WidgetChannel
- WidgetChannel remains a lightweight representation of a channel within a widget.
- Access common fields like id, name, and position.
- Use mention property to format a channel mention for messages.

  Example:

  .. code-block:: python

     channel = widget.channels[0]
     print(channel.mention)  # <#123456789012345678>

- If you need creation time, use created_at (requires datetime type import).

  Example:

  .. code-block:: python

     import datetime
     at = widget.channels[0].created_at
     assert isinstance(at, datetime.datetime)

3) Working with WidgetMember
- WidgetMember exposes user-like information as provided by the widget payload.
- Important fields include name, discriminator, bot, status, nick, avatar, activity, connected_channel.

  Example:

  .. code-block:: python

     member = widget.members[0]
     print(member.display_name)
     if member.connected_channel:
         print(member.connected_channel.name)

- The activity field may be a BaseActivity or a Spotify activity, depending on the payload.

  Example:

  .. code-block:: python

     if member.activity:
         print(member.activity.name)

4) Widget data construction and payloads
- Widget, WidgetChannel, and WidgetMember can be constructed from payloads similar to how REST/internals might deliver widget data.
- If you previously decoded widget payloads manually, you can continue to do so with the same keys, but now rely on nextcord’ s typing and utilities for better type safety and stability.

5) Type hints and runtime checks
- nextcord supports typing with TYPE_CHECKING and related hints as shown in the WidgetMember class.
- You can annotate WidgetMember.activity as Optional[Union[BaseActivity, Spotify]] to reflect possible activity types.

6) Utilities and helpers
- Utilities such as snowflake_time can be used to derive creation times from IDs, just as in the provided codebase.

Migration checklist
-------------------

- [ ] Update all imports from discord.* to nextcord.* for widget-related classes.
- [ ] Validate that widget payloads are consumed the same way, adjusting type hints as needed.
- [ ] Replace any direct references to deprecated or renamed widget fields with their nextcord equivalents.
- [ ] Run unit tests and integration tests involving widget data (Widget, WidgetChannel, WidgetMember).
- [ ] Update documentation strings and inline comments to reflect nextcord APIs.

Example: migrating a widget payload handling snippet
----------------------------------------------------

Suppose you previously processed a widget payload to create WidgetChannel and WidgetMember objects. A migration-safe approach is to rely on Nextcord type hints and ensure predictable attribute access.

  .. code-block:: python

     from nextcord import Widget, WidgetChannel, WidgetMember
     from nextcord import Status  # if you use status enums
     from nextcord.utils import snowflake_time

     # Example payload structure (adjust to your actual payload)
     payload = {
         "id": 987654321,
         "name": "Example Guild",
         "channels": [
             {"id": 111, "name": "general", "position": 1},
             {"id": 222, "name": "voice", "position": 2},
         ],
         "members": [
             {
                 "user": {"id": 333, "username": "Alice", "discriminator": "4242"},
                 "nick": "Al",
                 "status": "online",
                 "game": {"name": "Coding"},
             }
         ]
     }

     widget = Widget(state=None, data=payload)  # state may come from your existing connection state

     for ch in widget.channels:
         print(ch.name, ch.id)

     for m in widget.members:
         print(m.display_name, m.status)

Migration notes
---------------

- If your code relied on private or internal widget structures, prefer the public attributes on Widget, WidgetChannel, and WidgetMember for forward compatibility.
- Ensure you run tests in an environment that uses the nextcord package to catch API differences early.

This migration guide focuses on the widget-related aspects demonstrated by the provided code snippet. For broader migration guidance (commands, event handling, cogs, and other modules), consult the official nextcord migration guide and API reference.
