# Unified renderable interface (renderable.rst)

Overview
--------
This documentation describes the unified renderable interface provided by the
codebase. The interface centralizes rendering of objects into templates while
providing flexible hooks for context data, CSS class generation, and BEM-style
naming. Developers can build reusable renderable components by subclassing the
provided abstract base classes.

Key classes
-----------
- AbstractRenderable
  - Core rendering interface.
  - Responsibilities:
    - Define which template(s) to render via get_template_names().
    - Provide template context via get_context_data().
    - Render via render().
  - Basic contract:
    - template_name attribute (or override get_template_names()).
    - get_context_data(request=None) -> dict (default: {'me': self}).
    - render(request=None, extra_context_data=None) -> rendered string.

- AbstractRenderableWithCss (extends AbstractRenderable)
  - Unified CSS class API.
  - Responsibilities:
    - Build a list of base and extra CSS classes.
    - Expose a css_classes string suitable for HTML class attributes.
    - Optional test CSS suffixes for automated tests.
  - API highlights:
    - get_base_css_classes_list() -> list
    - get_extra_css_classes_list() -> list
    - get_css_classes_list() -> list
    - css_classes property (string)
    - get_test_css_class_suffixes_list() -> list
    - get_css_classes_string() is deprecated (use css_classes)

- AbstractBemRenderable (extends AbstractRenderable)
  - BEM-oriented renderables (block/element naming).
  - Responsibilities:
    - Provide a convenient way to specify a BEM block and/or element.
    - Compose CSS class suffixes and extra classes in a BEM-friendly manner.
  - API highlights:
    - __init__(bem_block=None, bem_element=None, bem_variant_list=None, extra_css_classes_list=None)
    - bem_block_or_element property
    - get_bem_block() -> str
    - get_bem_element() -> str
    - get_test_css_class_suffixes_list() -> list

Common utilities
----------------
- join_css_classes_list(css_classes_list)
  - Joins a list of CSS class strings into a single string.

Core rendering flow
-------------------
1. Determine template names via get_template_names().
2. Build context data via get_context_data(request=request) and any extra_context_data provided.
3. Render the selected template with render_to_string(template_name, context, request).

Usage patterns
--------------

1) Simple renderable

- Purpose: render a template for a given object without extra CSS or BEM concerns.
- Typical usage:
  - Set template_name on the class or override get_template_names().
  - Optionally override get_context_data() to add extra context.

Example:

```python
from django_cradmin.renderable import AbstractRenderable

class MySimpleRenderable(AbstractRenderable):
    template_name = 'myapp/my_template.html'

    def get_context_data(self, request=None):
        data = super().get_context_data(request=request)
        data['extra'] = {'foo': 'bar'}
        return data
```

2) Renderable with CSS classes

- Purpose: renderable that needs a unified CSS class surface.
- Use AbstractRenderableWithCss:
  - Override get_base_css_classes_list() and/or get_extra_css_classes_list() to build classes.
  - Access the final string via the css_classes property.

Example:

```python
from django_cradmin.renderable import AbstractRenderableWithCss

class MyCssRenderable(AbstractRenderableWithCss):
    template_name = 'myapp/my_template.html'

    def get_base_css_classes_list(self):
        return ['my-component']

    def get_extra_css_classes_list(self):
        return ['is-visible']

# Usage
renderable = MyCssRenderable()
class_attr = renderable.css_classes  # e.g. 'my-component is-visible'
```

3) Bem-renderable (BEM naming)

- Purpose: renderables that follow BEM conventions for CSS class naming.
- Use AbstractBemRenderable:
  - Provide bem_block and/or bem_element.
  - Optionally provide bem_variant_list for variants after --.
  - Include extra CSS classes as needed.

Example:

```python
from django_cradmin.renderable import AbstractBemRenderable

class MyBemRenderable(AbstractBemRenderable):
    template_name = 'myapp/bem_template.html'

    def __init__(self, *args, **kwargs):
        super().__init__(bem_block='card', bem_element='card__header', extra_css_classes_list=['custom'])
```

Notes and caveats
-----------------

- AbstractRenderable
  - get_template_names() returns the configured template_name by default.
  - If template_name is not set, get_template_names() raises NotImplementedError.

- AbstractRenderableWithCss
  - css_classes is the canonical, public API for CSS classes.
  - get_css_classes_string() is deprecated; use css_classes instead.
  - get_test_css_class_suffixes_list() enables test-specific CSS class suffixes that can be inserted into the final class string when the DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES setting is enabled (see below).

- DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES
  - If True, test CSS suffixes from get_test_css_class_suffixes_list() are appended to the css_classes string using a helper tag (cradmin_test_css_class).
  - This is controlled via crsettings and the cradmin template tags.

- AbstractBemRenderable
  - bem_block and bem_element can't both be supplied. If both are provided, a ValueError is raised.
  - bem_element must contain __ (double underscore) to denote a BEM element.
  - There is a known issue in the provided code: AbstractBemRenderable.get_bem_element() returns self._b (an undefined attribute). It should return self._bem_element. This is a bug in the current implementation; the documentation assumes correct behavior and a fix is recommended.

- get_context_data defaults
  - By default, the render context includes {'me': self}. Override get_context_data() to add or modify context data as needed.

Deprecations and compatibility
------------------------------
- get_css_classes_string() in AbstractRenderableWithCss is deprecated. Prefer accessing the css_classes property.
- If you rely on test CSS suffixes, enable DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES in settings to activate test CSS suffixes via cradmin_test_css_class.

Examples: complete usage
------------------------

- Simple renderable (no CSS or BEM)

```python
from django_cradmin.renderable import AbstractRenderable

class SimpleRenderable(AbstractRenderable):
    template_name = 'example/simple.html'
```

- Renderable with CSS

```python
from django_cradmin.renderable import AbstractRenderableWithCss

class CssRenderable(AbstractRenderableWithCss):
    template_name = 'example/css.html'

    def get_base_css_classes_list(self):
        return ['component', 'component--active']

    def get_extra_css_classes_list(self):
        return ['override']

# Usage
r = CssRenderable()
print(r.css_classes)  # 'component component--active override'
```

- Bem-renderable

```python
from django_cradmin.renderable import AbstractBemRenderable

class BemRenderable(AbstractBemRenderable):
    template_name = 'example/bem.html'

# Block with an element, plus extra classes
r = BemRenderable(bem_block='card', bem_element='card__header', extra_css_classes_list=['shadow'])
print(r.css_classes)  # e.g. 'card__header shadow' (actual output depends on full implementation)
```

Conclusion
----------
The Unified renderable interface provides a cohesive model for rendering templates with a consistent approach to context data, CSS class generation, and BEM naming. By extending AbstractRenderable, AbstractRenderableWithCss, and AbstractBemRenderable, developers can build reusable, well-structured UI components with predictable rendering behavior. Be mindful of the noted bug in AbstractBemRenderable.get_bem_element() and apply the recommended fix in production code.
