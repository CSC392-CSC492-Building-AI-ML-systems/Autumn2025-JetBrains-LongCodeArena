Using custom attributes
=======================

This document describes how to use custom attributes with the ftrack API. Custom attributes allow you to attach user-defined metadata to ftrack entities (such as Tasks, Assets, Shots, Versions, etc.). They are defined once on the server and values are stored per-entity.

Overview
--------

- Custom attributes are defined as AttributeDefinitions and can be of various data types (string, integer, boolean, date/time, enumeration, JSON, etc.).
- Each entity can have zero or more custom attribute values assigned to it.
- The ftrack_api.attribute module provides facilities to create, retrieve and manage these definitions, as well as to read and write per-entity attribute values.
- Proper permissions are required to create and modify attribute definitions. Reading and setting values on entities typically requires read/write access to the affected entities.

Key concepts
------------

- AttributeDefinition: A server-side definition for a custom attribute (name, data type, allowed values, default, constraints, etc.).
- AttributeValue: The per-entity value assigned to a particular AttributeDefinition.
- Resource type: The entity type to which the attribute is attached (e.g., Task, Asset, Shot).

Supported data types
--------------------

- String
- Integer / Float
- Boolean
- Date / Time
- Enumeration (a defined set of allowed string values)
- JSON or structured data (for complex metadata)

Creating and managing attribute definitions
-------------------------------------------

To use a custom attribute, you typically create or fetch an AttributeDefinition for the target resource type, then assign values on entities of that type.

Note: The exact API calls can vary between ftrack API versions. The following illustrates the general flow; adapt to the API surface available in your environment.

.. code-block:: python

    # Pseudo-code illustrating the typical flow with ftrack_api.attribute
    # 1) Acquire a session (already established in your application)
    from ftrack_api import Session

    session = Session(server_url='https://ftrack.example.com',
                      api_key='your_api_key',
                      api_user='your_user')

    # 2) Retrieve an existing attribute definition for Task named 'Priority'
    priority_def = session.attribute.get_definition(
        resource_type='Task', name='Priority'
    )

    # 3) If it does not exist, create the attribute definition
    if priority_def is None:
        priority_def = session.attribute.create_definition(
            resource_type='Task',
            name='Priority',
            data_type='string',
            allowed_values=['Low', 'Medium', 'High'],
            default_value='Medium'
        )

    # 4) (Optional) Inspect attributes and their constraints
    print(priority_def.name, priority_def.data_type, priority_def.allowed_values)

Defining attributes (flows and considerations)
----------------------------------------------

- Pick a clear, descriptive name for the attribute.
- Choose an appropriate data_type that matches the intended values.
- If applicable, constrain values with allowed_values (e.g., for enumerations).
- Consider setting a default_value to ensure new entities have a sensible initial value.
- Document the meaning and acceptable values for teammates.

Reading attribute definitions
-----------------------------

- Retrieve the AttributeDefinition for inspection (name, data_type, constraints, default, etc.).
- Use this information to guide value assignment and UI presentation.

Setting and getting attribute values on entities
-----------------------------------------------

Once a definition exists, you can set values on individual entities (e.g., a Task) and read them back.

.. code-block:: python

    # 1) Find a specific entity (example: a Task)
    task = session.query('Task', filters=[('id', 'TASK-123')]).first()

    # 2) Set the value for the custom attribute
    session.attribute.set_value(
        entity=task,
        attribute_definition=priority_def,
        value='High'
    )

    # 3) Read back the value
    current_value = session.attribute.get_value(
        entity=task,
        attribute_definition=priority_def
    )
    print('Current Priority:', current_value)

Notes on per-entity values
--------------------------

- Attribute values are stored per entity instance and are isolated from other entities.
- If you move an entity to a different project or context, ensure the attribute definition is in scope for that context; otherwise, value handling may vary by server configuration.

Querying by custom attributes
-----------------------------

You can query entities based on custom attribute values. The exact query syntax depends on the API version; common patterns include filtering on the attribute value by the related AttributeDefinition.

.. code-block:: python

    # Example: find all Tasks with Priority == 'High'
    high_priority_tasks = session.query(
        entity_type='Task',
        filters=[('attribute.Prority', 'High')]  # adapt to your API's attribute path
    )

    for t in high_priority_tasks:
        print(t['id'], t['name'])

Migration and changes
---------------------

- When updating attribute definitions (e.g., adding allowed values or changing data_type), coordinate with server administrators to ensure existing values remain valid.
- If an attribute is deprecated or removed, plan for migrating values or preserving historical data as needed.

Error handling and troubleshooting
----------------------------------

- Missing permission: Ensure the API user has rights to create/modify AttributeDefinitions and to read/write values on the affected entities.
- Validation failures: Ensure values conform to the attribute’s data_type and allowed_values.
- Attribute not found: Verify the resource_type and attribute name match the server definition, and that the attribute is visible in your current context (project, workspace, etc.).

Advanced topics
---------------

- Enum-like attributes: Use allowed_values to constrain possible values and potentially drive UI drop-downs.
- Complex data: For structured metadata, use JSON or equivalent data_type to store nested information.
- UI considerations: Custom attributes may be surfaced in the UI; ensure names and values are user-friendly.
