Header Interface Transition Guide
=================================

Purpose
-------
This document describes the plan, rationale, and practical steps for transitioning
the Header interface used by the project. The goal is to provide a stable path
from the current header API to a unified, extensible, and better-tested header
interface with minimal disruption for users and downstreams.

Scope
-----
- Applies to the public header API used for reading, writing, and mutating
  header data (e.g., FITS headers) within this project.
- Covers design goals, migration strategy, compatibility considerations,
  deprecation policy, and example migrations.
- Does not mandate any specific internal implementation details beyond the
  public surface behavior and expected semantics.

Key Concepts
------------
- Header: A container of key/value pairs with optional metadata (comments, units, etc.)
- Public surface: The parts of the header API that external users and downstreams rely on.
- Transition surface: The new, unified interface surface introduced for headers.
- Backward compatibility: The existing API behavior should continue to work during the transition
  with explicit deprecation warnings; removal is planned for a future release after the transition window.

Rationale for Transition
------------------------
- Consistency: Unify header access across different backends and formats.
- Extensibility: Allow richer value types, validation, and metadata (comments, units, history).
- Robustness: Improve error handling, type safety, and testability.
- Maintainability: Simplify future enhancements and reduce duplicated logic.

Target State of the Header Interface
------------------------------------
- A single, well-documented, backwards-compatibility-focused public API.
- Clear, predictable behavior for common operations:
  - Accessing values (read, with defaults)
  - Setting values (with validation)
  - Deleting items
  - Iteration over keys, values, items
  - Metadata handling (comments, units, etc.)
- Deterministic error handling with explicit exceptions.
- Deprecation paths for legacy patterns, with warnings and recommended alternatives.

Migration Strategy
------------------
Phases:
1) Parallel Support (Current API + New API)
   - Introduce the new header interface in parallel, keeping the old API intact.
   - Emit non-blocking deprecation warnings when the old API is used.
   - Provide a transitional wrapper that maps old calls to the new interface.

2) Developer Adoption
   - Update core codepaths to prefer the new interface.
   - Add tests that validate both old and new paths where needed, prioritizing the new path.

3) Deprecation Window
   - Enforce deprecation warnings for all usage of the legacy patterns.
   - Document migration paths and provide upgrade guidelines to downstreams.

4) Removal
   - Remove legacy code paths in a planned major release after the transition window.
   - Ensure users have had ample notice and compatibility guarantees via warnings.

Deprecation Policy
------------------
- Deprecated APIs will issue a clear warning (e.g., DeprecationWarning or a project-specific warning type).
- The warning will include:
  - The deprecated API surface
  - The recommended replacement
  - The planned removal timeline
- Documentation and migration guides will be updated to reflect deprecated usage and its alternatives.

Public API Surface (Illustrative)
---------------------------------
Note: The exact API names may evolve; the following represents the intended capabilities and semantics.

- Header access
  - get(key, default=None): Retrieve the value for key, or default if missing.
  - __getitem__(key): Read-value access (may raise KeyError if missing).
  - __setitem__(key, value): Set value for key (with validation).
  - pop(key, default=None): Remove item and return value, or default if missing.
  - keys(), items(), values(): Iteration utilities over header content.
  - contains(key): Check for key presence.

- Metadata handling
  - add_comment(key=None, comment): Attach a comment or a set of comments to a key (or header-wide comment).
  - get_comment(key=None): Retrieve comments for a key or header-wide comments.

- Validation and typing
  - validate_item(key, value): Hook for type validation and constraints.
  - to_dict(): Serialize header to a plain dictionary representation.

- Serialization/Deserialization
  - from_dict(d): Create header from a dictionary representation.
  - to_bytes()/from_bytes(): Serialize to and from a binary-like representation if applicable.

Migration Guidance for Implementers
-----------------------------------
- Start by exposing the new interface alongside the old one, wrapping old calls with warnings.
- Use adapters to map old method names to the new semantics without changing behavior.
- Centralize validation logic to ensure consistent behavior across both interfaces.
- Add comprehensive tests for the new surface first; extend coverage to ensure parity with legacy behavior.
- Document all caveats (case sensitivity, key normalization, and special handling of comments/metadata).

Examples
--------

1) Accessing a value (old vs new)
- Old (illustrative):
  value = header['NAXIS']
  header['NAXIS'] = 2

- New (illustrative):
  value = header.get('NAXIS')
  header.set('NAXIS', 2)

2) Handling missing keys
- Old:
  value = header['BITPIX']  # KeyError if missing

- New:
  value = header.get('BITPIX', default=None)

3) Adding comments/metadata
- Old (conceptual):
  header.add_comment('NAXIS', 'Number of data axes')

- New (conceptual):
  header.add_comment('NAXIS', 'Number of data axes')
  or header.set_comment('NAXIS', 'Number of data axes')

4) Serialization
- Old:
  d = dict(header)

- New:
  d = header.to_dict()

Notes on Behavior and Semantics
--------------------------------
- Key normalization: Define whether header keys are case-insensitive and how to treat aliases.
- Data types: Clarify how numeric types, strings, and special FITS-like values are represented.
- Comments and metadata: Decide on how to store, retrieve, and format comments consistently.
- Mutability: Specify whether certain header items are read-only (e.g., computed from data) and how to handle attempts to mutate them.

Testing and Validation
----------------------
- Add regression tests for both old and new interfaces during the transition window.
- Validate round-trip serialization and deserialization to ensure no loss of metadata.
- Validate error handling for invalid keys and values.
- Include tests for edge cases (empty headers, very large headers, Unicode comments, etc.).

Backward Compatibility Considerations
--------------------------------------
- Maintain the old API in a shim layer during the transition period.
- Ensure warning messages point to the new API and provide concrete migration steps.
- Avoid silent failures; prefer explicit errors or warnings for deprecated usage.

Rollout Plan
------------
- Documentation: Update developer guides and user-facing docs with migration instructions.
- Communication: Announce the transition in release notes and via deprecation notices.
- Code changes: Implement the new interface with a minimal, well-documented surface; maintain the shim for a defined timeline.
- Community feedback: Gather feedback from downstream projects and incorporate practical improvements.

Appendix: Developer Checklist
-----------------------------
- [ ] Define the final public API surface for the new header interface.
- [ ] Implement a compatibility shim that maps old calls to the new interface.
- [ ] Emit deprecation warnings for all legacy usage paths.
- [ ] Write comprehensive tests for both interfaces during the transition.
- [ ] Document migration steps and update release notes.
- [ ] Plan removal of legacy code in a future major release.
