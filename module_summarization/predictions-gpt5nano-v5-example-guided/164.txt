# Conditions in PyTables

## Overview

PyTables provides a flexible condition syntax to filter data when reading from Tables and EArrays. Conditions are expressed as text strings that reference column names and combine comparisons with logical operators. The evaluation is performed per row, and PyTables can leverage indexes to speed up condition evaluation. Internally, NaN values are handled in a nan-aware manner to avoid misleading results in numeric columns.

Key internal concept: PyTables maintains a Condition Cache to optimize repeated condition evaluations. When a column index becomes dirty (out of sync with the underlying data), the Condition Cache is notified to revalidate and refresh its results.

Example usage typically occurs via read-related methods that accept a where-like condition string, such as read_where.

> Note: Some low-level interactions with the condition cache and index dirtiness are internal. The following example illustrates the internal hook for educational purposes.

```python
# Internal (advanced) example showing interaction with the condition cache
idx.dirty = True   # internally nails the condition cache
idx.dirty = False  # internally unnails the condition cache
```

---

## Key Components

- Condition syntax: Python-like boolean expressions using column names, numeric/string literals, and operators.
- Logical operators: and, or, not
- Comparison operators: >, <, >=, <=, ==, !=
- Parentheses: Used to control precedence in complex expressions
- Column references: Refer to table columns by name in the condition string
- Nan-aware handling: Comparisons involving NaN are treated using nan-aware semantics to avoid misleading results
- Index integration: When a column has an index, PyTables can use it to accelerate condition evaluation
- Condition Cache: An internal mechanism that caches results of condition evaluations for performance

---

## Usage

- Applying a condition when reading data (typical usage):

```python
import tables

with tables.open_file("data.h5", "r") as f:
    tab = f.get_node("/my_table")
    # Example: select rows where age > 50 and country is "US"
    data = tab.read_where('(age > 50) and (country == "US")')
```

- More complex conditions:

```python
# Example: multiple criteria with grouping
data = tab.read_where('(age >= 18) and (status == "active") and (score < 100)')
```

- Handling strings and numbers in conditions:

```python
# String literal in condition
data = tab.read_where('(city == "New York") or (city == "San Francisco")')
# Numeric comparison
data = tab.read_where('(salary > 70000) and (department != "Intern")')
```

- Advanced note on internals (not required for daily use):
  - The condition evaluation path may involve the internal Index object and the Condition Cache.
  - If an index-backed column changes (becomes out of sync), the Condition Cache is notified to re-evaluate as needed, ensuring correct results.

---

## Details

- Supported syntax:
  - Comparisons: >, <, >=, <=, ==, !=
  - Logical operators: and, or, not
  - Grouping with parentheses for precedence
  - Column references by name
  - String literals must be quoted (single or double quotes)
- NaN handling:
  - Numeric comparisons involving NaN values are handled in a nan-aware manner to avoid incorrect filtering results.
- Performance considerations:
  - Using indexed columns in conditions can significantly speed up reads.
  - Repeated or complex conditions may benefit from the internal Condition Cache; internal mechanisms ensure correctness when data changes or indexes are updated.

- Internal interactions:
  - The Index class exposes properties and hooks used by the condition machinery.
  - For example, a dirty index toggles nails on the condition cache, prompting a re-evaluation when needed (see internal example above).
