# File Objects

## Overview

This module provides the low-level infrastructure for representing and managing HDF5 object identifiers within h5py. It centers around an ObjectID base class that wraps an HDF5 identifier (hid_t) and tracks its lifecycle, including locking for thread-safety, a registry of live objects, and a mechanism to invalidate stale identifiers when nonlocal events (e.g., closing a file) occur.

Key goals:
- Ensure only one thread at a time can perform low-level API actions via a locking mechanism.
- Maintain a registry of live ObjectID instances to detect and invalidate stale identifiers (zombie IDs).
- Provide a clean lifecycle for identifiers so high-level objects (like FileID, GroupID) can rely on consistent state.

The module includes both the locking infrastructure and the registry logic, which are used by higher-level file and object wrappers.

    - _phil is the lock (FastRLock when locking is enabled, otherwise a fallback)
    - with_phil decorator to apply the lock around helper functions

- Registry
  - A registry dictionary that tracks live ObjectID instances by their Python id
  - The registry stores weak references to ObjectID objects so deallocation can occur safely
  - print_reg utility for debugging the registry contents
  - nonlocal_close function to invalidate stale identifiers when nonlocal events occur

  Key details:
  - registry maps id(obj) -> weakref(obj)
  - Objects are added in ObjectID.__cinit__ and removed in ObjectID.__dealloc__
  - nonlocal_close traverses the registry to reset obj.id to 0 for stale objects

- ObjectID class
  - Base class representing an HDF5 identifier
  - Attributes:
    - id: the underlying HDF5 id (hid_t)
    - locked: flag indicating if the object cannot be closed
    - _hash, _pyid: internal tracking fields
  - Key properties and methods:
    - fileno: returns a tuple identifying the file object by querying HDF5
      through H5Gget_objinfo
    - valid: checks whether this h5py object is still valid
  - Lifecycle:
    - __cinit__(self, id_): registers the object in the registry and stores its id
      - with _phil: set id, locked, _pyid; add to registry
      - optional debug print when DEBUG_ID is enabled
    - __dealloc__(self): unregisters and optionally decrefs
      - with _phil: optional debug print; if object is valid and not locked, decrease reference count
      - emits a warning if a reference counting issue is detected

## Usage

- File and object identifiers
  - FileID, GroupID, and other object wrappers derive from ObjectID and share its lifecycle semantics.
  - The fileno property provides a low-level file-number pair that identifies the object for the HDF5 layer.

- Checking validity
  - Use the valid property to determine if the object is still a live HDF5 object:
    - if obj.valid: proceed with operations
- Handling nonlocal closures
  - When a nonlocal event occurs (e.g., closing a file or a strong close), the system may invalidate identifiers.
  - Invoke nonlocal_close() to walk the registry and reset stale IDs to 0:
    - nonlocal_close()

- Basic example (conceptual)
  - Accessing the file number for a file/object:
    - fileno_tuple = obj.fileno
  - Verifying an object is still valid:
    - if obj.valid:
        # perform operations
- Debugging and registry inspection
  - Use print_reg() to inspect current live ObjectID instances and their types
    - print_reg()

## Details

- Thread safety and lifecycle
  - The low-level HDF5 API interactions are serialized with a single recursive lock to prevent races when identifiers may be mutated or invalidated from different threads.
  - This approach avoids partially updated identifiers and ensures consistent state when objects are closed or replaced.

- Zombie identifiers and nonlocal invalidation
  - With HDF5 1.8+, identifiers can be reused soon after closure, which may lead to stale references if an ObjectID remains alive.
  - The registry keeps track of live ObjectID instances. When nonlocal events occur, nonlocal_close() iterates the registry and sets invalid identifiers (obj.id = 0) for non-locked, non-died objects, preventing incorrect behavior.

- Mapping and lifecycle
  - The registry maps Python-level identifiers to weak references of ObjectID instances:
    - Objects are added in ObjectID.__cinit__ and removed by ObjectID.__dealloc__.
  - On deallocation, the code handles reference counting and may raise a warning if issues arise:
    - H5Idec_ref(self.id) < 0 triggers a warning about a reference counting issue.

- High-level relationship
  - FileID objects (and other object wrappers) rely on this base class to manage their underlying HDF5 identifiers, ensuring safe lifecycle and predictable behavior in multi-threaded or complex usage scenarios.
