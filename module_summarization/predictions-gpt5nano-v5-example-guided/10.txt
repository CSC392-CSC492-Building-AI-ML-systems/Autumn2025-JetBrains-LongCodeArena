# Using metadata

## Overview
This documentation covers using metadata for authenticating and configuring interactions with an ftrack server via the metadata-enabled session framework. The core idea is to attach authentication context (metadata) to requests and manage session-wide configuration (caching, plugins, event hubs, and schema caching) through a consistent interface.

The metadata mechanism is implemented through:
- SessionAuthentication: attaches authentication metadata to HTTP requests.
- Session: central entry point for interacting with the ftrack server, including environment-based configuration, caching, plugins, and optional embedded event hub support.

Key behavior:
- Requests are augmented with authentication headers, enabling secure communication with the server.
- Sessions can be configured via explicit arguments or environment variables, and can opt into features such as auto-population of entity attributes, schema caching, and local/remote event hub support.

Code reference (highlights):
- Attaching authentication to requests
  - The SessionAuthentication class modifies each request to include the necessary metadata headers:
    - ftrack-api-key
    - ftrack-user
  - Example:
    - request.headers.update({'ftrack-api-key': self.api_key, 'ftrack-user': self.api_user})

- Core session configuration
  - The Session class handles server URL, API key, and API user configuration, with fallbacks to environment variables:
    - FTRACK_SERVER
    - FTRACK_API_KEY (or FTRACK_APIKEY)
    - FTRACK_API_USER

  - Example constructor parameters:
    - server_url, api_key, api_user
    - auto_populate, plugin_paths, cache, cache_key_maker
    - auto_connect_event_hub, schema_cache_path, plugin_arguments

  - If credentials are not provided, the session will raise a TypeError with guidance to supply via arguments or environment variables.

## Key Components
- SessionAuthentication
  - Purpose: Attach ftrack session authentication information to HTTP requests.
  - Core behavior: Before a request is sent, the class adds headers:
    - ftrack-api-key: <api_key>
    - ftrack-user: <api_user>

  - Reference snippet:
    - __call__(self, request):
        request.headers.update({
            'ftrack-api-key': self.api_key,
            'ftrack-user': self.api_user
        })
        return request

- Session
  - Purpose: Isolated session for interaction with an ftrack server.
  - Core responsibilities:
    - Manage server_url, api_key, and api_user (with environment variable fallbacks)
    - Support auto_populate of entity attributes
    - Provide a caching mechanism (with a top-level memory cache by default)
    - Load and manage plugins (via plugin_paths and plugin_arguments)
    - Optionally connect to an embedded event hub (auto_connect_event_hub)
    - Optional schema caching (schema_cache_path)
    - Track operations and control whether to record operations
  - Important default behavior:
    - If server_url is not provided, FTRACK_SERVER environment variable is used (or an error is raised if absent)
    - If api_key/api_user are not provided, environment variables (FTRACK_API_KEY, FTRACK_API_USER) are used or a best-effort guess is attempted (e.g., using getpass for the user)

## Usage

Basic session creation (explicit credentials)
```
import ftrack_api

session = ftrack_api.Session(
    server_url='https://my.ftrack.server',
    api_key='my_api_key',
    api_user='my_api_user'
)
```

Environment-based session creation (using environment variables)
- Set these in your environment:
  - FTRACK_SERVER=https://my.ftrack.server
  - FTRACK_API_KEY=your_api_key
  - FTRACK_API_USER=your_username

```
import ftrack_api

session = ftrack_api.Session()
```

Disabling auto-population of entity attributes
```
session = ftrack_api.Session(auto_populate=False)
```

Using a custom cache
```
from ftrack_api.cache import MemoryCache

session = ftrack_api.Session(cache=MemoryCache())
```

Enabling schema caching
```
session = ftrack_api.Session(schema_cache_path='/path/to/schema_cache')
```

Plugin configuration
```
session = ftrack_api.Session(
    plugin_paths=['/path/to/plugins'],
    plugin_arguments={'arg1': 'value1'}
)
```

Referencing the internal authentication mechanism
```
# Direct usage of authentication metadata (illustrative)
auth = ftrack_api.SessionAuthentication(api_key='my_api_key', api_user='my_api_user')
# This auth object can be used with HTTP requests to attach necessary headers.
```

## Details

- Authentication headers
  - Every request made through the session (via the underlying HTTP client) includes:
    - ftrack-api-key: <api_key>
    - ftrack-user: <api_user>

- Configuration sources
  - Server URL
    - Explicit server_url argument
    - Fallback to environment variable FTRACK_SERVER
  - API credentials
    - Explicit api_key and api_user arguments
    - Fallback to environment variables FTRACK_API_KEY (or FTRACK_APIKEY) and FTRACK_API_USER
    - If api_user cannot be inferred, an attempt may be made to derive it (e.g., via getpass), but a TypeError will be raised if it cannot be determined

- Features
  - Auto population of entity attributes (default: True)
  - Caching support (layered cache with top-level memory cache)
  - Event hub integration (embedded hub can be auto-connected)
  - Schema caching (path-based or disabled)
  - Plugin discovery and argument handling

- Behavior notes
  - The session is designed to operate safely even if a suitable cache cannot be configured (the constructor can continue with None for cache)
  - If a registered plugin requires a connected event hub, its status should be checked explicitly by the plugin
  - Subscribing to events does not require a connected event hub; connection is done in the background if enabled

This metadata-driven approach centralizes how authentication and configuration are propagated to all server interactions, enabling consistent and secure access to ftrack resources.
