# BaseUser — Generate documentation for this module

## Overview
BaseUser is a Piccolo ORM Table that provides a basic user model with authentication support. It defines common user fields (username, password, email, etc.) and a set of utilities for password hashing, validation, login, and password updates. The database table is named piccolo_user.

Key features:
- Passwords are stored in a salted PBKDF2-HMAC-SHA256 format.
- Password validation enforces configurable length requirements.
- Convenience methods for hashing, login, and password updates.
- Readable representation for debugging and admin interfaces.

Code reference (class skeleton):
- class BaseUser(Table, tablename="piccolo_user"):
- Fields: id, username, password, first_name, last_name, email, active, admin, superuser, last_login
- Password handling is integrated into attribute setting and updates

## Key Components

- Fields and schema
  - id: Serial
  - username: Varchar(length=100, unique=True)
  - password: Secret(length=255)
  - first_name: Varchar(null=True)
  - last_name: Varchar(null=True)
  - email: Varchar(length=255, unique=True)
  - active: Boolean(default=False)
  - admin: Boolean(default=False, help_text="An admin can log into the Piccolo admin GUI.")
  - superuser: Boolean(default=False, help_text="If True, this user can manage other users' passwords in the Piccolo admin GUI.")
  - last_login: Timestamp(null=True, default=None, required=False, help_text="When this user last logged in.")
- Password policy
  - _min_password_length = 6
  - _max_password_length = 128
  - _pbkdf2_iteration_count = 600_000
- Password hashing and salting
  - get_salt(): returns a hex salt
  - hash_password(password, salt="", iterations=None): returns a string in the format
    pbkdf2_sha256$<iterations>$<salt>$<hash>
- Readable representation
  - get_readable(): returns a Readable with template "%s" and columns [cls.username]
- Password validation
  - _validate_password(password): raises ValueError if invalid
- Password management
  - __init__(self, **kwargs): hashes provided password if not already hashed
  - __setattr__(self, name, value): hashes password on set if needed
  - split_stored_password(password): splits hashed password into [algorithm, iterations, salt, hash]
  - update_password_sync(user, password): sync wrapper for update_password
  - update_password(user, password): updates password for a user by id or username
  - login_sync(username, password): sync wrapper for login
  - login(username, password): validates credentials and updates last_login if matched

Notes about implementation:
- Passwords provided in plain text are automatically hashed on assignment or creation.
- The stored password format enables upgrading iteration counts transparently during login.
- Login requires that the username exists and the provided password, when hashed with the stored salt and iterations, matches the stored value.
- The module uses run_sync for synchronous wrappers around async operations.

## Usage

Create a user (password is hashed automatically if plain text is provided):
- Async:
  await BaseUser.create(
    username="alice",
    password="password123",
    email="alice@example.com"
  )

Hash a password directly:
- Async:
  hashed = BaseUser.hash_password("password123")
  print(hashed)

Login a user:
- Async:
  user_id = await BaseUser.login("alice", "password123")
  # user_id is an integer id on success, or None on failure

Sync wrappers (where you need synchronous calls):
- Sync password update:
  BaseUser.update_password_sync(user_identifier, "newpassword456")
- Sync login:
  BaseUser.login_sync("alice", "password123")

Update a user’s password (by id or username):
- Async:
  await BaseUser.update_password(user_id_or_username, "newpassword456")

Readable representation:
- Get a human-friendly representation for admin interfaces:
  readable = BaseUser.get_readable()
  print(readable.template % "alice")

## Details

Security and configuration notes:
- The password hashing uses PBKDF2-HMAC-SHA256 with a large iteration count (600,000) by default.
- The iteration count, minimum/maximum password lengths, and the salt size are defined by class attributes and can be adjusted to meet evolving security requirements.
- Passwords are stored with the format: pbkdf2_sha256$<iterations>$<salt>$<hash>
- When setting password values, the code ensures they are stored as hashed values, preventing accidental plaintext storage.
- The login method updates last_login upon successful authentication and includes safeguards against excessively long inputs.

Integration considerations:
- The table name for this model is piccolo_user.
- The module exposes a mix of async and sync helpers to fit both asynchronous and synchronous application codebases.
- When migrating to new iteration counts, login logic can upgrade stored passwords to the current standard during authentication.
