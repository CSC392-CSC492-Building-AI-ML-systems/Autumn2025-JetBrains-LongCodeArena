# Calculator Interface

## Overview
This module implements a Calculator Interface for converting between latitude-longitude coordinates and Guyou map coordinates using elliptic functions. It provides a programmable API that accepts scalar or array inputs and leverages SciPy's elliptic functions for computation.

Key public function:
- ellipticJi(u, v, m): returns the values [sn, cn, dn] for the complex argument u + i v with parameter m.

The implementation supports broadcasting over arrays and differentiates between scalar and non-scalar inputs for convenient results.

The code includes licensing and provenance notes from an external projection library and contains a small, commented-out alternative (mpmath-based) approach.

## Key Components
- ellipticJi(u, v, m)
  - Returns: [sn, cn, dn](u + i v | m)
  - Accepts scalars or arrays; handles broadcasting.
  - Internally delegates to helper functions for real, imaginary, and mixed cases.
- _ellipticJi_real(u, m)
  - Helper: computes ellipticJ(u, m) and returns (sn, cn, dn).
- _ellipticJi_imag(v, m)
  - Helper: computes the imaginary component using ellipticJ(v, 1 - m) and derives (1j*sn/cn, 1/cn, dn/cn).
- _ellipticJi(u, v, m)
  - Core computation for mixed (u, v) inputs using ellipticJ on u and v.

Note: The module uses SciPy’s elliptic functions (ellipj) and provides a compatibility layer to support both scalar and array inputs. There is a non-active, commented block showing a potential alternative implementation using mpmath.

## Usage

- Public API (example with scalars)
  - from calculator import ellipticJi
  - sn, cn, dn = ellipticJi(0.5, 0.0, 0.5)

- Public API (example with arrays)
  - import numpy as np
  - u = np.array([0.0, 0.2])
  - v = np.array([0.0, 0.0])
  - m = 0.5
  - sn, cn, dn = ellipticJi(u, v, m)

Notes:
- The return shape adapts to scalar vs. array inputs. When inputs are scalar, the function returns a 3-tuple (sn, cn, dn); otherwise it returns an array with the computed values.
- The implementation includes a placeholder for logging or expansion (e.g., handling more edge cases) and explicitly states that special-case handling is not currently implemented.

## Details

- Dependencies: NumPy, SciPy (scipy.special.ellipj as ellipticJ, ellipkinc as ellipticF)
- Technical approach:
  - Real inputs are computed via ellipticJ(u, m).
  - Imaginary inputs use ellipticJ(v, 1 - m) to derive the imaginary components.
  - Mixed inputs are handled by combining real and imaginary computations.
- Design considerations:
  - Broadcast support via np.broadcast_arrays.
  - Explicitly notes that special cases are ignored for now.
- Licensing and provenance:
  - The header includes BSD-style licensing text from d3/d3-geo-projection and attribution information.
  - The file contains a legacy, commented-out mpmath-based alternative implementation for reference.
