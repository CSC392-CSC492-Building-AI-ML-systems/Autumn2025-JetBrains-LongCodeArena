Unified renderable interface
============================

Overview
--------
This module defines a unified interface for renderable components. At its core is AbstractRenderable, which standardizes how an object can render a template using Django’s render_to_string, with an extensible context. Additional mixins provide CSS class handling (AbstractRenderableWithCss) and BEM-style naming support (AbstractBemRenderable).

Key Components
--------------

- join_css_classes_list (helper)
  - Joins a list of CSS class names into a single string using two spaces as separator.

  Example:
  - Provided in code as:
    - return '  '.join(css_classes_list)

- AbstractRenderable
  - The core interface for rendering. Subclasses typically specify a template and optionally customize context data.

  Primary features:
  - template_name attribute: default template name (or None to be provided by override)
  - get_template_names(self): returns the template name(s) to render. If template_name is not set, raises NotImplementedError.
  - get_context_data(self, request=None): returns the context for rendering. Defaults to {'me': self}.
  - render(self, request=None, extra_context_data=None): renders the template with the merged context. If request is provided, it is forwarded to get_context_data and render_to_string.

  Key code excerpts:
  - get_template_names
    - "Defaults to template_name. Raises NotImplementedError if not set."
  - get_context_data
    - "Defaults to {'me': self}"
  - render
    - Compiles context and calls render_to_string with template_name, context, and request.

  Example usage:
  - Subclass should define template_name and can override get_context_data:
    - class MyRenderable(AbstractRenderable):
        template_name = 'my/template.html'
        def get_context_data(self, request=None):
            ctx = super().get_context_data(request=request)
            ctx['extra'] = 'value'
            return ctx

- AbstractRenderableWithCss
  - Extends AbstractRenderable with a unified API for CSS classes.

  Primary features:
  - get_base_css_classes_list(self) -> list
  - get_extra_css_classes_list(self) -> list
  - get_css_classes_list(self) -> list:
    - Combines base and extra CSS classes. Intended to be overridden by implementing get_css_classes_list.
    - Returns a concatenated list of base and extra CSS classes.
  - get_test_css_class_suffixes_list(self) -> list:
    - Returns suffixes used for test CSS class generation (empty by default).
  - css_classes (property):
    - Joins the CSS classes from get_css_classes_list into a string.
    - If DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES is enabled, appends test CSS classes via cradmin_test_css_class.
  - get_css_classes_string(self) -> str:
    - Deprecated. Returns the same as css_classes with a deprecation warning.

  Important notes:
  - The css_classes property relies on a lazy import to avoid circular imports:
    from django_cradmin.templatetags import cradmin_tags
  - Use get_css_classes_list to define the classes; override only this method.
  - The module provides a deprecation warning for get_css_classes_string.

  Example:
  - class MyCssRenderable(AbstractRenderableWithCss):
      def get_base_css_classes_list(self):
          return ['my-base']

      def get_extra_css_classes_list(self):
          return ['extra']

      @property
      def css_classes(self):
          return super().css_classes  # or customize further

- AbstractBemRenderable
  - Base class for renderables using BEM (Block-Element-Modifier) naming.

  Constructor:
  - __init__(self, bem_block=None, bem_element=None, bem_variant_list=None, extra_css_classes_list=None)

  Behavior:
  - bem_block and bem_element are mutually exclusive. Providing both raises ValueError.
  - bem_element must contain '__' to denote an element (e.g., 'block__element').
  - Stores provided values for later usage (not all are exposed in this excerpt).

  Key methods:
  - get_test_css_class_suffixes_list(self) -> list: same purpose as in AbstractRenderableWithCss (empty by default).
  - bem_block_or_element property:
    - Returns get_bem_block() or falls back to get_bem_element().
  - get_bem_block(self):
    - Returns the BEM block string.
  - get_bem_element(self):
    - Returns the BEM element string. (Note: In the provided source, this method returns self._b, which appears to be a bug and should probably return self._bem_element.)

  Example usage:
  - class MyBemRenderable(AbstractBemRenderable):
      def __init__(self, bem_block='my-block', bem_element=None):
          super().__init__(bem_block=bem_block, bem_element=bem_element)

Usage
-----
Rendering an instance is straightforward:

- Subclass AbstractRenderable (or one of its siblings) and specify a template.
- Optionally customize context data via get_context_data.
- Use render(...) to produce the final string.

Code example (minimal):
- class MyRenderable(AbstractRenderable):
    template_name = 'my/template.html'
    def get_context_data(self, request=None):
        ctx = super().get_context_data(request=request)
        ctx['title'] = 'Hello'
        return ctx

- obj = MyRenderable()
- html = obj.render(request=request)

CSS and Test Classes
--------------------
If you need CSS class management, extend AbstractRenderableWithCss and implement:
- get_css_classes_list(): return the desired list of classes
- get_test_css_class_suffixes_list(): return any test CSS suffixes if you use test CSS tagging

The css_classes property provides the final string of class names, with optional test classes appended when DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES is enabled.

Details and Considerations
--------------------------
- The core rendering flow is centralized in AbstractRenderable.render, which forwards the optional HttpRequest to get_context_data and to Django’s render_to_string.
- get_template_names raises NotImplementedError unless template_name is set or overridden.
- Deprecations:
  - get_css_classes_string is deprecated in favor of the css_classes property.
- BEM rendering with AbstractBemRenderable uses bem_block_or_element to select the active BEM target, and enforces correct usage (only one of block/element, and element must include __).

Notes
-----
- There appears to be a minor bug in AbstractBemRenderable.get_bem_element() returning self._b instead of self._bem_element. Documentation reflects the code as-is; consider patching for correctness.
