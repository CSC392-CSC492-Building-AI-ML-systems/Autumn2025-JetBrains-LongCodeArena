# Table Data

## Overview
This document describes the table data handling utilities related to verification, error reporting, and change notification used when working with tabular data (e.g., FITS table data). It covers the verification framework, the structure for collecting and formatting errors, and the observer pattern that enables reacting to changes in table data.

## Key Components

- _Verify: Shared verification helpers
  - Purpose: Provide a consistent mechanism to verify values within a table-like object, produce user-facing messages, and optionally fix issues.
  - Core methods:
    - run_option(option="warn", err_text="", fix_text="Fixed.", fix=None, fixable=True)
      - Executes a specific verification option and returns a tuple (fixable, text).
      - Behavior:
        - If option is "warn" or "exception", the issue is not fixable (fixable becomes False).
        - If fixable is False, the text is prefixed with "Unfixable error: ".
        - If fixable and a fix callback is provided, it executes fix() and appends the fix_text to the message.
      - Example:
        - v = _Verify()
        - fixable, text = v.run_option(option="warn", err_text="Invalid value in table cell")
        - print(text)  # may be a warning message; no fix attempted
    - verify(option="warn")
      - Verifies all values in the instance using the specified option.
      - Accepts options in VERIFY_OPTIONS (including combinations like "fix+warn").
      - Breaks the option into fix-related and reporting-related parts if a "+" is present.
      - Collects error messages via self._verify(opt) and formats output according to the option.
      - Emits warnings or raises VerifyError as appropriate.
  - Notes:
    - VERIFY_OPTIONS lists the supported modes, such as "ignore", "warn", "exception", "fix", "silentfix", and combinations with "+ignore", "+warn", "+exception".
    - See the code block for the exact option set and the logic that determines how messages are reported and whether fixes are attempted.

- _ErrList: Nested error reporting structure
  - Purpose: Represent and format verification errors in a nested, hierarchical fashion.
  - Structure:
    - Inherits from list; each element can be either a simple error tuple or another _ErrList for nested sections.
    - Attributes:
      - unit: A label used when presenting nested error sections (e.g., "Element 0", etc.).
  - Key methods:
    - __str__(self)
      - Returns a formatted string of all error messages, flattened with appropriate indentation.
    - iter_lines(self, filter=None, shift=0)
      - Generator that yields (header, text) pairs for each line, applying indentation based on the nesting level.
      - First pass prints top-level messages; second pass traverses nested _ErrList items, optionally filtering, and adds headers for nested sections when present.
  - Example:
    - The nested error structure is used to present multiple related issues in an organized, readable way. See __str__ and iter_lines for formatting behavior.

- NotifierMixin: Change notification for table data
  - Purpose: Enable observer/listener communication so that changes to a table-like object can notify registered listeners.
  - Behavior (as described in the class docstring):
    - Listeners can be registered via an internal mechanism (e.g., _add_listener) using weak references to prevent memory leaks.
    - Changes on the object can be broadcast to listeners via _notify(change_type, *args, **kwargs), which will attempt to call corresponding update methods on listeners (e.g., listener._update_change_type(...)).
    - Listeners that do not implement the appropriate update method are ignored.
  - Example (from the class docstring):
    - class Widget(NotifierMixin):
        state = 1
        def __init__(self, name):
            self.name = name
        def update_state(self):
            self.state += 1
    - This pattern allows a table-like data structure to inform other components about changes (e.g., data updated, schema changed).

## Usage

- Basic verification of table data
  - Create a verification helper (or subclass that defines _verify) and use _Verify to report issues.
  - Example:
    - v = _Verify()
    - fixable, text = v.run_option(option="warn", err_text="Invalid value in table cell")
    - print(text)

- Verifying with reporting and fixing behavior
  - Options may be combined, such as "fix+warn" or "silentfix+ignore".
  - The verify flow splits the option into fix and report components, handles fix callbacks, and emits warnings or raises errors accordingly.
  - Refer to VERIFY_OPTIONS for all supported modes.

- Error reporting with nested structure
  - When multiple related issues are detected, _ErrList organizes messages in a hierarchical form.
  - Iterate and format messages with iter_lines or __str__ to present user-friendly output.

- Observing table data changes
  - By mixing in NotifierMixin, a table-like object can support listeners that respond to changes.
  - Register listeners using the mixin’s internal mechanism, and trigger updates via _notify.

Details
- The verification framework is designed to work with table-like data structures, providing standardized messaging, optional automatic fixes, and clear reporting.
- Error messages include a header note about indexing (e.g., “Note: astropy.io.fits uses zero-based indexing.”) to aid users when interpreting positions within tables.
- The NotifierMixin describes a robust observer pattern using weak references to avoid memory leaks, enabling responsive and decoupled components that react to table data changes.
