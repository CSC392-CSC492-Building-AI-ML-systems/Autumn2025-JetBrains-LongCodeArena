# Fields - Examples and usage

## Overview
This module defines the base Field type used to describe model attributes and how they map to database columns. It supports a mixin-based mechanism to specify the Python type (through field_type), per-DB overrides, and basic value conversion between Python and database representations.

Key features include:
- A mixin-based approach to set field_type (via _FieldMeta)
- Basic value conversion with to_db_value and to_python_value
- Validation for common constraints (e.g., primary key vs nullability, indexing)
- Per-DB overrides for dialect-specific behavior through inner _db__ dialect classes
- Descriptive class attributes (SQL_TYPE, GENERATED_SQL, etc.)

## Key Components
- Field: Base field type with common options (source_field, generated, pk, null, default, unique, index, description)
- _FieldMeta: Metaclass enabling field_type inference when mixing Field with a type
- Constants for referential actions (for describe/usage clarity): CASCADE, RESTRICT, SET_NULL, SET_DEFAULT
- to_db_value(value, instance): Converts Python value to a database-compatible value
- to_python_value(value): Converts DB value to Python value
- Field attributes used by ORMs and schema generation:
  - field_type: Python type for this field (set by _FieldMeta)
  - indexable, has_db_field, skip_to_python_if_native, allows_generated
  - function_cast: Optional casting term for DB emulation
  - SQL_TYPE, GENERATED_SQL: Dialect-specific SQL details
- Per-DB overrides: class _db__{dialect} with overrides for SQL_TYPE, skip_to_python_if_native, and function_cast

Code references of interest:
- to_db_value and to_python_value perform casts only when necessary:
  - If value is None or already of field_type, it is returned as-is
  - Otherwise, it is cast: self.field_type(value)
- Validation in __init__ prevents nonsensical configurations:
  - A field cannot be both pk=True and null=True
  - If pk is True, index and unique are automatically enabled

## Usage

- Defining a custom field type using a mixin (to set field_type)
  Example: create a field whose Python type is str (via mixin)
  ```python
  from tortoise.fields import Field

  class StringField(Field, str):
      pass
  ```
  This defines a new field type where field_type is str, inferred by the metaclass.

- Using a custom field in a model (illustrative; assumes a Model base is available)
  ```python
  from tortoise.models import Model

  class User(Model):
      # A string field with default behaviors
      name = StringField(null=False, unique=True)

      # An integer field using a mixin-based approach
      class IntField(Field, int):  # Optional inline definition
          pass

      age = IntField(null=True)
  ```

- Value conversion (to DB and back)
  ```python
  sf = StringField()  # field_type inferred from mixin (str)

  # Python to DB representation
  db_value = sf.to_db_value(123, None)  # "123" (string)

  # DB to Python representation
  py_value = sf.to_python_value("456")  # "456" (string)
  ```

- Constraints and validation
  The constructor enforces sensible constraints. For example, trying to create a field with pk=True and null=True will raise a ConfigurationError:
  ```python
  from tortoise.exceptions import ConfigurationError

  try:
      bad = StringField(pk=True, null=True)
  except ConfigurationError as e:
      print("Configuration error:", e)
  ```

- Per-DB overrides (dialect-specific behavior)
  You can customize behavior for a specific database dialect by providing an inner class named _db__<dialect>:
  ```python
  from typing import Optional
  from pypika.terms import Term

  class StringField(Field, str):
      pass

  class _db_sqlite:
      SQL_TYPE = "VARCHAR(40)"
      skip_to_python_if_native = False

      def function_cast(self, term: Term) -> Term:
          # Example of a custom cast for SQLite
          return functions.Cast(term, SqlTypes.NUMERIC)
  ```
  The inner _db_sqlite class demonstrates how to override type mappings and provide a custom function_cast for dialect-specific emulation.

- Generated fields and SQL generation
  If a field supports generation (autogenerated DB values), set allows_generated and GENERATED_SQL accordingly:
  ```python
  class AutoIncIntField(Field, int):
      allows_generated = True
      GENERATED_SQL = "GENERATED ALWAYS AS IDENTITY"
  ```
  This example illustrates intent; actual syntax depends on the ORM and DB dialect.

## Details

- Field attributes and behavior
  - field_type is a read-only property set by _FieldMeta. It determines how values are interpreted in Python and how they are converted to DB values.
  - indexable controls whether the field can be indexed by the database engine. If indexable is False, attempting to set index or unique on the field will raise a ConfigurationError.
  - has_db_field indicates whether there is a direct DB column for this field (some fields can be virtual).
  - skip_to_python_if_native can be used to optimize conversions if the DB driver natively supports the Python type.
  - allows_generated and GENERATED_SQL relate to DB-generated values (e.g., auto-increment fields).
  - function_cast is an optional callable that takes a Term and returns a Term, used when the DB needs emulation help for casting.

- Per-DB overrides
  The inner dialect-specific classes (e.g., _db_sqlite) allow you to tailor SQL type definitions, Python conversion behavior, and other dialect-specific details. This enables database-specific optimizations or compatibility work without changing the global field API.

- Primary key and indexing
  If pk is True, the field becomes indexed and unique automatically. A conflicting combination (pk=True and null=True) is rejected at initialization to prevent invalid schemas.

- Source field and documentation
  - source_field lets you map a model field to a specific DB column name if you want a name different from the model attribute.
  - description provides a human-readable explanation and can appear in schema descriptions and database comments.
