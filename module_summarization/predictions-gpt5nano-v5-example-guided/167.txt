# Library conventions

## Overview
This document captures the library-wide conventions used in the Python control systems library. It covers public API exposure, docstring style, module and class layout, and common patterns illustrated by the existing source (e.g., time-domain utilities like sample_system and c2d). Adhering to these conventions ensures consistency, readability, and ease of use across the codebase.

## Key Components

- Public API exposure
  - Public symbols are explicitly declared via __all__ in modules.
  - Example (from dtime.py):
    - __all__ = ['sample_system', 'c2d']
  - This makes it clear which functions are part of the module’s public interface.

- Time-domain utilities (examples)
  - sample_system(sysc, Ts, method='zoh', alpha=None, prewarp_frequency=None)
    - Converts a continuous-time system to discrete time by sampling.
    - Docstring follows NumPy/SciPy style with sections: Parameters, Returns, Notes, Examples.
    - Input validation is performed (e.g., ensuring the system is continuous time via isctime).
    - Implementation delegates to the underlying system object: sysc.sample(...)
  - c2d(sysc, Ts, method='zoh', prewarp_frequency=None)
    - A wrapper that delegates to sample_system to perform the conversion.
    - Docstring mirrors sample_system and Conveys the same behavior, returning a discrete-time system of the same class.

- Docstring conventions
  - NumPy/SciPy style docstrings with cross-references.
  - Use of Sphinx-like references for types and classes, e.g.:
    - sysc : LTI (:class:`StateSpace` or :class:`TransferFunction`)
  - Cross-references to methods using :meth:`...`, classes with :class:`...`.
  - Sections typically include: Parameters, Returns, Notes, Examples.
  - Examples demonstrate typical usage in doctest-like form.

- Class and API design
  - Functions operate on LTI objects (StateSpace, TransferFunction).
  - APIs are designed to be intuitive and consistent with existing methods (e.g., .sample on LTI objects).
  - Behavior and return types are described in the docstrings (e.g., sysd : linsys).

- Licensing and metadata
  - Files contain a licensing header with copyright and redistribution terms.
  - Metadata such as Author and Date is included in the header.
  - This header is part of the convention for all source files.

- Naming conventions
  - Public functions follow snake_case naming (e.g., sample_system, c2d).
  - Classes use CamelCase (e.g., StateSpace, TransferFunction).
  - Module names are descriptive (e.g., dtime for discrete-time utilities).

- Documentation style and references
  - Documentation uses cross-references to related components and methods.
  - See sections in docstrings linking to StateSpace.sample and TransferFunction.sample for deeper details.

- Error handling and validation
  - Input validation is explicit (e.g., continuous-time verification via isctime).
  - Meaningful error messages are raised (e.g., ValueError when the input is not continuous time).

- Examples and doctests
  - Examples illustrate typical usage and expected results.
  - Doctest-like snippets help with in-function testing and user guidance.

## Usage

- Documenting a new function in the library
  - Follow the same docstring structure as existing functions:
    - Provide a clear Parameters section with types.
    - Include a Returns section describing the output type.
    - Add Notes for any caveats or behavior details.
    - Include Examples showing typical usage, ideally using doctest-style code.
  - Export the function in __all__ to expose it as part of the public API.

- Example usage from existing conventions
  - Sampling a continuous-time system:
    ```
    >>> sysc = TransferFunction([1], [1, 2, 1])
    >>> sysd = sample_system(sysc, 1, method='bilinear')
    ```
  - The c2d function delegates to sample_system, maintaining a simple, consistent interface:
    ```
    #  Call the sample_system() function to do the work
    sysd = sample_system(sysc, Ts,
        method=method, prewarp_frequency=prewarp_frequency)
    ```

## Details

- Documentation style
  - NumPy/SciPy docstring conventions are used for consistency and readability.
  - Cross-references with :class:`StateSpace`, :class:`TransferFunction`, and :meth:`StateSpace.sample` help users navigate the API.

- Code structure alignment
  - Public API is explicitly declared via __all__ in each module.
  - Functions operate on well-defined types (LTI objects) and return similarly typed results.

- Example alignment
  - The provided examples demonstrate typical usage and reinforce the intended workflow (continuous to discrete conversion via sampling).

- Notes on extensibility
  - The conventions accommodate additional sampling methods (e.g., 'bilinear', 'zoh', 'gbt'), with clear documentation of method-specific arguments (e.g., prewarp_frequency only valid for bilinear).

If you add new modules or functions, mirror these conventions to maintain coherence across the library.
