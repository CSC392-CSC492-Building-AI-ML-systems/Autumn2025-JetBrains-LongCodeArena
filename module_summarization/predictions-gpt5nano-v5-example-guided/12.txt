A quick dive into using the API
================================

Overview
--------
This tutorial introduces the core entry point for interacting with an ftrack server: the Session class and its associated authentication helper. It covers how to configure authentication, establish a session, and how features such as lazy loading, caching, and event hub interaction are wired into the API.

Key Components
--------------

SessionAuthentication
~~~~~~~~~~~~~~~~~~~~~

Attaches ftrack authentication information to HTTP requests by adding the required headers.

.. code-block:: python

   class SessionAuthentication(requests.auth.AuthBase):
       '''Attach ftrack session authentication information to requests.'''

       def __init__(self, api_key, api_user):
           '''Initialise with *api_key* and *api_user*.'''
           self.api_key = api_key
           self.api_user = api_user
           super(SessionAuthentication, self).__init__()

       def __call__(self, request):
           '''Modify *request* to have appropriate headers.'''
           request.headers.update({
               'ftrack-api-key': self.api_key,
               'ftrack-user': self.api_user
           })
           return request

Session
~~~~~~~

An isolated session for interaction with an ftrack server.

.. code-block:: python

   session = ftrack_api.Session(
       server_url='https://ftrack.example.com',
       api_key='your_api_key',
       api_user='your_username',
       auto_populate=True,           # optional
       plugin_paths=None,              # optional
       cache=None,                     # optional
       cache_key_maker=None,           # optional
       auto_connect_event_hub=None,    # optional
       schema_cache_path=None,           # optional
       plugin_arguments=None           # optional
   )

Key constructor parameters (high level)
- server_url: URL of the ftrack server (including port). If omitted, FTRACK_SERVER env var is used.
- api_key: API key for authentication. If omitted, FTRACK_API_KEY or FTRACK_APIKEY env vars are used.
- api_user: Username for recording operations. If omitted, FTRACK_API_USER or a system user may be used.
- auto_populate: If True, accessing attributes will fetch missing data from the server on demand.
- auto_connect_event_hub: Controls whether a background event hub connection is started automatically.
- schema_cache_path: Path to enable schema caching. If unset, FTRACK_API_SCHEMA_CACHE_PATH is used, or a temporary dir is created. Set to False to disable caching.
- cache, cache_key_maker: Caching configuration. By default a MemoryCache is used at the top level along with a StringKeyMaker.

Usage
-----

Basic usage

.. code-block:: python

   import ftrack_api
   session = ftrack_api.Session(
       server_url='https://ftrack.example.com',
       api_key='your_api_key',
       api_user='your_username'
   )

Advanced usage and behavior
- Environment-driven configuration: If server_url, api_key, or api_user are not supplied explicitly, the constructor will look for FTRACK_SERVER, FTRACK_API_KEY, and FTRACK_API_USER (or related alternatives) in the environment.
- Auto-population: With auto_populate enabled (default), entity attributes will be fetched from the server on access if not already loaded.
- Event hub: If auto_connect_event_hub is True, an embedded event hub will be connected to the server in a background thread, enabling non-local event publishing/subscription. Subscribing to events does not require a connected hub.
- Schema caching: Enable by setting schema_cache_path to a folder. If not set, the environment variable FTRACK_API_SCHEMA_CACHE_PATH is checked, else a temporary directory is used. Set to False to disable schema caching entirely.
- Plugins: plugin_arguments allow passing initial arguments to plugin discovery; incompatible signatures are gracefully reduced with a warning.

Details
-------

- Authentication
  - SessionAuthentication attaches the API key and user to outbound requests via headers:
    - ftrack-api-key
    - ftrack-user

- Session lifecycle
  - The Session class encapsulates server interaction, including:
    - Validation of required credentials (server_url, api_key, api_user) and helpful errors if missing.
    - Initialization of a layered cache with a top-level MemoryCache, simplifying in-process caching behavior.
    - Provision of a default cache_key_maker (StringKeyMaker) to generate stable keys for cached objects.

- Network and concurrency
  - The event hub connection (when enabled) runs in a background thread to improve startup performance.
  - Subscribing to events and publishing local events are supported independently of a connected hub, depending on configuration.

- Extensibility
  - plugin_paths and plugin_arguments allow discovery and configuration of plugins at runtime.
  - The design supports lazy loading of entities (via auto_populate) and pluggable caching strategies.
