# Header Interface Transition Guide

## Overview
This guide documents the core utilities exposed by the header transition machinery, focusing on verification of header values and notification of changes during interface transitions. The provided code implements:

- A verification framework to validate header values and optionally fix, warn, or raise on issues.
- A nested error list structure for formatting and reporting multiple verification errors.
- A notifier mixin to enable decoupled communication between components when header-related state changes occur.

These components support a robust migration path for header interfaces by enabling consistent validation, clear error reporting, and event-driven updates across related objects.

## Key Components

- _Verify
  - Purpose: Shared verification helpers used to validate values during header interface transitions.
  - Primary methods:
    - run_option(option="warn", err_text="", fix_text="Fixed.", fix=None, fixable=True)
      - Executes verification with the selected option.
      - Behavior:
        - If option is "warn" or "exception", fixable is set to False.
        - If the value is not fixable, the message is marked as unfixable.
        - If fix is provided and fixable, the fix is executed and the message appends fix_text.
      - Returns a tuple (fixable, text).
      - This method is intended to be used by concrete verifiers that supply the specific validation logic.
    - verify(option="warn")
      - Verifies all values in the instance.
      - Accepts options such as "ignore", "warn", "exception", "fix", "silentfix" and combinations with "+ignore", "+warn", "+exception" (e.g., "fix+warn").
      - Validates the option against VERIFY_OPTIONS and delegates to the instance-specific _verify method.
      - Handles splitting fix-related and reporting-related options, including special handling for "silentfix".
      - Aggregates messages and reports them as warnings or raises an exception depending on the chosen options.
  - VERIFY_OPTIONS
    - A list of all supported verification options, including:
      - "ignore", "warn", "exception", "fix", "silentfix",
      - "fix+ignore", "fix+warn", "fix+exception",
      - "silentfix+ignore", "silentfix+warn", "silentfix+exception"

  - Notes: The _Verify class is designed as a mixin-like utility; concrete verifiers should implement the internal _verify method to return an _ErrList of issues. The provided methods handle the reporting and optional fixing behavior.

- _ErrList
  - Purpose: A specialized list structure to hold verification errors in a nested, hierarchical format.
  - Key features:
    - __init__(val=(), unit="Element"): Initialize with optional values and a unit label for substructures.
    - __str__(): Render the error list as a string by joining top-level messages.
    - iter_lines(filter=None, shift=0): Iterate through the nested structure, yielding lines with appropriate indentation for each level.
      - The method emits lines in two passes:
        - First pass: top-level messages (non-_ErrList items).
        - Second pass: nested _ErrList items, indenting and optionally inserting a header like "<unit> <index>:" when appropriate.
      - This structure supports readable formatting of multi-level verification results.

- NotifierMixin
  - Purpose: Mixin providing services to register listeners and notify them of changes on an object.
  - Key behavior (as described in its docstring):
    - Listeners can be registered via an internal _add_listener method (weak-referenced, so they do not prevent garbage collection).
    - Changes are propagated by calling _notify('change_type', *args, **kwargs), which triggers listener methods named accordingly (e.g., listener._update_change_type(*args, **kwargs)) if present.
  - Usage concept:
    - Attach listeners to an object to receive updates when its state changes.
    - This supports decoupled components during header interface transitions, where multiple parts of the system need to react to header state changes.

- Supporting utilities
  - path_like, cmp, all_integer_types: Small helpers used elsewhere in the module to support header transition logic.

## Usage

- Basic verification flow
  - Subclass or compose a verifier that implements _verify(self, opt) to return an _ErrList of issues.
  - Use ver.verify(option) to drive the verification process, leveraging the option semantics described above.
  - Example (conceptual):
    - class MyHeaderVerifier(_Verify):
        def _verify(self, opt):
          # Build and return an _ErrList with nested messages
          return _ErrList([(_ErrList(...), "Header value missing")], unit="Header Field")
    - v = MyHeaderVerifier(...)
    - v.verify("warn")  # or "fix+warn", "silentfix", etc.

- Formatting verification results
  - Use _ErrList.iter_lines(...) to produce nicely indented strings representing issues at multiple levels.
  - The formatted messages can be emitted as warnings or raised as exceptions depending on the chosen verify option.

- Event-driven header transitions
  - Attach listeners to objects using NotifierMixin to respond to header-related changes.
  - Use _notify to broadcast change events; listener methods named _update_<change_type> will be invoked if present.

Notes:
- The verification system mirrors patterns found in astropy.verify, enabling flexible handling of header issues during a transition (reporting, fixing, or raising as needed).
- The notifier mechanism supports clean decoupling between components that react to header state changes during the transition process.
