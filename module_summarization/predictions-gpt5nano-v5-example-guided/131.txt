Scheme to verify FITS data
===========================

Overview
--------
This module implements the verification framework used to check consistency and validity of FITS-related data structures in PyFITS. It defines a standard set of options for reporting verification results, as well as mechanisms to fix up fixable issues and to report unfixable problems.

Key components
--------------
- VerifyError
  - Exception raised when verification fails under certain option configurations.
- VerifyWarning
  - Warning emitted for verification issues (subclass of AstropyUserWarning).
- VERIFY_OPTIONS
  - List of valid verification reporting/fixing options, including:
    - "ignore", "warn", "exception", "fix", "silentfix",
    - "fix+ignore", "fix+warn", "fix+exception",
    - "silentfix+ignore", "silentfix+warn", "silentfix+exception"
- _Verify
  - Core verification helper used by classes that implement verification logic.
  - run_option(option="warn", err_text="", fix_text="Fixed.", fix=None, fixable=True)
    - Executes verification for a single issue with a chosen option.
    - Returns a tuple (fixable, text) indicating whether the issue is fixable and the message to report.
    - Behavior varies by option (warn/exception disable fix, fix applies a fix function, etc.).
  - verify(option="warn")
    - Runs verification for all values in the instance.
    - Parses the option, delegates to _verify, and handles reporting/fixing behavior.
    - Examples of supported options:
      - "warn", "exception", "ignore"
      - Combinations like "fix+warn", "silentfix+ignore", etc.
    - If fixable issues exist, they may be fixed and reported according to the chosen option.
    - Note: prints a header and a note about zero-based indexing when reporting results.
- _ErrList
  - Nested list structure for collecting verification error messages.
  - __init__(val=(), unit="Element")
    - Creates an error list with an optional unit label.
  - __str__()
    - Returns a string representation of all error messages.
  - iter_lines(filter=None, shift=0)
    - Iterates the nested error structure, yielding (fixable, message) pairs with proper indentation.
    - Supports filtering and hierarchical formatting.
  - Used by verification implementations to accumulate and format messages.

Usage
-----
- Typical workflow
  - Implement a verifier by subclassing _Verify and providing a _verify method that returns an _ErrList of issues.
  - Call the instance’s verify method with an option to report/fix issues.

Example
-------
A minimal demonstration of how a custom verifier might be structured and used:

.. code-block:: python

    from verification import _Verify, _ErrList

    class MyVerifier(_Verify):
        def __init__(self, value):
            self.value = value

        def _verify(self, opt):
            # Build an error list with fixable and unfixable issues
            errs = _ErrList([], unit="Value")
            if self.value < 0:
                errs.append((True, "value must be non-negative"))
            if self.value > 100:
                errs.append((False, "value exceeds maximum"))
            return errs

    # Example usage
    v = MyVerifier(-5)
    v.verify(option="warn")  # emits warnings for fixable issues

Notes on options and behavior
-----------------------------
- Options are case-insensitive and must be one of VERIFY_OPTIONS.
- If option is "ignore", verification is skipped.
- The option can be a combination of fixing and reporting:
  - Example: "fix+warn", "silentfix+exception"
- For "silentfix" with "ignore", the results are not reported.
- When fix is requested and issues are fixable, the provided fix function is executed and the message is augmented with a fixed note.
- If issues are unfixable and the option specifies reporting as "exception", a VerifyError is raised with a consolidated message.
- A note is appended indicating zero-based indexing in FITS as used by PyFITS during reporting.

Details
-------
- The verification framework is designed to be extensible via the _Verify base class. Implementations should override _verify to return an _ErrList describing issues found.
- The reporting mechanism formats nested error information clearly, using indentation to reflect structure, and includes a header (“Verification reported errors:”) and a closing note about zero-based indexing.
- Typical usage involves emitting warnings (VerifyWarning) for fixable issues or raising VerifyError for unfixable or explicitly fatal conditions, depending on the chosen option.
