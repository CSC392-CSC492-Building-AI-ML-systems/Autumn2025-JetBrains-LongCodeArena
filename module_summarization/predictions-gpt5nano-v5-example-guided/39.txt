# Model usage

## Overview
This module provides internal utilities and helpers used to define and manage ORM models. It includes facilities for:
- Normalizing meta options (e.g., unique constraints, indexes)
- Determining default ordering
- Handling relationship fields (foreign keys, one-to-one, many-to-many)
- Extracting field documentation from source for better model introspection
- Storing and exposing model metadata via the MetaInfo class

The utilities are intended for use by the ORM core when a Model subclass is defined and its inner Meta class is processed.

## Key Components

- get_together(meta, together)
  - Normalizes meta options like unique_together and indexes into a consistent structure.
  - Returns a tuple of tuples representing field groups.
  - Example:
    - unique_together = (("field1", "field2"),)
    - indexes = (("field3",),)
  - Implementation note: if the value is a list/tuple of strings, it is wrapped into a single tuple.

- prepare_default_ordering(meta)
  - Converts the Meta ordering specification into a parsed, machine-friendly representation using QuerySet._resolve_ordering_string.
  - Example:
    - ordering = ("name",)
  - The result is a tuple of (field_name, Order) pairs suitable for query construction.

- _fk_setter(self, value, _key, relation_field, to_field)
  - Internal helper to synchronize a ForeignKey value and its corresponding key field on a model instance.
  - Sets both the relation field and the raw key field when assigning a related object.

- _fk_getter(self, _key, ftype, relation_field, to_field)
  - Internal helper to fetch the related object for a ForeignKey field.
  - Returns either the cached value or a lazy query for the related object if not yet loaded.

- _rfk_getter(self, _key, ftype, frelfield, from_field)
  - Internal helper for reverse ForeignKey relations.
  - Creates or retrieves a ReverseRelation instance to access related objects.

- _ro2o_getter(self, _key, ftype, frelfield, from_field)
  - Internal helper for reverse one-to-one relations.
  - Returns a single related object or a lazy query for it.

- _m2m_getter(self, _key, field_object)
  - Internal helper for Many-to-Many relations.
  - Returns a ManyToManyRelation proxy object that manages related items.

- _get_comments(cls)
  - Extracts inline field documentation from the model source.
  - Look for lines preceding a field definition in the form:
    #: Comment text
    field_name = FieldType(...)
  - Supports multiline comments and replaces the placeholder "{model}" with the actual model class name.
  - Example:
    #: The user's id
    id = IntField(pk=True)
    #: The user's name
    name = CharField(max_length=255)

- MetaInfo
  - Core data structure that aggregates model metadata after a Model subclass is processed.
  - Tracks:
    - abstract, db_table, app
    - fields, db_fields, m2m_fields, fk_fields, o2o_fields
    - backward relations (backward_fk_fields, backward_o2o_fields)
    - fetch_fields, fields_db_projection, fields_map
    - unique_together, indexes
    - default_connection, basequery, basetable
    - pk_attr, generated_db_fields, table_description
    - _default_ordering, _ordering_validated
  - Purpose: central repository for model metadata used by the ORM when constructing queries, validating schemas, and managing relations.

- get_together, prepare_default_ordering (usage overview)
  - The module provides helpers to interpret and prepare model meta options in a consistent way for the ORM to consume.

## Usage

- Declaring model meta options
  - Define a model with common meta options such as table name, ordering, unique constraints, and indexes.

  Example:
  - Define a model with a custom table name and unique constraint
    - class User(Model):
        class Meta:
            table = "user_table"
            unique_together = (("email", "username"),)

  - Define default ordering
    - class Product(Model):
        class Meta:
            ordering = ("name",)

  - Define indexes
    - class Log(Model):
        class Meta:
            indexes = (("created_at",),)

- Documenting fields using source comments
  - You can attach documentation to fields using specially formatted comments before the field definition. The _get_comments function parses these comments and replaces "{model}" with the actual model name.

  Example:
    class User(Model):
        #: The user's id
        id = IntField(pk=True)

        #: The user's login name
        username = CharField(max_length=150)

  In the above, the comments will be associated with the corresponding fields and can be used for docs or metadata extraction.

- Relationship handling (conceptual)
  - The internal helpers (_fk_setter, _fk_getter, _rfk_getter, _ro2o_getter, _m2m_getter) are used by the ORM to manage:
    - ForeignKey fields and their underlying key attributes
    - Reverse relations for ForeignKey and OneToOne
    - Many-to-Many relations
  - While these are internal, understanding their existence can help in debugging model behavior related to relation loading and assignment.

## Details

- Meta options and normalization
  - get_together(meta, "unique_together" / "indexes") normalizes input into a uniform structure, which simplifies downstream processing for the database schema and query generation.

- Default ordering
  - prepare_default_ordering(meta) reads the ordering specification from Meta and converts it into a structured representation suitable for populating QuerySets by default.

- Documentation extraction
  - _get_comments(cls) scans the model source for field-level comments using a defined pattern and supports the "{model}" placeholder, enabling per-field documentation to be collected automatically.

- Metadata container
  - MetaInfo serves as the canonical container for all model metadata used by the ORM during:
    - Table creation and schema validation
    - Query building and execution
    - Relationship resolution and loading

Note: The examples above illustrate typical usage patterns and internal concepts. The actual implementation details are used by the ORM core to process Model subclasses and should not normally be required for day-to-day model usage, which focuses on defining models and Meta options, and declaring relationships via fields.
