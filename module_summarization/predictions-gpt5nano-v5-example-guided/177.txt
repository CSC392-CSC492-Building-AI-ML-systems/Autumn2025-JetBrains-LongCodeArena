# Pipelines and composite estimators

## Overview
The pipeline module provides utilities to build composite estimators as a chain of transforms followed by a final estimator. Transforms in the chain must implement fit and transform, while the final estimator only needs to implement fit. Pipelines enable you to:
- chain multiple processing steps in a single estimator
- cross-validate all steps together
- tune parameters across steps using the familiar ``__``-delimited syntax
- optionally cache intermediate results for efficiency

The module also includes utilities for combining features in parallel (e.g., FeatureUnion) and convenient constructors (make_pipeline, make_union).

## Key Components

- Pipeline: Core class for chaining transforms with a final estimator
  - Semantics: a list of (name, transform) pairs; the last step is the final estimator
  - Transforms must implement fit and transform; the final estimator must implement fit
  - Memory caching for fitted transformers is supported
  - Parameters can be set using the ``step__parameter`` naming convention
  - Access steps via the ``named_steps`` attribute

- FeatureUnion: Combines features produced by several transformer objects in parallel

- make_pipeline / make_union: Convenience functions to construct pipelines or unions without naming every step manually

- _final_estimator_has: Internal helper used with available_if to require certain attributes exist on the final estimator

## Usage

- Basic pipeline example
  - The pipeline applies a scaler, then a support vector classifier
  - You can treat the pipeline as a single estimator

```python
from sklearn.svm import SVC
from sklearn.preprocessing import StandardScaler
from sklearn.datasets import make_classification
from sklearn.model_selection import train_test_split
from sklearn.pipeline import Pipeline

X, y = make_classification(random_state=0)
X_train, X_test, y_train, y_test = train_test_split(X, y, random_state=0)

pipe = Pipeline([('scaler', StandardScaler()), ('svc', SVC())])

# The pipeline can be used as any other estimator
pipe.fit(X_train, y_train).score(X_test, y_test)

# Example of setting a parameter across steps
pipe.set_params(svc__C=10).fit(X_train, y_train).score(X_test, y_test)
```

- Parameter setting and inspection
  - Use the double-underscore syntax to set parameters of inner steps, e.g., `pipeline__stepname__param`.

- Replacing or removing steps
  - A step can be replaced by another estimator, or removed by setting it to `'passthrough'` or `None` (to bypass that stage).

- Caching with memory
  - The `memory` argument enables caching of fitted transformers. The last step (the final estimator) is not cached. When a string is provided, it is treated as a path to a caching directory.

- Output configuration
  - The `set_output` method configures the output container for transform and fit_transform calls. It propagates the requested output format to all steps.

```python
# Example: set the output format for transforms
pipe.set_output(transform="pandas")
```

## Details

- Structure
  - Steps: A list of (name, transformer) tuples. The last step must be an estimator (i.e., implement fit).
  - Memory: Used to cache fitted transformers. When enabled, a clone of the transformer is used during fitting to avoid side effects.
  - Verbose: If True, prints elapsed time for each step during fitting.

- Important attributes
  - named_steps: A dict-like object to access steps by name
  - steps: The list of (name, transform) tuples
  - memory, verbose: Configuration options

- Parameter access and validation
  - The pipeline and its sub-estimators expose parameters via `get_params`, including those from nested steps when `deep=True`.
  - Parameters can be updated using `set_params` with the `step__param` notation.

- Internal helpers
  - `_final_estimator_has(attr)`: Ensures that the final estimator provides a given attribute, enabling conditional parameter validation via mechanisms like `available_if`.

- Example with grid search
  - Pipelines integrate naturally with cross-validation and hyperparameter tuning. Use `'__'` to address parameters inside steps when exploring parameter grids.

- Notes
  - The final step is required to be a fitted estimator (i.e., one that implements `fit`).
  - Transform steps must implement both `fit` and `transform`.
  - The pipeline can be inspected via its `named_steps` or `steps` attributes to review constituent estimators and their parameters.
