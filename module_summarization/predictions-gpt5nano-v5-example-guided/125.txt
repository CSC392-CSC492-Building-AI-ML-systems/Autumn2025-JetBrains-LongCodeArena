# FITS Headers

## Overview
This documentation covers the verification and error-reporting mechanisms used for FITS headers in the library. The header verification framework provides a structured way to check header values, report issues, and optionally fix problems.

Key concepts implemented in the header verification module include a set of verification options, a verification engine, and a nested error-list structure for presenting issues.

## Key Components

- Verify exception and warning classes
  - VerifyError: exception raised when verification fails in non-warn modes.
  - VerifyWarning: warning category used for verification issues.
  - See examples in the codebase where these classes are defined:
    - VerifyError
    - VerifyWarning

  Example definitions:
  - VerifyError class
    - "Verify exception class."
  - VerifyWarning class
    - "Verify warning class."

- Verification options
  - A predefined set of options controlling verification behavior:
    - ignore, warn, exception, fix, silentfix
    - fix+ignore, fix+warn, fix+exception
    - silentfix+ignore, silentfix+warn, silentfix+exception

  These options can be combined (e.g., "fix+warn") to determine how errors are reported and whether fixable issues are applied automatically.

  Source reference (constants):
  - VERIFY_OPTIONS = [
      "ignore", "warn", "exception", "fix", "silentfix",
      "fix+ignore", "fix+warn", "fix+exception",
      "silentfix+ignore", "silentfix+warn", "silentfix+exception",
    ]

- _Verify: the core verification engine
  - run_option(self, option="warn", err_text="", fix_text="Fixed.", fix=None, fixable=True)
    - Executes the verification with the selected option.
    - Determines whether an issue is fixable and what message to surface.
    - Returns a tuple (fixable, text).
  - verify(self, option="warn")
    - Runs verification for all values in the instance.
    - Validates the option, delegates to self._verify(opt), and handles reporting vs. fixing behavior.
    - Supports combinations like "fix+warn" and "silentfix+ignore" by splitting the option into fix and report parts.
    - Handles reporting logic (warnings) or raising VerifyError as appropriate.

  Example snippet (as implemented in code):
  - The verify method documents how options are interpreted and how messages are emitted or exceptions are raised.

- _ErrList: nested verification error container
  - A list subclass used to hold verification errors in a hierarchical structure.
  - __init__(self, val=(), unit="Element")
    - Creates the error list, with units used for headers in reports.
  - __str__(self)
    - Produces a human-readable string representation of all errors.
  - iter_lines(self, filter=None, shift=0)
    - Iterates through the nested error structure and yields lines with indentation appropriate to their level.
    - Supports an optional filter to select which lines to yield.

  The nested structure is designed to present errors in a readable, hierarchical form, with indentation indicating levels and units labeling sections (e.g., "Element 0:", "Header 1:", etc.).

- Notes on reporting
  - When fixable errors are present, the verification process can either warn, fix, or raise an exception, depending on the chosen option.
  - Special handling exists for silent modes, which suppresses output for certain conditions.
  - The library prints a header line “Verification reported errors:” and a trailing note about zero-based indexing in FITS headers:
    - Note: astropy.io.fits uses zero-based indexing.

## Usage

- Basic verification (warns on issues)
  - Example (conceptual):
    - v = _Verify()
    - v.verify(option="warn")

- Verification with fixing
  - Example (conceptual):
    - v.verify(option="fix+warn")
  - The first part ("fix") fixes fixable issues; the second part ("warn") surfaces issues via warnings. If issues cannot be fixed, behavior depends on the specific option (e.g., "fix+exception" may raise).

- Silent verification
  - Example (conceptual):
    - v.verify(option="silentfix+ignore")
  - Silent modes skip reporting under certain conditions while still attempting fixes.

Details

- Purpose
  - The FITS header verification framework ensures headers conform to expected standards and semantics. It collects issues, formats them for human consumption, and provides mechanisms to fix where possible.

- Reporting semantics
  - The behavior of reporting and fixing is governed by the chosen option (e.g., "warn" vs. "exception" vs. "fix+warn").
  - In silent modes, the framework can suppress or selectively suppress output while still performing fixes.

- User note
  - The verification system is designed to work with nested error information, making it easier to present complex header issues in an organized fashion.
  - When presenting messages, the note about zero-based indexing helps users interpret header positions correctly.

- Acknowledgment
  - The implementation is designed to be consistent with the overall ASTROPY verification interface, and you can refer to the broader verify documentation for additional context and usage patterns.
