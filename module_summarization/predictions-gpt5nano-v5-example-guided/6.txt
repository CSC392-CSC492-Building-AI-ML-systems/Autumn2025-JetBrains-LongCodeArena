Working with user security roles
===============================

Overview
--------
Security roles control what actions a user can perform on the ftrack server. This documentation covers how the client-side API establishes a session and authenticates requests, which is a prerequisite for any role-based operations performed on the server. The code in this module does not implement client-side role management; instead, it provides the authentication and session infrastructure used to interact with the server where roles are managed.

Key Components
--------------
- SessionAuthentication
  - Attaches authentication information to outgoing HTTP requests.
  - Injects ftrack-api-key and ftrack-user headers to authenticate with the server.

  Example (extracted from source):
  .. code-block:: python

     class SessionAuthentication(requests.auth.AuthBase):
         '''Attach ftrack session authentication information to requests.'''

         def __init__(self, api_key, api_user):
             '''Initialise with *api_key* and *api_user*.'''
             self.api_key = api_key
             self.api_user = api_user
             super(SessionAuthentication, self).__init__()

         def __call__(self, request):
             '''Modify *request* to have appropriate headers.'''
             request.headers.update({
                 'ftrack-api-key': self.api_key,
                 'ftrack-user': self.api_user
             })
             return request

- Session
  - Encapsulates an isolated session for interaction with an ftrack server.
  - Handles required credentials, environment fallbacks, and optional features (caching, plugins, event hub, etc.).
  - Requires server_url, api_key, and api_user. If not provided, these values may be sourced from environment variables:
    - server_url from FTRACK_SERVER
    - api_key from FTRACK_API_KEY or FTRACK_APIKEY
    - api_user from FTRACK_API_USER or the current system user (via getpass)

  Notable behavior (from the source):
  - The session will raise a TypeError if essential credentials are missing.
  - It supports environment-variable-based configuration to simplify usage in scripts and deployments.

Usage
-----
Important: Direct management of security roles (creating, assigning, or altering user roles) is performed on the server side. The client-side API described here focuses on establishing authenticated sessions and performing operations permitted by those roles.

How to start a session
- Provide the server URL, API key, and API user (via arguments or environment variables).

  Example:
  .. code-block:: python

     from ftrack_api import Session

     session = Session(
         server_url='https://myserver.example',
         api_key='YOUR_API_KEY',
         api_user='your_username'
     )

  Once the session is created, you can interact with the server (subject to your role) through the session's facilities, such as operations tracking and other API features.

Related capabilities
- The session contains features like logging, caching, and an embedded event hub (optional). Detailed usage of those features is outside the scope of role management and would be governed by server-side permissions and API endpoints.

Details
-------
- Authentication mechanism
  - Requests are authenticated by attaching headers:
    - ftrack-api-key: the API key
    - ftrack-user: the API user
  - This behavior is implemented in the SessionAuthentication class and is used to authorize actions against the ftrack server.

- Credential sourcing and validation
  - server_url, api_key, and api_user are required.
  - If not provided, the module attempts to read them from environment variables:
    - FTRACK_SERVER for server_url
    - FTRACK_API_KEY or FTRACK_APIKEY for api_key
    - FTRACK_API_USER for api_user
  - If api_user is not specified, the module will attempt to use the current system user as a fallback.

- Practical note on security roles
  - Roles are managed on the server. Use the server administrator interface or server-side APIs to modify user roles.
  - This client-side API must be supplied with credentials that have the appropriate permissions to perform requested operations.
