# Tensor Attributes

## Overview
Tensor Attributes refer to metadata associated with tensor values within the OneFlow MLIR dialect. This documentation describes how Tensor Attributes are integrated into the OneFlow build system, exposed via MLIR dialect libraries, and wired into the runtime. The Tensor Attributes components live as part of the MLIR dialect libraries and are built when MLIR support is enabled.

The build system for these components is defined in the CMake configuration, notably through helper functions that package MLIR libraries and dialects, manage include paths, and ensure correct runtime search paths.

Key CMake helpers related to Tensor Attributes:
- oneflow_add_mlir_library
- oneflow_add_mlir_dialect_library
- oneflow_add_llvm_tool
- update_rpath

These helpers standardize how Tensor Attribute-related code is compiled and linked, and how its runtime dependencies are resolved.

## Key Components
- Tensor Attribute definitions (conceptual)
  - Represent metadata attached to tensor values in the dialect (e.g., shape, dtype, layout). The actual attribute definitions are part of the dialect source and are built into MLIR dialect libraries.

- oneflow_add_mlir_library
  - Declares and wires an MLIR library target.
  - Example behavior from source:
    - add_mlir_library(${ARGV})
    - set_compile_options_to_oneflow_target(${ARGV0})
    - update_rpath(${ARGV0})

- oneflow_add_mlir_dialect_library
  - Declares and wires an MLIR dialect library target for Tensor Attributes.
  - Example behavior from source:
    - add_mlir_dialect_library(${ARGV})
    - set_compile_options_to_oneflow_target(${ARGV0})
    - update_rpath(${ARGV0})

- oneflow_add_llvm_tool
  - Declares and wires an LLVM tool (utility) associated with the Tensor Attributes workflow.
  - Example behavior from source:
    - add_llvm_tool(${ARGV})
    - llvm_update_compile_flags(oneflow-runner)
    - set_compile_options_to_oneflow_target(${ARGV0})
    - update_rpath(${ARGV0})

- update_rpath
  - Ensures build-time and install-time runtime search paths include LLVM and the OneFlow build root.
  - Example:
    function(update_rpath)
      set_property(TARGET ${ARGV0} APPEND PROPERTY BUILD_RPATH "${LLVM_LIBRARY_DIR}")
      set_property(TARGET ${ARGV0} APPEND PROPERTY BUILD_RPATH "${ONEFLOW_BUILD_ROOT_DIR}")
      set_property(TARGET ${ARGV0} APPEND PROPERTY INSTALL_RPATH "${LLVM_LIBRARY_DIR}")
      set_property(TARGET ${ARGV0} APPEND PROPERTY INSTALL_RPATH "${ONEFLOW_BUILD_ROOT_DIR}")
    endfunction(update_rpath)

- Build gating for MLIR-related components
  - When WITH_MLIR is enabled, the build adds subdirectories for include, lib, test, oneflow-translate, oneflow-runtime, oneflow-extension, oneflow-opt, oneflow-runner, and oneflow-lite, enabling Tensor Attributes functionality as part of the MLIR dialect.

## Usage

- Enabling Tensor Attributes support
  - Enable MLIR-related features by configuring with WITH_MLIR (and related options as needed):
    - cmake -DWITH_MLIR=ON -DWITH_PYTHON=ON -DCMAKE_BUILD_TYPE=Release ...
  - This triggers building the MLIR-based Tensor Attributes components, along with the dialect library and related tooling.

- Declaring a Tensor Attributes dialect library (example)
  - In your CMakeLists.txt (illustrative usage), you would declare a dialect library for Tensor Attributes:
    - oneflow_add_mlir_dialect_library(MyTensorAttrs
        TensorAttrs.cpp
        TensorAttrsOps.cpp)
  - This leverages the underlying helper to configure compile options, link correctly, and set RPATH for runtime.

- Typical build steps
  - cmake -DWITH_MLIR=ON -DWITH_PYTHON=ON -DCMAKE_BUILD_TYPE=Release -S . -B build
  - cmake --build build -j

## Details

- The Tensor Attributes module relies on the LLVM/MLIR toolchain integration:
  - The project uses variables such as MLIR_TABLEGEN_EXE and MLIR_PDLL_TABLEGEN_EXE to drive table-gen related code generation.
  - It includes MLIR and LLVM headers and libraries via include_directories and link_directories.
  - It propagates include paths to consuming targets and to Python packaging when building Python bindings.

- Subdirectory structure and build flow (when WITH_MLIR is true)
  - add_subdirectory(include)
  - add_subdirectory(lib)
  - add_subdirectory(test)
  - add_subdirectory(oneflow-translate)
  - add_subdirectory(oneflow-runtime)
  - add_subdirectory(oneflow-extension)
  - add_subdirectory(oneflow-opt)
  - add_subdirectory(oneflow-runner)
  - add_subdirectory(oneflow-lite)

- Python packaging considerations
  - When building Python bindings, the system installs a curated set of LLVM headers into ONEFLOW_INCLUDE_DIR, excluding a number of headers to avoid conflicts or bloat.

- Policy and LLVM provider
  - The CMake configuration sets policy CMP0068, CMP0075, CMP0077, CMP0116 and handles LLVM_PROVIDER values ("in-tree" or "install"), ensuring correct integration with the LLVM toolchain.
