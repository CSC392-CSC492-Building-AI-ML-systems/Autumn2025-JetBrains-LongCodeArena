autowrap
==========
Generate documentation for this code

Overview
--------
This module provides a common interface for wrapping code generated by the codegen backends (e.g., f2py, fwrap, Cython, SWIG). It compiles the generated code into a binary Python callable that can be used for efficient numerical evaluation. The intended workflow is to take a Diofant/SymPy expression, generate code for it, wrap that code into a binary function, and then use it directly or inside symbolic expressions.

Key Components
--------------
- CodeWrapError: Generic error raised during the code-wrapping process.
- CodeWrapper: Base class for code wrappers. Manages file naming, module naming, code generation, and the execution of the external build/wrap process.
  - filename: Property returning the generated code filename for the current module counter.
  - module_name: Property returning the Python module name for the current module counter.
  - include_header / include_empty: Flags controlling whether header/empty files are included in the build.
  - wrap_code(routine, helpers): Orchestrates code generation, file preparation, compilation, dynamic import, and retrieval of the wrapped function.
  - _process_files(routine): Executes the external command used to compile the generated code and reports errors.
- DummyWrapper: A simple, backend-agnostic wrapper used for testing without relying on an external backend.
  - _generate_code: Produces a minimal Python module that returns a string representation of the expression.
  - _get_wrapped_function: Retrieves the wrapper function from the dummy module.
- CythonCodeWrapper: Wrapper that uses Cython for compilation and wrapping (implementation details partially shown in the source).
  - setup_template: Part of the setup for building with Cython (excerpt shown in the source).
  
Usage
-----
Typical usage involves taking a Diofant/SymPy expression and obtaining a binary callable that can be evaluated numerically:

Example (from docstring):
.. code-block:: python

    >>> expr = ((x - y)**25).expand()
    >>> binary_callable = autowrap(expr)
    >>> binary_callable(1, 2)
    -1.0

If you prefer to keep a symbolic Function object in expressions, use binary_function to wrap the expression
into a Diofant Function and then combine it with other symbolic terms:

.. code-block:: python

    >>> f = binary_function('f', expr)
    >>> 2*f(x, y) + y
    y + 2*f(x, y)
    >>> (2*f(x, y) + y).evalf(2, subs={x: 1, y: 2}, strict=False)
    0.e-190

Notes
-----
- The wrapper returns a binary Python function (not a Diofant object). The compiled function is attached as the _imp_ attribute and is invoked during numerical evaluation (evalf()) or when using lambdify().
- This approach is particularly beneficial for:
  - Large array computations where Python loops are slow or where numpy advanced indexing is advantageous.
  - Repeatedly called long expressions where compiled code is significantly faster than Diofant's evalf().
  - Quick testing of generated binaries from within Diofant via the binary wrappers.
  - Creating customized ufuncs for numpy integration.
- This module is not always the best choice if extreme speed or memory optimizations are required; consider using low-level wrapper tools directly for highly optimized workflows. Temporary files may be left intact if a tempdir path is provided.

Doctest dependencies
---------------------
This module declares doctest dependencies to ensure tests run in the right environment:

- Executables: f2py, gfortran, gcc
- Modules: numpy

These dependencies are required to build and test the wrapped backends.

Details
-------
- The module relies on a code generation backend abstraction (CCodeGen and related helpers) to produce the necessary C/C++ (or other) code, which is then compiled and wrapped into a Python callable.
- The design emphasizes a one-button experience for obtaining high-performance numerical callables from symbolic expressions, with a seamless path back into symbolic computation when needed.
