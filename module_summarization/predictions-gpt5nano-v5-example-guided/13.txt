Understanding sessions
======================

Overview
--------
A Session represents an isolated connection and context for interacting with an ftrack server. It encapsulates server connection details, authentication, caching, optional event hub integration, plugin discovery, and schema caching. Creating a Session provides a configured environment for performing operations against the server, with features such as automatic population of entity attributes, layered caching, and optional embedded event hub support.

Key Components
--------------

- SessionAuthentication
  Attaches ftrack authentication information to outgoing HTTP requests.

  Example:
  .. code-block:: python

     class SessionAuthentication(requests.auth.AuthBase):
         '''Attach ftrack session authentication information to requests.'''

         def __init__(self, api_key, api_user):
             '''Initialise with *api_key* and *api_user*.'''
             self.api_key = api_key
             self.api_user = api_user
             super(SessionAuthentication, self).__init__()

         def __call__(self, request):
             '''Modify *request* to have appropriate headers.'''
             request.headers.update({
                 'ftrack-api-key': self.api_key,
                 'ftrack-user': self.api_user
             })
             return request

- Session
  An isolated context for interacting with an ftrack server. Key configuration options include:

  - server_url: URL of the ftrack server (including port).
  - api_key and api_user: credentials used for authentication.
  - auto_populate: if True, accessing entity attributes fetches data from the server on demand.
  - plugin_paths: list of paths to search for plugins.
  - cache: a cache instance or a callable returning a cache; intended to be layered under a MemoryCache at the top level.
  - cache_key_maker: key-maker for cache entries (default: StringKeyMaker).
  - auto_connect_event_hub: if True, embed and connect an event hub in the background.
  - schema_cache_path: path to cache schemas; False disables schema caching.
  - plugin_arguments: optional mapping of arguments passed to plugin register functions.

  The __init__ signature (summary):
  .. code-block:: python

     def __init__(
         self, server_url=None, api_key=None, api_user=None, auto_populate=True,
         plugin_paths=None, cache=None, cache_key_maker=None,
         auto_connect_event_hub=None, schema_cache_path=None,
         plugin_arguments=None
     ):
         ...
  The session provides a number of behaviors described in the constructor docstring, including environment-based fallbacks and warnings for incompatible plugin arguments.

  Important behavior notes from the implementation:
  - If server_url is not provided, it is read from the FTRACK_SERVER environment variable.
  - If api_key is not provided, FTRACK_API_KEY (or FTRACK_APIKEY for backwards compatibility) is used.
  - If api_user is not provided, FTRACK_API_USER or the current system user (via getpass) may be used.
  - The session initializes with a top-level memory cache and a layered caching strategy.
  - The embedded event hub can be connected automatically in a background thread when auto_connect_event_hub is True.
  - Environment-based and runtime configuration options are documented in the constructor docstring.

Usage
-----

Create a session with explicit credentials
  - Provide server URL, API key, and API user.
  - Optional features include enabling the embedded event hub and custom caching.

  .. code-block:: python

     from ftrack_api import Session

     session = Session(
         server_url='https://ftrack.example.com',
         api_key='YOUR_API_KEY',
         api_user='your_username'
     )

Using environment variables
  - If you prefer environment-based configuration, set:
    - FTRACK_SERVER
    - FTRACK_API_KEY (or FTRACK_APIKEY for backwards compatibility)
    - FTRACK_API_USER

  Then you can create a session with minimal arguments:

  .. code-block:: python

     session = Session()

  - You can additionally enable or disable schema caching via schema_cache_path.
  - You can provide a specific cache or leave it to the default layered caching.

Customize caching and plugins
  - You may pass a custom cache implementation or a function that returns one.
  - You may specify plugin_paths to discover and load plugins.
  - You may pass plugin_arguments to influence plugin registration behavior.

  .. code-block:: python

     from ftrack_api.cache import MemoryCache
     session = Session(
         server_url='https://ftrack.example.com',
         api_key='YOUR_API_KEY',
         api_user='your_username',
         cache=MemoryCache(),
         plugin_paths=['/path/to/plugins'],
         plugin_arguments={'foo': 'bar'}
     )

Notes on usage
  - auto_populate controls whether entity attribute access triggers on-demand data fetches.
  - The event hub connection runs in a background thread by default. If you disable it, you can still publish/subscribe to local events and later manually connect to the remote hub.
  - The top-level MemoryCache is enforced as part of a layered caching strategy to ensure consistent object identity and cache behavior.

Details
-------

- Environment-variable fallbacks
  - server_url: FTRACK_SERVER
  - api_key: FTRACK_API_KEY (or FTRACK_APIKEY)
  - api_user: FTRACK_API_USER
  - schema cache path: FTRACK_API_SCHEMA_CACHE_PATH (if provided; otherwise the path is determined as described in the docstring)

- Caching
  - The session adds the specified cache to a pre-configured layered cache with a MemoryCache at the top level. This pattern avoids the need to construct a separate memory cache for typical usage.

- Event hub
  - If auto_connect_event_hub is True (default), the embedded event hub connects to the event server in a background thread, enabling non-local event publishing and subscribing. Subscribing to events does not require a connected hub, and plugins may check hub status prior to subscribing.

- Plugins
  - plugin_paths allows discovery of plugins. If a discovered plugin signature does not accept the provided plugin_arguments, the discovery mechanism attempts to reduce arguments to those accepted, logging a warning if necessary.

- Operations
  - The session tracks pending operations using recorded_operations and record_operations flags.

- API details
  - The provided snippet shows the core structure of SessionAuthentication and Session, including the headers added for authentication and the validation of required fields in Session.__init__.

This documentation focuses on Understanding sessions and the main aspects of the Session class, including configuration options, environment variable fallbacks, and the role of the embedded event hub and caching.
