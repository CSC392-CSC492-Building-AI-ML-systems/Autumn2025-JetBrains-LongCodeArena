# Compatibility Policy

## Overview
This policy defines how compatibility is maintained for the compatibility.rst component, focusing on stable APIs/ABIs, cross-platform behavior, and reproducible builds. It ties the build configuration and packaging choices in the source to long-term compatibility with NumPy, Python, and downstream users.

## Key Components

- API and ABI compatibility
  - Preserve access to NumPy’s C-API by enabling deprecated features where necessary to avoid breaking existing code.
  - Use explicit build-time flags to control API exposure and compatibility surface.
  - Example: allow deprecated API usage for broader compatibility
    - c_args_random includes: `'-DNPY_NO_DEPRECATED_API=0'` (NumPy C-API compatibility)
    
- Platform and toolchain compatibility
  - Include platform-specific handling to ensure consistent behavior across environments.
  - Example: Windows/Cygwin handling to preserve symbol visibility
    - `if host_machine.system() == 'cygwin'`
      - add: `'-Wl,--export-all-symbols'`
    
- Build-time consistency and reproducibility
  - Generate and integrate essential C/Python interfaces in a controlled manner to avoid drift.
  - Use custom targets to produce and install interface files when needed (e.g., pxd files).
  - Example: _bounded_integers interface generation
    - Custom target to generate `_bounded_integers.pxd` from `_bounded_integers.pxd.in`
    - Custom target to generate `_bounded_integers.pyx` from `_bounded_integers.pyx.in`
  - Centralized management of sources and dependencies to maintain ABI stability across builds.

- Packaging, distribution, and import stability
  - Install libraries and Python modules under stable subdirectories to preserve import paths.
  - Example:
    - npyrandom library installed to: `np_dir / 'random/lib'`
    - Python extension modules installed under: `subdir: 'numpy/random'`
  - Ensure tests and data are packaged consistently to verify compatibility across environments.

- Licensing and policy notice
  - Document licensing clearly to avoid ambiguities in downstream usage.
  - Note: The source indicates a dual-license model with the University of Illinois/NCSA Open Source License; ensure license terms are respected in distributions.

## Usage

- When building to preserve compatibility:
  - Keep deprecated API access enabled unless a deliberate break is planned.
  - Use platform-aware linker flags (e.g., for Cygwin) to maintain symbol exports.
  - Maintain stable module layout and install paths so existing importers continue to work.
- Example build snippets reflecting compatibility choices:
  - Enable deprecated NumPy API access:
    - c_args: `['-DNPY_NO_DEPRECATED_API=0']`
  - Cywin symbol export guidance:
    - if host: `cygwin` then append `'-Wl,--export-all-symbols'`
  - Install locations for compatibility:
    - `install_dir: np_dir / 'random/lib'`
    - Python modules under: `subdir: 'numpy/random'`

- API surface management
  - Avoid unnecessary breaking changes by retaining access to deprecated APIs unless intentionally deprecated in a controlled release.

- Build reproducibility
