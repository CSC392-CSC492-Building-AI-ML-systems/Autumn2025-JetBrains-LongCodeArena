Delete view (Preview+Confirm+Delete) for Form-based Delete Workflow
====================================================================

Overview
--------
This module provides a form-based delete workflow for Django Cradmin views. It introduces:
- DeleteViewMixin: the core behavior to render a preview/confirm page and perform the deletion.
- WithinRoleDeleteView: a concrete view class wired for use with a Cradmin role, including the standard within-role template and context augmentation.

The workflow implemented is:
- Preview: Present a preview of the object to be deleted, with a confirm message.
- Confirm: Show a delete button (and a cancel button) to confirm or abort.
- Delete: On submission, perform the deletion, then display a success message and redirect to the app index.

Core API
--------
DeleteViewMixin
- Purpose: Encapsulates the preview/confirm/delete workflow for a Django DeleteView-based interface.
- Import path note: You should import via from django_cradmin import viewhelpers and refer to it as viewhelpers.formview.DeleteViewMixin.

Key methods and defaults
- get_pagetitle()
  - Returns the page title for the delete confirmation view.
  - Default: "Confirm delete"
- get_action_label()
  - Name of the action being performed; used as the page title prefix and default for the delete button label.
  - Default: "Delete"
- get_delete_button_label()
  - Label for the delete button.
  - Default: same as get_action_label()
- get_cancel_button_label()
  - Label for the cancel button.
  - Default: "Cancel"
- get_object_preview()
  - Textual preview of the object to be deleted. By default, uses str(self.get_object()).
- get_confirm_message()
  - Message shown in the focus area asking the user to confirm deletion.
  - Default: 'Are you sure you want to delete "%(object_preview)s"?' formatted with the object preview.
- get_success_url()
  - Redirect URL after a successful deletion.
  - Default: app index URL for the current Cradmin app (request.cradmin_app.reverse_appindexurl()).
- get_success_message(object_preview)
  - Optional user-facing success message after deletion.
  - Default: 'Deleted "%(what)s"' formatted with the object_preview.
- add_success_messages(object_preview)
  - Hook to add success messages after deletion.
  - If get_success_message returns a value, a Django messages.success is emitted.
- add_delete_view_mixin_context_data(context)
  - Populates common context data used by templates:
    - model_verbose_name: object's verbose name
    - success_url: get_success_url()
    - object_preview: get_object_preview()
    - pagetitle: get_pagetitle()
    - confirm_message: get_confirm_message()
    - delete_button_label: get_delete_button_label()
    - cancel_button_label: get_cancel_button_label()
- delete(request, *args, **kwargs)
  - Overrides the DeleteView.delete method to capture object_preview before deletion, perform the deletion, then add success messages, and return the response.

WithinRoleDeleteView
- Purpose: A concrete, role-aware delete view with sane default context and template for Cradmin “Within Role” scenarios.
- Inheritance: QuerysetForRoleMixin, DeleteViewMixin, DjangoDeleteView, CommonCradminViewMixin, javascriptregistry.viewmixin.WithinRoleViewMixin
- Import path note: You should import via from django_cradmin import viewhelpers and refer to it as viewhelpers.formview.WithinRoleDeleteView.
- Template: template_name = 'django_cradmin/viewhelpers/formview/within_role_delete_view.django.html'
- Context data augmentation:
  - get_context_data() extends the base context with:
    - javascriptregistry component IDs
    - common Cradmin view mixin data
    - delete view mixin context data (as above)

How the workflow is realized in practice
----------------------------------------
- Rendering the confirmation page (GET):
  - The view prepares context via get_context_data(), including:
    - object (the target instance)
    - object_preview and confirm_message for user visibility
    - pagetitle and labels for UI buttons
  - The template (within_role_delete_view.django.html) presents the preview, the confirm message, and the action buttons.
- Submitting the confirmation (POST):
  - delete() is invoked, which:
    - captures object_preview before deletion
    - delegates to the base Django DeleteView to perform the deletion
    - calls add_success_messages(object_preview) to show a success message
    - returns the response (redirect to get_success_url())

Context data exposed to templates
---------------------------------
- object: the object being deleted
- model_verbose_name: verbose name of the model
- success_url: URL to redirect to after deletion
- object_preview: string representation for preview
- pagetitle: page title (from get_pagetitle)
- confirm_message: confirmation message (from get_confirm_message)
- delete_button_label: label for the delete button
- cancel_button_label: label for the cancel button

Template considerations
-----------------------
- Default template path for WithinRoleDeleteView:
  django_cradmin/viewhelpers/formview/within_role_delete_view.django.html
- This template relies on the context keys listed above to render:
  - A focused confirmation area including object_preview and confirm_message
  - Action buttons using delete_button_label and cancel_button_label
  - A visible pagetitle derived from pagetitle

Usage examples
--------------
Minimal example using WithinRoleDeleteView:

.. code-block:: python

    from django_cradmin import viewhelpers
    from myapp.models import MyModel

    class MyModelDeleteView(viewhelpers.formview.WithinRoleDeleteView):
        model = MyModel
        # Optionally customize, e.g. queryset, permissions, etc.

    # URL conf (example)
    # path('mymodel/<int:pk>/delete/', MyModelDeleteView.as_view(), name='mymodel-delete')

Notes on customization
----------------------
- Override individual hooks to tailor text and behavior:
  - get_pagetitle(), get_action_label(), get_delete_button_label(), get_cancel_button_label()
  - get_object_preview(), get_confirm_message()
  - get_success_url(), get_success_message()
  - add_success_messages() for custom success handling
  - add_delete_view_mixin_context_data() to inject extra data into the template context
- For role-aware views, use WithinRoleDeleteView to gain proper Cradmin context processing and template usage.

Why and when to use
-------------------
- Use DeleteViewMixin when you need a reusable, consistent preview/confirm/delete flow for deleting objects in a Cradmin app.
- Use WithinRoleDeleteView when your delete view is tied to a specific Cradmin role and you want the standard within-role template and context augmentation.

This documentation covers the essential behavior and customization points for the preview+confirm+delete workflow implemented by viewhelpers_formview_deleteview.rst.
