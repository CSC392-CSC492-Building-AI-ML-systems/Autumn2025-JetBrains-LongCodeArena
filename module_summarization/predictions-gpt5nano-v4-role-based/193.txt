The Metadata Routing API
========================

Overview
--------

The Metadata Routing API provides a compact, composable set of utilities that enable meta-estimators to route operations to one or more sub-estimators based on per-sample or per-dataset metadata. This mechanism is used internally by multioutput and chain-like estimators to apply different sub-estimators to distinct data groups, enabling flexible and granular modeling workflows.

Public API surface (imported from sklearn.utils.metadata_routing)
---------------------------------------------------------------

- MetadataRouter
  - Role: Central routing component that holds sub-estimators and routing rules and delegates calls (e.g., fit, predict) to the appropriate sub-estimator(s) according to the routing configuration.
  - Typical use: instantiate with a map of sub-estimators and a routing policy, then call fit/predict with metadata guiding the routing decisions.

- MethodMapping
  - Role: Helper to declare and validate how methods should be routed across sub-estimators.
  - Purpose: provide a structured way to describe which methods are routable and how they map to sub-estimators.

- process_routing
  - Role: Utility to construct and validate a routing configuration from user input.
  - Output: a normalized, internal routing structure that MetadataRouter can consume.

- _routing_enabled
  - Role: Internal predicate indicating whether routing is active for a given object.
  - Usage: typically used within the routing machinery to decide if routing logic should be engaged.

- _raise_for_params
  - Role: Internal helper that validates routing configuration against estimator parameters.
  - Behavior: raises informative errors when routing configuration is incompatible with the provided estimators or their parameters.

Notes on internal API
---------------------

- Names prefixed with an underscore (_routing_enabled, _raise_for_params) are intended as internal utilities. External users should primarily interact with MetadataRouter, and optionally process_routing for configuring routing.

Concepts and workflow
---------------------

- Routing scope: Routing decisions can be driven by per-sample metadata (e.g., group labels, target types) or per-dataset metadata. This allows different sub-estimators to handle distinct portions of the data within a single higher-level estimator.
- Composition: MetadataRouter works with a map of sub-estimators and a routing policy. It delegates appropriate method calls to the chosen sub-estimator(s) based on the routing configuration.
- Validation: process_routing and _raise_for_params ensure that routing definitions are consistent with the capabilities and parameters of the contained estimators, helping catch misconfigurations early.

Usage example (illustrative)
-----------------------------

Note: The exact constructor signatures and method calls depend on the current implementation. The following is an illustrative pattern to convey intended usage.

- Conceptual example of wiring two sub-estimators with routing:

  from sklearn.utils.metadata_routing import MetadataRouter

  estimators = {
      "est_A": EstimatorA(),
      "est_B": EstimatorB(),
  }

  routing_config = {
      "group_by": {"A": "est_A", "B": "est_B"}
  }

  router = MetadataRouter(estimators, routing=routing_config)

  # Use per-sample routing metadata to guide fitting/prediction
  router.fit(X, y, sample_metadata=metadata)
