Library conventions
===================

This section describes the conventions used in the codebase to promote consistency, readability, and maintainability across the library. The examples reference the provided code (dtime.py and related modules) to illustrate how these conventions are applied in practice.

Purpose
-------

- Establish a uniform approach to API design, documentation, imports, error handling, and code structure.
- Aid developers in understanding how to contribute new functionality that integrates cleanly with the existing architecture.

Public API and exports
----------------------

- Expose a minimal, stable surface via __all__ in each module.
  - Example: In dtime.py, the public API is declared as:
    - __all__ = ['sample_system', 'c2d']
- Keep public functions small and focused; prefer wrappers that delegate to core functionality.
  - Example: c2d delegates to sample_system to perform the work, providing a thin wrapper pattern.

Docstrings and documentation style
----------------------------------

- Use NumPy/SciPy-style docstrings with explicit sections:
  - Parameters
  - Returns
  - Notes
  - Examples
- Cross-reference related types and classes using Sphinx roles.
  - Example: sysc : LTI (:class:`StateSpace` or :class:`TransferFunction`).
- Provide practical examples in the Examples section that demonstrate typical usage.
  - Example from sample_system:
    - >>> sysc = TransferFunction([1], [1, 2, 1])
    - >>> sysd = sample_system(sysc, 1, method='bilinear')
- Maintain consistency in unit/parameter naming and constraints.
  - Example: alpha is a parameter used only with method="gbt" (as documented).

Input validation and error handling
-----------------------------------

- Validate inputs and raise clear exceptions when preconditions are not met.
  - Example: sample_system checks that the input is a continuous-time system using isctime, and raises ValueError if not.
- Use standard exception types (e.g., ValueError) for invalid arguments to aid user debugging.

Interfaces and consistency
--------------------------

- Prefer a consistent interface for time-domain conversion:
  - A continuous-time LTI system (sysc) is converted to a discrete-time system (sysd) given a sampling period Ts.
  - The conversion method is delegated to the system’s own sampling implementation (sysc.sample(...)).
- Ensure the return type is the same family of objects as the input (e.g., StateSpace/TransferFunction) to preserve user expectations.
- Maintain consistent naming conventions for core functionality:
  - sample_system for direct sampling-based conversion
  - c2d as a conventional alias/wrapper (documented as delegating to sample_system)

Code structure and imports
--------------------------

- Use relative imports for internal modules to reflect package structure:
  - Example: from .namedio import isctime; from .statesp import StateSpace
- Keep imports narrow and explicit to minimize side effects and improve readability.
- Place module-level __all__ to clearly define the public surface of the module.

Licensing, headers, and metadata
-------------------------------

- Preserve licensing and copyright headers as required by project policy.
- Include author/date information where relevant to maintain historical context.
- Ensure the module docstring (or file header) provides a brief description of purpose and contents.

Examples from the codebase
--------------------------

- dtime.py demonstrates:
  - A tight, well-documented API for time-domain conversions.
  - A clear public surface via __all__.
  - Input validation using isctime and a simple delegation to the object’s sample method.
  - Docstrings with sections that guide users through usage and options (e.g., method, alpha, prewarp_frequency).

Illustrative conventions in practice
------------------------------------

- Documentation style for a function:
  - Use a numpy-style docstring with Parameters, Returns, Notes, and Examples.
  - Example pattern in sample_system:
    - Parameters include sysc with a precise type description (LTI, StateSpace or TransferFunction).
    - Returns describes the discrete-time system with consistent terminology (sysd, linsys).
    - Notes references related methods for further details.
    - Examples show a minimal, reproducible usage scenario.
- Wrapper implementation pattern:
  - c2d is a thin wrapper around sample_system, reinforcing the convention that specialized operations delegate to a common core mechanism when possible.

Practical guidance for contributors
------------------------------------

- Follow NumPy docstring conventions for all public functions.
- Keep __all__ declarations in sync with actual public API.
- Validate input types and states early; fail fast with informative messages.
- Document non-obvious parameter interactions (e.g., alpha only used with certain methods).
- Retain licensing headers and provide clear module-level descriptions.

Notes
-----

- The conventions described here are intended to complement, not replace, project-specific guidelines. When in doubt, align with the established patterns demonstrated by the provided code (dtime.py and related modules).
