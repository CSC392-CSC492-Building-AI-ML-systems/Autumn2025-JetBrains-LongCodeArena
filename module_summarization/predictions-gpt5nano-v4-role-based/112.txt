# metadata.rst Documentation

Overview
--------

This module provides metadata-aware planning utilities used to orchestrate data acquisition
with detectors and motors. It defines high-level plans that assemble rich metadata dictionaries
along with the actual data collection logic. The supported plans are:

- count: take a specified number of readings from one or more detectors.
- list_scan: perform a multi-dimensional scan (inner product) over one or more motors.
- rel_list_scan: scaffold for a relative multi-motor scan (Not implemented yet).

Key dependencies and concepts
-----------------------------

- Bluesky-inspired plan patterns and decorators:
  - plan_patterns: provides helpers for complex plan patterns (e.g., inner list-product scans).
  - preprocessors as bpp: stage and run decorators to annotate plans with metadata and staging logic.
  - plan_stubs as bps: utility stubs like one-shot hooks and repeat generators used inside plans.
- partition (from cytools or toolz): used to split interleaved (motor, positions) pairs.
- Detectors are expected to be “readable” objects with a name attribute, which is used to populate metadata.
- Metadata structure is populated in a consistent shape to support downstream tooling and UI.

Public API
----------

### count(detectors, num=1, delay=None, *, per_shot=None, md=None)

Take one or more readings from detectors.

Parameters
- detectors (list): List of readable detector objects. Each detector must have a name attribute; the plan uses det.name to populate metadata.
- num (int, optional): Number of readings to take. If None, the plan will run indefinitely (i.e., until canceled). Default is 1.
- delay (iterable or scalar, optional): Time delay in seconds between successive readings. A scalar delay applies to every interval; an iterable provides per-interval delays. If an iterable is provided, it must have at least num - 1 entries or a ValueError will be raised during plan iteration.
- per_shot (callable, optional): Hook for customizing actions of the inner loop. The hook should have the following signature:
  - def f(detectors: Iterable[OphydObj]) -> Generator[Msg]: ...
  If not provided, the default per-shot hook is bps.one_shot.
- md (dict, optional): Metadata dictionary to augment the generated plan metadata.

Returns
- A generator-based plan. The plan yields to the inner loop via:
  - inner_count() which uses bps.repeat(partial(per_shot, detectors), num=num, delay=delay)

Metadata details
- Detectors field: _md['detectors'] = [det.name for det in detectors]
- Plan identification: _md['plan_name'] = 'count'
- Points and timing:
  - _md['num_points'] = num
  - _md['num_intervals'] = num - 1 (or None if num is None)
- Plan arguments for reproducibility: _md['plan_args'] = {'detectors': list(map(repr, detectors)), 'num': num}
- Pattern hints:
  - _md['plan_pattern'] = None (not used for 'count')
  - _md['plan_pattern_module'] may be omitted or set by higher-level tooling
- Initial hints: _md['hints'] is initialized as an empty dict, then defaulted to include dimensions for timing:
  - _md['hints'].setdefault('dimensions', [(('time',), 'primary')])

Example
-------
- Basic count of 5 measurements from two detectors, with 0.2s delay:
  - plan = count([det1, det2], num=5, delay=0.2)
  - RE(plan)

Notes
- The count plan is straightforward and primarily focuses on assembling and propagating metadata alongside detector readings.

### list_scan(detectors, *args, per_step=None, md=None)

Scan over one or more variables in steps simultaneously (inner product).

Parameters
- detectors (list): List of readable detector objects (same requirements as for count).
- *args: Pairs specifying the motor (or settable object) and a list of target points. The structure must be:
  - motor1, [point1, point2, ...],
  - motor2, [point1, point2, ...],
  - ...
  Each motor/points pair is processed as an axis in the scan.
- per_step (callable, optional): Hook for customizing actions of the inner loop per step. Signature:
  - def f(detectors, motor, step) -> plan (generator)
- md (dict, optional): Metadata dictionary to augment the generated plan metadata.

Returns
- A generator-based plan that executes a multi-motor inner-product scan by delegating execution to Bluesky’s scan_nd, passing a cycler generated by plan_patterns.inner_list_product(args).

Implementation notes
- Argument validation:
  - The number of *args must be even (motor, positions) for each motor. If not, a ValueError is raised.
  - All position lists must have the same length. If not, a ValueError is raised with a detailed message.
- Metadata construction:
  - _md includes detectors, motors (motor names), num_points (length of the position lists), num_intervals (length - 1).
  - plan_args captures the detectors and the flattened args as repr strings and lists.
  - plan_name: 'list_scan'
  - plan_pattern: 'inner_list_product'
  - plan_pattern_module: plan_patterns.__name__
  - plan_pattern_args: {'args': md_args}
- Hints and dimensions:
  - Extracts motor hints.fields to determine which axes should be labeled as primary dimensions.
  - If any axes have fields, adds a dimensions hint: dimensions=[(fields, 'primary')].
  - Merges any user-provided hints from md['hints'].
- Execution:
  - Builds full_cycler via plan_patterns.inner_list_product(args) and defers to scan_nd(detectors, full_cycler, per_step=per_step, md=_md).

Example
-------
- Two motors with three points each:
  - RE(list_scan([det1], motor1, [0, 1, 2], motor2, [-1, 0, 1]))
- With custom per-step behavior:
  - RE(list_scan([det1], motor1, [0, 1], motor2, [0, 1], per_step=my_step_hook))

Metadata details
- Detectors: _md['detectors'] = [det.name for det in detectors]
- Motors: _md['motors'] = motor_names
- Points: _md['num_points'] = length
- Intervals: _md['num_intervals'] = length - 1
- Plan arguments:
  - _md['plan_args'] = {'detectors': [...], 'args': md_args, 'per_step': repr(per_step)}
  - _md['plan_pattern'] = 'inner_list_product'
  - _md['plan_pattern_module'] = plan_patterns.__name__
  - _md['plan_pattern_args'] = {'args': md_args}
- Hints:
  - _md['hints'] initialized and updated to include dimensions based on motor hints.fields if present.
  - User-supplied hints (md.get('hints')) are merged last.

See Also
- plan_patterns.inner_list_product: utility to generate the cycler for inner-list-product scans.
- bluesky.plans.scan_nd: underlying mechanism used to execute multi-axis scans.
- preprocessors (bpp) and plan_stubs (bps): decorators and stubs used to annotate and run these plans in a RunEngine.
- Msg: message type used for per-step hooks.

Notes
- This module relies on cytools.partition (falling back to toolz.partition if cytools is unavailable) to split interleaved motor/points arguments.
- rel_list_scan is declared but not implemented yet (TODO). It is intended to provide relative multi-axis scans.

Rel_list_scan
-------------

- Currently a placeholder with a TODO.
- Intended behavior: scan over variables relative to the current position, mirroring list_scan semantics but using relative steps.
- NotImplemented: As of this version, use count or list_scan for functional scanning.

Usage Tips
----------

- Ensure detectors have a meaningful name attribute, as the metadata uses det.name to populate _md['detectors'].
- When constructing list_scan, provide equal-length point lists for each motor; otherwise a ValueError will be raised with a helpful message.
- Use md to add custom metadata; it will be merged with the generated plan metadata and can influence downstream dashboards and logging.

Implementation details (for maintainers)
---------------------------------------

- Core functions are decorated with:
  - @bpp.stage_decorator(detectors)
  - @bpp.run_decorator(md=_md)
- count uses a per-shot hook (default: bps.one_shot) and a repeat generator to iterate over the specified number of points with optional delays.
- list_scan derives a cycler from plan_patterns.inner_list_product and delegates execution to scan_nd, supplying a detailed metadata object and potential per-step hooks.
- rel_list_scan remains a scaffold and should be completed to provide relative-scanned functionality similar to list_scan.

This documentation captures the public-facing API, metadata semantics, and usage patterns for the code in metadata.rst.
