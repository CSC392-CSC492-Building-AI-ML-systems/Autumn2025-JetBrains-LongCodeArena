Using review sessions
=====================



Overview
--------

A review session provides an isolated context for interacting with an ftrack server. It is built around the Session class and exposes features that are useful when reviewing work across entities, such as on-demand population of attributes, caching strategies, and optional event hub support for non-local events. This guide describes how to configure and use a review session effectively for code review workflows and related tasks.

Prerequisites
-------------

- Access to an ftrack server.
- Credentials: server URL, API key, and API user.
  - These can be provided directly or sourced from environment variables:
    - FTRACK_SERVER
    - FTRACK_API_KEY (or FTRACK_APIKEY for backward compatibility)
    - FTRACK_API_USER
- Optional: environment for plugin discovery and schema caching if you want to customize behavior.

Key concepts
------------

- Auto-population (auto_populate)
  - When enabled (default), accessing an entity’s attributes will automatically fetch missing data from the server.
  - Disable to fetch data only when you explicitly request data, which can improve startup time for large sessions.
- Event hub (auto_connect_event_hub)
  - If enabled, an embedded event hub is started in the background to enable non-local event publishing/subscription.
  - Local-only mode remains available until you connect to a full event hub explicitly.
- Caching
  - Sessions can be configured with a cache to improve performance and reduce server load.
  - The top-level cache is a MemoryCache by default; you can supply custom caches or a callable that returns a cache.
- Schema cache
  - Schema information can be cached on disk for faster startup and reduced server queries.
- Plugins
  - Plugins can extend session capabilities; you can pass keyword arguments to plugin discovery to tailor their initialization.

Creating a review session
-------------------------

The Session class requires connection details and offers sensible defaults via environment variables. You can create a session in several ways:

- Minimal (uses environment variables):
  - The session will read FTRACK_SERVER, FTRACK_API_KEY (or FTRACK_APIKEY), and FTRACK_API_USER to establish authentication.

  .. code-block:: python

     import ftrack_api
     session = ftrack_api.Session()

- Explicit configuration:
  - Provide server URL, API key, and API user directly.

  .. code-block:: python

     import ftrack_api
     session = ftrack_api.Session(
         server_url="https://ftrack.example.com",
         api_key="your_api_key",
         api_user="your_username",
         auto_populate=True,
         auto_connect_event_hub=True
     )

- Advanced options
  - Customize caching, schema cache, and plugin behavior as needed.

  .. code-block:: python

     import ftrack_api
     from ftrack_api.cache import MemoryCache, DiskCache
     session = ftrack_api.Session(
         server_url="https://ftrack.example.com",
         api_key="your_api_key",
         api_user="your_username",
         cache=MemoryCache(),
         schema_cache_path="/path/to/schema/cache",
         auto_connect_event_hub=False,
         plugin_paths=["/path/to/plugins"],
         plugin_arguments={"arg1": "value1"}
     )

Authentication and server selection
-----------------------------------

- The session requires a server URL and credentials.
- If not provided, environment variables are consulted:
  - FTRACK_SERVER for the server URL
  - FTRACK_API_KEY / FTRACK_APIKEY for the API key
  - FTRACK_API_USER for the API user (falls back to the system user if not provided)

- If any required information is missing, a TypeError is raised with guidance on how to supply it.

Behavioral flags and configuration
----------------------------------

- auto_populate (default: True)
  - When set to True, missing attributes on entities will be loaded on-demand from the server as you access them.
  - Set to False to fetch data only when you explicitly request it.

  .. code-block:: python

     session = ftrack_api.Session(..., auto_populate=False)

- auto_connect_event_hub (default: depends on session setup)
  - When True, an embedded event hub is connected in the background, enabling non-local event publishing and subscribing.
  - When False, only local events are supported until you explicitly connect to an event hub.

  .. code-block:: python

     session = ftrack_api.Session(..., auto_connect_event_hub=True)

- cache
  - A cache instance implementing the ftrack_api.cache.Cache interface, or a callable that receives the session and returns a cache or None.
  - The session adds the provided cache to a layered cache stack (with MemoryCache at the top by default).

  .. code-block:: python

     from ftrack_api.cache import MemoryCache
     session = ftrack_api.Session(cache=MemoryCache())

- cache_key_maker
  - Used to generate keys for objects stored in the cache. If not supplied, a StringKeyMaker is used.

- schema_cache_path
  - Path to a folder for persistent schema caching.
  - If not set, environment variable FTRACK_API_SCHEMA_CACHE_PATH is used.
  - If neither is set, a temporary directory is used.
  - Set to False to disable schema caching entirely.

  .. code-block:: python

     session = ftrack_api.Session(schema_cache_path="/tmp/ftrack/schema_cache")

- plugin_arguments
  - A mapping of keyword arguments passed to plugin registration functions during discovery.
  - If a plugin signature does not accept some arguments, those arguments are reduced to the ones the plugin accepts with a warning.

  .. code-block:: python

     session = ftrack_api.Session(plugin_arguments={"verbose": True})

Using review sessions
---------------------

Typical review-session workflows revolve around inspecting and possibly modifying entities in a controlled, efficient manner. The Session API provides the foundation for these workflows with on-demand data loading, caching, and event handling.

Practical guidelines
- Start with a minimal session (prefer environment variables) to reduce boilerplate for quick reviews.
- Use auto_populate to control when data is fetched:
  - For read-heavy reviews, keep auto_populate enabled to simplify access patterns.
  - For performance-critical reviews over large datasets, disable auto_populate and fetch data explicitly as needed.
- Leverage the event hub for collaboration or triggering external review events:
  - Keep auto_connect_event_hub enabled if you rely on remote events, otherwise connect manually when needed.
- Configure a cache to speed up repeated reviews:
  - A persistent or in-memory cache can dramatically reduce server load during iterative reviews.
- Enable schema caching to speed up session startup in large environments.
- If you rely on plugins for review-specific behavior, pass appropriate plugin_arguments to ensure compatibility.

Example review session workflow
-------------------------------

1) Create a session (prefer environment-driven startup for simplicity):

.. code-block:: python

   import ftrack_api
   session = ftrack_api.Session()

2) Inspect an entity and its attributes
   - Access attributes; if auto_populate is enabled, missing data will be loaded on demand.

   .. code-block:: python

      # Pseudo-example: actual entity retrieval depends on the session's query/lookup APIs
      entity = session.find('task', 'abc123')  # illustrative; adjust to actual API usage
      name = entity.name  # triggers fetch if not populated

3) Review changes with recorded operations
   - The session tracks pending operations in recorded_operations; you can inspect or flush these as part of your review workflow.

   .. code-block:: python

      ops = session.recorded_operations
      # Inspect or apply ops as appropriate for your workflow

4) Enable non-local event awareness (optional)
   - If your review requires reacting to or publishing events across the network, ensure the event hub is connected.

   .. code-block:: python

      # Example hint (exact invocation may vary by version)
      session.event_hub.connect('wss://ftrack.example.com/events')

5) Persist schema and improve startup performance (optional)
   - Configure a schema_cache_path to persist schema information between sessions.

   .. code-block:: python

      session = ftrack_api.Session(schema_cache_path="/path/to/schema/cache")

Best practices
--------------

- Use environment-based startup for quick review sessions in CI or local development.
- Prefer controlled auto_populate settings when reviewing large or complex datasets.
- Consider a dedicated cache for review sessions to avoid stale data across reviews.
- Document any plugin-based extensions used during reviews for reproducibility.

Troubleshooting
---------------

- Missing server_url, api_key, or api_user errors indicate misconfigured environment or missing explicit parameters.
- If auto_populate is unexpectedly loading data slowly, review your cache configuration and the size of the data being fetched.
- When using the event hub, ensure the hub is connected and that network access to the event server is permitted.
