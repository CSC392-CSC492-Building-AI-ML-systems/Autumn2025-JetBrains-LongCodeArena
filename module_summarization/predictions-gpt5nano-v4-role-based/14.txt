Querying
========

Overview
--------

Querying in ftrack API provides a structured way to retrieve data from a connected ftrack server. The querying subsystem is exposed via the ftrack_api.query module and is designed to work hand-in-hand with a Session instance. Queries are constructed, filtered, and executed to return entities and their requested fields from the server, with eager or lazy loading controlled by the session's configuration.

Key concepts
------------

- Session bound queries: All queries are executed against a specific Session, which carries authentication, server URL, cache configuration, and auto-populate behavior.
- Models and fields: A query targets a specific entity type (e.g., tasks, assets, shots) and specifies which fields to return.
- Filtering and conditions: Queries support filtering by values, ranges, membership, and related fields to narrow results.
- Projection: Field selection (projection) limits data returned to only what you need, improving performance.
- Ordering and paging: Results can be ordered and paged to handle large data sets.
- Lazy loading vs. eager fetching: Depending on session configuration, related data may be loaded on demand or pre-fetched as part of the query.

Getting started
--------------

To use querying, you first create a Session and then construct a query through the ftrack_api.query interface, providing the session for connectivity and caching context.

Example (conceptual)
- Create a session
- Build a query for a specific entity type
- Specify fields to fetch
- Apply filters
- Execute and iterate results

Notes:
- The Session class handles server URL, API key, user, and optional auto-population of related data.
- The ftrack_api.query module is the entry point for constructing and running queries.

Building a query
----------------

A query is constructed in the context of a Session. You typically:

- Instantiate a Query object bound to your session.
- Specify the target entity type (e.g., Task, Shot, Asset).
- Declare which fields to return (projection).
- Apply one or more filters to constrain results (e.g., name contains "render", id in a set).
- Optionally specify ordering and paging (limit/offset).

Filtering and conditions
------------------------

Filters narrow results to entities matching given criteria. Typical filter capabilities include:

- Equality and inequality
- String containment and pattern matching
- Membership (in / not in)
- Comparison operators (greater than, less than, etc.)
- Filtering on related fields (dot notation to traverse relationships)

Projection and fields
---------------------

Projection controls which fields are included in the returned objects. Project only what you need to minimize data transfer and processing time. Returned entities expose the selected fields as attributes.

Execution
---------

Executing a query yields results that can be iterated in Python. Each item is typically an entity object with attributes corresponding to the projected fields. Depending on the session's auto_populate setting, related data may be loaded lazily or eagerly as part of the query.

Performance considerations
--------------------------

- Projection: Always project only the fields you need.
- Filtering: Push filters to the server to reduce the data volume transferred.
- Caching: Take advantage of the session’s cache (in-memory, or configured caches) to avoid repeated fetches for identical queries or entities.
- Paging: For large result sets, fetch results in pages or batches.

Error handling
--------------

Querying interacts with the server and may raise exceptions related to connectivity, authentication, or invalid query construction. Catch and handle ftrack_api.exception.Subclassed exceptions as appropriate for robust tooling.

Examples
--------

Simple query: fetch IDs and names of Task entities containing "render" in their name

- Concept:
  - Target: Task
  - Projection: id, name
  - Filter: name contains "render"
  - Execute and print results

Pseudo-code (illustrative):

from ftrack_api import Session
from ftrack_api import query as ftrack_query

session = Session(
    server_url='https://ftrack.example.com',
    api_key='YOUR_API_KEY',
    api_user='YOUR_USERNAME'
)

Q = ftrack_query.Query(session)
Q.set_type('Task')
Q.set_fields(['id', 'name'])
Q.filter_by('name', 'contains', 'render')
results = Q.execute()

for task in results:
    print(task.name, task.id)

Complex query: fetch recent Tasks assigned to a specific user with related project data

- Concept:
  - Target: Task
  - Projection: id, name, due_date, project.name
  - Filters: assignee == 'jdoe@example.com', due_date >= today
  - Joins/relations: include related project information
  - Ordering: due_date ascending
  - Execution and iteration

Pseudo-code (illustrative):

from datetime import date
today = date.today()

Q = ftrack_query.Query(session)
Q.set_type('Task')
Q.set_fields(['id', 'name', 'due_date', 'project.name'])
Q.filter_by('assignee', 'equals', 'jdoe@example.com')
Q.filter_by('due_date', 'greater_than_or_equal', today)
Q.order_by('due_date')
results = Q.execute()

for t in results:
    print(f"{t.name} (Due: {t.due_date}) in project {t.project.name}")

Best practices
--------------

- Prefer projection of only necessary fields to minimize payload.
- Push filters to the server to reduce data transfer and processing.
- Leverage session caching for frequently accessed entities.
- Use pagination for large result sets to maintain responsiveness.

Relation to other components
----------------------------

- Queries rely on the Session for server connectivity and authentication (server URL, api_key, api_user) as demonstrated in the Session class initialization.
- Auto-population and lazy loading settings affect how and when related data is retrieved during or after a query.
- The results of queries are typically entities from ftrack_api.entity.* modules, which integrate with the broader object model (dependencies, relationships, and attributes).
