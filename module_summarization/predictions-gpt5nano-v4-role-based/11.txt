Working with entities
===================== 

This document describes how to work with entities in the ftrack API via the Session entry point, including lazy loading (auto-population), caching, authentication, and basic operation tracking. The examples reflect the behavior and options exposed by the Session class and related components in the provided code.

Overview
--------

- An entity represents a resource in ftrack (for example, tasks, assets, or other managed objects) that you interact with through a Session.
- The Session is the primary interface for connecting to a server, retrieving entities, and performing operations on them.
- By default, entity attributes are loaded lazily. When you access an attribute that hasn’t been loaded yet, the session can automatically fetch the data from the server (auto-populate).

Key concepts in this code

- Auto-population (lazy loading)
  - The Session parameter auto_populate controls whether attribute access triggers remote population automatically.
  - Default: True. Accessing an attribute that is not yet loaded will fetch the value from the server.
  - You can disable automatic loading by setting session.auto_populate to False to avoid implicit server calls.

- Session initialization and required credentials
  - server_url: The ftrack server URL (including port if needed). If not provided, the code looks for FTRACK_SERVER in the environment.
  - api_key and api_user: Credentials used for authentication. If not provided, the code looks for environment variables FTRACK_API_KEY and FTRACK_API_USER (or equivalent fallbacks).
  - If these required values are not supplied, TypeError is raised with guidance on how to provide them.

- Caching
  - The Session supports a layered, pluggable caching system. A top-level MemoryCache is used by default to back a layered cache.
  - cache: The session can accept a cache object implementing ftrack_api.cache.Cache or a callable that returns such a cache when invoked with the session.
  - cache_key_maker: Used to generate stable keys for objects stored in the cache. If not supplied, a StringKeyMaker is used.

- Authentication for requests
  - SessionAuthentication attaches ftrack-specific authentication headers to outgoing HTTP requests:
    - ftrack-api-key
    - ftrack-user
  - You can use this class to customize how requests are authenticated, or to perform ad-hoc authenticated requests.

- Operations tracking
  - The Session maintains recorded_operations (an Operations collection) representing pending actions.
  - The flag record_operations controls whether operations are recorded. This is useful for batching or auditing actions taken through the session.

Working with entities: practical guidance
----------------------------------------

- Creating a Session
  - Use explicit values or environment variables (as the code supports) to provide server URL, API key, and API user.
  - Example (explicit values):
    - server_url: The ftrack server URL (e.g., "https://ftrack.example.com")
    - api_key: Your API key
    - api_user: Your user name in ftrack

- Example: basic session setup (conceptual)
  - This demonstrates how a session is configured; it mirrors the behavior in the code where credentials can be supplied directly or read from the environment.
  - The example also implies that you will typically obtain and work with entities through the session after it is created.

.. code-block:: python

    from ftrack_api import Session

    session = Session(
        server_url="https://ftrack.example.com",
        api_key="your_api_key",
        api_user="your_user"

- Auto-population behavior
  - With session.auto_populate set to True (default), accessing an attribute such as entity.name will fetch the data from the server if it has not been loaded yet.
  - To avoid implicit server calls when accessing attributes, disable auto-population:
  
.. code-block:: python

    session.auto_populate = False
    # entity should be obtained via an explicit retrieval method
    # name = entity.name  # This will not trigger a fetch automatically

- Caching considerations
  - By supplying a cache or a cache factory, you can influence how entities and related data are cached locally.
  - If you need custom caching behavior, pass a cache object implementing ftrack_api.cache.Cache or a callable returning such an object.
  - You can also customize the key generation used to store/retrieve items in the cache with cache_key_maker.

- Authentication customization
  - If you need to customize how requests are authenticated, you can wrap your requests with SessionAuthentication or provide a custom authentication object that follows the same interface.
  
.. code-block:: python

    import requests
    from ftrack_api import SessionAuthentication

    auth = SessionAuthentication(api_key="your_api_key", api_user="your_user")
    response = requests.get("https://ftrack.example.com/api/some-endpoint", auth=auth)

- Operations tracking and write workflows
  - The session keeps track of operations performed during a session in recorded_operations.
  - The record_operations flag allows you to enable/disable operation recording, which can be useful for batching, auditing, or dry-run scenarios.

- Schema caching and environment configuration
  - The session supports enabling schema caching by design (via schema_cache_path). If set, schema data may be cached to speed up repeated interactions.
  - If the schema_cache_path is not provided, the environment variable FTRACK_API_SCHEMA_CACHE_PATH (if defined) is used. If neither is set, a temporary directory is used.
  - You can disable schema caching by setting schema_cache_path to False.

Notes and caveats
-----------------

- The provided code excerpt focuses on the Session class, its initialization, and related components (authentication, caching, and operation tracking). Actual entity creation, update, or query operations are implemented via other modules (e.g., ftrack_api.query, ftrack_api.entity, etc.) and are used in conjunction with the Session.
- If required credentials are missing, the Session raises TypeError with guidance on how to provide the missing information either as arguments or via environment variables.
- The lazy-loading behavior depends on the auto_populate setting and the underlying server response. Network availability will affect the ability to populate attributes on-demand.

Appendix: Key classes and their roles
-------------------------------------

- Session
  - Entry point for server interaction, entity retrieval, lazy loading, caching, and operation tracking.
  - Parameters:
    - server_url, api_key, api_user: Authentication and server location
    - auto_populate: Enable/disable automatic attribute loading
    - cache, cache_key_maker: Caching configuration
    - auto_connect_event_hub, schema_cache_path, plugin_arguments: Additional behavior controls (see code and docstrings for details)

- SessionAuthentication
  - Small helper to attach ftrack-specific headers to HTTP requests, enabling server authentication.

- ftrack_api.cache
  - Interfaces and implementations used to store entities locally and speed up repeated access.

- ftrack_api.operation
  - Mechanism for tracking pending operations performed during a session.

- ftrack_api.query and ftrack_api.entity
  - Modules responsible for retrieving and manipulating entities. This doc focuses on how sessions enable and control their usage.

This documentation reflects the capabilities available in the provided code for working with entities through the Session object, including lazy loading, caching, and authentication aspects that influence how you interact with ftrack resources.
