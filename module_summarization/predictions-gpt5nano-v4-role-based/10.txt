Using metadata

Overview
- Metadata is a flexible, per-entity key-value store that allows you to attach arbitrary contextual information to any trackable entity managed by the session (for example, tasks, assets, or other ftrack entities). Keys are strings and values are JSON-serializable data.
- Metadata is accessed through the entity object. It is designed to be lazily loaded on first access (consistent with the session’s auto-populate behavior) and persisted as part of the session’s operation stream.

Key concepts from the code base
- Session-based lifecycle
  - The Session class coordinates communication with the ftrack server and tracks operations that should be sent to the server. Metadata changes are part of this operation stream and will be transmitted when the session commits/persists changes.
  - The session supports lazy loading and optional schema caching to improve performance when querying metadata definitions.
- Lazy loading and caching
  - By default, accessing attributes (and metadata) can trigger a server fetch if the information is not already loaded. This behavior is controlled by the session’s auto_populate setting, which applies to attribute and metadata access.
  - Schema caching can be enabled via schema_cache_path to speed up metadata key validation and discovery without repeated round-trips to the server.

Accessing metadata
- Reading metadata
  - Access the metadata on an entity using the metadata interface, typically exposed as a mapping-like object on the entity.
  - Example:
    - value = entity.metadata.get('priority')
    - value = entity.metadata['priority']  # may raise KeyError if not present
  - If the key does not exist and a default is desired, use get with a default: entity.metadata.get('priority', default_value)

- Enumerating metadata
  - You can inspect available metadata keys on an entity:
    - keys = list(entity.metadata.keys())
    - for key in keys: value = entity.metadata[key]

- Metadata schema awareness
  - Metadata keys may be defined by a server-side schema. When schema caching is enabled, keys and their expected types can be discovered more efficiently from the local cache.

Modifying metadata
- Writing metadata
  - Metadata is mutated via a mapping-like interface on the entity:
    - entity.metadata['priority'] = 'high'
    - entity.metadata['department'] = {'name': 'R&D', 'code': 4}  # JSON-serializable value
- Deleting metadata
  - Remove a metadata item:
    - del entity.metadata['priority']
- Persisting changes
  - Metadata changes are recorded as operations in the session (Session.recorded_operations). They will be transmitted to the server as part of the session’s lifecycle. Depending on your session configuration, changes may be automatically flushed or require an explicit commit/flush in your workflow.

Best practices
- Use stable, namespaced keys
  - Prefer prefixes (e.g., team:, priority:, reviewer:) to avoid collisions and aid discoverability.
- Keep metadata JSON-friendly
  - Store values that are JSON-serializable (strings, numbers, lists, dicts) to ensure smooth transport and storage.
- Be mindful of schema hints
  - If your server defines a metadata schema, rely on it for validation and consistency. Leverage schema caching to reduce round-trips.
- Performance considerations
  - Enable schema caching (schema_cache_path) to reduce repeated metadata-schema lookups.
  - Be aware that accessing metadata may trigger server calls if not already loaded, consistent with auto-populate behavior.

Examples

Reading a metadata value
- Retrieve a task’s priority, with a safe default if not set:
  - priority = getattr(task, 'metadata').get('priority', 'normal')
  - Alternatively: priority = task.metadata.get('priority', 'normal')

Setting metadata
- Annotate a task with a custom flag:
  - task.metadata['custom_flag'] = True
  - The change will be persisted when the session updates are committed.

Deleting metadata
- Remove an obsolete metadata key:
  - if 'obsolete_key' in task.metadata:
      del task.metadata['obsolete_key']

Notes
- Auto-population behavior: Accessing metadata may trigger a server fetch if it is not already loaded, depending on the session’s auto_populate setting.
- Persistence: Metadata changes flow through the session’s operation tracking. Ensure your workflow includes the necessary steps to flush or commit these operations to the server as required by your application.

Version/compatibility
- Metadata handling relies on the same session lifecycle and caching mechanisms as other entity attributes in the framework; behavior may vary with changes to session persistence and schema caching features.
