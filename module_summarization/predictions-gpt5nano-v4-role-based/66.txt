Pipelines Tutorial (using sklearn.pipeline.Pipeline)

Overview
--------
The Pipeline class provides a convenient way to chain together a sequence of data preprocessing
(transforms) steps and a final estimator. Each intermediate step must implement fit and transform,
while the final estimator must implement fit. A Pipeline can be tuned end-to-end by treating the
whole chain as a single estimator, enabling parameter searches across steps with a single
parameter grid.

Key ideas:
- Sequential composition: transforms are applied in order, then the final estimator is fit.
- Parameter routing: parameters are set using a double-underscore separator, e.g. svc__C=1.0.
- Caching: intermediate transform results can be cached to speed up repeated fits.
- Flexibility: steps can be replaced or removed by adjusting parameters (e.g., setting a step to
  'passthrough' or None).

Basic usage
-----------
Create a simple pipeline with a standard scaler and a classifier:

.. code-block:: python

   from sklearn.svm import SVC
   from sklearn.preprocessing import StandardScaler
   from sklearn.datasets import make_classification
   from sklearn.model_selection import train_test_split
   from sklearn.pipeline import Pipeline

   X, y = make_classification(random_state=0)
   X_train, X_test, y_train, y_test = train_test_split(X, y, random_state=0)

   pipe = Pipeline([('scaler', StandardScaler()), ('svc', SVC())])
   pipe.fit(X_train, y_train)
   score = pipe.score(X_test, y_test)
   print(score)

Parameter routing: setting parameters for steps
------------------------------------------------
You can tune parameters for any step by naming the step and using a double-underscore to
separate the step name from the parameter name.

Example: set the C parameter of the SVC step:

.. code-block:: python

   pipe.set_params(svc__C=10)
   pipe.fit(X_train, y_train)
   score = pipe.score(X_test, y_test)
   print(score)

Replacing or removing steps
---------------------------
A step can be replaced or removed by adjusting parameters:

- Replace a step by providing another estimator for that step name:
  e.g., replace the final estimator (svc) with a different classifier.

.. code-block:: python

   from sklearn.linear_model import LogisticRegression
   pipe.set_params(svc=LogisticRegression())

- Remove a transformer step by setting it to 'passthrough' (or None):
  This makes the pipeline skip that transform while preserving the rest of the chain.

.. code-block:: python

   pipe.set_params(scaler='passthrough')
   # Now the pipeline will pass data directly to the final estimator

Caching and memory management
-----------------------------
Pipeline supports caching of fitted transformers via the memory argument. If a string is provided,
it is treated as a path to a directory used for caching. When caching is enabled, transformers are
 cloned before fitting to avoid side effects, and only non-final steps are cached.

.. code-block:: python

   pipe = Pipeline([('scaler', StandardScaler()), ('svc', SVC())], memory='/tmp/pipeline_cache')
   pipe.fit(X_train, y_train)

Note: The last step (the final estimator) is never cached, even if it is a transformer.

Output configuration and set_output
-----------------------------------
The set_output method can configure how transform outputs are produced, such as the default
array-like output or a pandas DataFrame. This affects the transform and fit_transform paths.

.. code-block:: python

   pipe.set_output(transform='pandas')
   X_out = pipe.transform(X)

Accessing steps and parameters
------------------------------
- Access steps by name via named_steps (a Bunch-like object), e.g., pipe.named_steps.scaler.
- Get a full parameter map with get_params(deep=True).

.. code-block:: python

   params = pipe.get_params(deep=True)
   print(list(params.keys())[:5])

Advanced usage patterns
-----------------------
- Grid search or randomized search across the pipeline: Combine with
  cross-validation utilities to optimize parameters across multiple steps in a single search.
  Parameters across steps are addressed using the '__' syntax, as shown above.
- Extendability: The Pipeline is a _BaseComposition, so it can be composed with other estimators
  via utilities like FeatureUnion or make_pipeline/make_union for more complex
  architectures.

Common pitfalls and tips
-------------------------
- Ensure intermediate steps implement fit and transform; the final estimator must implement fit.
- When caching, be mindful that the final estimator is not cached and that cloning may affect
  inspector access to the original step instances.
- Use the __ syntax for nested parameter tuning rather than manually re-wiring the pipeline.

Related utilities
-----------------
- FeatureUnion: combine features from multiple transformer outputs in parallel.
- make_pipeline / make_union: convenience constructors for common pipeline patterns.
These are part of the same compose module and can simplify pipeline construction.

API highlights (high level)
---------------------------
- Pipeline(steps, memory=None, verbose=False)
  - steps: list of (name, transformer/estimator) tuples; last step must be an estimator.
  - memory: caching configuration (None or path or memory-like object).
  - verbose: print timing information for each step if True.

- set_output(transform=None)
  - Configure output type for transform/fit_transform. Options include "default" and "pandas".

- get_params(deep=True)
  - Retrieve constructor parameters and nested estimator parameters, enabling parameter grids for search.

- named_steps
  - Access individual steps by name (read-only dictionary-like container).

Examples in practice
--------------------
- A typical workflow is to assemble preprocessing and modeling steps, fit on training data, and evaluate on test data. Then, use a GridSearchCV or RandomizedSearchCV over parameters of various steps to optimize the whole pipeline.

.. note::

   The Pipeline class is designed to enable cross-validated parameter tuning across multiple steps
   while keeping a clean, auditable workflow. It also helps prevent data leakage by ensuring
   that all preprocessing is performed within each cross-validation fold.
