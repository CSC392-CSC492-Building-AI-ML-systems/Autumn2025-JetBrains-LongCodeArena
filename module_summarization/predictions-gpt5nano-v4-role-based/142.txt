Delayed API
===========

Overview
--------
The dask.delayed interface provides a lightweight, lazy API to build and manipulate Dask task graphs. It exposes:

- Delayed objects: lightweight wrappers around a portion of a Dask graph that represent a computation to be performed.
- delayed: a factory/decorator used to create Delayed objects from Python values or functions.
- Internal helpers (for advanced users): utilities to normalize Python objects into Dask tasks and graphs (unpack_collections, finalize, and a deprecated to_task_dask).

Key concepts
------------
- Lazy evaluation: computations are described as tasks and only executed when a scheduler computes the result.
- Graph construction: operations on Delayed objects (e.g., arithmetic, function calls) produce new Delayed objects that depend on prior keys.
- Normalization: inputs containing Delayed objects, Python literals, and common containers are normalized into a task description and a merged Dask graph via internal helpers.

Creating delayed objects
------------------------
- delayed(value, name=None) creates a Delayed object that represents a computation described by value. If a name is provided, it can be used as the key in the underlying graph.

- The delayed interface also supports using delayed as a decorator to lift functions into Delayed computations.

- Example:
  - a = delayed(1, "a")
  - b = delayed(2, "b")
  - You can combine Delayed objects with Python operators or function calls to build more complex graphs.

Working with delayed results
---------------------------
- Delayed objects participate in the same Python semantics as native values in expressions, enabling you to build graphs naturally via operations like addition, multiplication, or calling functions.

- The computed results are realized when you invoke a scheduler (e.g., compute()) on the resulting Delayed graph (or via dask.compute on a collection of Delayed objects).

Internal helpers for advanced usage
-----------------------------------
Note: These helpers are primarily for advanced users and internal use, not required for everyday Delayed usage.

- Unpack_collections(expr)
  - Purpose: Normalize a Python object containing Delayed objects and/or nested containers into a single task and a merged collection of sub-graphs.
  - Behavior:
    - Replaces Delayed objects with their keys.
    - Converts literals to scheduler-friendly forms.
    - Extracts graphs from all enclosed values, returning a pair (task, collections).
  - Examples
    - a = delayed(1, "a")
      b = delayed(2, "b")
      task, collections = unpack_collections([a, b, 3])
      task -> ['a', 'b', 3]
      collections -> (Delayed('a'), Delayed('b'))
    - task, collections = unpack_collections({a: 1, b: 2})
      task -> (<class 'dict'>, [['a', 1], ['b', 2]])
      collections -> (Delayed('a'), Delayed('b'))

- finalize(collection)
  - Purpose: Create a finalizing task for a Dask collection (e.g., after a sub-graph has been computed).
  - Returns a Delayed corresponding to applying the finalize operation to the collection's graph keys.
  - Example: finalize(expr) yields a Delayed that, when computed, produces the finalized result of the collection.

- to_task_dask(expr)
  - Deprecated: This function was used to normalize an object into a task and a merged Dask graph.
  - Behavior mirrors unpack_collections but warns that it is deprecated in favor of unpack_collections.
  - If you encounter this function, migrate to unpack_collections.

Code examples (from the API)
----------------------------
- Basic creation and normalization
  - Example:
    >>> import dask
    >>> a = delayed(1, 'a')
    >>> b = delayed(2, 'b')
    >>> task, collections = unpack_collections([a, b, 3])
    >>> task
    ['a', 'b', 3]
    >>> collections
    (Delayed('a'), Delayed('b'))

    >>> task, collections = unpack_collections({a: 1, b: 2})
    >>> task
    (<class 'dict'>, [['a', 1], ['b', 2]])
    >>> collections
    (Delayed('a'), Delayed('b'))

- Deprecated path
  - Example:
    >>> task, dasks = to_task_dask([a, b, 3])
    >>> task
    ['a', 'b', 3]
    >>> dict(dasks)
    {'a': 1, 'b': 2}
  - Note: Use unpack_collections instead.

API reference (surface)
-----------------------
- delayed (factory / decorator)
  - Create Delayed objects from values or functions; support explicit naming of keys.
  - Enables lazy construction of computation graphs via operator overloading and function calls.

- Delayed
  - Represents a single node (or a compound node via operators) in a Dask graph.
  - Supports lazy evaluation, composition, and integration with Dask schedulers.

- Helper functions (advanced)
  - unpack_collections(expr): Normalize and merge sub-graphs from nested Python objects containing Delayed objects.
  - finalize(collection): Create a finalizing Delayed for a Dask collection.
  - to_task_dask(expr): Deprecated alias of unpack_collections with a deprecation warning.

Important notes
---------------
- Unpack_collections and finalize are intended for internal use, but are useful for understanding how Delayed objects are normalized and merged into graphs.
- The primary user-facing API is the delayed factory/decorator and the Delayed objects created from it.
- The to_task_dask function is deprecated; prefer unpack_collections for normalization.

Examples recap
--------------
- Constructing and combining delays
  - a = delayed(1, "a")
  - b = delayed(2, "b")
  - c = a + b  # creates a new Delayed representing the sum, with dependencies on a and b

- Normalizing structures
  - task, collections = unpack_collections([a, b, 3])
  - task -> ['a', 'b', 3]
  - collections -> (Delayed('a'), Delayed('b'))

- Finalizing a collection
  - finalized = finalize(some_dask_collection)
  - finalized.compute() will yield the finalized result after dependencies are resolved

This documentation focuses on the dask.delayed interface: how to create and compose delayed computations, how normalization of inputs works under the hood, and how to use the provided helpers (noting deprecation of to_task_dask) to understand and troubleshoot the delayed graph construction.
