Compatibility Policy

Overview
This document defines the compatibility commitments for the npyrandom integration within the numpy.random package, as implemented by the Meson build configuration shown in the code. It covers public Python API stability, binary/ABI compatibility, build and packaging expectations, platform considerations, and testing and release practices.

Scope
- Public Python API: The numpy.random extensions and modules exposed by this build (e.g., _bounded_integers, _mt19937, _philox, _pcg64, _sfc64, bit_generator, mtrand, and related test and example resources).
- Binary/ABI: The compiled extension modules and the static library npyrandom that they link against.
- Build and packaging: The Meson-based build, generated interface files, and installed artifacts under numpy/random (and numpy/random/tests, numpy/random/tests/data).
- Platform considerations: Special handling for platform-specific builds (e.g., Windows/Cygwin symbol export behavior).

Public API compatibility
- Stability principle: Public Python-level APIs and extension module entry points are expected to remain stable within a major release series.
- Deprecations: When APIs are slated for removal or replacement, they are deprecated in advance in a backwards-compatible manner and remain available for the duration of the current major/minor release cycle. Users should migrate to the recommended alternatives before removals occur in a future major release.
- Additions: Non-breaking additions (new modules, new functions, or new optional parameters with safe defaults) may be introduced without breaking existing code.
- Documentation: All public API changes are to be reflected in release notes and compatibility documentation.

Binary/ABI compatibility
- C API usage: The code uses the NumPy C-API with NPY_NO_DEPRECATED_API=0, allowing access to deprecated APIs to maximize compatibility with older NumPy versions. This aligns with maintaining broad compatibility across a range of NumPy releases, while still supporting newer functionality where feasible.
- Extension modules: The Cython-generated Python extensions are built to link with the static library npyrandom and, per module, with additional libraries as needed. Extensions are installed into numpy/random and are expected to work with the corresponding Python interpreter and NumPy binary.
- Platform specifics: On Windows/Cygwin targets, symbol export behavior is adjusted (e.g., -Wl,--export-all-symbols) to ensure binary compatibility and proper symbol resolution in those environments.

Build, packaging, and distribution compatibility
- Build system: Meson-based build (as shown) produces a static library npyrandom and a set of Python extension modules under numpy/random. Artifacts install to numpy/random (and its subpaths) to preserve a consistent layout across environments.
- Generated interfaces: The build generates and uses _bounded_integers.pxd via a templated, custom_target-based process to ensure that interface definitions stay in sync with the corresponding Python and C sources.
- Resource packaging: Tests, test data, and examples are installed under numpy/random/tests and numpy/random/tests/data, ensuring reproducible compatibility checks across environments.
- Licensing: The software is dual-licensed under the University of Illinois/NCSA Open Source License (NC...), and compatibility documentation should reflect licensing expectations for downstream integrations.

Platform considerations
- Cygwin/Windows: Special linker flags are applied to ensure symbol visibility and compatibility on Windows-like environments. This helps maintain ABI compatibility when distributing prebuilt binaries or building in cygwin-like environments.

Deprecation and removal policy (high-level)
- Deprecations should be announced in advance (e.g., in release notes) and implemented with a clear migration path.
- Removals of public APIs or breaking changes occur in major releases, with an accompanying migration guide and sufficient transition time.

Testing and validation
- The repository includes a comprehensive test suite installed under numpy/random/tests, along with test data under numpy/random/tests/data. Compatibility is validated by running these tests against supported Python/NumPy configurations to ensure the public API and binary interfaces behave consistently.

Migration guidance (high-level)
- When upgrading to a new major version, consult release notes for any deprecations or removals.
- If relying on internal/private modules (e.g., orthogonal or distribution-specific components), validate that there are public equivalents or update code to use supported API surfaces.

Contributing to compatibility
- Report any ABI/API incompatibilities or packaging issues via the project’s issue tracker.
- When introducing changes, include a compatibility note in the changelog and update any relevant documentation (including this compatibility policy) as needed.

Notes
- This policy reflects the build and packaging practices demonstrated in the Meson configuration (static library npyrandom, cython extension modules, template-generated pxds, and installed test/data resources) and the platform-specific accommodations (e.g., Cygwin symbol export).
