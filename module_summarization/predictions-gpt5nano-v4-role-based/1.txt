Using custom attributes
=======================

Overview
--------
Custom attributes extend entities with user-defined metadata that goes beyond the native fields of the entity. They are defined on the server (per entity type) and can be read or updated from client code through the ftrack API. This guide explains how to discover, read and write custom attribute values from Python code using the ftrack_api.

Key concepts
------------
- Custom attribute: a user-defined attribute attached to one or more entity types (e.g., Task, Shot). Each custom attribute has a unique key, a human-friendly name, a value type (string, integer, float, boolean, date/time, selection, etc.), and optional constraints (required, default, allowed values, etc.).
- Attribute key: the identifier used to reference a custom attribute in code. In client usage, custom attributes are accessed via fields named following the pattern: custom_attribute_<attribute_key>.
- Entity: the ftrack object to which custom attribute values are attached (e.g., Task, Shot). The value of a custom attribute is read from or written to the corresponding field on the entity.

Discovering custom attributes
-----------------------------
To work with custom attributes, you first need to discover which custom attributes exist for a given entity type and what their metadata (name, type, constraints) is. This is typically done via the ftrack API’s attribute-related surface.

- Retrieve the set of CustomAttribute definitions for an entity type (server-defined schema).
- Inspect each CustomAttribute to determine its key, display name, type, and constraints.
- Use the attribute key to read or write values on entities.

Example (illustrative; adjust to your API usage patterns)
.. code-block:: python

    # Pseudo-example: obtain custom attribute definitions for Task
    # and print their keys and names
    for ca in session.query(
        'CustomAttribute',
        where='scope_type.name = "Task"'
    ):
        print(ca.key, ca.name, ca.type)

Reading custom attribute values
-------------------------------
Custom attribute values are exposed on entities in a dictionary-like fashion using a field named after the attribute key.

- Access pattern: entity['custom_attribute_<attribute_key>']
- Return value: the value stored for that custom attribute, converted to a Python type when possible
- If the attribute has no value set on the entity, the read may return None

Example
.. code-block:: python

    task = session.find_one('Task', {'id': '<task-id>'})
    estimated_hours = task.get('custom_attribute_estimated_hours')
    release_ready   = task.get('custom_attribute_release_ready')

Writing/updating custom attribute values
---------------------------------------
You can assign values to custom attributes on an entity just like standard dictionary fields.

- Access pattern: entity['custom_attribute_<attribute_key>'] = <value>
- Persisting changes: use the standard entity update/save flow provided by the SDK (e.g., via the session’s update/commit mechanism or your typical update flow)

Example
.. code-block:: python

    task = session.find_one('Task', {'id': '<task-id>'})
    task['custom_attribute_release_ready'] = True
    task['custom_attribute_estimated_hours'] = 12.5
    # Persist changes using your normal update flow
    task.update()  # or session.commit(), depending on your SDK usage

Notes on typing and validation
------------------------------
- The value type should match the custom attribute’s declared type (string, integer, float, boolean, date/time, or selection).
- If a value is set that does not conform to the attribute’s type or constraints, the server-side validation will typically reject the update and raise an error.
- Some attributes may be single-valued, while others may support multiple selections (arrays/lists). The exact semantics are defined by the server-side CustomAttribute.

Best practices
--------------
- Always discover and review the metadata (name, key, type, constraints) for the custom attributes you plan to read or write.
- Prefer using the attribute key (custom_attribute_<key>) rather than hard-coding names, to remain robust against display-name changes.
- When writing values, validate basic types client-side if possible to reduce round-trips and server validation errors.
- Consider caching frequently read custom attribute metadata to minimize repeated schema lookups.

Error handling
--------------
- Attempting to read or write a non-existent custom attribute key will typically yield a missing-value result (None) or raise an attribute-not-found error, depending on the SDK version.
- Type mismatches or violations of server-enforced constraints will raise server-side exceptions when the update is persisted.

Troubleshooting
---------------
- If you don’t see a custom attribute you expect, confirm:
  - The attribute is defined for the entity type you’re working with.
  - You’re using the correct attribute key (not the display name).
  - Your user has permission to read/write the attribute.
- If updates fail with validation messages, verify the value type and any constraints (e.g., allowed values, required, max/min).

API references
--------------
- ftrack_api.attribute: API surface for custom attributes (definitions and metadata).
- ftrack_api.entity: Entity access patterns for dynamic fields like custom_attribute_<key>.
- Session/connection: Persisting changes with your normal update/commit flow.

  - task['custom_attribute_<key>'] = <value>
  - Persist via the standard update/commit flow.
