Using lists

Overview
The codebase demonstrates several common patterns for using Python lists as multi-value inputs and simple collections. The most explicit example in the shown code is the handling of plugin search paths, which are expected to be provided as a list of strings. Lists are also used to represent collections of pending operations and other multi-item configurations in typical usage patterns.

Key concepts

- Multi-value configuration via lists
  - plugin_paths: A list of directory paths to search for plugins. This is the primary explicit usage of a list in the shown code.
  - If plugin_paths is not provided, the system falls back to an environment variable (FTRACK_EVENT_PLUGIN_PATH), which may contribute multiple paths when populated.

- Collections of pending items
  - recorded_operations: A collection that holds operations to be processed. It represents a list-like container of items awaiting execution.
  - In practice, this pattern is used to accumulate multiple items (e.g., operations) before processing them in a batch.

- General guidance
  - When a parameter or API expects multiple values, consider passing a Python list (or constructing a list) to provide those values in a predictable order.
  - Lists can be easily combined, extended, or sliced using standard Python techniques (concatenation, list comprehensions, itertools, etc.) to compose the final configuration or batch to be processed.

Practical examples

1) Passing multiple plugin directories
- Purpose: Allow the session to discover plugins from several locations.
- Example:
  session = Session(
      server_url="http://ftrack.local",
      api_key="your_api_key",
      api_user="your_username",
      plugin_paths=["/opt/ftrack/plugins", "~/my_plugins"]
  )
- Notes:
  - plugin_paths is a list of strings.
  - If you don’t supply plugin_paths, the system reads FTRACK_EVENT_PLUGIN_PATH from the environment.

2) Extending plugin search paths (conceptual pattern)
- If you need to combine a default set of paths with user-provided extra paths, you can construct a single list before passing it to the constructor:
  default_paths = ["/usr/local/ftrack/plugins"]
  extra_paths = ["/home/user/custom_plugins"]
  session = Session(
      server_url="http://ftrack.local",
      api_key="your_api_key",
      api_user="your_username",
      plugin_paths=default_paths + extra_paths
  )
- This demonstrates how Python lists can be composed to form a final configuration.

3) Working with a collection of pending operations
- Purpose: Accumulate operations to be executed in a batch.
- Conceptual pattern (based on recorded_operations):
  - recorded_operations acts as a list-like container for operations.
  - You append or add operations to this collection and then process them in a batch (the exact API for the collection is not shown in the snippet).
- Example (conceptual):
  op1 = ftrack_api.operation.Operation(...)

  session = Session(...)
  session.recorded_operations.append(op1)
  # Later, the batch is executed or transmitted to the server
- Note:
  - The exact methods depend on the implementation of ftrack_api.operation.Operations, but the concept is that of a list-like collection for batched work.

Best practices

- Prefer explicit lists for multi-value inputs to keep configuration clear and deterministic.
- When constructing lists for configuration (like plugin_paths), consider composing them from defaults and user-provided values to maintain predictable behavior.
- Be mindful of mutability:
  - If you reuse or modify lists across contexts, ensure you understand whether you’re mutating a shared object or working with a freshly constructed list.

References in code

- plugin_paths parameter in Session.__init__: describes a list of paths to search for plugins and the default behavior if not provided.
  - docstring excerpt hint: "*plugin_paths* should be a list of paths to search for plugins. If not specified, default to environment variable FTRACK_EVENT_PLUGIN_PATH."
- recorded_operations in Session.__init__: a collection intended to hold pending operations, illustrating the list-like use of batching.
