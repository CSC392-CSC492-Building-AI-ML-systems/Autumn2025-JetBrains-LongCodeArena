Calculator Interface
====================

Overview
--------
The Calculator Interface provides a lightweight, self-contained CLI to discover which parameters are used across available models. It introspects model metadata to produce a parameter-to-model mapping and supports optional filtering by parameter kind. This utility is implemented in the calculator-facing module that exposes two core functions (find_pars and list_pars) and a command-line interface (main).

Public API
----------
- find_pars(kind=None) -> dict
  - Purpose: Build a reference table mapping each parameter name to the list of models that use it.
  - Return: 
    - A dictionary where keys are parameter names (str) and values are lists of model names (list[str]).
  - Filtering: If kind is provided, only parameters whose type matches kind are included. If kind is None, all parameters are included.
  - Notes: Iterates over all models returned by list_models(), loads each model’s info via load_model_info(name), and collects kernel_parameters.

- list_pars(names_only=True, kind=None) -> None
  - Purpose: Print the parameter-to-model mapping.
  - Behavior:
    - If names_only is True (default), prints a sorted, columnized list of parameter names only.
    - If names_only is False, prints one line per parameter in the form: "parameter: model1, model2, ...".
  - Filtering: Delegates to find_pars(kind) to respect the kind filter.
  - Output formatting: Uses columnize(...) to format a single-column list for compact display.

CLI and Usage
-------------
The module can be invoked as a standalone script:

- Basic usage (print only parameter names):
  - python -m sasmodels.list_pars

- Verbose usage (print parameter names along with the models that use them):
  - python -m sasmodels.list_pars -v
  - In verbose mode, each line shows "parameter: model1, model2, ..."

- Filter by parameter kind:
  - python -m sasmodels.list_pars volume
  - python -m sasmodels.list_pars orientation
  - python -m sasmodels.list_pars sld
  - python -m sasmodels.list_pars none
  - python -m sasmodels.list_pars any
  - Note: The implementation treats 'any' and 'none' as None, effectively disabling filtering (i.e., no kind constraint).

Implementation details (how it works)
-----------------------------------
- list_models(): Retrieves all model names available in the SASModels repository.
- load_model_info(name): Loads metadata for a given model, exposing model_info.parameters.kernel_parameters.
- Each parameter p in model_info.parameters.kernel_parameters has at least:
  - p.name: the parameter’s name (str)
  - p.type: the parameter’s kind/type (e.g., 'volume', 'orientation', 'sld')
- The core data structure produced is partable: dict[str, list[str]] mapping parameter name to the list of models that use it.
- The columnize helper (from .compare) is used to render a clean single-column list when names_only is requested.

Data Model and Semantics
------------------------
- Model metadata is accessed via load_model_info(name).
- kernel_parameters represents the set of parameters that affect the model’s kernel computation.
- Each parameter object has:
  - name: unique identifier within the parameter space of kernel parameters
  - type: categorical descriptor used for filtering (e.g., volume, orientation, sld)
- The interface is intended for discovery and auditing of parameter usage across models, not for numerical calculation.

Examples
--------
- List all parameter names used by any model (compact, single column):
  - python -m sasmodels.list_pars

- List parameter names with model usage (verbose):
  - python -m sasmodels.list_pars -v

- List parameters filtered by kind (e.g., volume) without showing models:
  - python -m sasmodels.list_pars volume

- List parameters filtered by kind with model usage:
  - python -m sasmodels.list_pars -v volume

Design notes
------------
- The interface relies on existing core introspection utilities (list_models, load_model_info) to remain decoupled from specific model implementations.
- Filtering by kind is optional and uses the parameter’s type attribute.
- The verbose option is designed to aid debugging and parameter-tracing across models.
