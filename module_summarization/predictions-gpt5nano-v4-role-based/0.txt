Working with assignments and allocations
======================================

Overview
--------

This document describes how to work with assignments and allocations using the ftrack Python API, as exposed by the Session class in the provided code. It explains the concepts of assignments (ownership/responsibility of an entity) and allocations (resource or time allocations) at a high level and how they are typically manipulated through a session, including how changes are staged as pending operations before being sent to the server.

Key concepts from the code
--------------------------

- Session: The primary context for interacting with an ftrack server. It manages authentication, lazy population of data, caching, and the lifecycle of operations performed during a session.
- SessionAuthentication: Attaches ftrack authentication information to every HTTP request (api_key and api_user).
- Pending operations: The session keeps a collection of pending operations (session.recorded_operations) that represent changes to be applied to the server. You can control whether changes are recorded in real time by toggling session.record_operations.
- Auto-population: If enabled, accessing entity attributes will fetch data from the server on demand.
- Event hub: The session can connect to an event hub to publish/subscribe to events beyond the local process.

Working with assignments
------------------------

Assignments represent who is responsible for or associated with a particular entity (e.g., a task, asset, or shot). The API supports manipulating assignments through standard entity operations and via the session’s operation-tracking mechanism.

Typical workflow:
- Retrieve the target entity.
- Inspect or modify its assignment-related attributes or related collections.
- Changes are staged as operations in session.recorded_operations (pending commit to the server).

Notes:
- The exact attributes or collection names used to represent assignments depend on the specific entity type (Task, Asset, Shot, etc.) and the version of the API you are using. The codebase provides a generic Session and operation-tracking mechanism rather than hard-coded attribute names for assignments.
- You can disable or enable batching of changes by toggling session.record_operations:
  - sess.record_operations = False  # Stop recording new operations
  - sess.record_operations = True   # Resume recording

Example (conceptual)
- Fetch an entity and inspect or modify its assignment-related data.
- Add, remove, or change the assignee via the standard entity API or through operations collected in session.recorded_operations.

Working with allocations
------------------------

Allocations refer to time, capacity, or resource allocations assigned to entities (e.g., how much time is allocated to a task). Like assignments, allocations are managed via the API by manipulating entity data and/or by recording operations in the session.

Typical workflow:
- Retrieve the target entity.
- Inspect its allocation information (time ranges, assigned user, capacity, etc.).
- Modify allocations (add/remove time blocks, change allocation amounts, reassign capacity).
- Changes are staged in session.recorded_operations and later synchronized with the server.

Notes:
- Allocation-related attributes and methods are entity-specific. The API uses a general operation-tracking mechanism rather than requiring a single, universal allocation API surface.

Example (conceptual)
- Fetch an entity and read its allocations.
- Add or adjust allocations by updating the relevant allocation-related attributes or by creating allocation operations that are captured by session.recorded_operations.

Batching and committing changes
------------------------------

- Pending operations are collected in:
  - session.recorded_operations (an instance of ftrack_api.operation.Operations)
- The session maintains a flag:
  - session.record_operations (default is True)
  - Setting it to False stops new changes from being recorded into the pending operations collection.
- To apply assignments and allocations to the server, you rely on the API’s standard commit/apply mechanism (which processes the queued operations). The code base indicates an operation-based flow, but the exact commit mechanism is part of the broader ftrack API and may be provided by the session’s methods in the complete library.

Example (conceptual)
- Disable batching, perform local changes, then re-enable batching and flush:
  - sess.record_operations = False
  - Perform local, non-queued changes
  - sess.record_operations = True
  - Call the API’s commit/apply mechanism to push queued operations to the server (refer to the complete ftrack API for the exact method).

Practical considerations
------------------------

- Authentication: Use Session and SessionAuthentication to securely connect to the server with an API key and user.
- Environment defaults: The code reads server URL and credentials from environment variables (FTRACK_SERVER, FTRACK_API_KEY, FTRACK_API_USER) if not provided directly.
- Auto-population: Enable auto_populate for convenient lazy loading of assignment/allocation data as you access it.
- Caching: Schema and data caching can improve performance; configure schema_cache_path if you rely on it.
- Event hub: If you need non-local event publishing/subscription, enable auto_connect_event_hub and ensure the hub is connected.

Error handling and debugging
----------------------------

- If required connection details are missing, the Session constructor raises a TypeError with guidance on how to provide server_url, api_key, and api_user (or via environment variables).
- When working with assignments and allocations, failures will raise standard API exceptions as defined in the ftrack_api.exception hierarchy. Use try/except blocks around operations and check error messages for guidance.

API references (from the codebase)
----------------------------------

- Session
  - Manages server connection, authentication, caching, and operation batching.
  - Attributes used for assignments/allocations workflow:
    - session.recorded_operations: collection of pending operations to apply to the server.
    - session.record_operations: flag to enable/disable recording new operations.
    - auto_populate: whether to lazily fetch data from the server when attributes are accessed.
  - Initialization parameters include server_url, api_key, api_user, auto_populate, plugin_paths, cache, cache_key_maker, auto_connect_event_hub, schema_cache_path, plugin_arguments.

- SessionAuthentication
  - Attaches ftrack-api-key and ftrack-user headers to requests.

- ftrack_api.operation.Operations
  - Represents pending operations (including those related to assignments and allocations) that will be synchronized with the server.

Best practices
--------------

- Use session.record_operations = True (default) to batch changes for efficiency; disable only when you need immediate, local-only changes.
- Rely on auto_populate to minimize upfront data loading; it will fetch necessary data on demand.
- When dealing with assignments and allocations, prefer updating entity attributes through the standard API where possible, and rely on the session’s operation tracking for batching and consistency.
- Inspect session.recorded_operations if you need to audit or inspect what will be sent to the server in the next commit.
