Schema generation
===================

Overview
--------
This module provides helpers to generate and optionally apply the SQL schema for a given asynchronous database client. It separates the concerns of producing the SQL string and executing it against the database.

Core components
--------------
- get_schema_sql(client, safe) -> str
  - Returns the SQL that would create the current schema for the provided client.
  - The optional safe flag controls existence checks in the generated SQL:
    - safe = True: CREATE TABLE IF NOT EXISTS semantics (avoid errors if tables already exist).
    - safe = False: may attempt to create tables unconditionally, depending on the generator.
  - Implementation detail: obtains a schema generator via client.schema_generator(client) and calls generator.get_create_schema_sql(safe).

- generate_schema_for_client(client, safe) -> None
  - Generates and immediately applies the SQL schema to the database.
  - Steps:
    1) Obtain the generator with client.schema_generator(client).
    2) Retrieve the SQL via get_schema_sql(client, safe).
    3) Log the SQL at debug level.
    4) If the SQL is non-empty, execute it with generator.generate_from_string(schema) (awaited).

Key concepts
------------
- BaseDBAsyncClient: The client type expected by these helpers. The actual client provides a schema_generator(client) method returning a generator object.
- Schema generator object:
  - get_create_schema_sql(safe) -> str: Produces the SQL string to create the schema.
  - generate_from_string(schema) -> awaitable: Executes the provided SQL against the database.
- Logging: The module uses a logger named "tortoise" to debug-print the generated SQL.

Usage examples
--------------

1) Inspect the SQL without applying
.. code-block:: python

   sql = get_schema_sql(client, safe=True)
   print(sql)

2) Generate and apply the schema to the database
.. code-block:: python

   await generate_schema_for_client(client, safe=True)

Notes on behavior
-----------------
- The safe flag (safe) governs whether the generated SQL uses IF NOT EXISTS semantics to prevent errors when the schema already exists.
- If the generated SQL is empty, no action is performed.
- The actual application of SQL is performed by the generator via generate_from_string(schema); any exceptions raised will propagate to the caller.

Implementation highlights
-------------------------
- Runtime imports are minimized by using TYPE_CHECKING for the BaseDBAsyncClient type hinting.
- The code path for generating the SQL is centralized in the generator object, enabling customization per backend/client.
- Pragmas:
  - TYPE_CHECKING: pragma: nocoverage for the import block.
  - In get_schema_sql, the empty-schema branch is marked as pragma: nobranch to simplify coverage analysis.
