Async plans
=============

This module provides asynchronous plan helpers for performing batched readings and coordinated scans with multiple detectors and motion variables. It integrates with Bluesky preprocessors and plan patterns to create generator-based plans that can be run in an asynchronous I/O context.

Overview
--------

- The primary entry points are count and list_scan, which construct well-defined Bluesky plans with metadata that describe the plan configuration.
- count(detectors, num, delay, per_shot, md) returns a generator plan that records a number of readings from the provided detectors, optionally spacing readings with a delay and allowing a per-shot hook to customize actions between readings.
- list_scan(detectors, *args, per_step, md) scans one or more motors over specified point lists in an inner product (Cartesian) manner, combining detectors with the generated motor positions in an efficient, extensible way.
- rel_list_scan is declared but not implemented (TODO); it is intended to scan relative to the current position.

Key integration points
----------------------

- Stage and run decorators: Plans are wrapped with bpp.stage_decorator(detectors) and bpp.run_decorator(md=_md) to ensure detectors are staged and plan metadata is attached.
- Plan patterns: Uses plan_patterns (inner_list_product) to compose complex multi-motor, multi-point scans from simple inputs.
- Detectors and metadata:
  - Detectors are expected to be readable objects (e.g., Ophyd objects). Their names are recorded in metadata.
  - The _md dictionary captures plan name, plan arguments, detector/motor information, and hints (dimensions).
- Partition utility: The code uses partition to group pairs of (motor, positions) from the *args input, enabling consistent handling of multiple motors with independent position lists.

Detailed API

count(detectors, num=1, delay=None, *, per_shot=None, md=None)
--------------------------------------------------------------

Take one or more readings from detectors.

Parameters
- detectors: list
  List of readable detector objects (e.g., Ophyd detectors).
- num: int, optional
  Number of readings to take. If None, capture data indefinitely (until canceled).
- delay: iterable or scalar, optional
  Time delay in seconds between successive readings. If iterable, it must have at least num - 1 entries, otherwise a ValueError is raised during iteration.
- per_shot: callable, optional
  Hook for customizing the action of the inner loop (messages per step). Expected signature:
    def f(detectors: Iterable[OphydObj]) -> Generator[Msg]: ...
  If None, a default one-shot hook (bps.one_shot) is used.
- md: dict, optional
  Metadata dictionary to augment plan metadata.

Returns
- A generator plan: yield from inner_count(), which yields messages produced by the per_shot hook (if provided) and coordinates detector reads.

Notes
- If num is None, num_intervals is set to None (unbounded).
- _md includes:
  - detectors: list of detector names
  - num_points: num
  - num_intervals: num - 1 (if num is not None)
  - plan_args: representation of detectors and num
  - plan_name: 'count'
  - hints: with default dimensions set to time as primary
- The per_shot hook is invoked for every step (inner loop) via partial(per_shot, detectors).

Examples
- Basic usage with a single detector:
  yield from count([det], num=5, delay=0.5)

- Unbounded reading until cancellation:
  yield from count([det1, det2], num=None)

- Using a custom per_shot hook:
  def my_per_shot(detectors):
      yield Msg('read', detectors=[d.name for d in detectors])
  yield from count([det], num=10, per_shot=my_per_shot)

md is optional and can include additional plan metadata, e.g., hints for dimensions, etc.

list_scan(detectors, *args, per_step=None, md=None)
-----------------------------------------------

Scan over one or more motors with one point list per motor, performing an inner product of positions.

Parameters
- detectors: list
  List of detectors to read at each point in the scan.
- *args: pairs of (motor, [point1, point2, ...])
  For N motors, provide N pairs. Motors can be any settable object (e.g., Ophyd motors, temperature controllers).
- per_step: callable, optional
  Hook for customizing the action of the inner loop (messages per step). Expected signature:
    f(detectors, motor, step) -> plan (a generator)
- md: dict, optional
  Metadata dictionary to augment plan metadata.

Returns
- A generator plan: yield from scan_nd(detectors, full_cycler, per_step=per_step, md=_md)

Notes
- The number of *args must be even (motor, positions) pairs. If not, a ValueError is raised.
- All position lists must be the same length. If not, a ValueError is raised with details.
- Detectors’ names are captured in metadata under 'detectors'.
- Motors’ names (from their objects) are captured under 'motors'.
- The number of points is inferred from the length of the position lists.
- The plan uses plan_patterns.inner_list_product(args) to generate the full cycler.
- Default hints are derived from motor hints: if any motors provide a fields list in hints, those fields are added as primary dimensions in the metadata.

Metadata and plan structure
---------------------------

- _md (metadata dict) contains:
  - detectors: [det.name for det in detectors]
  - motors: [motor names]
  - num_points: length
  - num_intervals: length - 1
  - plan_args: {'detectors': [repr(det) for det in detectors], 'args': md_args, 'per_step': repr(per_step)}
  - plan_name: 'list_scan'
  - plan_pattern: 'inner_list_product'
  - plan_pattern_module: plan_patterns.__name__
  - plan_pattern_args: {'args': md_args}
  - hints: starts with an optional dimensions specification (default_dimensions)
- Dimensions (hints.dimensions) are derived from motors’ hints.fields if provided; if present, they are used to set default primary axes.

Examples
- Simple two-motor scan:
  yield from list_scan([det], motor1, [0, 1, 2], motor2, [10, 20, 30])

- With per_step customization:
  def my_per_step(detectors, motor, step):
      # yield or perform actions for the current motor/step
      yield Msg('step', motor=motor.name, step=step)
  yield from list_scan([det], motor1, [0, 1, 2], motor2, [10, 20, 30], per_step=my_per_step)

Note: If multiple motors are provided, all position lists must be the same length to form a proper inner product.

rel_list_scan(detectors, *args, per_step=None, md=None)
------------------------------------------------------

Scan over one variable in steps relative to the current position.

Status: TODO (not implemented)

Parameters
- detectors: list
  Readable detectors to capture at each step.
- *args: motor, [points], ... 
  For each motor, provide a motor and a list of relative steps.
- per_step: callable, optional
  Hook for customizing action per step (same signature as in list_scan).
- md: dict, optional
  Metadata dictionary to augment plan metadata.

Returns
- Currently not implemented; this function is a placeholder for future relative scans.

Notes
- The intended behavior is to execute scans where each position is relative to the current motor position, rather than absolute coordinates.

What to know when using these plans
-----------------------------------

- Detectors must be readable objects compatible with Bluesky plans.
- The per_shot and per_step hooks enable custom messaging and side effects between steps, enabling rich logging or progress reporting.
- The md metadata is merged with any user-supplied metadata and is used to document the plan in dashboards and logs.
- plan_patterns and plan_pattern arguments allow sophisticated composition of scanning strategies; inner_list_product is used for multi-motor, multi-point scans.

Error handling
--------------

- In list_scan, a ValueError is raised if the total number of (motor, points) pairs is not even.
- A ValueError is raised if the lengths of the point lists across motors are not all equal.
- In count, the delay iterable must have enough entries to cover num - 1 steps; otherwise the behavior is determined by the underlying generator (ValueError during iteration).

Notes on architecture
---------------------

This module demonstrates a small, cohesive subset of Blender/Bluesky-style asynchronous plans:

- It relies on high-level plan composition primitives (inner_list_product) to build complex scans from simple inputs.
- It separates concerns: motion planning (motors/points) is independent of data acquisition (detectors), but wired together through a metadata-rich plan.
- It leverages the Bluesky staging and run decorators to ensure detectors are prepared and plan metadata is attached consistently.
- It provides hooks (per_shot/per_step) to customize per-step behavior without modifying the overall plan structure.
