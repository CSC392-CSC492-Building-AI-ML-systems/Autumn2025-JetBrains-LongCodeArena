# metrics.rst

Module overview
- The metrics utilities module provides input validation and dtype handling helpers used by pairwise distance metrics. It ensures that input arrays (X and Y) are properly formatted, typed, and shaped before distance computations are performed. It also supports precomputed distance matrices and sparse inputs with configurable validation options.

Public helpers
- _return_float_dtype(X, Y)
  - Purpose
    - Determine an appropriate floating-point dtype for X and Y, preferring float32 only when both inputs are float32; otherwise use a standard floating dtype (usually float64).
    - Normalize input types so downstream code can assume numpy arrays or sparse matrices as appropriate.
  - Behavior
    - If X or Y are not numpy arrays, they are converted to numpy arrays (unless they are sparse).
    - If Y is None, Y_dtype is taken from X. If Y is provided, its dtype is used (converting to numpy array if needed).
    - Returns:
      - X: possibly converted input
      - Y: possibly converted input (or the original X if Y is None)
      - dtype: the chosen floating dtype (np.float32 if both inputs are float32; otherwise a general float dtype)
  - Typical usage
    - Used internally to decide the numeric dtype when validating pairwise inputs.

- check_pairwise_arrays(X, Y, *, precomputed=False, dtype=None, accept_sparse="csr", force_all_finite=True, copy=False)
  - Purpose
    - Validate and sanitize inputs for pairwise distance computations.
    - Ensure that X and Y are conformant and safe to use for distance calculations.
  - Behavior
    - Determines dtype using _return_float_dtype if dtype is not provided.
    - If Y is X or Y is None, both X and Y are validated via check_array (producing safe, finite arrays) and Y may be a reference to X (no copy).
    - If Y is provided and distinct from X, both X and Y are validated separately.
    - If precomputed is True, checks that X.shape[1] == Y.shape[0] (i.e., X is a (n_queries, n_indexed) matrix and Y is a (n_indexed, n_indexed?) reference in distance computations) and validates shape compatibility accordingly.
    - If precomputed is False, checks that X.shape[1] == Y.shape[1], ensuring the feature dimensions match.
    - Sparse handling
      - accept_sparse controls allowed sparse formats (e.g., 'csr', 'csc', etc.). If inputs are sparse but not in an allowed format, they are converted to the first allowed format.
    - force_all_finite
      - Controls whether infinite or NaN values are allowed. Accepted values:
        - True: require all values to be finite
        - False: allow NaN and Inf
        - 'allow-nan': allow NaN (and pd.NA converted to NaN), but do not allow infinite values
    - copy
      - If True, a forced copy may occur during conversion; otherwise, in-place conversion may occur.
    - Returns
      - safe_X: array-like, shape (n_samples_X, n_features)
        - X guaranteed as a numpy array (or suitable sparse matrix, depending on context)
      - safe_Y: array-like, shape (n_samples_Y, n_features)
        - Y guaranteed as a numpy array (or pointer to X if Y was None)
  - Practical notes
    - This function is a core helper used by various pairwise distance implementations to ensure consistent input handling and error messages.

- check_paired_arrays(X, Y)
  - Purpose
    - Validate inputs for paired distance metrics (where two distinct inputs X and Y are involved).
  - Behavior
    - Ensures both X and Y are array-like with compatible shapes and numeric types.
    - Intended to be the first validation step for paired-distance computations to confirm correctness and safety of inputs.
  - Notes
    - The function’s full implementation details are not shown here, but its role is to prepare X and Y for subsequent distance computations by performing initial type and dimensional checks.

API reference (signatures)
- _return_float_dtype(X, Y)
  - Returns: (X, Y, dtype)
- check_pairwise_arrays(X, Y, precomputed=False, dtype=None, accept_sparse='csr', force_all_finite=True, copy=False)
  - Returns: (safe_X, safe_Y)
- check_paired_arrays(X, Y)
  - Returns: None (validation side effects; typically ensures X and Y are suitable for downstream use)

Usage examples
- Example 1: Basic validation for X and Y
  - Given two datasets X and Y, validate inputs for pairwise distance computation:
  - Code:
    - X = [[1.0, 2.0], [3.0, 4.0]]
    - Y = [[5.0, 6.0], [7.0, 8.0]]
    - safe_X, safe_Y = check_pairwise_arrays(X, Y)
  - Result: safe_X and safe_Y are numpy arrays ready for distance calculations.

- Example 2: Using precomputed distances
  - For a metric that accepts precomputed distances on X, provide precomputed=True and ensure shapes align with Y:
  - Code:
    - X = distance_matrix_of_queries  # shape (n_queries, n_indexed)
    - Y = index_set  # depends on metric
    - safe_X, safe_Y = check_pairwise_arrays(X, Y, precomputed=True)
  - Result: validated precomputed matrices with consistent dimensions.

- Example 3: Sparse input handling
  - If X is a sparse CSR matrix and Y is provided as dense or sparse in a compatible format:
  - Code:
    - from scipy.sparse import csr_matrix
    - X = csr_matrix([[0, 1], [2, 0]])
    - Y = csr_matrix([[1, 0], [0, 1]])
    - safe_X, safe_Y = check_pairwise_arrays(X, Y, accept_sparse='csr')
  - Result: inputs converted as needed and validated for finite values.

Notes and caveats
- Sparse formats
  - accept_sparse can be a string, boolean, or list/tuple of strings. If a sparse input is not in an allowed format, it will be converted to the first allowed format in the list.
- NaN and Inf handling
  - force_all_finite controls whether NaN/Inf are allowed. The option 'allow-nan' allows NaN values but forbids Inf values; pd.NA is converted to NaN.
- Data type inference
  - When dtype is None, the module uses _return_float_dtype to determine an appropriate floating type, typically preferring float32 only when both inputs are float32.
- In-place vs copy
  - The copy flag indicates whether a copy might be made during validation. If copy=False, some in-place conversions may occur to optimize performance.
- Interplay with higher-level distance functions
  - These helpers are designed to be used by pairwise distance implementations (e.g., Euclidean, Manhattan, cosine) to enforce consistent input validation, shape compatibility, and dtype handling across the library.
