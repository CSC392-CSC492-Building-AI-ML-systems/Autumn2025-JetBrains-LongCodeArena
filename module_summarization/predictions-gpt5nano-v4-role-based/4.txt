Sync LDAP users
================

Overview
--------

This guide describes how to implement and operate a workflow that synchronizes users from an LDAP directory into the ftrack user registry. The aim is to ensure that LDAP users (and their LDAP group memberships) are consistently reflected in ftrack, enabling unified authentication, permissions, and project access management.

Key concepts
- LDAP source of truth: Users and their group memberships are maintained in LDAP.
- ftrack registry: Users and their roles/permissions in ftrack are updated to reflect LDAP data.
- Mapping: A configurable mapping between LDAP groups and ftrack roles/groups determines what permissions a user receives in ftrack.
- Idempotent sync: The process creates or updates ftrack users based on LDAP data and can remove or disable users who are no longer present in LDAP, depending on configuration.

Prerequisites
-------------

- A running ftrack server and access to its API via a valid API key and user.
- A Python environment with the ftrack_api package installed.
- An LDAP server reachable from the environment running the sync process.
- An LDAP client library (e.g., ldap3 or python-ldap) available in the environment.
- A defined mapping between LDAP groups and ftrack roles/groups.
- Optional: a scheduling mechanism (cron, systemd timer, or equivalent) to run the sync periodically.

Configuration
-------------

Environment and connection details for both systems must be provided. Typical configuration items include:

- ftrack connection
  - server_url: The URL of the ftrack server (e.g., https://ftrack.example.com)
  - api_key: An API key with permissions to read and update users
  - api_user: The user associated with the API key

- LDAP connection
  - LDAP_URI: LDAP server address (e.g., ldaps://ldap.example.com:636)
  - LDAP_BASE_DN: Base distinguished name for user search (e.g., dc=example,dc=com)
  - LDAP_BIND_DN: Distinguished name used to bind to LDAP (e.g., cn=binduser,dc=example,dc=com)
  - LDAP_BIND_PASSWORD: Password for the bind DN
  - LDAP_USER_FILTER: Filter to locate user entries (e.g., (&(objectClass=person)(|(mail=*)(uid=*))))
  - LDAP_USER_ATTRIBUTES: List of LDAP attributes to read (e.g., uid, mail, displayName, memberOf)

- Mapping and synchronization options
  - LDAP_GROUP_TO_FTRACK_ROLE: Mapping from LDAP group DNs or common names to ftrack roles or groups
  - Sync_mode: e.g., allow_create=True/False, update_existing=True/False
  - Remove_missing: If True, disable or remove ftrack users not present in LDAP during the latest sync
  - Attribute mapping: How LDAP attributes map to ftrack user fields (e.g., login, email, name)

- Logging and error handling
  - Log file path and log level
  - Retry or backoff strategy for temporary LDAP or network failures

Workflow overview
-----------------

1. Establish authenticated sessions with both systems:
   - Connect to ftrack using the provided server URL, API key, and API user.
   - Bind to LDAP using LDAP_BIND_DN and LDAP_BIND_PASSWORD.

2. Retrieve LDAP users:
   - Search LDAP under LDAP_BASE_DN using LDAP_USER_FILTER.
   - For each user entry, collect required attributes (uid/login, mail/email, displayName/full name, groups).

3. Reconcile users in ftrack:
   - For each LDAP user, determine if a corresponding ftrack user exists (by login/email).
   - If the user does not exist and Sync_mode allows creation, create a new ftrack user with mapped attributes.
   - If the user exists, update attributes as needed (name, email, etc.).

4. Sync group memberships:
   - Read LDAP group memberships for the user.
   - Map LDAP groups to ftrack roles/groups using LDAP_GROUP_TO_FTRACK_ROLE.
   - Update the user’s group memberships in ftrack accordingly.

5. Optional cleanup:
   - If Remove_missing is enabled, identify ftrack users not present in the latest LDAP dump and disable or remove them per policy.

6. Logging and reporting:
   - Produce a summary of actions taken (created, updated, failed, removed) and any conflicts or warnings.

Practical considerations
------------------------

- Identity matching: Choose a reliable LDAP attribute for user identity (commonly uid or mail) to map to ftrack’s user login or email. Ensure uniqueness to avoid conflicts.
- Group mapping fidelity: LDAP group naming and nesting may require normalization. Consider handling nested groups if LDAP supports them.
- Permissions and governance: Align ftrack roles with your organization’s access control model. Ensure that removing a user from LDAP triggers an appropriate change in ftrack (e.g., deactivation, revocation of API access).
- Error handling: Implement robust retry logic for transient LDAP or network errors and maintain a clear audit trail of changes.
- Performance: For large LDAP directories, batch processing with paging and incremental sync windows can improve performance and reduce load.
- Security: Protect credentials (LDAP bind password, API keys) using secure storage (environment variables, secret management) and restrict the sync process to minimal required privileges.

Testing and debugging
---------------------

- Validate the LDAP connection independently before integrating with ftrack.
- Run with a dry-run mode to preview actions without applying changes to ftrack.
- Use a dedicated test LDAP namespace and test ftrack project/users to avoid impacting production data.
- Enable verbose logging to trace user matching, creation, and group mapping decisions.

Security and governance
-----------------------

- Limit the privileges of the LDAP bind account to only what is necessary for reading user data.
- Store and access credentials securely; avoid hard-coding secrets.
- Maintain an audit trail of changes to ftrack users and their permissions from LDAP synchronization.
