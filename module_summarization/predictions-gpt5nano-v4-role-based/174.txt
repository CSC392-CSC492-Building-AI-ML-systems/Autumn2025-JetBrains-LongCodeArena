Array API support
===================

Overview
--------

This document describes how the codebase interacts with the Array API ecosystem. The current implementation relies on NumPy (and SciPy where appropriate) as the primary array representation. There is no dedicated Array API abstraction layer in the core estimators, so Array API backends are not dispatched or swapped in automatically. Users working with Array API-compatible backends should convert their data to NumPy arrays before feeding them to estimators.

Scope of support
----------------

- Internal data representation: NumPy ndarrays (with optional SciPy sparse matrices for sparse inputs).
- Input acceptance: Array-like inputs are accepted and converted to NumPy arrays as needed by the estimators.
- Backends: No explicit Array API backend dispatch; the codebase targets NumPy/SciPy semantics.
- Tests and docs: Dependency constraints include specific versions of NumPy, SciPy, and related tools to ensure numerical behavior remains consistent with NumPy-based arrays.

Dependency and environment cues (relevant to Array API considerations)
--------------------------------------------------------------------

From the minimum dependency configuration:

- NumPy
  - PyPy: minimum version 1.19.2
  - Other interpreters: minimum version 1.17.3
- SciPy: minimum version 1.5.0
- Other utilities: joblib, threadpoolctl, Cython, pytest, etc. (documentation and tests ecosystems)

Note: These constraints ensure consistent numeric behavior across environments that use NumPy arrays as the core data type.

Array representations and data flow
-----------------------------------

- Primary array type: NumPy ndarrays
- Optional inputs: SciPy sparse matrices (when models or transformers support sparse data)
- Data validation path: The estimators rely on NumPy array semantics and associated utilities (e.g., check_random_state, clone, fit/predict interfaces) rather than an Array API dispatch mechanism.

What this means for Array API users
----------------------------------

- If you produce data via an Array API backend (e.g., backends that expose an Array API interface but can present NumPy-like arrays), you should convert to a standard NumPy ndarray before calling fit, predict, or transform.
- There is no automatic routing to alternative array backends; all computations flow through NumPy/SciPy paths.

Practical guidance for developers
---------------------------------

- Estimator inputs:
  - Accept array-like inputs and convert to NumPy arrays internally (as is standard in scikit-learn).
  - Avoid relying on backend-specific array methods; use NumPy-compatible operations.
- New estimators:
  - Ensure inputs can be converted to NumPy arrays (np.asarray) and that numeric dtype semantics are preserved where relevant.
  - Do not assume the presence of a non-NumPy Array API at the core of the estimator.
- Multi-output estimators:
  - The codebase includes meta-estimators such as MultiOutputRegressor and MultiOutputClassifier, which internally clone and fit sub-estimators.
  - Example (conceptual):
    - Create a base estimator that adheres to fit/predict.
    - Wrap it with a meta-estimator:
      - mor = MultiOutputRegressor(base_estimator)
      - mor.fit(X, y)
    - Internally, sub-estimators are cloned and trained on each target dimension.

Code-derived examples
---------------------

- Fitting a multi-output estimator (typical usage pattern):

  .. code-block:: python

     from sklearn.linear_model import LinearRegression
     from sklearn.multioutput import MultiOutputRegressor
     import numpy as np

     X = np.random.randn(100, 3)
     y = np.random.randn(100, 2)

     base = LinearRegression()
     mor = MultiOutputRegressor(base)
     mor.fit(X, y)

- Internal fit helpers (illustrative, not part of the public API):

  .. code-block:: python

     def _fit_estimator(estimator, X, y, sample_weight=None, **fit_params):
         estimator = clone(estimator)
         if sample_weight is not None:
             estimator.fit(X, y, sample_weight=sample_weight, **fit_params)
         else:
             estimator.fit(X, y, **fit_params)
         return estimator

     def _partial_fit_estimator(
         estimator, X, y, classes=None, partial_fit_params=None, first_time=True
     ):
         partial_fit_params = {} if partial_fit_params is None else partial_fit_params
         if first_time:
             estimator = clone(estimator)

         if classes is not None:
             estimator.partial_fit(X, y, classes=classes, **partial_fit_params)
         else:
             estimator.partial_fit(X, y, **partial_fit_params)
         return estimator

Limitations and current status
------------------------------

- No native Array API dispatch layer is present in the core estimators.
- Data must be provided as NumPy arrays (or convertible to NumPy arrays) for consistent behavior.
- If integrating with separate Array API backends, implement an explicit conversion step to NumPy before using estimators.

Future considerations
---------------------

- Potential addition of a compatibility layer to directly accept Array API arrays and dispatch to NumPy under the hood.
- Tests validating cross-backend interoperability could be introduced if such a layer is pursued.

Reference
---------

- Core array handling relies on NumPy (and SciPy for sparse support) with minimum version constraints that ensure consistent numerical behavior across environments.
