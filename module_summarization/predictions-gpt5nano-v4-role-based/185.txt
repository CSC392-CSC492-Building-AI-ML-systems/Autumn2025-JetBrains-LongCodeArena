Grid Search: Tuning the hyper-parameters of an estimator

Overview
- Grid search is a systematic method to explore a predefined set of hyper-parameter values for an estimator.
- It uses cross-validated performance to identify the best combination of hyper-parameters.
- The result is a trained estimator (or a pipeline) configured with the best parameters, ready for evaluation or deployment.

Key concepts
- Estimator: The machine learning model or pipeline whose hyper-parameters you want to tune.
- Parameter grid (param_grid): A dictionary (or list of dictionaries) specifying parameter names and the values to try. Parameter names use the double-underscore syntax to address parameters of nested estimators (e.g., pipeline steps or enclosers).
- Cross-validation (cv): The strategy for splitting the data into training and validation sets during evaluation of each parameter combination.
- Scoring: The metric used to compare parameter combinations. Can be any supported scorer (e.g., accuracy, precision, recall, AUC). You can pass a string or a callable to GridSearchCV.
- Refit: Whether the best performing parameter set is re-fitted on the full training data after grid search.

API overview
- GridSearchCV(estimator, param_grid, scoring=None, cv=None, refit=True, n_jobs=None, verbose=0, return_train_score=False, **kwargs)
  - estimator: The model or pipeline to tune.
  - param_grid: Dict[str, List[Any]] or List[Dict[str, List[Any]]] defining parameter values to explore.
  - scoring: Scorer callable or string to evaluate performance.
  - cv: Cross-validation strategy or number of folds.
  - refit: If True, refit the best parameters on the full training data; if a string, refit on the estimator with that parameter.
  - n_jobs: Number of parallel jobs (parallelize over parameter combinations when possible).
  - verbose: Verbosity level for progress messages.
  - return_train_score: If True, include train scores in the results.
- cv_results_: A dict-like object containing detailed results for each parameter combination, including:
  - mean_test_score, std_test_score
  - mean_fit_time, std_fit_time
  - params (the parameter settings for each combination)
  - Optional: mean_train_score, std_train_score (when return_train_score is True)

Typical workflow
1) Define your estimator (or a Pipeline if you need preprocessing steps).
2) Define param_grid with the hyper-parameters you want to tune.
3) Create and run GridSearchCV.
4) Inspect results to understand which parameters performed best.
5) Use best_estimator_ (or the fitted grid search object) for predictions on new data, or continue training on the full dataset if desired.

Practical guidelines for designing param grids
- Start coarse: choose a small, diverse set of values that covers the likely useful ranges.
- Use log-spaced values for scale-sensitive hyper-parameters (e.g., C, gamma in SVMs; n_estimators, max_features in tree methods).
- For pipelines, prefix nested estimator parameters with the step name and double underscore, e.g., "clf__C" or "preprocessor__normalize".
- If the grid grows too large, consider RandomizedSearchCV for a fixed number of random samples, which can be more efficient and still effective.

Example: Tuning a Support Vector Classifier in a Pipeline
- Objective: Tune C and gamma for an RBF kernel SVM within a standard plug-in pipeline.

Code snippet:
- from sklearn.pipeline import Pipeline
- from sklearn.preprocessing import StandardScaler
- from sklearn.svm import SVC
- from sklearn.model_selection import GridSearchCV

pipeline = Pipeline([
    ('scaler', StandardScaler()),
    ('svc', SVC(kernel='rbf'))
])

param_grid = {
    'svc__C': [0.5, 1.0, 2.0],
    'svc__gamma': [0.01, 0.1, 1.0]
}

grid = GridSearchCV(
    estimator=pipeline,
    param_grid=param_grid,
    scoring='accuracy',
    cv=5,
    n_jobs=-1,
    verbose=1
)

grid.fit(X, y)

Print and interpret:
- best_params_ -> {'svc__C': 1.0, 'svc__gamma': 0.1}
- best_score_ -> mean cross-validated accuracy with best params
- best_estimator_ -> a Pipeline configured with the best parameters, ready for predictions
- cv_results_ -> detailed performance per parameter combination

Example: Tuning a RandomForestClassifier
- Objective: Find the best combination of n_estimators and max_depth.

Code snippet:
- from sklearn.ensemble import RandomForestClassifier
- from sklearn.model_selection import GridSearchCV

rf = RandomForestClassifier(random_state=42)

param_grid = {
    'n_estimators': [100, 200],
    'max_depth': [None, 10, 20]
}

grid = GridSearchCV(
    estimator=rf,
    param_grid=param_grid,
    scoring='accuracy',
    cv=4,
    n_jobs=-1
)

grid.fit(X, y)

Results:
- best_params_ example: {'n_estimators': 200, 'max_depth': 20}
- best_score_: best cross-validated accuracy
- best_estimator_: RandomForestClassifier configured with these hyper-parameters
-cv_results_: matrix of results across all combinations

Inspecting results and diagnostics
- Access best results:
  - best_params_, best_score_, best_estimator_
- Review cross-validation performance for all combinations:
  - grid.cv_results_
  - key fields commonly used: mean_test_score, std_test_score, params
- Visualizing grid search outcomes:
  - Create custom plots (e.g., heatmaps) using grid.cv_results_ to understand how different hyper-parameters affect performance.
  Note: The plotting utilities in this codebase focus on learning curves visualization. For grid search, you typically plot metrics from cv_results_ or export to visualization tools.

Best practices and caveats
- Avoid leakage: Ensure cross-validation splits are created from training data only; do not use test data in parameter tuning.
- Reproducibility: Set random_state for estimators that support it; use a fixed cv splitter when possible.
- Compute and memory costs: Grid search is combinatorial; large grids can be expensive. Use a smaller grid, random sampling, or parallelization (n_jobs).
- Early stopping and resource constraints: For very large models or datasets, consider constrained search or early-stopping-like heuristics.
- Nested cross-validation: For unbiased performance estimates, consider nested CV if you must report the performance of your tuned model on held-out data.

Advanced topics
- Refitting strategy: The refit parameter determines whether GridSearchCV refits on the entire training data using the best parameters. You can set refit to a boolean or to a scoring metric to select the best estimator by a specific criterion.
- Scoring variants: You can optimize for a specific metric by passing a scoring callable; you can also optimize multiple metrics by combining them or using a custom scorer.
- Pipeline-aware parameter naming: When using pipelines, parameter names use the format "<step_name>__<param_name>" to address inner estimator parameters.

Recommended workflow summary
- Define estimator or pipeline
- Create a carefully designed param_grid
- Run GridSearchCV with a reasonable cv and scoring
- Use results to select and deploy the best estimator
- Optionally assess the chosen model with learning curves and external validation datasets to confirm generalization

Notes
- GridSearchCV is robust and integrates seamlessly with pipelines and cross-validation strategies.
- For extremely large hyper-parameter spaces, consider RandomizedSearchCV or successive halving variants to balance thoroughness and computational cost.
