Tensor Attributes
=================

Overview
--------

Tensor attributes are MLIR attributes that describe, carry, or configure tensor data within the OneFlow MLIR dialect. They are used to encode constant tensor values, shapes, and element types, and to annotate operations with tensor-related metadata. The OneFlow build system exposes and integrates these attributes as part of the MLIR dialect libraries.

Key Concepts
------------

- TensorType and ShapedType
  - Describe the shape and element type of a tensor (e.g., rank, dimensions, element type like i32, f64).
- DenseElementsAttr
  - A concrete, dense representation of a tensor’s values. Used to embed literal tensor data as an attribute.
- DenseIntElementsAttr / DenseFPElementsAttr
  - Typed specializations of DenseElementsAttr for integer or floating-point data.
- ArrayAttr
  - A sequence of attributes, often used to represent ancillary tensor-related metadata or a list of tensor attributes.
- Attribute binding on ops
  - Tensor attributes can be attached to operations (operands, results, or attributes) to convey constant tensors, shape hints, or other tensor-specific metadata.

Tensor Attributes in OneFlow MLIR Dialect
------------------------------------------

- Build and availability
  - Tensor attributes live within the OneFlow MLIR dialect libraries.
  - The CMake configuration provides helpers to build these components:
    - oneflow_add_mlir_library: for MLIR libraries
    - oneflow_add_mlir_dialect_library: for dialect libraries
  - With MLIR enabled (WITH_MLIR), the build includes the dialact library, headers, and runtime integration as part of the OneFlow project structure.

- Typical usage patterns
  - Embedding literal tensor data in an attribute:
    - Use DenseElementsAttr (e.g., DenseIntElementsAttr) to store the values of a tensor.
  - Attaching tensor metadata to ops:
    - Ops can carry attributes such as a weights tensor, bias tensor, or other tensor-related data via attributes.

Practical Examples
------------------

MLIR textual representation

- A dense tensor value as an attribute:
  - dense< [1, 2, 3, 4] > : tensor<4xi32>

- Attaching a tensor attribute to an op (example syntax; op names may vary in the OneFlow dialect):
  - %Y = oneflow.dialect.some_op %X, attributes { "weights" = dense< [1, 2, 3, 4] > : tensor<4xi32> }

C++ API usage

- Creating a DenseElementsAttr (example for a 1D int32 tensor of length 4):
  - #include "mlir/IR/BuiltinTypes.h"
  - #include "mlir/IR/BuiltinAttributes.h"
  - mlir::MLIRContext context;
  - auto tensorType = mlir::RankedTensorType::get({4}, context.getI32Type());
  - auto denseAttr = mlir::DenseIntElementsAttr::get(tensorType, {1, 2, 3, 4});
  - // Attach to an operation state as an attribute:
  - mlir::OperationState state(location, "some_op");
  - state.addAttribute("weights", denseAttr);

Notes and Best Practices
------------------------

- Prefer DenseElementsAttr for constant/tensor literal data to ensure compact and efficient representation.
- Use DenseIntElementsAttr or DenseFPElementsAttr when the element type is known to be integer or floating-point, respectively.
- When constructing attributes in C++, ensure the tensor type matches the actual data layout you intend to encode (shape and element type must align with the provided values).
- Attributes for tensors are typically combined with the dialect's operations to drive kernel selection, shape inference, and constant propagation.

Relation to the Codebase
------------------------

- The tensor attribute infrastructure is part of the OneFlow MLIR dialect integration.
- The CMake setup provides:
  - oneflow_add_mlir_library: to publish core MLIR libraries used by the dialect.
  - oneflow_add_mlir_dialect_library: to publish dialect-specific components (including tensor-related attributes).
- WITH_MLIR gating ensures the necessary directories (include, lib) and tests related to tensor attributes are built when MLIR support is enabled.
