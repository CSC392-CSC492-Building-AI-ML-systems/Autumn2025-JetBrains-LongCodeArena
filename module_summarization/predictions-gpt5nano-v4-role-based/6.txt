Working with User Security Roles
================================

Overview
--------

In the ftrack API, security roles define what actions a user (identified by api_user) is permitted to perform on resources. All API requests are authenticated and authorized using the session’s identity; permissions are enforced on the server side according to the user’s roles.

Authentication context
----------------------

Requests are authenticated by attaching two headers to every HTTP request:

- ftrack-api-key: the API key for the user
- ftrack-user: the username associated with the API key

These headers are injected automatically by the SessionAuthentication component when requests are sent.

Configuring a session for a specific user
-----------------------------------------

To operate under a particular security context, create a Session with the following required parameters:

- server_url: the ftrack server URL (including port if needed)
- api_key: the API key for the acting user
- api_user: the username of the user to perform actions as

If any of these are omitted, the client will attempt to resolve them from environment variables, or fall back to the current OS user for api_user.

Environment variable resolution (order of precedence)
-----------------------------------------------------

- server_url: FTRACK_SERVER
- api_key: FTRACK_API_KEY, or FTRACK_APIKEY (backwards compatible)
- api_user: FTRACK_API_USER, or the current system user via getpass.getuser() as a fallback

Examples
--------

Explicit credentials:

.. code-block:: python

    from ftrack_api import Session

    session = Session(
        server_url='https://my.ftrack.server',
        api_key='0123456789abcdef',
        api_user='alice'
    )

Using environment variables:

.. code-block:: python

    # Environment variables:
    # export FTRACK_SERVER=https://my.ftrack.server
    # export FTRACK_API_KEY=0123456789abcdef
    # export FTRACK_API_USER=alice
    session = Session()

Behavior when credentials are missing
--------------------------------------

If required credentials cannot be determined, a TypeError is raised during Session construction, indicating which value is missing. Specifically:

- Missing server_url leads to a TypeError about the required server_url not being specified
- Missing api_key or api_user similarly raises a TypeError

Notes on security scope
-----------------------

- The security context (api_user and api_key) established at session creation determines the permissions for all subsequent operations performed through that session.
- To operate as a different user or with a different role, create a new Session instance with the appropriate api_user and api_key.

See also
--------

- SessionAuthentication: how authentication headers are attached to requests
- Session: core entry point for establishing a security context and interacting with the server

This documentation focuses on the mechanism by which user identity and security roles are applied in the client, enabling developers to work with the correct user security context when performing operations.
