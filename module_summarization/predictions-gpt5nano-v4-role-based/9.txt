Using timers
============

Overview
--------
The code in this module does not expose a dedicated timer API. Timers or scheduled tasks are not part of a dedicated “timer” component here. Instead, any timer-like or asynchronous behavior is achieved via Python’s standard threading and background execution. A notable example within this project is the event hub connection, which is performed in a background thread to avoid delaying session startup.

Key takeaways
- There is no built-in timer utility in this codebase.
- Background work, such as the event hub connection, uses a separate thread to stay non-blocking.
- If you need periodic or delayed work, you should use Python’s standard threading or scheduling tools (e.g., threading.Timer, sched, or external schedulers) in your own code around the library.

Background timers in ftrack_api
-------------------------------
- Session initialization may start an embedded event hub in a background thread when auto_connect_event_hub is True. This design improves startup time by not blocking while establishing a connection to the event server.
- Subscribing to events does not require the hub to be connected, but publishing to non-local events may require the hub to be connected (the actual connection happens asynchronously).

Example: background hub connection
-----------------------------------
This is conceptual and demonstrates how background work is used rather than providing a timer API.

.. code-block:: python

   from ftrack_api import Session

   # Background event hub connection is started if enabled
   session = Session(
       server_url='https://your-ftrack-server',
       api_key='your_api_key',
       api_user='your_username',
       auto_connect_event_hub=True  # hub connects in a background thread
   )

   # The session can be used to publish/subscribe without blocking startup

Using Python timers for periodic work
-------------------------------------
If you need to perform periodic tasks while using ftrack_api, you should use Python’s standard timer/scheduler facilities. The library itself does not provide a timer API, so you can compose periodic tasks around a Session object as needed.

Common patterns
- Periodic polling or refresh
- Delayed execution of a task after some event

Example: a simple repeating task with threading.Timer
---------------------------------------------------
.. code-block:: python

   import threading
   import time

   def periodic_task(session, interval):
       # Your logic here, e.g., refresh or fetch data
       print("Periodic task running with session:", session)
       # Schedule next run
       t = threading.Timer(interval, periodic_task, args=(session, interval))
       t.daemon = True
       t.start()
       return t

   # Example usage (adjust interval as needed)
   # session = Session(server_url='...', api_key='...', api_user='...')
   # t = periodic_task(session, 60)  # run every 60 seconds

Notes
- Timers created with threading.Timer are daemon threads; they won't prevent program exit.
- Keep a reference to Timer objects if you need to cancel them later (call cancel()).
- Ensure thread-safety when interacting with ftrack_api.Session or any shared resources from within timer callbacks.

Cleanup and shutdown
--------------------
- When shutting down an application, ensure that any active timer threads are canceled to avoid background work after exit.
- The library uses atexit in some contexts; if you create your own periodic timers, align their lifecycle with your application’s shutdown sequence.

Best practices
- Prefer using the library’s asynchronous capabilities (e.g., the background hub) for non-blocking behavior, but implement your own timers for domain-specific periodic tasks.
- Avoid heavy or blocking operations inside timer callbacks; offload work to worker threads or tasks.
- Document timer-driven behavior clearly in your integration code so future maintainers understand the timing guarantees and lifecycle.

Relation to the codebase
------------------------
- The Session class handles authentication, server interaction, caching, and optional background event hub connectivity.
- The presence of a background thread for the event hub is the primary “timer-like” behavior exposed by default in this codebase.
- No explicit timer API is defined in the provided excerpt; timers are to be implemented by users of this API with Python’s standard tools if periodic actions are required.
