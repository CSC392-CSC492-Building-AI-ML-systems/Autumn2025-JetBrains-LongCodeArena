w_states.rst

Overview
- The w_states command displays or updates basis (initial) states and target (recycling) states used by a WEST simulation.
- By default, states are displayed (dumped to files or stdout). Modes:
  - --show: display current basis and target states for the current iteration.
  - --replace: replace the current basis/target states for the next iteration.
  - --append: append new target states to the list for the next iteration (basis states should not be appended).
- Supports reading from and writing to state files:
  - Basis states: --bstate-file and --bstate
  - Target states: --tstate-file and --tstate

Command synopsis
- w_states [options]
- Default mode: show

Modes of operation
- show
  - Display current basis states and target states for the current WEST iteration.
  - If --bstate-file is provided, write basis states to that file; otherwise write to stdout.
  - If --tstate-file is provided, write target states to that file; otherwise write to stdout.
  - Output format is governed by BasisState.states_to_file and TargetState.states_to_file, with headers indicating the iteration.
- replace
  - Replace the current basis states and/or target states for the next iteration with those provided via files and/or command-line arguments.
  - Basis states:
    - Read from --bstate-file and/or --bstate (the latter can be specified multiple times).
    - If any basis states are provided, ensure their probabilities sum to unity (within a tolerance). If not, rescale all probabilities to sum to 1.
    - Assign progress coordinates for basis states, create the ibstate group for the next iteration, and report the basis states to the simulation manager.
  - Target states
    - (In the provided code snippet, target-state handling is not fully shown; the documented behavior expects similar processing for target states as for basis states—reading from --tstate-file and/or --tstate, assigning coordinates, and updating the next-iteration target-state list.)
- append
  - Append new target states to the existing list for the next iteration.
  - Basis states are not appended (as noted in the documentation string); to extend basis states, use --replace after editing a separate file.
  - Target states are appended in the order provided (via --tstate and/or --tstate-file) to the existing next-iteration target-state list.

Arguments and options
- --show
  - Action: store_const mode='show'
  - Description: Display current basis/target states (or dump to files).
- --append
  - Action: store_const mode='append'
  - Description: Append the given target states to the next-iteration target-state list.
- --replace
  - Action: store_const mode='replace'
  - Description: Replace current basis/target states with the supplied ones for the next iteration.
- --bstate-file BSTATE_FILE
  - Read or write basis state information (names, probabilities, and optional auxref) to/from BSTATE_FILE.
- --bstate
  - Append a basis state string to the list, format: label,probability[,auxref]
  - Can be specified multiple times; appended after any states read from --bstate-file.
- --tstate-file TSTATE_FILE
  - Read or write target state names and representative progress coordinates to/from TSTATE_FILE.
- --tstate
  - Append a target state string to the list, format: label,pcoord0[,pcoord1[,...]]
  - May be specified multiple times; appended after any states read from --tstate-file.

Format details and formats
- Basis state file format (bstates):
  - Each line typically contains: label,probability[,auxref]
  - Lines starting with '#' are comments (e.g., headers like “# Basis states for iteration N”)
  - Example line: S1,0.25
  - Example with auxref: S2,0.75,auxref_name
- Target state file format (tstates):
  - Each line typically contains: label,pcoord0[,pcoord1[,...]]
  - No explicit probability is stored for target states in this format; the line lists the label and one or more progress coordinates
  - Example line: T1,0.123,4.56
- Inline specifications (command line):
  - Basis: label,probability[,auxref]
  - Target: label,pcoord0[,pcoord1[,...]]

Implementation notes (behavioral details)
- Only the master process in the WEST parallel environment performs changes; workers act as pass-through.
- When in replace mode:
  - If any segments in the current iteration are complete, the new states apply to the next iteration (iteration number is incremented before applying the changes).
  - Basis states are validated for total probability sum and rescaled if needed to enforce unity.
  - Basis state probabilities are used to assign progress coordinates (via sim_manager.get_bstate_pcoords).
  - Basis states are written into the next iteration via data_manager.create_ibstate_group and reported to the simulation manager.
  - Target states handling is intended to mirror this process for the target-state list (the provided code snippet shows the basis-state handling path in detail; target-state handling is indicated by the surrounding documentation and argument structure but is not fully shown in the snippet).
- In show mode:
  - Basis states for the current iteration are retrieved and written to --bstate-file or stdout.
  - Target states for the current iteration are retrieved and written to --tstate-file or stdout.
- Basis state file note:
  - Appending basis states is discouraged; instead, read existing basis states with --bstate-file, edit the file to include additional basis states, and use --replace with the edited file to apply changes.
- Defaults:
  - If no mode is specified, w_states defaults to show.

Practical examples
- Show current basis and target states to stdout (default behavior):
  - w_states
- Show and write basis/target states to files:
  - w_states --show --bstate-file=bstates.txt --tstate-file=tstates.txt
- Replace basis and target states for the next iteration using files:
  - w_states --replace --bstate-file=bstates.txt --tstate-file=tstates.txt
- Append new target states for the next iteration:
  - w_states --append --tstate-file=new_tstates.txt
  - w_states --append --tstate=NewTarget1,0.25,1.0 --tstate=NewTarget2,0.75,2.0
- Add a new basis state (with a specific auxref) for the next iteration (using a file for the rest, and a single inline addition):
  - w_states --replace --bstate-file=bstates_existing.txt --bstate=ExtraState,0.1,auxrefX

Notes
- The w_states command is designed to operate primarily on the master process; worker processes do not apply state changes directly.
- The code enforces that basis-state probabilities sum to unity; if not, probabilities are rescaled uniformly.
- The documented formats for --bstate-file and --tstate-file align with the internal BasisState and TargetState representations and the associated states_to_file/state_from_file methods.
