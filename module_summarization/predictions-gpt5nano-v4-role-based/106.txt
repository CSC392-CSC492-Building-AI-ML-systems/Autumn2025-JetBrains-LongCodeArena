HyperExpand: Expanding Hypergeometric and Meijer G Functions into Named Special Functions

Overview
- This module constructs a Sphinx-friendly docstring at import time. Its sole purpose is to provide a readable, automatically generated set of LaTeX math blocks for the Sphinx documentation.
- It imports core utilities and builds a collection of known hypergeometric representations, translating them into named special-function forms.
- The generated documentation is exposed via the module’s __doc__ attribute, which Sphinx can render as math blocks.

Key components
- Dependencies
  - from ..core import Eq
  - from ..functions import hyper
  - from ..printing import latex
  - from .hyperexpand import FormulaCollection
- Docstring generation
  - The module creates a FormulaCollection (c = FormulaCollection()).
  - It iterates over c.formulae and, for each formula f, constructs an equation:
    obj = Eq(hyper(f.func.ap, f.func.bq, f.z), f.closed_form.rewrite('nonrepsmall'))
  - The resulting LaTeX is inserted into the doc with a math directive:
    doc += f'.. math::\n  {latex(obj)}\n'
  - Finally, __doc__ is set to the generated doc string.

Public API
- __doc__
  - A string containing a sequence of Sphinx math blocks (LaTeX) representing the known hypergeometric/Meijer G expansions.
  - This is the primary surface for documentation consumers; importing the module populates this string automatically.
- _mod1(x)
  - Internal helper used for modular arithmetic with a special handling for numbers vs symbolic expressions.
  - Purpose: compute x mod 1 with nuanced behavior for symbolic expressions (see the implementation for details).
- add_formulae(formulae)
  - Internal function that builds the knowledge base of formulae.
  - Uses an inner helper Matrix-based structure to store complex identities.
  - Adds base cases such as 0F0, 1F0, 2F1, and more, into the provided formulae collection.
- add(ap, bq, res)
  - Internal helper to register a formula of the form Hyper(ap, bq) = res.
  - Creates a Hyper_Function and appends a Formula to the formulae list.
- addb(ap, bq, B, C, M)
  - Internal helper to register a matrix-based formula for a hypergeometric identity.
  - Stores additional structural information (B, C, M) used for matching and instantiation of the identity.

What this code represents (high level)
- Expand Hypergeometric (and Meijer G) functions into named special functions.
- The approach relies on a curated collection of lookup tables for hypergeometric functions and their properties, enabling expansions into named functions.
- It references the Roach (1997) paper for the theoretical basis and is described to a degree in the Sphinx documentation.
- The internal commentary outlines a large ecosystem of shift operators (A_J, B_J, C_J, D_J), order reductions, reachability of index quadruples, and related concepts that drive the expansion strategy. The implementation itself uses hand-built routines to instantiate and match applicable formulas.

Algorithmic and design notes (high level)
- Bucketing and modular handling
  - _mod1(x) attempts to compute x mod 1 with careful handling of integers vs. symbolic parameters.
- Formula registration
  - add_formulae builds a knowledge base of standard hypergeometric representations and their “named” counterparts.
  - The knowledge base includes:
    - 0F0: exp(z)
    - 1F0: HyperRep_power1(-a, z)
    - 2F1: A matrix of representations with HyperRep_power2
    - Additional multi-parameter cases are prepared via addb (matrix cases and their polynomial coefficients)
- Documentation generation
  - For each formula, the left-hand side is the hypergeometric function with its ap, bq, and z, and the right-hand side is the rewritten closed form (nonrepsmall).
  - The LaTeX rendering of the equation is appended to the module’s __doc__ string as a Sphinx math block.

Practical usage and examples
- Import-time doc generation
  - When the module is imported, hyperexpand.rst’s corresponding module (or its documentation build) will include a generated __doc__ containing many math blocks.
  - Example (conceptual) content you might see in the generated docs:
    - 0F0 expansion
      .. math::
         e^z
      (represents hyper((), (), z) rewritten to exp(z))
    - 1F0 expansion
      .. math::
         HyperRep_power1(-a, z)
      (represents hyper(a; ; z) rewritten to HyperRep_power1(-a, z))
    - 2F1 expansion
      .. math::
         Hyper([a, a-1/2], [2a], z) = [HyperRep_power2(a, z),
                                      HyperRep_power2(a + 1/2, z)/2]
      (represents a 2x2 matrix identity with named representations)

Notes for maintainers
- The module is designed to be introspective for Sphinx; it relies on a pre-populated FormulaCollection and a consistent formatting pipeline via latex() to render math.
- The heavy lifting for expansion is performed by the underlying formula database and the rewrite mechanism (rewrite('nonrepsmall')), so changes to the algorithm should primarily occur in the data and the rewrite rules, with the documentation layer remaining intact.
