Library Conventions
===================

This document describes the conventions used in the library for code structure, documentation, and implementation.

Module Structure
----------------
Each module in the library follows a consistent structure:

1. **Module docstring**: A brief description of the module's purpose and list of routines
2. **Copyright and license information**: BSD 3-clause license with California Institute of Technology copyright
3. **Author and date information**: Includes author name, creation date, and version identifier
4. **Imports**: Relative imports for internal modules, absolute imports for external dependencies
5. **__all__ declaration**: Explicit list of public API functions
6. **Function definitions**: With comprehensive docstrings following numpy style

Documentation Standards
-----------------------
All functions include detailed docstrings with the following sections:

- **Parameters**: Type annotations and descriptions for all arguments
- **Returns**: Description and type of return value
- **Notes**: Additional implementation details and references
- **Examples**: Working code examples showing usage

Function names use snake_case convention. Public API functions are explicitly listed in __all__.

Error Handling
--------------
The library uses ValueError for invalid argument values and ControlNotImplemented for unimplemented features. Input validation is performed at function boundaries.

Configuration System
--------------------
Module-specific default parameters are defined in configuration dictionaries (e.g., _optimal_defaults) that can be modified by users.

Time Conventions
----------------
- Discrete time systems use sampling period Ts (float > 0)
- Frequency parameters use radians per second (rad/s)
- Time points are specified as 1D arrays

Constraint Specification
------------------------
Constraints follow SciPy optimize module conventions, using LinearConstraint and NonlinearConstraint objects with standard parameter passing.

Code Style
----------
- Class names use CamelCase convention
- Private class members use leading underscore
- Extensive use of type hints in documentation
- Consistent spacing around operators and after commas
