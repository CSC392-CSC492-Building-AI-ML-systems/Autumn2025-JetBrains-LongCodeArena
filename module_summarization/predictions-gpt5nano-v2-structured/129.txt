# Table Data

Table data refers to data organized in a tabular form, typically stored in FITS files as Binary Tables (BTABLE) or ASCII Tables, and accessed or manipulated in memory through numpy arrays or higher-level table abstractions (e.g., astropy.table.Table). This page describes the structure of table data, how to read and write it, and common practices for working with table columns, rows, and metadata.

## Table data model

- Tables consist of rows and columns.
- Each column has:
  - A name (TTYPE keyword in FITS headers).
  - A data type and shape (TFORM in FITS headers) that determine how data is stored on disk.
  - Optional metadata such as units (TUNIT), scale, and null/missing-value specifications (e.g., TNULL or fill values).
- Rows represent records; each row contains one value per column.
- Multi-dimensional columns are allowed, so a single column can hold arrays per row (e.g., an image per row or a vector per row).
- Metadata in the header describes the table format and per-column properties.

## Access patterns

- Reading from a FITS file:
  - Use astropy.io.fits to open the file and access the table data as a structured array.
  - Access columns by name (e.g., data['flux']).
  - Iterate rows or operate column-wise for efficiency.

- In-memory representations:
  - Data can be loaded into a numpy structured array or converted to an astropy.table.Table for richer, convenient manipulation.

- Writing:
  - Create a BinTableHDU (binary table) via from_columns or by constructing Columns, then add to an HDUList and write to disk.

## Working with columns

- Column names should be descriptive and follow naming conventions.
- Data types map to numpy dtypes; FITS TFORM codes describe the on-disk representation and must be compatible with the desired in-memory dtype.
- Units (TUNIT) can be attached to columns to preserve physical units via astropy.units.

## Handling missing data

- Integer columns commonly use TNULL to indicate missing values.
- Floating-point columns typically use NaN to represent missing data, but FITS supports explicit null values in some contexts.
- In-memory, missing data can be represented with masked arrays or numpy.nan as appropriate.

## Examples

- Example 1: Read a binary table from a FITS file and access a column
  ```python
  from astropy.io import fits

  with fits.open('sample.fits') as hdul:
      data = hdul[1].data  # first extension after the primary HDU
      # Access a column named 'time'
      times = data['time']
      # Iterate over rows
      for row in data:
          print(row['time'], row['flux'])
  ```

- Example 2: Create and write a simple binary table
  ```python
  import numpy as np
  from astropy.io import fits

  times = np.arange(5, dtype=np.float64)
  flux  = np.random.random(5).astype(np.float32)

  col_time = fits.Column(name='time', format='D', array=times)  # 8-byte float
  col_flux = fits.Column(name='flux', format='E', array=flux)   # 4-byte float

  hdu = fits.BinTableHDU.from_columns([col_time, col_flux])
  hdu.writeto('table.fits', overwrite=True)
  ```

- Example 3: Convert a FITS table to an in-memory table and back
  ```python
  from astropy.io import fits
  from astropy.table import Table

  with fits.open('table.fits') as hdul:
      data = hdul[1].data
      tbl = Table(data)          # convert to astropy.table.Table

  # Manipulate using Table methods
  tbl['flux'] = tbl['flux'] * 1.1

  # Convert back to a FITS-like structured array
  data_back = tbl.as_array()
  # (data_back can be used to create a new BinTableHDU if needed)
  ```

## Data types and conversion

- on-disk types (TFORM) map to numpy dtypes when loaded into memory.
- Common mappings:
  - Floating-point: float32, float64
  - Integers: int8, int16, int32, int64 (with appropriate on-disk codes)
  - Boolean, character strings, and multi-dimensional fields are supported via appropriate formats.
- When possible, preserve units and metadata during read/write operations to maintain scientific provenance.

## Units and metadata

- Use TUNIT for per-column units to capture physical quantities.
- Metadata in the FITS header (keywords like TTYPE, TFORM, TUNIT) describes the structure; astropy.io.fits handles parsing and loading into convenient Python objects.
- In-memory tables (e.g., astropy.table.Table) can attach units using astropy.units for column data.

## Multi-dimensional and nested data

- Columns may contain arrays (e.g., per-row vectors or images). Access with standard numpy indexing, e.g., data['column'][row_index] or data['column'][row_index, ...].
