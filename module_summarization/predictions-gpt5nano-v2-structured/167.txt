Library Conventions
===================

This document summarizes the conventions used throughout the library to ensure consistency, readability, and maintainability. All files in the library should adhere to these guidelines.

Introduction
------------
The library is designed with a focus on clear, documented public APIs, consistent naming, and reusable components. Conventions apply to code structure, documentation, testing, licensing, and contribution workflows.

Public API and Module Structure
-------------------------------
- Public API is defined via explicit exports in each module using __all__.
  - Example:
    __all__ = ['sample_system', 'c2d']
- Modules should expose a small, coherent surface area. Internal helpers may be private (leading underscore) but should not be part of the public API.
- Each Python module should have a module-level docstring describing its purpose and contents.
- Top-level packages should have an __init__.py that documents package exports and versioning.

Docstrings and Documentation Style
----------------------------------
- Docstrings should use the NumPy/SciPy style for consistency:
  - Short summary line describing the object.
  - Parameters, Returns, and Notes sections as appropriate.
  - Examples section with runnable Python code.
- Use explicit type hints in the docstring when helpful (and import-time types if available), while keeping runtime compatibility.
- For internal API notes, include a Notes or See Also section to aid discoverability.
- All public functions and classes must have docstrings describing purpose, inputs, outputs, and behavior.

Example: Typical function docstring (NumPy style)
- Example:
  .. code-block:: python

     def sample_system(sysc, Ts, method='zoh', alpha=None, prewarp_frequency=None):
         """
         Convert a continuous time system to discrete time by sampling.

         Parameters
         ----------
         sysc : LTI
             Continuous time system to be converted (StateSpace or TransferFunction).
         Ts : float
             Sampling period (positive).
         method : str
             Sampling method ('zoh', 'bilinear', etc.).
         alpha : float, optional
             Generalized bilinear transformation parameter (used with method='gbt').
         prewarp_frequency : float, optional
             Frequency for prewarping (used with method='bilinear').

         Returns
         -------
         sysd : LTI
             Discrete time system with sampling rate Ts.

         Notes
         -----
         See :meth:`StateSpace.sample` or :meth:`TransferFunction.sample` for details.

         Examples
         --------
         >>> sysc = TransferFunction([1], [1, 2, 1])
         >>> sysd = sample_system(sysc, 1, method='bilinear')
         """

- File-level docstrings should include purpose, usage context, and a minimal example when helpful.

Code Style and Formatting
-------------------------
- Adhere to PEP 8 for general style; prefer readability over cleverness.
- Line length: aim for 88 characters as a practical limit; wrap long lines gracefully.
- Naming conventions:
  - Functions and methods: lowercase_with_underscores
  - Classes: CamelCase
  - Constants: ALL_CAPS
- Imports should be grouped: standard library, third-party, local imports, each separated by blank lines.
- Use explicit exceptions for invalid inputs and provide informative error messages.
- Include a short header in each source file with copyright/license information (as shown in the example in the repository).

- The header should be preserved in all distributions and derivatives.

Testing and Quality Assurance
-----------------------------
- Tests should cover public APIs and critical internal helpers where applicable.
- Use pytest for unit tests; name test modules as test_*.py and include descriptive test names.
- Doctests and in-code examples are encouraged where helpful (via Examples in docstrings).
- Tests should be runnable with minimal external dependencies; mock where necessary.

Versioning and Release Management
--------------------------------
- Follow semantic versioning: MAJOR.MINOR.PATCH
  - BREAKING changes increment MAJOR
  - NEW features increment MINOR
  - Bug fixes and minor improvements increment PATCH
- Document API changes in a changelog with clear rationale.

Contribution and Maintenance
----------------------------
- Contributions should be submitted via pull requests with a clear description.
- Follow the repository’s contribution guidelines, including code style checks and tests.
- Update relevant docs and tests when changing public API or behavior.

Example Module Layout
---------------------
- Each module contains:
  - A module-level docstring
  - Public API exposed via __all__
  - Public classes and functions with well-documented docstrings
  - Internal helper functions prefixed with a single underscore when not part of the public API

Tree example:
  package/
  ├── __init__.py
  ├── core.py
  ├── dtime.py
  ├── optimal.py
  └── tests/
      ├── test_core.py
      └── test_dtime.py

Backward Compatibility
----------------------
- Maintain backward compatibility where feasible. If deprecations are necessary, provide a clear deprecation path and timeline.
- Document deprecated features in the release notes and update code paths gradually.
