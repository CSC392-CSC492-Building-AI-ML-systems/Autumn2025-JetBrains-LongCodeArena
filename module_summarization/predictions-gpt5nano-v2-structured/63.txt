# Commands Module Documentation

This module provides the core command implementations used by the environment for Alembic-style script management. It exposes three public operations:

- List available templates
- Initialize a new script environment
- Create a new revision file

The functions are designed to be used both from a higher-level CLI and programmatically.

## Public API

### list_templates(config: Config) -> None

List available templates that can be used to initialize a new script environment.

- Parameters
  - `config`: a Config object. Used to access template directory information and to print user-facing messages.

- Returns
  - `None`

- Behavior
  - Prints a header: "Available templates:"
  - Iterates over the directories in the configured template directory and prints each template name with its synopsis (read from the template's README).
  - After listing, prints guidance on using templates via the `init` command.

- Usage (example)
  - Programmatic:
    ```python
    from yourmodule import commands
    from yourmodule.util import Config  # adapt import as needed

    cfg = Config(...)
    commands.list_templates(cfg)
    ```
  - CLI usage is typically handled by the higher-level command interface; this function is the underlying implementation.

- Notes
  - Assumes each template directory contains a README with a synopsis line.
  - Demonstrates how templates are surfaced to end users.

### init(config: Config, directory: str, template: str = "generic", package: bool = False) -> None

Initialize a new scripts directory using a specified template.

- Parameters
  - `config`: a Config object.
  - `directory`: target directory path for the new scripts environment.
  - `template`: template name to use (default: `"generic"`).
  - `package`: if `True`, also create `__init__.py` files in the environment location as well as in `versions/`.

- Returns
  - `None`

- Behavior
  - Validates that the target directory does not exist or is empty.
  - Validates the template directory exists.
  - Creates the target directory and a `versions/` subdirectory if needed.
  - Copies template files into the target directory; for `alembic.ini.mako`, generates the actual `alembic.ini` (unless it already exists).
  - Optionally creates `__init__.py` files when `package=True`.
  - Prompts the user to edit configuration/connection/logging settings in the generated `alembic.ini` before proceeding.

- Usage (example)
  - Programmatic:
    ```python
    from yourmodule import commands
    cfg = Config(...)
    commands.init(cfg, directory="/path/to/new_env", template="generic", package=True)
    ```
  - CLI usage (conceptual):
    - `alembic init /path/to/new_env --template generic --package`

- Notes
  - If the target directory exists and is not empty, an error is raised.
  - If the chosen template does not exist, a `CommandError` is raised.

### revision(
  config: Config,
  message: Optional[str] = None,
  autogenerate: bool = False,
  sql: bool = False,
  head: str = "head",
  splice: bool = False,
  branch_label: Optional[str] = None,
  version_path: Optional[str] = None,
  rev_id: Optional[str] = None,
  depends_on: Optional[str] = None,
  process_revision_directives: Optional[ProcessRevisionDirectiveFn] = None,
) -> Union[Optional[Script], List[Optional[Script]]]

Create a new revision file.

- Parameters
  - `config`: a Config object.
  - `message`: optional message for the revision (the `-m` option).
  - `autogenerate`: if `True`, generate the revision script from the database state (the `--autogenerate` option).
  - `sql`: if `True`, dump the script to stdout as SQL (the `--sql` option).
  - `head`: the parent/head revision to build upon (the `--head` option).
  - `splice`: if `True`, create a new head for the revision (the `--splice` option).
  - `branch_label`: optional label for the branch (the `--branch-label` option).
  - `version_path`: optional version path (the `--version-path` option).
  - `rev_id`: optional explicit revision identifier (the `--rev-id` option).
  - `depends_on`: optional dependencies (the `--depends-on` option).
  - `process_revision_directives`: a callable to customize the generated revision structure programmatically (not exposed via CLI).

- Returns
  - `Union[Optional[Script], List[Optional[Script]]]` — a single `Script` or a list of `Script`s depending on how the revision creation is invoked.

- Behavior
  - Builds a `ScriptDirectory` from the configuration.
  - Creates a `RevisionContext` using the provided parameters.
  - Reads the `revision_environment` setting from the config and enables autogeneration when requested.
  - Validates incompatible options (e.g., `--autogenerate` with `--sql` raises an error).
  - Proceeds to generate and/or apply the revision(s) according to the provided arguments and the environment context.

- Usage (example)
  - Programmatic:
    ```python
    from yourmodule import commands
    cfg = Config(...)
    script = commands.revision(cfg, message="initial revision", autogenerate=True)
    ```
  - CLI usage (conceptual):
    - `alembic revision -m "initial revision" --autogenerate`
    - If you prefer SQL output: `alembic revision -m "initial" --autogenerate --sql`

- Notes
  - The function is designed for both CLI-driven and programmatic usage.
  - The exact revision generation process is delegated to the `RevisionContext` and the autogeneration machinery.

## How to Use These Commands

- Programmatic usage (example):
  - Initialize a new environment:
    ```python
    from yourmodule import commands
    from yourmodule.config import Config  # adapt to your project layout

    cfg = Config(...)
    commands.init(cfg, "/path/to/env", template="generic", package=True)
    ```
  - List available templates:
    ```python
    commands.list_templates(cfg)
    ```
  - Create a new revision:
    ```python
    commands.revision(cfg, message="start", autogenerate=True)
    ```

- CLI usage (conceptual, handled by the top-level CLI wiring):
  - Initialize:
    - alembic init /path/to/env --template generic --package
  - Revision:
    - alembic revision -m "start" --autogenerate

## Design Notes

- The module is designed to be used both from a command-line interface and programmatically within Python code.
- It relies on a Config object to determine template locations, the active environment, and messaging options.
- The autogeneration path is enabled when requested and when not in conflict with SQL output.

## Related Concepts

- Template management: templates are read from a configured template directory, each template providing a set of files to copy into a new environment.
- ScriptDirectory: the core abstraction used to manage script locations, versions, and associated metadata.
- RevisionContext: coordinates the revision creation process, including autogeneration and directive processing.
