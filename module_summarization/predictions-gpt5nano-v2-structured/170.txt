Handling Large Datasets
=======================

Overview
--------

Computing moment maps (zeroth, first, and higher-order moments) from large spectral cubes can easily exceed available memory if the entire dataset is loaded at once. This documentation describes strategies implemented in the moment computation utilities to handle large datasets efficiently, with a focus on memory usage, performance trade-offs, and practical usage.

Key strategies for moment computation
-------------------------------------

The moment computation utilities offer three primary processing strategies:

- Cubewise (load-all): Load the full data into memory and compute moments. Fastest for small to moderate datasets but memory-intensive.
- Slicewise: Process one 2D plane (slice) at a time along the chosen axis, accumulating results to form the moment map. Moderate memory usage; a good balance for medium-to-large datasets.
- Raywise: Process one ray (line along the collapsed axis for each pixel) at a time. Minimal memory usage; can be slower but scales well to very large datasets.

All strategies return a moment map with the same two spatial dimensions as the cube, i.e. the moment map shape equals the cube shape with the collapsed axis removed.

Core functionality
------------------

- moment_auto(cube, order, axis)
  Builds a moment map by selecting a processing strategy automatically based on the data and axis. The available strategies map to:
  - cube: full in-memory cubewise computation
  - slice: slicewise computation
  - ray: raywise computation

- moment_slicewise(cube, order, axis)
  Compute moments by processing one 2D plane at a time along the specified axis. Uses per-slice data loading and accumulation.

- moment_raywise(cube, order, axis)
  Compute moments by accumulating results one ray at a time. Minimizes peak memory usage.

- moment_cubewise(cube, order, axis)
  Compute moments using the entire data cube loaded into memory. Fastest when memory is not a constraint.

- _slice0(cube, axis), _slice1(cube, axis)
  Specialized implementations for the 0th and 1st moments when slicing along an axis.

- _moment_shp(cube, axis)
  Utility to determine the shape of the resulting moment map when collapsing along a given axis.

- _get_filled_data, _pix_size_slice, _pix_cen, _iter_rays, and masking behavior
  The underlying cube-handling methods used by the different strategies to fetch data, handle masks, and compute weights.

Usage examples
--------------

Basic usage with automatic strategy selection:

.. code-block:: python

   from spectral_cube import SpectralCube
   from spectral_cube.moments import moment_auto

   cube = SpectralCube.read("my_cube.fits")
   # Compute zeroth moment along the spectral axis (axis=0)
   mom0 = moment_auto(cube, order=0, axis=0)

   # Compute first moment along the spectral axis
   mom1 = moment_auto(cube, order=1, axis=0)

Explicitly selecting a strategy (for advanced control):

.. code-block:: python

   from spectral_cube.moments import moment_raywise

   mom0_ray = moment_raywise(cube, order=0, axis=0)

Notes
-----

- The moment maps have the shape of the cube with the collapsed axis removed. For a cube with shape (nz, ny, nx) and axis=0, the moment maps shape is (ny, nx).

- The code supports masked and partially filled data. For sliced computations, masks are respected to avoid bias from invalid data.

- For higher-order moments (order > 1), the computation may involve intermediate steps (e.g., First moment) to stabilize results and ensure proper normalization.

Memory and performance considerations
-------------------------------------

- Cubewise (cube strategy)
  - Pros: Fastest for small datasets; straightforward implementation.
  - Cons: Requires loading the entire dataset into memory, which may be prohibitive for large cubes.

- Slicewise (slice strategy)
  - Pros: Lower memory footprint than cubewise; suitable for moderately large datasets.
  - Cons: May be slower than cubewise due to repeated loading of planes and more complex accumulation.

- Raywise (ray strategy)
  - Pros: Minimal peak memory usage; scales to very large datasets.
  - Cons: Often the slowest approach due to per-ray overhead and more complex data access patterns.

Choosing a strategy
--------------------

If you are unsure which strategy to use, prefer moment_auto to let the system select an appropriate balance between speed and memory usage. Consider the following guidelines:

- If the cube is large and memory is limited, prefer slicewise or raywise.
- If you have ample memory and need speed, cubewise may be the best option.
- If you are working with masks or partially filled data, slicewise or raywise can better accommodate data integrity checks on the fly.

Implementation notes
--------------------

- The underlying strategy selection relies on cube_utils.iterator_strategy, which inspects the cube and axis to determine an appropriate processing approach.

- allbadtonan is used to ensure consistent handling of all-bad data when reducing along an axis, particularly in cubewise computations.

- Masks: Moment calculations respect the cube’s masks. In raywise mode, the code checks which rays are includable before processing to avoid unnecessary work.
