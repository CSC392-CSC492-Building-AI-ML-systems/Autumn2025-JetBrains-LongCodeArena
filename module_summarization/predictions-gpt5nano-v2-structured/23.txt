w_fork
======

Overview
--------

w_fork prepares a new weighted ensemble (WE) simulation by forking from an existing WestPA run at a specified iteration. It generates a new HDF5 file containing the new simulation, including target states, basis states, and a transformed set of segments that become initial states for the forked run. A flat text mapping file is produced to relate old segments to the new initial states, facilitating restart and traceability.

Key concepts
------------

- Each old segment is mapped to a single new initial state and a corresponding new segment in the forked simulation.
- The new initial states inherit the final coordinates (pcoords) from the old segments at the chosen iteration.
- The new segments start from the corresponding initial state and carry forward the old segment weight.
- A textual state map (ISTATE_MAP) records the correspondence: old iteration, old segment ID, and new initial state ID.

Usage
-----

w_fork [OPTIONS]

Options
-------

- -i, --input
  - Input HDF5 file to fork from (default behavior reads from the configuration file if not provided).

- -I, --iteration
  - The iteration to fork from. If omitted, the last complete iteration is used.

- -o, --output
  - Output HDF5 file for the forked simulation (default: forked.h5).

- --istate-map
  - Path to the ISTATE_MAP text file describing the mapping from old segments to new initial states (default: istate_map.txt).

- --no-headers
  - Do not write header lines to ISTATE_MAP.

Examples
--------

Fork from a specific iteration and write outputs to custom locations:

.. code-block:: text

   w_fork -i old_run.h5 -I 25 -o forked_run.h5 --istate-map my_istate_map.txt

Fork without headers in the mapping file:

.. code-block:: text

   w_fork -i old_run.h5 -I 25 -o forked_run.h5 --istate-map my_map.txt --no-headers

What w_fork does
----------------

1. Open the existing (old) WE HDF5 file.
2. Determine the iteration to fork from (explicit or last complete iteration).
3. Create and open a new HDF5 file for the forked simulation.
4. Copy target states from the old iteration to the new file.
5. Copy basis states for the new simulation’s ibstate group (basis states for the new run).
6. Transform each old segment into:
   - a new initial state (one per old segment),
   - a corresponding new segment for the forked run.
   The new initial state is configured as a restart, prepared, with iter_created = 0 and iter_used = 1.
   The new segment uses the old segment’s weight and references the new initial state as its parent.
   The first pcoord of the new segment is set to the final pcoord of the old segment at the fork iteration.
7. Update the new run’s initial states and prepare the first iteration (n_iter=1) with the new segments.
8. Write a state map file mapping each old segment to the corresponding new initial state (old_n_iter, old_seg_id, new_istate_id).
9. Close both HDF5 files.

Output artifacts
----------------

- A new HDF5 file containing the forked simulation data (default: forked.h5 or as specified with -o/--output).
- An ISTATE_MAP text file describing the mapping from old segments to new initial states (default: istate_map.txt or as specified with --istate-map). Can include headers unless disabled with --no-headers.

Notes
-----

- For executable propagation workflows, you may need to prepare the new simulation directory so that restart data from the old iteration is available as the new simulation’s initial state data.
- The ISTATE_MAP provides a straightforward way to trace how each old segment corresponds to its new initial state, aiding restart and debugging.

Related concepts
----------------

- Initial states (istates) and their role as restart points for new segments.
- Basis states and target states copying to maintain consistency with the original simulation configuration.
- Pcoord handling: the new segments inherit the final pcoords from theOld segments to preserve dynamic consistency at the fork point.

Reference
---------

This command is implemented in w_fork.rst (docs accompanying the WES Python package) and relies on WestPA’s data management interfaces (DataManager) to manipulate HDF5 files, initial states, and segments.
