Compatibility Policy
====================

This document describes the compatibility policy for the npyrandom project, focusing on how public APIs, binary interfaces, packaging, and build systems are treated across releases.

Scope
-----
- Public Python API exposed by the numpy.random integration (modules, classes, functions, and constants that users import and rely on).
- C extension ABI compatibility exposed to Python (shared libraries built for numpy.random).
- Packaging and distribution metadata (wheel/sdist compatibility) and build system expectations.
- Supported Python versions, NumPy versions, and platform coverage as they relate to compatibility.

Key Principles
--------------
- Semantic compatibility for public APIs within a major version: minor/patch releases should preserve behavior and signatures unless a deprecation path is provided.
- Binary (ABI) compatibility for C extensions across minor/patch releases when feasible; incompatible changes require a major version bump.
- Deprecations are signaled in a prior release and allowed to be removed only in a subsequent major release.
- Changes that would break compatibility must be clearly documented in release notes and versioned according to the project’s versioning policy.

Versioning and Breaking Changes
-------------------------------
- The project follows a semantic-like versioning approach:
  - MAJOR version: breaking changes to public API, ABI, or packaging compatibility.
  - MINOR/patch: bug fixes, performance improvements, and non-breaking enhancements.
- When a breaking change is introduced, the release must:
  - Increment the MAJOR version.
  - Provide a migration guide in the release notes.
  - Prefer deprecation where possible to ease transition.

Public API Compatibility
------------------------
- Public API changes should preserve existing callables, signatures, and expected behavior.
- Additions to the API are allowed in minor releases but should be backwards-compatible.
- Removals and renamed symbols require a major release and a deprecation period prior to removal if feasible.
- Deprecation policy:
  - Deprecated features should emit a clear warning (e.g., DeprecationWarning) at runtime.
  - Document deprecated items with guidance on the preferred alternatives.
  - Remove deprecated items only in a major release.

Binary Compatibility (C Extensions)
------------------------------------
- The built C extensions expose the runtime ABI used by Python and NumPy.
- Maintain ABI stability across minor/patch releases when possible.
- If an ABI change is unavoidable (e.g., due to fundamental data structure changes or API surface changes in the extension module), a major version bump is required.
- Platform-specific considerations (Windows, macOS, Linux) are accounted for in CI to ensure cross-platform compatibility within the supported version sets.

Python, NumPy, and Platform Compatibility
-------------------------------------------
- Supported Python versions: align with the project’s policy (see contributor docs or setup metadata for exact ranges).
- Supported NumPy versions: the library requires a NumPy API compatible with the extensions (as determined by the C/Python interface and NumPy headers used during build).
- Platform support: builds for major desktop/server environments (Windows, macOS, Linux) are validated in CI; any platform-specific issues should be documented.

Build System and Packaging
--------------------------
- Build system: Meson-based build for C extensions with Python bindings.
- Packaging: wheels and source distributions should preserve compatibility guarantees stated above.
- Changes to the build configuration that affect the public surface (e.g., new default build flags affecting ABI, or removal of a public build target) count as compatibility-breaking changes and require a major version bump.

Testing for Compatibility
-------------------------
- Run the full test suite across supported Python and NumPy combinations to verify compatibility.
- CI should exercise:
  - Public API behavior (unit and integration tests).
  - C-extension ABI behavior (binary tests where applicable).
  - Packaging and installation paths (wheel/sdist).
- If a breaking change is introduced, tests should include a migration scenario or a dedicated compatibility test in the release.

Migration Guide (Example)
-------------------------
- When upgrading from 0.x.y to 1.0.0:
  - Review deprecations introduced in the 0.x.y releases; update code to use new APIs.
  - Update import paths, function names, or behavior to align with the 1.x API.
  - Rebuild extensions against the latest NumPy headers and Python runtime.
  - Run the full test suite and verify performance characteristics.

Code Examples
-------------
- Deprecation pattern (maintain compatibility while guiding users to new API):

.. code-block:: python

   import warnings

   def old_api(*args, **kwargs):
       warnings.warn(
           "old_api is deprecated and will be removed in a future major release. "
           "Use new_api instead.",
           DeprecationWarning,
           stacklevel=2
       )
       return new_api(*args, **kwargs)

   def new_api(*args, **kwargs):
       # new implementation
       pass

- Checking compatibility constraints at runtime (example snippet for internal use):

.. code-block:: python

   from importlib.metadata import version, PackageNotFoundError

   TRY_PY_VERSION = ">=3.8"
   try:
       import sys
       py_ver_ok = sys.version_info >= (3, 8)
   except Exception:
       py_ver_ok = False

   try:
       numpy_ver = version("numpy")
       # Implement any numpy version gating logic as needed
   except PackageNotFoundError:
       numpy_ver = None

   if not py_ver_ok or not numpy_ver:
       raise SystemError("Incompatible environment: requires Python >= 3.8 and NumPy installed.")

Documentation and Release Notes
-------------------------------
- Every release must include release notes describing:
  - What changed (API/ABI/packaging)
  - Whether the change is breaking or non-breaking
  - Deprecations and migration guidance
  - Any changes to minimum/maximum supported environments

Conclusion
----------
This Compatibility Policy aims to provide clear expectations for users and contributors regarding how npyrandom maintains compatibility across versions, how breaking changes are managed, and how to perform migrations with minimal disruption. For any edge cases or clarifications, refer to the project’s contributing guidelines and release management documents.
