# Conditions in PyTables

This section documents how to express and use conditions to filter data in PyTables. Conditions enable efficient row selection when reading from Table-like objects (Table, EArray, etc.) and when using indexed columns.

## Overview

Conditions are boolean expressions evaluated against the rows of a table or a related array. They are most commonly used with:

- read operations that return only the rows matching the condition (e.g., `Table.read_where`),
- iterators that yield rows meeting the condition (e.g., `Table.where`).

PyTables evaluates these expressions using the column names of the target table. When an index exists on a column used in the condition, PyTables will leverage the index to prune the search space for better performance.

Key ideas:
- Expressions compare column values against constants or other column values.
- Logical composition is done with bitwise operators for clarity and performance: use `&` for logical AND, `|` for logical OR, and `~` for logical NOT.
- Parentheses control precedence.
- String constants are written in Python string syntax inside the condition.

NaN-aware behavior: PyTables uses internal helpers that handle NaN values consistently in floating-point comparisons. This helps produce predictable results when data contain NaNs.

## Supported syntax

- Basic comparisons
  - `column > value`
  - `column >= value`
  - `column < value`
  - `column <= value`
  - `column == value`
  - `column != value`
- Logical operations
  - `&` for AND
  - `|` for OR
  - `~` for NOT
- Grouping
  - Use parentheses to control precedence, e.g. `(A > 5) & ((B < 10) | (C == 0))`
- Membership tests
  - `column in [val1, val2, ...]`
  - Example: `status in ["Active", "Pending"]`
- Constants
  - String constants: use quotes, e.g. `name == "John"`
  - Numeric constants: plain numbers, e.g. `age >= 18`
- Column references
  - Use the column name as it appears in the table. For tables with nested groups, reference the column by its path within the table (the common, user-facing column name is usually sufficient).

Notes:
- Use `&` and `|` instead of the Python keywords `and`/`or`. Do not rely on Python boolean short-circuiting for performance.
- If a column stores bytes, compare with a bytes literal, e.g. `b"Active"`.

## Examples

Code examples assume you have an opened HDF5 file and a table with columns such as `age`, `name`, `status`, and optionally indices on some of these columns.

- Simple filter: ages between 18 and 30
  - Condition: `(age >= 18) & (age <= 30)`
  - Usage:
    ```python
    data = table.read_where('(age >= 18) & (age <= 30)')
    ```

- Filtering with strings
  - Condition: `name == "John"`
  - Usage:
    ```python
    data = table.read_where('(name == "John")')
    ```

- Combining numeric and string criteria
  - Condition: `age > 21` and `status` is `"Active"`
  - Usage:
    ```python
    data = table.read_where('(age > 21) & (status == "Active")')
    ```

- Membership test
  - Condition: `category` is one of several values
  - Usage:
    ```python
    data = table.read_where('(category in ["A", "B", "C"])')
    ```

- Using NOT
  - Condition: select rows where `flag` is false
  - Usage:
    ```python
    data = table.read_where('~(flag)')
    ```

- Using an index to accelerate a range query
  - If an index exists on `age`, queries like `(age >= 50) & (age < 70)` will be accelerated by the index.

- Row-wise iteration with a condition
  - You can iterate rows that satisfy a condition using:
    ```python
    for row in table.where('(age >= 18) & (status == "Active")'):
        # process row
        pass
    ```

## Best practices

- Favor indexed columns for large tables: place conditions on indexed columns to maximize pruning.
- Keep conditions simple and avoid expensive expressions inside the condition (e.g., function calls) when possible.
- Use parentheses to make intent explicit and ensure correct precedence.
- When dealing with strings, ensure quotes are properly escaped in Python strings, e.g. `'(name == "O\'Brien")'`.

## Limitations and notes

- The condition syntax is evaluated against the table’s column data; it does not execute Python code or arbitrary expressions.
- Column names must match exactly as stored; if a name contains special characters, refer to it by its exact column path.
- NaN handling in floating-point columns is managed by internal nan-aware comparison helpers to provide stable behavior.
