Unified renderable interface
============================

This document describes the unified renderable interface provided by renderable.rst. It defines a small, composable set of classes and helpers to render templates with a consistent input object, along with optional CSS-class handling and BEM-friendly naming.

Overview
--------
- The core concept is a renderable object that knows how to render a template using data derived from itself.
- Rendering is performed via render(), which delegates to Django's render_to_string and forwards an optional HttpRequest to provide request-aware context and template rendering.
- The interface is designed to be extended with CSS class management (AbstractRenderableWithCss) or BEM-style naming (AbstractBemRenderable).

Key components
--------------

join_css_classes_list(css_classes_list)
- Purpose: Convert a list of CSS class names into a single string.
- Signature: join_css_classes_list(css_classes_list) -> str
- Behavior: Joins items with two spaces between classes (as implemented in the code).

AbstractRenderable
------------------
Base class for renderables. It defines the core rendering contract and a minimal context.

- Template handling
  - template_name: The default template name to render if get_template_names is not overridden.
  - get_template_names(): Return the template name(s) to render. Defaults to template_name. If neither is provided, raises NotImplementedError.

- Context data
  - get_context_data(request=None): Returns the context dictionary used for rendering. Defaults to { 'me': self }.
  - The request argument is forwarded to this method when provided to render().

- Rendering
  - render(request=None, extra_context_data=None): Renders the template returned by get_template_names() using the context from get_context_data(), merged with any extra_context_data. If a request is provided, it is forwarded to render_to_string.

Usage example
-------------
Minimal renderable

.. code-block:: python

   from django.template.loader import render_to_string
   from django_http import HttpRequest  # if needed
   from django_cradmin.renderable import AbstractRenderable

   class MyRenderable(AbstractRenderable):
       template_name = 'my_template.html'

       def get_context_data(self, request=None):
           data = super().get_context_data(request=request)
           data['custom'] = 'value'
           return data

   # Usage
   req = HttpRequest()
   render_str = MyRenderable().render(request=req)

Notes
-----
- The render() method forwards the request to both get_context_data and render_to_string when provided.
- If you need a different template selection strategy, override get_template_names().

AbstractRenderableWithCss
-------------------------
Extends AbstractRenderable with a unified API for CSS classes.

- CSS class sources
  - get_base_css_classes_list(): Returns a list of base CSS classes. Defaults to [].
  - get_extra_css_classes_list(): Returns a list of additional CSS classes. Defaults to [].
  - get_css_classes_list(): Combines base and extra CSS classes. Returns a list. You should override this to define component-specific classes. The default implementation concatenates the results of get_base_css_classes_list() and get_extra_css_classes_list().

- Test CSS helpers
  - get_test_css_class_suffixes_list(): Returns a list of suffixes to include when running automatic tests. These suffixes are processed by a template tag in tests.

- CSS class rendering
  - css_classes (property): Joins the list returned by get_css_classes_list() into a single string. If the DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES setting is enabled, additional test CSS classes are appended using the cradmin_test_css_class tag.
  - get_css_classes_string(): Deprecated alias for css_classes; emits a deprecation warning and returns css_classes.

- Example
.. code-block:: python

   from django_cradmin.renderable import AbstractRenderableWithCss

   class MyCssRenderable(AbstractRenderableWithCss):
       def get_css_classes_list(self):
           return ['my-component', 'is-visible']

   # Access in template via: {{ my_renderable.css_classes }}

Notes
-----
- To enable test CSS classes, set DJANGO_CRADMIN_INCLUDE_TEST_CSS_CLASSES to True.
- get_css_classes_string() is deprecated; prefer the css_classes property.

AbstractBemRenderable
---------------------
Base class for renderables that follow BEM (Block-Element-Modifier) naming for CSS.

- Constructor
  - __init__(bem_block=None, bem_element=None, bem_variant_list=None, extra_css_classes_list=None)
    - bem_block: The BEM block name. Cannot be supplied together with bem_element.
    - bem_element: The BEM element name. Must contain a double underscore '__' if supplied.
    - bem_variant_list: A list of BEM variants (the part after --).
    - extra_css_classes_list: Extra CSS classes to include.

  - Validation rules
    - If both bem_block and bem_element are provided, a ValueError is raised.
    - If bem_element is provided, it must contain '__' (to separate block and element in the BEM naming scheme).

- BEM accessors
  - bem_block_or_element: Returns the block or element to use, preferring the block if present, otherwise the element.
  - get_bem_block(): Returns the bem_block string.
  - get_bem_element(): Returns the bem_element string.

- Test CSS suffixes
  - get_test_css_class_suffixes_list(): As with AbstractRenderableWithCss, supports test CSS suffixes for testing via template tags.

Usage example
-------------
Bem-based renderable

.. code-block:: python

   from django_cradmin.renderable import AbstractBemRenderable

   class MyBemRenderable(AbstractBemRenderable):
       def __init__(self, **kwargs):
           super().__init__(bem_block='card', bem_element='card__header',
                            bem_variant_list=['large'], extra_css_classes_list=['shadow'])

       # The resulting CSS classes can be accessed via the css_classes property
       # and will reflect the BEM naming plus any extra classes.

Notes
-----
- This class provides a structured approach to naming and class composition following BEM conventions.
- If you are not using BEM, consider using AbstractRenderableWithCss for simpler CSS composition.

Notes and considerations
------------------------
- The renderable interface is designed to be extended to provide a consistent rendering workflow across components.
- CSS-class management is centralized in AbstractRenderableWithCss, with optional test CSS integration.
- BEM support is provided via AbstractBemRenderable for projects adopting BEM conventions.
- The codebase includes a deprecated get_css_classes_string() method; prefer using the css_classes property.

Reference implementation highlights
----------------------------
- Rendering core:
  - render() composes context_data from get_context_data(request) and extra_context_data, then renders with render_to_string(template_name=get_template_names(), context=context_data, request=request).
- Template selection:
  - get_template_names() returns template_name by default, otherwise raises NotImplementedError to enforce explicit template specification.

This documentation reflects the unified renderable interface and its extension points for CSS and BEM-based styling.
