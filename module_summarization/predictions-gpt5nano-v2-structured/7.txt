# Using lists

This section explains how to work with lists when using the ftrack_api library. Lists are the primary way to collect, manipulate and process multiple entities (such as tasks, events, or projects) returned from queries or collected from operations.

## Overview

- Lists are standard Python list-like collections used to hold multiple ftrack entities.
- They can be created by collecting results from a query, aggregating multiple results, or by combining existing lists.
- You can apply typical Python list operations (indexing, slicing, filtering, sorting, etc.) to work efficiently with many entities at once.
- For large result sets, prefer lazy processing or streaming where possible and convert to a list only when you need random access or full iteration.

## Creating lists from queries

Collect the results of a query into a Python list for further processing.

```python
# Example: collect all Task entities for a given project
project_id = 'abc123'

tasks = [task for task in session.query(
    "select id, name, priority from Task where project.id = :project_id",
    project_id=project_id
)]
```

Notes:
- The exact query syntax depends on the underlying query interface. The example shows a typical pattern: iterate over a query result and build a Python list.
- You can also build a list using a straightforward comprehension if your query returns an iterable of objects.

## Basic list operations

- Indexing and slicing

```python
# Get the first task
first_task = tasks[0] if tasks else None

# Get a subset of tasks
subset = tasks[2:5]
```

- Transforming lists

```python
# Create a list of task names
task_names = [t.name for t in tasks]

# Create a list of IDs
task_ids = [t.id for t in tasks]
```

- Filtering

```python
# Tasks marked as high priority
high_priority_tasks = [t for t in tasks if getattr(t, 'priority', None) == 'high']
```

- Sorting

```python
# Sort by name
tasks_sorted = sorted(tasks, key=lambda t: getattr(t, 'name', ''))
```

- Combining lists

```python
more_tasks = [...]  # another list of Task-like objects
all_tasks = tasks + more_tasks  # concatenation
```

- Removing items

```python
# Remove a specific task
if some_task in tasks:
    tasks.remove(some_task)

# Remove by index
del tasks[0]
```

## Working with large result sets

- Avoid constructing a large list if you only need to process items sequentially.

```python
for task in session.query(
    "select id, name from Task where project.id = :project_id",
    project_id=project_id
):
    process(task)
```

- If you need random access later, you can materialize a list when appropriate:

```python
tasks = list(session.query("...", project_id=project_id))
```

## Practical examples

- Example: find all tasks in a project that are incomplete and return their IDs in a list.

```python
incomplete_task_ids = [
    t.id for t in session.query(
        "select id from Task where project.id = :project_id and status != 'done'",
        project_id=project_id
    )
]
```

- Example: group tasks by priority (producing a dictionary of lists)

```python
from collections import defaultdict

tasks_by_priority = defaultdict(list)
for t in tasks:
    priority = getattr(t, 'priority', 'unknown')
    tasks_by_priority[priority].append(t)
```

## Advanced: working with list-like collections

Some ftrack_api interfaces may return list-like collection objects that are lazy or support streaming. When you need a standard Python list, you can convert them:

```python
# If result is a collection-like object
collection_like = session.query("select id from Task where project.id = :pid", pid=project_id)
tasks = list(collection_like)  # materialize into a Python list
```

Be mindful of memory usage when materializing very large collections.

## Best practices

- Prefer list comprehensions for clear, concise transformations.
- Use slicing or iterators for large data to avoid unnecessary memory usage.
- Convert to a list only when you need random access or to pass the data to APIs that require a concrete list.
- Leverage Python’s built-in tools (filter, map, itertools) for readable processing pipelines.
