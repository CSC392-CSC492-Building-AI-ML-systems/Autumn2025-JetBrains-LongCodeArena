Working with user security roles
===============================

Overview
--------
Security roles define the level of access a user has within an ftrack instance. Roles can be assigned to users and may be scoped to contexts such as projects or workspaces. This guide covers how to discover roles, assign and revoke roles for users, and how to inspect a user’s current permissions via the API.

Prerequisites
-------------
- A working ftrack API session
- Access to a valid server, API key, and API user
- Knowledge of the user you want to manage (by login/username)

Example session setup
---------------------
The following demonstrates creating a session, which is required for all role management operations.

.. code-block:: python

    from ftrack_api import Session

    # Replace with your server details and credentials
    session = Session(
        server_url='https://your-ftrack-server',
        api_key='YOUR_API_KEY',
        api_user='YOUR_API_USER'
    )

    # Ensure the session is connected and ready
    session.connect()  # if applicable in your setup
    # Or rely on the lazy connection behavior of the Session

Usage overview
--------------
Common tasks when working with user security roles:

- List all available security roles
- Retrieve a role by name
- Retrieve a user by login or ID
- Assign a role to a user
- Revoke a role from a user
- Inspect a user’s current roles and effective permissions

Listing available roles
-----------------------
Retrieve all known security roles in the system.

.. code-block:: python

    # Get all security roles
    roles = session.query('SecurityRole').all()

    for role in roles:
        print(role.name)

Getting a role by name
----------------------
Fetch a specific role by its name for assignment or inspection.

.. code-block:: python

    role_name = 'Project Manager'
    role = session.query('SecurityRole where name is "%s"' % role_name).one()

Assigning a role to a user
--------------------------
Obtain the user object and attach the desired role. Depending on the API version, you may have one of several supported patterns (both shown below are common approaches).

.. code-block:: python

    # Option 1: Add role to the user's roles collection
    user = session.query('User where login is "alice"').one()
    user.roles.append(role)

    # Option 2: Add user to the role's users collection
    role.users.append(user)

    # Persist changes
    session.commit()

Notes:
- If the user already has the role, the operation will be a no-op or raise a helpful message depending on the API version.
- Some deployments require explicit context (e.g., per-project roles). See the Contextual roles section below.

Revoking a role from a user
----------------------------
Remove the association between the user and the role.

.. code-block:: python

    user = session.query('User where login is "alice"').one()
    role = session.query('SecurityRole where name is "Viewer"').one()

    # Option 1: Remove from user's roles
    user.roles.remove(role)

    # Option 2: Remove user from role's users
    role.users.remove(user)

    session.commit()

Inspecting a user’s roles and permissions
-----------------------------------------
You can inspect which roles a user currently has and infer their permissions. Depending on API surface, you may access direct relationships or query for the user’s roles.

.. code-block:: python

    user = session.query('User where login is "alice"').one()

    # List current roles
    current_roles = getattr(user, 'roles', [])
    print("Current roles for {}: {}".format(user.login, [r.name for r in current_roles]))

    # Example: determine effective permissions (implementation varies by API version)
    # This is a conceptual example; adapt to your API surface if needed.
    try:
        effective_perms = user.get_effective_permissions()
        print("Effective permissions:", effective_perms)
    except AttributeError:
        # Fallback: inspect permissions via role-based attribute if available
        perms = set()
        for r in current_roles:
            perms.update(getattr(r, 'permissions', []))
        print("Aggregated permissions:", sorted(perms))

Contextual and project-scoped roles
-----------------------------------
Some deployments support scoping roles to a specific project or context. When working with scoped roles:

- Retrieve or create the role within the target context
- Assign the context-scoped role to the user as described above
- Verify the user’s access within that context

Example (conceptual)
.. code-block:: python

    project = session.query('Project where id is "P123"').one()
    scoped_role = session.query('SecurityRole where name is "Project Contributor" and project_id is "%s"' % project.id).one()

    user = session.query('User where login is "alice"').one()
    user.roles.append(scoped_role)
    session.commit()

Best practices
--------------
- Prefer explicit, minimal role assignments to follow the principle of least privilege.
- Use descriptive role names and maintain an up-to-date mapping of roles to permissions.
- When possible, rely on project- or context-scoped roles to minimize unintended access.
- Always commit changes to persist role assignments.

Notes
-----
- The exact class names and relationship attributes (for example, SecurityRole, User, roles, or users) may vary between API versions. Adapt the code samples to match your API surface.
- If you enable or rely on an embedded event hub or additional plugins, ensure the hub is connected before performing role-related operations in distributed setups.
