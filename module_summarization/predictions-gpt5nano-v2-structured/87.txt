Running migrations (Backward)
=============================

This module provides functionality to reverse previously executed migrations. It exposes a BackwardsMigrationManager that can reverse migrations for a single app, as well as helper functions to reverse migrations across one or multiple apps. A lightweight status-checker (CheckMigrationManager) is included as a scaffold for querying migration statuses.

Public API
----------

BackwardsMigrationManager
~~~~~~~~~~~~~~~~~~~~~~~~

Description
    Orchestrates reversing (undoing) migrations for a single app. It determines which migrations have already been run,
    selects the appropriate subset to reverse, and executes the reversal in reverse order. After reversing, it can
    optionally delete migration records from the database and/or delete the corresponding migration files from disk.

Initialization
    __init__(self, app_name: str, migration_id: str, auto_agree: bool = False, clean: bool = False, preview: bool = False)

Parameters
    - app_name: The name of the app whose migrations should be reversed.
    - migration_id: Specifies the migrations to reverse. Accepts:
        - "all" to reverse all migrations for the app
        - "1" to reverse only the most recent migration
        - a specific migration_id to reverse up to and including that migration
    - auto_agree: If True, automatically proceed without prompting the user.
    - clean: If True, delete the corresponding migration file from disk after reversal.
    - preview: If True, do not apply changes; instead, print or collect the SQL/operations to be executed.

Key methods
    - run(self) -> MigrationResult
        Creates the migration table (if needed), loads app config, and runs backwards for the specified app.
        Returns a MigrationResult indicating success or failure.

    - run_migrations_backwards(self, app_config: AppConfig)
        Core implementation that:
        - Loads migration modules from the app's migrations folder
        - Determines migrations that have already run for the app
        - Validates the requested migration_id against the ran migrations
        - Builds the list of migrations to reverse, in reverse order
        - Executes each reversal, possibly in preview mode
        - Optionally cleans up records/files after reversal
        Returns a MigrationResult indicating success or failure

Usage example
    # Programmatic usage
    from piccolo.apps.migrations.running import BackwardsMigrationManager
    import asyncio

    async def main():
        manager = BackwardsMigrationManager(
            app_name="blog",
            migration_id="1",
            auto_agree=True,
            clean=False,
            preview=False,
        )
        result = await manager.run()
        print(result)

    asyncio.run(main())

    # Alternative: use run_backwards for a quick one-off
    from piccolo.apps.migrations.running import run_backwards
    import asyncio

    async def main():
        result = await run_backwards(
            app_name="blog",
            migration_id="1",
            auto_agree=True,
            clean=False,
            preview=False,
        )
        print(result)

    asyncio.run(main())

Notes
    - If no migrations have run yet for the app, a success message is returned: "🏁 No migrations to reverse!".
    - When reversing, non-preview runs will remove the corresponding Migration record, and optionally delete the
      migration module file from disk if clean is True and the module file path is available.
    - For migrations that return a MigrationManager when invoked, the backwards flow is delegated to that manager
      via its run(backwards=True) method. If preview is enabled, the manager is put into preview mode before running.

Return value
    MigrationResult: Indicates success or failure and may include a message describing the outcome.

run_backwards
-------------

Description
    Helper function that orchestrates backwards migration for either a single app or all apps. It handles
    user confirmation when reversing migrations for multiple apps and dispatches to BackwardsMigrationManager
    for the actual work.

Parameters
    - app_name: The app to reverse migrations for, or "all" to reverse migrations for all apps.
    - migration_id: Same semantics as BackwardsMigrationManager (e.g., "all", "1", or a specific id).
    - auto_agree: If True, automatically confirm prompts.
    - clean: If True, delete migration files after reversal.
    - preview: If True, do not execute migrations; only print/collect the operations.

Behavior
    - If app_name == "all":
        - Determine the sorted list of app names, reverse it, and prompt the user (unless auto_agree).
        - For each app, print a heading and run a BackwardsMigrationManager with migration_id="all".
    - Else:
        - Run a BackwardsMigrationManager for the specified app and the given parameters.

Return value
    MigrationResult: Indicates success or failure.

Usage example
    # Reversing all apps
    from piccolo.apps.migrations.running import run_backwards
    import asyncio

    async def main():
        result = await run_backwards(app_name="all", migration_id="all", auto_agree=True)
        print(result)

    asyncio.run(main())

    # Reversing a single app
    async def main_single():
        result = await run_backwards(app_name="blog", migration_id="1", auto_agree=True)
        print(result)

    asyncio.run(main_single())

backwards
---------

Description
    Public entry point used by CLI or higher-level orchestration to undo migrations.
    It delegates to run_backwards and exits the process with a non-zero status if the result is unsuccessful.

Parameters
    - app_name: The app to reverse migrations for, or "all" to reverse migrations for all apps.
    - migration_id: Migrations will be reversed up to and including this migration_id ('all' or '1' or a specific id).
    - auto_agree: If True, automatically proceed without prompts.
    - clean: If True, delete migration files after reversal.
    - preview: If True, do not apply changes; only print/collect the operations.

Behavior
    - Calls run_backwards with the provided parameters.
    - If the result is not successful, exits the process with status code 1.

Return value
    None (the function exits the process on failure; otherwise completes successfully).

Notes
    - Intended as a CLI-friendly entry point. The function is asynchronous; when used from a script, wrap it with an asyncio loop.

Additional references
---------------------

- MigrationResult: Returned by the core operations indicating success or failure.
- Migration table: Used to track which migrations have run.
- Migration modules: Loaded from each app's migrations folder to execute backwards/forwards logic.
- Preview mode: Allows generating SQL/operations without applying changes.

Code excerpts
-------------

The core flow (high-level) is represented by the following simplified outline:

.. code-block:: python

    class BackwardsMigrationManager(BaseMigrationManager):
        def __init__(self, app_name, migration_id, auto_agree=False, clean=False, preview=False):
            ...

        async def run(self) -> MigrationResult:
            await self.create_migration_table()
            app_config = self.get_app_config(self.app_name)
            return await self.run_migrations_backwards(app_config=app_config)

        async def run_migrations_backwards(self, app_config: AppConfig):
            migration_modules = self.get_migration_modules(app_config.migrations_folder_path)
            ran_migration_ids = await Migration.get_migrations_which_ran(app_name=self.app_name)
            if len(ran_migration_ids) == 0:
                message = "🏁 No migrations to reverse!"
                print(message)
                return MigrationResult(success=True, message=message)

            if self.migration_id == "all":
                earliest_migration_id = ran_migration_ids[0]
            elif self.migration_id == "1":
                earliest_migration_id = ran_migration_ids[-1]
            else:
                earliest_migration_id = self.migration_id

            if earliest_migration_id not in ran_migration_ids:
                message = (
                    "Unrecognized migration name - must be one of "
                    f"{ran_migration_ids}"
                )
                print(message, file=sys.stderr)
                return MigrationResult(success=False, message=message)

            latest_migration_id = ran_migration_ids[-1]
            start_index = ran_migration_ids.index(earliest_migration_id)
            end_index = ran_migration_ids.index(latest_migration_id) + 1
            subset = ran_migration_ids[start_index:end_index]
            reversed_migration_ids = list(reversed(subset))

            n = len(reversed_migration_ids)
            _continue = (
                "y" if self.auto_agree else input(f"Reverse {n} migration{'s' if n!= 1 else ''}? [y/N] ")
            )
            if _continue in "yY":
                for migration_id in reversed_migration_ids:
                    migration_module = migration_modules[migration_id]
                    response = await migration_module.forwards()
                    if isinstance(response, MigrationManager):
                        if self.preview:
                            response.preview = True
                        await response.run(backwards=True)
                    if not self.preview:
                        await Migration.delete().where(Migration.name == migration_id).run()
                        if self.clean and migration_module.__file__:
                            os.unlink(migration_module.__file__)
                    print("ok! ✔️")
                return MigrationResult(success=True)
            else:
                message = "Not proceeding."
                print(message, file=sys.stderr)
                return MigrationResult(success=False, message=message)

Notes on usage
--------------

- Use backwards() for a CLI-oriented workflow or to expose this functionality via a higher-level CLI.
- Use run_backwards in programmatic contexts where you need direct control over the result and behavior (e.g., test suites or scripts).
- The auto_agree flag is useful for automation; omit it to interactively confirm reversals.
