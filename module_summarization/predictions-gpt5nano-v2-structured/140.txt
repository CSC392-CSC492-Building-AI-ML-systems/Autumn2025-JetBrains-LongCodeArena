Short overview of Dask best practices
======================================

Dask provides parallel, out-of-core execution for pandas-like workloads by operating on partitioned data. The typical workflow remains lazy: build up a task graph through transformations, then trigger computation when needed. This overview highlights practical guidelines to write robust, scalable Dask code and avoid common pitfalls.

Key principles
- Partition-aware operations: Dask DataFrames are composed of partitions. Choose partition sizes that fit memory and balance task overhead.
- Lazy evaluation: Build your computation as a graph; call compute/persist only when you need the result.
- Appropriate scheduler: Start locally with the default scheduler, then scale to a cluster or the distributed scheduler as needed.
- Minimize Python loops: Prefer vectorized operations and map_partitions for custom logic; avoid row-wise Python loops.
- Monitor and profile: Use diagnostics (ProgressBar, profiler) and graph visualization to understand performance.

Recommended practices
- Control partition size from the start
  - Read data with sensible block sizes to create partitions of a manageable size.
  - Example: read a CSV with a target block size
    .. code-block:: python

       import dask.dataframe as dd
       ddf = dd.read_csv("data/*.csv", blocksize="64MB")

- Use lazy operations and persist selectively
  - Build the computation, then persist only the parts you reuse to avoid recomputation.
    .. code-block:: python

       ddf = ddf.assign(new_col=ddf["a"] + ddf["b"])
       ddf = ddf.persist()

- Repartition to balance load
  - If operations become bottlenecked by skewed partitions, repartition to an optimal number of partitions.
    .. code-block:: python

       ddf = ddf.repartition(npartitions=200)

- Set indices and sorts thoughtfully
  - For large groupby/sort workloads, use set_index with the appropriate shuffle method to control memory and I/O.
    .. code-block:: python

       ddf = ddf.set_index("timestamp", shuffle="tasks")

- Favor optimized I/O patterns
  - When dealing with many small files, consider consolidating or using blocksize-aware reads.
  - For Parquet, leverage partitioned datasets for efficient queries.

- Leverage vectorized operations and map_partitions
  - For custom logic, prefer map_partitions over apply(row-wise) to keep tasks efficient.

- Profile, debug, and visualize
  - Use progress bars and dashboards to understand task graphs and memory usage.
    .. code-block:: python

       from dask.diagnostics import ProgressBar
       with ProgressBar():
           result = ddf.mean().compute()

- Plan for memory usage
  - Monitor worker memory, avoid unbounded growth, and let Dask spill to disk when needed.
  - Use distributed Scheduler for large workloads to distribute memory and compute.

Common pitfalls to avoid
- Too many small partitions leading to high scheduling overhead.
- Row-wise Python loops in transformations; prefer vectorized or map_partitions.
- Large shuffle operations without adequate resources or a distributed scheduler.
- Re-reading data or recomputing expensive steps due to unnecessary recomputation.

Quick-start example
- Read, transform, and compute a simple result
  .. code-block:: python

     import dask.dataframe as dd
     ddf = dd.read_csv("data/*.csv", blocksize="64MB")
     ddf = ddf[ddf["value"] > 0]
     result = ddf.groupby("category").value.mean().compute()
