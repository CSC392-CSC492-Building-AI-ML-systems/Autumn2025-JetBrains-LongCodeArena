# Querying

Querying is the primary mechanism for retrieving data from an ftrack server. The ftrack_api.query module provides a programmatic way to construct and execute queries, enabling you to fetch only the fields you need, apply filters, sort results, and paginate large result sets. Queries are executed in the context of a Session and can benefit from session-level caching and efficient server communication.

## Core concepts

- Entity type: The kind of object you want to retrieve (for example, Task, Shot, User, AssetVersion, etc.).
- Fields (projection): The subset of fields to return for each result (e.g. id, name, status, created_on).
- Restrictions (filters): Conditions used to narrow down results (e.g. status = "In Progress", created_on > 2020-01-01).
- Sorting: The order of the results based on one or more fields.
- Pagination: Limiting the number of results and/or offsetting the starting point for large result sets.
- Results: The data returned by a query; items typically expose the requested fields and may include related data depending on the query.

## Building a query

To query data, you typically:

1. Create a query for a specific entity type.
2. Specify the fields you want to retrieve (projection).
3. Add restrictions to filter the results.
4. Optionally set sorting and pagination parameters.
5. Execute the query within a Session.

The exact API for constructing a query is provided by the ftrack_api.query module. Use the Query object (and related helpers) to define the query, then execute it through your Session.

## Filtering and restrictions

Filters restrict the results to entities that match given conditions. Common operators include equality, inequality, and range comparisons. You can combine multiple restrictions to form more complex queries (e.g., AND/OR combinations).

Tips:
- Always restrict the number of fields returned to what you actually need to minimize data transfer.
- Combine multiple restrictions to precisely target the data you require.
- Use indexing-friendly conditions (such as equality on indexed fields) when possible to improve performance.

## Field projection

Projection controls which fields are returned for each result. Specifying only the necessary fields reduces memory usage and network traffic. If you need related data, you may be able to fetch it in the same query depending on the API’s capabilities.

## Sorting and pagination

- Sorting allows you to order results by one or more fields, in ascending or descending order.
- Pagination (limit/offset) helps you manage large result sets by returning them in chunks.

Best practice is to:
- Start with a small limit while developing queries.
- Increase limit or use pagination for production data retrieval.
- Apply stable sort keys to ensure deterministic results.

## Executing queries

Queries are executed in the context of a Session. The session handles authentication, network I/O, and caching as configured. The result set is typically iterable, yielding objects that expose the requested fields. Access patterns may vary slightly depending on the version of the API and the shape of the results (e.g., entity objects vs. dictionaries).

Common usage pattern (conceptual):

- Create a Session with server URL and credentials.
- Build a Query for the desired entity type.
- Specify fields, filters, and optional sorting/pagination.
- Execute the query via the Session and iterate the results.

Example (conceptual):

```python
# Note: Adapt this to your version of the API's exact Query API.
from ftrack_api import Session, query

session = Session(server_url='https://my-ftrack.example.com',
                  api_key='your_api_key', api_user='your_user')

q = query.Query('Task')
q.fields = ['id', 'name', 'status']
q.filters = [('status', 'eq', 'In Progress')]
q.limit = 100

for t in session.execute_query(q):
    print(t['id'], t['name'], t['status'])
```

Replace the attribute names and method names with the exact ones provided by your installed version of ftrack_api.query.

## Results and data handling

- Result items expose the requested fields. Depending on the API version and query configuration, results may be returned as dictionaries or as lightweight entity objects.
- When related data is requested, the query may fetch nested structures or provide references to related entities, reducing the need for additional server calls.

## Performance and best practices

- Limit the amount of data returned by each query (use limit and, if needed, offset).
- Projection should be minimal: fetch only the fields you actually need.
- Prefer queries that filter early on server-side rather than retrieving large datasets and filtering client-side.
- Cache usage: rely on the session’s caching strategy to minimize redundant server requests.

## Troubleshooting

- Authentication or connectivity errors: check server_url, api_key, and api_user, and ensure network access to the ftrack server.
- Permissions: ensure the authenticated user has the necessary permissions for the target entity type and fields.
- Performance: examine the query’s filters and field projections; consider indexing strategies on the server side if available.

## API reference

- ftrack_api.query: Main module for building and executing queries.
- Query: Core class used to specify entity type, fields, filters, and other query options.
- Restrictions/filters: Helpers to compose conditional expressions for filtering.
- Sorting and pagination: Utilities to control the order and slice of the results.
