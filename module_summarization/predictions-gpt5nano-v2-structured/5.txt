Publishing versions
==================

Overview
--------

Publishing versions in ftrack refers to creating, annotating, and making available new published states of assets (or related entities) within a project. This process typically involves creating a new Version entity linked to a target asset, attaching the produced deliverables and metadata, and then recording or pushing the version to the server where it can be reviewed, approved, or consumed by downstream workflows.

Key concepts
------------

- Asset: The source item for which new versions are created (e.g. a shot, asset, or sequence).
- Version: A published state of an asset, often including metadata, descriptions, and references to deliverables (files, renders, etc.).
- Publish: The action of finalising and uploading a Version so it becomes the current trackable state of the asset in the project.
- Deliverables: Files or references (such as binaries, images, or metadata) associated with a Version.

Prerequisites
-------------

- Access to an ftrack server.
- Authentication via API key and user (or equivalent environment configuration):
  - server URL (FTRACK_SERVER)
  - API key (FTRACK_API_KEY or FTRACK_APIKEY)
  - API user (FTRACK_API_USER)
- An existing Asset to publish versions for, or appropriate permission to create versions.

Typical workflow
---------------

1. Prepare or select the target asset.
2. Create a new Version associated with the asset.
3. Attach deliverables and metadata (descriptions, notes, references).
4. Publish the Version to make it available in the project.
5. Optionally, attach review notes or trigger downstream publishing steps.

Using the Python API (illustrative)
-----------------------------------

The exact method names may vary between API versions. The following illustrates a common pattern: create a Version for an Asset, attach deliverables, and publish. Adapt to your installation’s API surface.

.. code-block:: python

    # Pseudo-code: replace with actual API calls for your environment
    from ftrack_api import Session

    session = Session(
        server_url='https://ftrack.example.com',
        api_key='YOUR_API_KEY',
        api_user='YOUR_USERNAME'
    )

    # Retrieve the asset to publish a new version for
    asset = session.query('Asset').filter('name is "MyAsset"').one()

    # Create a new Version linked to the asset
    version = asset.versions.create(
        name='v2.0',
        description='Rendered new sequence for lighting pass',
        # additional metadata as required
    )

    # Attach deliverables (e.g., file paths or references)
    version.attach_deliverable('/path/to/my_asset_v2.mov')
    version.attach_deliverable('/path/to/notes.txt')

    # Publish the version to the server
    version.publish()

Notes:
- The exact API for creating a Version and attaching deliverables will depend on your ftrack_api version and any custom plugins.
- If your workflow requires approvals, ensure the Version passes any review checks before or during publishing.

Best practices
--------------

- Use clear, semantic version names (e.g., v2.0, v2.1-rc1) and descriptive descriptions.
- Attach all relevant deliverables and metadata at publish time to avoid missing context.
- Validate permissions and project scope before publishing to prevent cross-project contamination.
- Use a consistent naming and tagging convention to facilitate searching and filtering of versions.

Error handling
--------------

- Publishing operations may fail due to permissions, missing assets, or invalid metadata. Catch and log exceptions using the ftrack_api exception types appropriate to your installation.
- If publishing is asynchronous, poll for status updates and handle timeouts gracefully.

Migration and compatibility
---------------------------

- API surface evolves between ftrack_api versions. When upgrading, review the Version creation and publish methods for any renamed or moved APIs.
- If you rely on plugins for publishing, ensure they are compatible with the upgraded API and adjust any signature changes accordingly.
