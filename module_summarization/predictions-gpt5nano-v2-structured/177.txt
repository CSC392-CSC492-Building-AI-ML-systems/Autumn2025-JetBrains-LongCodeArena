Pipelines and Composite Estimators
===================================

Overview
--------

A Pipeline is a composite estimator that chains together multiple steps: a sequence of transformers (which implement fit and transform) followed by a final estimator (which implements fit). Pipelines enable you to:

- Treat a sequence of preprocessing steps and an estimator as a single estimator.
- Perform cross-validation and hyperparameter tuning across all steps using the familiar parameter syntax (step_name__param).
- Cache intermediate results to speed up repeated fits when the preprocessing is expensive.

Key ideas

- Steps are a list of (name, estimator) tuples. All steps except the last must be transformers (fit/transform); the last step is an estimator (fit).
- Parameter access uses the double-underscore syntax: for example, to set a parameter of the second step, you use second_step_name__param.
- The final estimator is the only step that needs to implement fit; all preceding steps must implement fit and transform.
- The pipeline supports memory caching to speed up repeated fits, with cloning of transformers.

Creating a Pipeline
-------------------

Use a list of (name, estimator) pairs. The last element must be an estimator.

Example: Standardize features then train a classifier

```python
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline

pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', LogisticRegression())
])

# Fit and predict as a single estimator
pipe.fit(X_train, y_train)
y_pred = pipe.predict(X_test)
```

Convenience: make_pipeline
--------------------------

For a shorter, less verbose construction, you can use make_pipeline. It assigns default names to steps automatically.

```python
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression

pipe = make_pipeline(StandardScaler(), LogisticRegression())

pipe.fit(X_train, y_train)
```

Accessing and Setting Parameters
--------------------------------

- Parameters of any step can be accessed or set using the __ syntax.

```python
# Set a parameter on the classifier
pipe.set_params(clf__C=0.5)

# Get the full parameter tree
params = pipe.get_params()
```

- You can inspect or modify steps via the named_steps attribute:

```python
scaler = pipe.named_steps['scaler']
```

- You can replace a step completely by setting it to a different estimator, or remove a step by setting it to 'passthrough' or None.

Caching and Memory
------------------

Pipeline supports memory caching to avoid recomputing expensive transformations during multiple fits.

```python
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', LogisticRegression())
], memory='cache_dir')
```

- If memory is a string path, a Joblib.Memory cache is used.
- The last step is never cached, even if it is a transformer.
- When memory caching is enabled, transformers are cloned before fitting to ensure isolation between pipeline instances.

Output Configuration
--------------------

You can configure the output type of transform calls across all steps using set_output. This is useful when transformers support pandas DataFrame outputs.

```python
pipe.set_output(transform='pandas')
```

- transform: choose among "default" (the transformer’s default), "pandas" (DataFrame output), or None (leave unchanged).

Step Semantics
--------------

- Intermediate steps: must implement fit and transform.
- Final estimator: only needs to implement fit (and typically predict or predict_proba when used for inference).

Common Patterns
---------------

- Inside a grid search, you can tune parameters for any step:

```python
from sklearn.model_selection import GridSearchCV

pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', LogisticRegression())
])

param_grid = {
    'clf__C': [0.1, 1.0, 10.0],
    'scaler__with_mean': [True, False]
}

grid = GridSearchCV(pipe, param_grid, cv=5)
grid.fit(X, y)
```

- Inspect results and parameters after fitting:

```python
best_params = grid.best_params_
best_pipe = grid.best_estimator_
```

Advanced usage
--------------

- Multiple composites: FeatureUnion and make_union allow parallel application of different feature extraction steps, later combined and fed into an estimator.
- Pipeline compatibility with cross-validation and meta-estimators (e.g., GridSearchCV) enables robust model selection across preprocessing and modeling steps.

Attributes and Notes
--------------------

- named_steps: dictionary-like container of step_name to estimator
- Classes, feature names, and other attributes come from the final estimator when applicable.
- Verbose mode prints progress during fitting of each step when True.

See Also
--------

- make_pipeline: Convenience function for simple, unnamed pipelines.
- make_union and FeatureUnion: Composing features from multiple transformers.
- sklearn.model_selection.GridSearchCV: Hyperparameter tuning across pipeline steps.

Examples
--------

1) Basic pipeline with scaling and logistic regression

```python
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression

pipe = Pipeline([('scaler', StandardScaler()), ('clf', LogisticRegression())])
pipe.fit(X_train, y_train)
score = pipe.score(X_test, y_test)
```

2) Pipeline with memory and parameter tuning via GridSearchCV

```python
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import GridSearchCV

pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('clf', LogisticRegression())
], memory='cache_dir')

param_grid = {'clf__C': [0.1, 1.0, 10.0]}
grid = GridSearchCV(pipe, param_grid, cv=5)
grid.fit(X, y)

best_score = grid.best_score_
best_params = grid.best_params_
```

Notes
-----

- The final estimator in a Pipeline is what you typically use to make predictions.
- You can combine Pipelines with other composite estimators (e.g., FeatureUnion) to build sophisticated preprocessing and modeling workflows.
