Fields
======

Examples and usage
------------------

Overview
--------

The Field class is the base type for all model fields. It defines how Python values are converted to and from their database representations, as well as common metadata (nullability, default values, indexing, primary keys, etc.). Concrete field types are typically created by mixing Field with a Python type, allowing the ORM to infer the Python type from the mixin while still sharing the same core behavior.

Key concepts
- field_type: The Python type for the field (set automatically when using a mixin).
- null, default, pk, unique, index: Core field constraints that influence DDL and runtime behavior.
- to_db_value / to_python_value: Conversion between Python values and DB values.
- Per-dialect overrides: You can customize behavior for specific SQL dialects via inner classes named _db_<dialect>.

Examples
-------

1) Create a simple custom field by mixing Field with a Python type

This pattern lets you define a field that the ORM will treat as a specific Python type, while reusing the common Field logic.

Code (example):
```py
from tortoise.fields import Field

class CharField(Field, str):
    """A simple string field type that uses Field as the base."""

    # Optional per-dialect overrides can go here as an inner class
    class _db_sqlite:
        SQL_TYPE = "VARCHAR(255)"
```

Usage in a model:
```py
from tortoise.models import Model

class User(Model):
    name = CharField(null=False, default="")
```

2) Define a primary key field

Marking a field as a primary key enforces uniqueness and creates the implicit index. If none is specified, Tortoise will generate a default primary key named id.

Code (example):
```py
class User(Model):
    id = CharField(pk=True)  # or use an appropriate integer-based field type
```

3) Non-null fields with defaults and indexing

Example showing nullability, a default value, and an index.

Code (example):
```py
class Product(Model):
    sku = CharField(pk=False, null=False, unique=True, index=True, default="UNKNOWN")
    name = CharField(null=False)
```

4) Per-dialect overrides

You can customize field attributes per database dialect by defining an inner class named _db_<dialect>.

Code (example):
```py
class EmailField(CharField):  # inherit from the CharField example above
    pass

class _db_sqlite:
    SQL_TYPE = "VARCHAR(128)"
    skip_to_python_if_native = False

    def function_cast(self, term):
        # Example of a dialect-specific cast; replace with real logic as needed
        return term
```

5) Dialect-specific value conversion

You can override to_db_value / to_python_value for a given dialect to adjust how values are converted for that DB.

Code (example):
```py
class MyIntField(Field, int):
    pass

class _db_postgresql:
    def to_db_value(self, value, instance):
        # Example: ensure integers are stored as int
        if value is None:
            return None
        return int(value)
```

6) On-delete behavior for relations

The module defines CASCADE, RESTRICT, SET_NULL, SET_DEFAULT constants that are commonly used with relational fields to specify delete behavior.

Usage example:
```py
from tortoise.models import Model
from tortoise.fields import ForeignKeyField
from tortoise.fields import CASCADE

class User(Model):
    id = Field(pk=True)  # or any suitable primary key field

class Post(Model):
    author = ForeignKeyField("models.User", related_name="posts", on_delete=CASCADE)
```

What to configure and verify
----------------------------

- Validation: A field cannot be both non-nullable and non-indexable if indexing is required for a constraint. The base Field enforces consistency at initialization.
- Field attributes: Many attributes (field_type, indexable, has_db_field, skip_to_python_if_native, allows_generated, function_cast, SQL_TYPE, GENERATED_SQL) are intended to be overridden on a per-field basis or per-dialect via inner classes.
- Database value conversions: Use to_db_value and to_python_value to customize how values are stored and retrieved when the default type conversion is not sufficient.

Notes
-----

- The base Field class is designed to be extended; real-world usage typically involves concrete field types provided by the ORM (e.g., CharField, IntegerField) or user-defined mixins like CharField above.
- Per-dialect overrides enable database-specific optimizations or compatibility adjustments without changing application code.
