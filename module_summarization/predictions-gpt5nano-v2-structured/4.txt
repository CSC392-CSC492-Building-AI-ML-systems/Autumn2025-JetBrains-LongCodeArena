Sync users with LDAP
====================

Overview
--------
This documentation describes how to keep your ftrack users synchronized with an LDAP directory. The LDAP sync process provisions and updates user accounts in ftrack based on LDAP data, maps LDAP groups to ftrack groups/roles, and can deactivate users who are no longer present in LDAP.

Key concepts
- Provisioning: Create ftrack users for LDAP identities that exist in your directory but are not yet in ftrack.
- Attribute synchronization: Keep critical user attributes (e.g. username, email, full name) in sync with LDAP.
- Group/role mapping: Align LDAP group membership with corresponding ftrack groups or roles.
- Deactivation: Optionally deactivate ftrack users when they are removed from LDAP (or no longer found during a sync).
- Scheduling: Run sync on a schedule or on-demand to keep data up to date.

Prerequisites
- An LDAP directory reachable from your ftrack server (or the component that performs the sync).
- A bind account with read access to the LDAP directory.
- A mapping plan from LDAP attributes to ftrack user properties (e.g., uid → ftrack username, mail → email).
- Administrative access to configure the LDAP sync in ftrack/server settings.
- If using LDAP groups, a defined strategy for mapping LDAP groups to ftrack groups/roles.

Configuration
---------------
Configure the LDAP synchronization with a config file or environment, depending on your deployment. The example below shows a typical configuration layout in YAML. Adapt paths, server addresses, and attribute mappings to your environment.

Example: ldap_sync.yaml
.. code-block:: yaml

   ldap:
     uri: "ldap://ldap.example.com:389"
     bind_dn: "cn=readonly,dc=example,dc=com"
     bind_password: "secret"
     base_dn: "dc=example,dc=com"
     user_filter: "(objectClass=inetOrgPerson)"
     user_attributes:
       uid: "username"       # LDAP attribute -> ftrack username
       mail: "email"         # LDAP attribute -> ftrack email
       displayName: "full_name" # LDAP attribute -> ftrack full name
     group_attribute: "memberOf"  # LDAP attribute that lists groups
     group_mappings:
       "cn=developers,ou=Groups,dc=example,dc=com": "Developers"
       "cn=qa,ou=Groups,dc=example,dc=com": "QA"
     sync_interval_minutes: 60     # Frequency of automatic sync

Configuration notes
- uri: Use ldap:// for unencrypted or ldaps:// for SSL/TLS. Consider StartTLS if your environment requires it.
- bind_dn / bind_password: Credentials for a read-only bind user. Do not use an admin bind if not necessary.
- base_dn: The subtree to search for users.
- user_filter: LDAP filter to identify user entries. Example: "(objectClass=inetOrgPerson)".
- user_attributes: Mapping from LDAP attributes to ftrack user fields.
- group_attribute and group_mappings: Optional. Define how LDAP groups map to ftrack groups or roles.
- sync_interval_minutes: How often the sync should run automatically.

Usage
-----
Manual sync
- Trigger a one-off synchronization from the ftrack server or your deployment tooling.
- Ensure the LDAP connection credentials are loaded and the mapping configuration is valid.

Example (conceptual)
.. code-block:: python

   from ldap_sync import LDAPSync  # hypothetical module
   from ftrack_api import Session

   # Load configuration from ldap_sync.yaml
   config = load_config("/path/to/ldap_sync.yaml")

   session = Session(
       server_url="https://ftrack.example.com",
       api_key="YOUR_API_KEY",
       api_user="YOUR_API_USER",
   )

   ldap_sync = LDAPSync(session, config)
   ldap_sync.run_sync()

Automated scheduling
- Run the sync at regular intervals (e.g., hourly) using your infrastructure’s scheduling facility (cron, systemd timers, or a task queue).
- Ensure the sync process logs appropriately for auditing and troubleshooting.

CLI usage (if provided by your deployment)
- A command-line tool may be available to perform LDAP sync with a given config.
- Example invocation:
.. code-block:: console

   $ ftrack-ldap-sync --config /path/to/ldap_sync.yaml

Security considerations
- Use a dedicated LDAP read-only bind account for synchronization.
- Do not store plain-text credentials where they are not securely protected.
- Limit the scope of the LDAP search to the necessary base DN.
- Audit logs of user provisioning, updates, and deprovisioning.

Error handling and troubleshooting
- Connection errors: Verify LDAP server reachability, network ACLs, and TLS settings.
- Authentication errors: Validate bind DN and password.
- Attribute mapping mismatches: Confirm LDAP attribute names and their presence on user entries.
- Group mapping mismatches: Ensure LDAP group DNs match the configured group mappings.
- Deprovisioning without authorization: Ensure you have explicit policy to deactivate users and an approval workflow if required.

Best practices
- Start in a read-only test mode where changes are not applied to ftrack to validate your mappings.
- Incrementally provision users before enabling deactivation to avoid accidental removals.
- Regularly review synchronization logs to detect drift or misconfigurations.

API reference (conceptual)
- LDAPSync(session, config): Creates a sync manager bound to a session and the LDAP configuration.
- LDAPSync.run_sync(): Performs a full synchronization cycle: fetches LDAP users, creates/updates ftrack users, applies group mappings, and optionally deactivates missing users.
- Mapping rules: Defined in the configuration under user_attributes and group_mappings; customize to your LDAP schema.
- Scheduling: Optional automatic runs configured via sync_interval_minutes in the config and your deployment’s scheduler.

Notes
-----
- The exact module and class names for the LDAP synchronization may vary with your deployment. Refer to your installation’s extension documentation or contact your administrator for the precise integration module names and commands.
