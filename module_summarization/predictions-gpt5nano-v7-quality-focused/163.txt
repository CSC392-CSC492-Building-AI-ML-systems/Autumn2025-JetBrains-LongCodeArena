Tensor Attributes
=================

Overview
--------

Tensor Attributes describe metadata associated with tensors in the OneFlow MLIR dialect. They enable ops to carry additional information about tensor properties (such as shape, element type, and layout) in a structured, MLIR-native form. Tensor Attributes are defined and wired into the dialect through the OneFlow MLIR build and code-generation flow.

What you will find in this area
------------------------------

- Definitions of tensor-related attributes in the OneFlow MLIR dialect.
- How tensor attributes are generated and wired into the build using MLIR TableGen and CMake integration.
- How to extend the set of tensor attributes by adding new TableGen definitions and rebuilding.

Key concepts
------------

- Tensor Attribute: A metadata container attached to tensors in the dialect, describing properties such as shape and element type, and potentially layout or other tensor-centric metadata.
- TableGen integration: Tensor Attributes are defined in TableGen within the dialect, enabling automatic generation of C++ attribute classes and builders.
- Build integration: Tensor Attributes are compiled as part of the MLIR dialect build when WITH_MLIR is enabled. The build relies on mlir-tblgen and related MLIR tooling.

Declaring tensor attributes
---------------------------

- Define tensor attribute structures in the dialect’s TableGen definitions.
- The build process uses MLIR’s TableGen-based code generation to produce the corresponding C++ attribute classes.
- The generated code is integrated into the OneFlow MLIR library targets via the standard MLIR build flow defined in the CMake configuration.

Using tensor attributes in MLIR
--------------------------------

- Tensor Attributes attach to tensor operands or results of dialect operations.
- You access and manipulate them through the MLIR attribute system in C++ (and through bindings where available).
- Typical uses include querying tensor shape, element type, and layout metadata to drive optimization or code generation passes.

Example (illustrative)
----------------------

Note: The following is a schematic example to illustrate usage in MLIR text format. The exact syntax for your dialect may vary.

- Attaching a tensor attribute to an operation (illustrative):

  // op that consumes a tensor and carries tensor metadata
  oneflow.op_name %arg0 : tensor<2x3xf32> {
    tensor_attr = { shape = [2,3], dtype = f32, layout = "NCHW" }
  }

- Querying a tensor attribute in C++ (illustrative):

  // Pseudocode: replace with actual API names from your dialect
  if (auto ta = op.getAttrOfType<TensorAttr>("tensor_attr")) {
    auto shape = ta.getShape();        // e.g., [2, 3]
    auto dtype = ta.getDtype();        // e.g., f32
    auto layout = ta.getLayout();      // e.g., "NCHW"
  }

Practical notes
--------------

- Tensor Attributes rely on the MLIR infrastructure included with the OneFlow MLIR build. The attributes’ definitions participate in the build when MLIR is enabled (WITH_MLIR).
- The code generation for tensor attributes uses mlir-tblgen and is wired into the dialect via the standard OneFlow MLIR build targets (oneflow_add_mlir_dialect_library and related utilities).
- Tensor Attributes definitions live in the OneFlow MLIR include directory set by ONEFLOW_MLIR_SOURCE_INCLUDE_DIRS during the CMake configure phase.

Build and integration
---------------------

- ENABLE MLIR: Ensure WITH_MLIR is enabled to build the MLIR dialect, including tensor attributes.
- Code generation: The build runs MLIR TableGen to generate attribute classes and builders, which are then compiled into the dialect library.
- RPATH and runtime: The build includes custom RPATH updates for MLIR and OneFlow libraries to ensure runtime resolution of the generated.attributes.

References in the codebase
--------------------------

- The build infrastructure creates and wires:
  - oneflow_add_mlir_library
  - oneflow_add_mlir_dialect_library
  - update_rpath
- Tensor attributes definitions are integrated through the MLIR include directories and TableGen-based code generation as part of the OneFlow MLIR dialect.
- The project’s MLIR-related subdirectories (include, lib, test, oneflow-translate, oneflow-runtime, oneflow-extension, oneflow-opt, oneflow-runner, oneflow-lite) participate in or depend on the tensor attribute definitions when WITH_MLIR is on.
