plans.rst
==========

Overview
--------

This module provides plan generators used to drive data acquisition in Bluesky-like experiments. It includes:

- count: Reads one or more detectors a specified number of times (with optional per-shot customization and delays).
- list_scan: Scans one or more motors over specified point lists in an inner-product fashion, producing a grid of readings.
- rel_list_scan: Intended to scan relative to the current position, but currently not implemented in this code.

API Reference
-------------

count(detectors, num=1, delay=None, *, per_shot=None, md=None)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Take one or more readings from detectors.

Signature
- detectors: sequence of readables (e.g., detectors) to read
- num: int or None, number of readings to take; None captures data until canceled (continuous)
- delay: iterable or scalar, optional; time delay in seconds between successive readings (default 0)
- per_shot: callable, optional; hook that customizes the action of the inner loop (messages per step)
- md: dict, optional; metadata to attach to the plan

Returns
- A generator-based plan; yields messages generated by the inner loop. The plan ends after producing the requested number of readings.

Behavior and details
- If num is None, the plan captures data indefinitely (num_intervals remains None).
- The internal metadata (_md) includes:
  - detectors: list of detector names
  - num_points: num
  - num_intervals: num - 1 (or None when num is None)
  - plan_args: representation of detectors and num
  - plan_name: "count"
  - hints: a dictionary (with default dimensions set to time/primary if not overridden)
- If per_shot is not provided, it defaults to bps.one_shot.
- The plan constructs an inner generator inner_count that yields from bps.repeat(partial(per_shot, detectors), num=num, delay=delay).
- The public plan is returned by yielding from inner_count().

Examples
- Basic usage (5 readings from two detectors with 0.1 s between readings):
  .. code-block:: python

     def my_plan(detectors):
         yield from count(detectors, num=5, delay=0.1)

  Calling as part of a runbook:
  .. code-block:: python

     yield from count([det1, det2], num=5, delay=0.1)

- Using a custom per_shot hook (advanced usage):
  .. code-block:: python

     def my_per_shot(detectors):
         for det in detectors:
             # yield a Msg or other plan-specific message per detector
             yield Msg('read', det.name)

     yield from count([det1, det2], num=10, per_shot=my_per_shot)

Notes
- The per_shot hook must conform to the signature def f(detectors) -> Generator[Msg].
- The plan name in metadata is "count".

list_scan(detectors, *args, per_step=None, md=None)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Scan over one or more variables in steps simultaneously (inner product).

Signature
- detectors: list of readables (detectors)
- *args: pairs of (motor, [points]) for each dimension. Each motor is a settable object; each points list contains positions to visit for that motor.
- per_step: callable, optional; hook that customizes the action of the inner loop (messages per step)
- md: dict, optional; metadata to attach to the plan

Notes on arguments
- The number of positional arguments after detectors must be even (motor, [points]) pairs. If not, a ValueError is raised.
- All point lists must have the same length; otherwise a ValueError is raised.
- Each motor’s pos_list is converted to a finite Python list to ensure consistency.

Internal metadata and structure
- Detectors meta: detectors: [det.name for det in detectors]
- Motors meta: motors: [motor names]
- num_points: number of points per motor (length of position lists)
- num_intervals: length - 1
- plan_args: includes string representations of detectors and args, plus per_step
- plan_name: "list_scan"
- plan_pattern: "inner_list_product"
- plan_pattern_module: plan_patterns.__name__
- plan_pattern_args: {'args': md_args}
- hints: may include default dimensions derived from motor hints

Dynamic dimension hints
- The plan collects x_fields from each motor’s hints.fields (if present) and builds a default dimension hint:
  - dimensions: [(x_fields, 'primary')] when there are any fields
- The provided md hints are merged with these defaults (md provided by the caller overrides as appropriate)

Construction flow
- Builds a flat md_args list from the (motor, points) pairs
- Validates lengths of all point lists are equal
- Builds full_cycler via plan_patterns.inner_list_product(args)
- Delegates execution to scan_nd(detectors, full_cycler, per_step=per_step, md=_md)

Return value
- A generator-based plan that yields from scan_nd with the prepared cycler and metadata.

Examples
- Simple 1D scan with a single motor:
  .. code-block:: python

     yield from list_scan([det1], motor1, [0, 1, 2, 3], md={'plan_name': 'my_list_scan'})

- 2D scan over two motors:
  .. code-block:: python

     yield from list_scan([det1], motorA, [0, 1, 2], motorB, [10, 20, 30])

- With a per_step hook:
  .. code-block:: python

     def per_step(detectors, motor, step):
         # customize per-step behavior
         yield Msg('set', motor, step)

     yield from list_scan([det1, det2], motor1, [0, 1], motor2, [0, 5], per_step=per_step)

Notes
- The function references plan_patterns.inner_list_product to generate the inner-product cycler.
- The default hints mechanism helps describe the expected dimensions when motors expose fields in their hints.

rel_list_scan(detectors, *args, per_step=None, md=None)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Scan over one variable in steps relative to the current position.

Parameters
- detectors: list of readables (detectors)
- *args: pairs of (motor, [points]) where points are relative to the current location
- per_step: callable, optional; hook for customizing action of inner loop (messages per step)
- md: dict, optional; metadata

Notes
- This function is declared with a placeholder implementation (TODO) in the current codebase.
- The intended behavior is to perform a scan where the specified points are offsets from the motor’s current position, rather than absolute positions, and to yield a plan consistent with list_scan/rel_list_scan patterns.
- See Also: list_scan, list_grid_scan, rel_list_grid_scan

Example (intended usage)
- As implemented, this function is not yet available for execution. When implemented, usage would resemble:
  .. code-block:: python

     # Intended usage (example sketch)
     yield from rel_list_scan([det1], motor1, [0, 1, -1], per_step=some_hook)

Caveats
- rel_list_scan currently raises a NotImplementedError or remains unfunctional due to the TODO in the codebase. Users should monitor updates for a full implementation.

Appendix: Related Utilities
---------------------------

- plan_patterns: Provides plan patterns used to compose complex scans (e.g., inner_list_product).
- bpp (preprocessors): Provides decorators like stage_decorator and run_decorator to wrap plan stages and attach metadata.
- bps (plan_stubs): Supplies plan stubs such as one_shot used by count to implement per-shot behavior.
- Msg: Message type used to communicate plan actions within the Bluesky framework.
