Review Sessions
===============

Using review sessions
---------------------

A review session is an isolated interaction with an ftrack server represented by a Session object. It encapsulates authentication, caching, event hub connectivity, and other user-specific configurations needed when reviewing and manipulating ftrack data. This guide covers how to configure and use review sessions effectively.

Overview
--------

- A session connects to a single ftrack server URL and authenticates with an API key and API user.
- If not supplied, server URL and credentials can be read from environment variables:
  - SERVER: FTRACK_SERVER
  - API key: FTRACK_API_KEY (or FTRACK_APIKEY for backwards compatibility)
  - API user: FTRACK_API_USER (or fall back to the system user via getpass)
- Sessions support automatic population of entity data on attribute access, a layered in-process cache, and optional background event hub connectivity for real-time updates.
- You can customize the behavior via several parameters, including caching, schema caching, and plugin discovery.

Quick start
-----------

Create a basic review session using environment defaults where possible:

.. code-block:: python

   import ftrack_api

   session = ftrack_api.Session(
       server_url='https://ftrack.example.com',
       api_key='your_api_key',
       api_user='your_username'
   )

Accessing entities will auto-populate data if auto_populate is enabled (default).

.. code-block:: python

   # Example: access an entity attribute; data is fetched on demand if missing
   # (actual entity usage depends on subsequent queries and operations)
   # entity = session.query(...).one()
   # print(entity.name)

Key concepts and defaults
-------------------------

- Required parameters
  - server_url: URL of the ftrack server (including port if needed)
  - api_key: API key for authentication
  - api_user: User name in ftrack

- Environment defaults
  - If server_url is not provided, the environment variable FTRACK_SERVER is used.
  - If api_key is not provided, FTRACK_API_KEY or FTRACK_APIKEY is used.
  - If api_user is not provided, FTRACK_API_USER is used, otherwise the system user is attempted via getpass.

- Validation
  - If any required value is missing, a TypeError is raised indicating the missing parameter.

- Core features created by the session
  - A recorded_operations collection for pending operations (session.recorded_operations)
  - A flag to indicate whether to record operations (session.record_operations)
  - A cache_key_maker, defaulting to a StringKeyMaker if not provided
  - A pre-configured layered cache with a MemoryCache at the top level

Configuration options
---------------------

You can customize the session using the following parameters:

- server_url: ftrack server URL (required)
- api_key: API key for authentication (required)
- api_user: User name to perform operations as (required)
- auto_populate: Enable automatic data population on attribute access (default: True)
- plugin_paths: List of paths to search for plugins (default reads FTRACK_EVENT_PLUGIN_PATH)
- cache: Cache instance conforming to ftrack_api.cache.Cache, or a callable that returns a cache or None
- cache_key_maker: Instance implementing KeyMaker for cache keys (default: StringKeyMaker)
- auto_connect_event_hub: Automatically connect to the event hub in the background (default: None implies True)
- schema_cache_path: Path to a folder for schema caching. Set to False to disable caching entirely. If not provided, FTRACK_API_SCHEMA_CACHE_PATH is used, or a temporary directory is used.
- plugin_arguments: Optional mapping of keyword arguments passed to plugin registration functions

Notes
-----

- The session’s event hub connection is performed in a background thread to improve startup time. If a plugin requires a connected hub, it should check the hub status explicitly.
- Subscribing to events does not require a connected hub; publishing/subscribing to non-local events can be enabled via the hub.
- The top-level cache is designed to be MemoryCache, with the session cache integrated into a layered approach.

Examples
--------

1) Full-featured session with environment-driven defaults

.. code-block:: python

   import ftrack_api
   import os

   session = ftrack_api.Session(
       server_url=os.environ.get('FTRACK_SERVER'),
       api_key=os.environ.get('FTRACK_API_KEY'),
       api_user=os.environ.get('FTRACK_API_USER'),
       auto_populate=True,
       schema_cache_path='/path/to/schema/cache',
       plugin_paths=['/path/to/plugins'],
       plugin_arguments={'argA': 'valueA'}
   )

2) Session with custom cache and key maker

.. code-block:: python

   from ftrack_api.cache import MemoryCache, StringKeyMaker
   from ftrack_api import Session

   cache = MemoryCache()
   key_maker = StringKeyMaker()

   session = Session(
       server_url='https://ftrack.example.com',
       api_key='your_api_key',
       api_user='your_username',
       cache=cache,
       cache_key_maker=key_maker,
       auto_populate=True
   )

3) Inspecting pending operations

.. code-block:: python

   # Access the recorded operations container
   ops = session.recorded_operations
   print(type(ops))
   # You can inspect or extend operations before you commit them to the server

Lifecycle and cleanup
---------------------

- Sessions raise TypeError if required connection details are missing.
- Sessions maintain an internal closed flag; ensure proper cleanup in application shutdown.
- When you no longer need a session, let it go out of scope or implement explicit cleanup as per your application’s lifecycle.

API reference (public attributes relevant to review workflows)
--------------------------------------------------------------

- server_url: The server URL used to connect.
- api_key: API key for authentication.
- api_user: User performing operations.
- auto_populate: Determines whether attributes trigger on-demand fetching.
- recorded_operations: Container for operations performed through the session.
- record_operations: Flag indicating whether to record operations.
- cache_key_maker: Tool used to generate cache keys for objects stored in the cache.
- (Additional internal attributes related to caching, event hub, and plugin handling are available through the session but are not required for basic review tasks.)

Troubleshooting
---------------

- TypeError: Missing required configuration. Ensure server_url, api_key, and api_user are provided or available via environment variables.
- Environment variables are set but values are not used. Verify precedence and that the expected environment variables are exported in the running process.
