g-functions
===========

Overview
--------

This document describes the module that generates a documentation string for g-functions at import time. The string is intended for use in Sphinx as the module’s documentation and is rendered with math directives.

What it documents
------------------

- Elementary functions (header produced when the category key is empty).
- Functions involving other symbols (header includes the function names extracted from the data).

Generation happens automatically when the module is imported. The resulting text is stored in the module’s __doc__ attribute and can be included in Sphinx via standard mechanisms (for example, automodule).

How the doc is generated
------------------------

- A name-to-entries mapping t is created as a defaultdict(list).
- The table _create_lookup_table(t) fills t with data describing formulas, their generated forms, conditions, and hints.
- The data is sorted by a stable sort key (default_sort_key) to produce a deterministic order.
- For each entry category:
  - If the category key is empty, the header “Elementary functions” is emitted.
  - Otherwise, the header “Functions involving …” is emitted. The list of function names is obtained by inspecting the first item of each category and extracting the function symbols from their atoms, then rendering each with LaTeX.
- Each entry within a category yields a math directive:
  - If gs (the generated structure) is not a list, g is set to the symbolic placeholder \text{generated}.
  - If gs is a list, g is built as Add(*[fac*f for (fac, f) in gs]), i.e., a sum of products of factors and functions.
  - obj becomes the equation formula = g.
  - cond is either empty (if there is no condition) or a LaTeX-rendered condition prefixed by “, \text{ if }”.
  - The doc line for the entry is a math block containing the LaTeX of obj, plus the conditional suffix if present.

Data structures
---------------

- t: A dict mapping a tuple of types to a list of entries. Each entry is a 4-tuple:
  - formula: The left-hand side of the equation to be documented.
  - gs: Either a list of pairs (fac, f) describing a generated form, or a non-list value signaling a generated form is not specified.
  - cond: A condition under which the formula holds; may be True (no condition) or a LaTeX-friendly predicate.
  - hint: Extra information (not directly used in rendering).

- Rendering details:
  - If gs is a list, g is Add(*[fac*f for (fac, f) in gs]); otherwise g is Symbol('\text{generated}').
  - obj = Eq(formula, g).
  - cond is transformed to a textual suffix only when it is not True.

Output format
-------------

The generated documentation uses reStructuredText with embedded Sphinx math directives:

- Headers separate sections of the generated documentation.
- Each formula is rendered as a math block:
  .. math::
    <LaTeX representation of Eq(formula, g)>[, \text{ if } <LaTeX(cond)>]

- If there is no condition, the math block ends after the equation.

Illustrative example (representative, not actual data)
-----------------------------------------------------

Elementary functions:

.. math::
  f(x) = x

Functions involving \emph{sin} and \emph{cos}:

.. math::
  \sin(x) = \sin(x),\ \text{ if } x \neq 0

Note: The exact content depends on the data produced by _create_lookup_table and the formulas collected there.

API and usage
--------------

- The module exposes its generated documentation via its __doc__ attribute.
- In Sphinx, you typically include this content by documenting the module (for example, using automodule) so that the generated doc appears under g-functions.
- To regenerate the content during development, import the module (e.g., by importing the package) so that _create_lookup_table(t) runs and sets __doc__ accordingly.

Practical tips
--------------

- If you modify the underlying data source for the lookup table (via _create_lookup_table or related code), re-import the module to update __doc__.
- The module uses LaTeX rendering for function symbols and conditions, which Sphinx can render with the math directive.
- The "Elementary functions" header provides a quick entry point for readers to identify base cases before exploring more complex combinations.

Notes
-----

- The documentation content is generated at import time and stored in __doc__.
- The structure emphasizes readability and consistency for Sphinx-based documentation workflows.
