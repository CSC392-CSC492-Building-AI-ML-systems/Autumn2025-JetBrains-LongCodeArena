# Querying

This section describes how querying works in the ftrack API client and how to configure a Session for efficient data retrieval.

Overview
- Querying is performed through the Session, which handles server connection, authentication, caching, and on-demand population of entity data.
- Accessing entity attributes can trigger server requests if the data is not yet loaded (auto-population).
- Queries leverage a layered caching strategy to improve performance and reduce repeated requests.

Getting started: creating a session
- A Session represents an isolated connection to an ftrack server. It requires the server URL, API key, and API user.
- If you do not provide these values explicitly, the client will try to read them from environment variables; otherwise, you must supply them.

Example: create a session (credentials provided directly)
- This example shows the typical session initialization. Adjust parameters as needed for your environment.

- from ftrack_api import Session
- session = Session(
-     server_url='https://my-ftrack-server',
-     api_key='my_api_key',
-     api_user='my_user'
- )

Important session configuration options
- server_url: The ftrack server URL, including port if needed. If omitted, the client reads FTRACK_SERVER from the environment.
- api_key: The API key used for authentication. If omitted, the client reads FTRACK_API_KEY or FTRACK_APIKEY from the environment.
- api_user: The user name for operations. If omitted, the client reads FTRACK_API_USER from the environment or uses the current system user via getpass when available.
- auto_populate: When True (default), accessing an entity attribute automatically fetches missing data from the server. Set to False to disable on-demand loading.
- plugin_paths: List of paths to search for plugins. If not set, the system falls back to environment-based plugin discovery.
- cache: A cache instance or a callable that returns a cache. The provided cache is integrated into a layered cache with a MemoryCache at the top. This makes it unnecessary to construct a separate memory cache for typical use.
- cache_key_maker: Object that generates cache keys. Defaults to StringKeyMaker if not provided.
- auto_connect_event_hub: If True, the embedded event hub is connected automatically to the event server, enabling publishing and subscribing to non-local events. If False, only local events can be published/subscribed until you connect the hub manually.
- schema_cache_path: Path to a folder used for schema caching. If not set, the environment variable FTRACK_API_SCHEMA_CACHE_PATH is used; if that is also not set, a temporary directory is used. Set to False to disable schema caching.
- plugin_arguments: Optional mapping of keyword arguments passed to plugin register functions. If a plugin’s signature does not accept a supplied argument, the discovery mechanism reduces the arguments to those that are accepted and logs a warning.

Authentication and request headers
- Requests sent to the server include authentication headers provided by SessionAuthentication.
- The SessionAuthentication class attaches the following headers to every request:
  - ftrack-api-key: your API key
  - ftrack-user: your API user

Caching and performance considerations
- The Session uses a layered cache strategy, with a MemoryCache at the top level.
- A user-supplied cache can be integrated into this layered cache to improve performance and reduce round-trips to the server.
- Schema caching can significantly speed up repeated queries by avoiding repeated schema lookups.

Querying workflow (conceptual)
- Create a Session with appropriate credentials and optional performance optimizations (e.g., caching, schema caching, auto-populated loading).
- Use the ftrack_api.query module to construct and execute queries against the connected server.
- Access to attributes may trigger lazy loading if auto_populate is enabled, reducing initial data transfer but potentially causing subsequent requests when data is accessed.

Example: enabling schema caching and lazy loading
- session = Session(
-     server_url='https://my-ftrack-server',
-     api_key='my_api_key',
-     api_user='my_user',
-     schema_cache_path='/path/to/schema/cache',
-     auto_populate=True
- )
- # Accessing an entity attribute may fetch data on demand if not cached.

Practical tips
- Prefer enabling schema caching in environments with frequent queries to reduce overhead.
- Use a suitable cache implementation to minimize server requests for repeated queries.
- If your application only requires local event handling, consider setting auto_connect_event_hub to False and connecting manually when needed.

Common pitfalls
- Missing credentials: Ensure server_url, api_key, and api_user are provided or available via environment variables.
- Disabling auto_populate: If you rely on lazy loading, ensure the code handles potential on-demand loading delays.
- Misconfigured cache: Incompatible or misconfigured cache backends can degrade performance or lead to stale data; verify cache integration when supplying a custom cache.

References to core components
- Session: Central class for server interaction, authentication, caching, and query-related configurations.
- SessionAuthentication: HTTP header injection for ftrack-api-key and ftrack-user.
- ftrack_api.query: Module responsible for building and executing queries against the server.
- ftrack_api.cache: Caching infrastructure used by the session (including MemoryCache and StringKeyMaker).

This section centers on how to configure and use a Session for querying in a practical, production-oriented way, leveraging automatic data loading, caching, and optional schema caching to optimize data retrieval.
