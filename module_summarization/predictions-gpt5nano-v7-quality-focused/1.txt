Using custom attributes
=======================

This document explains how to work with custom attributes in the ftrack API. Custom attributes are user-defined metadata fields that you attach to entities on the server. They appear as typed fields defined for a given entity type and are accessible through the API via the attribute subsystem.

Overview
--------
- Custom attributes are defined on the server for an entity type (for example, a Task, Shot, or Asset).
- The ftrack_api.attribute module exposes API surface for discovering attribute definitions and working with attribute values on entities.
- Accessing or mutating a custom attribute value causes the client to interact with the server as needed, respecting the attribute’s type (string, integer, boolean, date/time, selections, etc.).
- If auto_populate is enabled on the session, attribute values may be lazily loaded when you access them.

Working with custom attribute definitions
-----------------------------------------
To interact with custom attributes, you typically begin by obtaining the definitions for the relevant entity type. Attribute definitions describe the available custom attributes (code/name, type, possible values, etc.).

- Retrieve definitions for an entity type (conceptual example):
  - Use the session’s attribute-related interface to query or fetch definitions for the target entity type.
  - Each definition provides metadata such as the attribute code, display name, and type.

Code example (conceptual):
.. code-block:: python

    # Obtain an attribute definition for a given entity type and attribute code
    # (Exact API surface may vary; refer to the ftrack_api.attribute module)
    definition = session.attribute.get_definition(entity_type='Task', code='priority')

- Inspect available attributes:
  - You can enumerate definitions and inspect their codes and types to understand what attributes exist on the entity type.

Code example (conceptual):
.. code-block:: python

    for defn in session.attribute.list_definitions(entity_type='Task'):
        print(defn.code, defn.name, defn.type)

Reading custom attribute values
-------------------------------
Once an entity (for example, a Task) is loaded, you can read the value of its custom attributes.

Code example (conceptual):
.. code-block:: python

    task = session.query('Task', id=1234)
    # Read a custom attribute value by its code
    value = task.get('priority')  # returns a Python value appropriate to the attribute type

Notes:
- The returned value type depends on the attribute’s defined type (string, integer, float, boolean, date/time, etc.).
- If the value has not been populated yet and auto_populate is enabled, the value may be loaded from the server on access.

Writing custom attribute values
-------------------------------
To update a custom attribute, assign a new value to the attribute on the entity and persist the change.

Code example (conceptual):
.. code-block:: python

    task = session.query('Task', id=1234)
    task.set('priority', 'High')
    session.commit()  # Persist changes to the server

Notes:
- The value you assign must conform to the attribute’s defined type.
- Depending on the server and client configuration, you may need to commit or flush changes to persist updates.

Working with attribute values types and constraints
---------------------------------------------------
Custom attribute definitions declare the expected type and, in some cases, constraints (for example, a finite set of allowed values or a date/time range).

Guidance:
- Always validate values against the attribute type before setting.
- For selection-type attributes, provide one of the allowed values.
- For date/time attributes, ensure correct timezone handling if applicable.

Example: updating a date attribute
.. code-block:: python

    task = session.query('Task', id=1234)
    task.set('due_date', datetime.date(2025, 12, 31))
    session.commit()

Example: reading a boolean attribute
.. code-block:: python

    task = session.query('Task', id=1234)
    archived = task.get('archived')
    if archived:
        ...

Practical considerations
------------------------
- Server-defined attributes: Custom attributes are defined on the server. The client reflects these definitions and exposes values on entities accordingly.
- Lazy loading: If a custom attribute value is not yet loaded, accessing it may trigger a lazy load from the server, depending on session configuration (notably auto_populate).
- Persistence: To persist changes to custom attributes, you typically commit the session or issue a save/flush operation as provided by the API in your environment.
- Migration and compatibility: When upgrading server versions, verify that custom attribute definitions remain compatible and that attribute codes remain stable.

Examples
--------
- Read and print a task’s priority custom attribute:
  .. code-block:: python

      task = session.find_one('Task', id=1234)
      priority = task.get('priority')
      print('Priority:', priority)

- Set a new value for a custom attribute and save:
  .. code-block:: python

      task = session.find_one('Task', id=1234)
      task.set('priority', 'Medium')
      session.commit()
