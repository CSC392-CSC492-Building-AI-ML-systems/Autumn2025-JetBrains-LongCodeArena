Computational Performance
=========================

Overview
--------

The multi-output estimators in this module (MultiOutputRegressor, MultiOutputClassifier, ClassifierChain, RegressorChain) wrap a single base estimator to handle multiple targets. Performance depends on the cost of fitting the base estimator, the number of targets, and how well the work can be parallelized. The implementation relies on parallel utilities to distribute work across CPUs when feasible.

Parallelism and Concurrency
---------------------------

- The meta-estimators expose an n_jobs parameter (or rely on the base estimator’s parallel capabilities) to control parallel execution.
- When n_jobs > 1, fitting across the multiple outputs can run in parallel, using the Parallel and delayed utilities from the utilities module.
- The degree of speedup depends on:
  - The base estimator’s training cost
  - The number of outputs (targets)
  - Overhead from cloning estimators and inter-process communication
- Do not set n_jobs to a very large number; overhead and memory usage may offset benefits on smaller machines.

Cloning and Fit Cost
--------------------

- Fitting uses a clone of the base estimator to avoid mutating the original instance.
- The function _fit_estimator performs:
  - Clone(estimator)
  - Fit on X and y (with optional sample_weight)
- Cloning incurs overhead proportional to the size of the estimator’s parameter state. For lightweight estimators, this cost is negligible; for heavyweight models, it contributes to total training time and memory usage per target.

Partial Fit and Incremental Learning
------------------------------------

- The helper _partial_fit_estimator supports incremental learning via partial_fit.
- First-time calls clone the estimator to avoid reusing state from previous fits.
- Partial fitting can reduce peak memory usage and enable online or chunked learning on large datasets.

Internal Helpers and Rules
--------------------------

- _available_if_estimator_has returns a conditionally exposed parameter based on whether sub-estimators implement certain methods or attributes.
- This mechanism lets the interface adapt to the capabilities of the wrapped estimator without incurring runtime penalties in typical use.

Data and Dependencies
---------------------

- The module relies on NumPy and SciPy (and uses sparse representations when applicable).
- Performance is influenced by data layout (dense vs. sparse) and memory footprint of the base estimator.

Practical Guidelines
--------------------

- Prefer base estimators that support parallelism or are thread-safe if you plan to use n_jobs > 1.
- For large-scale problems, enable parallelism to utilize multiple cores, but monitor memory usage since cloning multiple estimators increases memory overhead.
- If incremental learning is suitable, use partial_fit to manage memory and potentially improve throughput on streaming data.

Examples
--------

Example 1: Basic usage with parallel fitting

.. code-block:: python

   from sklearn.multioutput import MultiOutputRegressor
   from sklearn.ensemble import RandomForestRegressor
   from sklearn.datasets import make_regression

   X, y = make_regression(n_samples=1000, n_features=20, n_targets=5, random_state=0)
   base = RandomForestRegressor(n_estimators=100, random_state=0, n_jobs=2)
   mor = MultiOutputRegressor(base, n_jobs=2)
   mor.fit(X, y)

Example 2: Incremental fitting path with partial_fit

.. code-block:: python

   import numpy as np
   from sklearn.linear_model import SGDRegressor
   from sklearn.multioutput import MultiOutputRegressor

   X = np.random.randn(1000, 10)
   y = np.random.randn(1000, 3)

   base = SGDRegressor(max_iter=1000, tol=1e-3)
   mor = MultiOutputRegressor(base)
   mor.partial_fit(X, y)  # suitable when the base estimator supports partial_fit

Notes
-----

- The documented behavior reflects the design of the internal helpers that enable parallelism, cloning, and incremental fitting in multi-output estimators.
