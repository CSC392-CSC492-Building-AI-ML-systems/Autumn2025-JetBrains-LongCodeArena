Internal organization of NumPy arrays
=====================================

Overview
--------
NumPy arrays (ndarray) are the central data structure for numerical computing in NumPy. The internal organization combines a well-defined in-memory layout with a rich type system that describes element types, shapes, and memory strides. The code sample shown in internals/code-explanations highlights how internal typing, low-level interfaces, and array metadata come together to support efficient array storage and operations.

Core components
--------------
- ndarray: The primary array container that holds a pointer to a contiguous or strided data buffer, along with metadata describing shape, strides, and dtype.
- dtype: Describes the type of each element stored in the array, including scalar types (integers, floating points, booleans) and structured/void types. The internal representation exposes many primitive type aliases (e.g., bool_, byte, short, int_, longlong, ubyte, ushort, uint, ulonglong, single, double, longdouble, void) to model a wide range of element kinds.
- flagsobj: Manages array flags that describe contiguity, alignment, writeability, and ownership of the underlying data buffer.
- _internal.ctypes: Bridges Python-level NumPy objects with C types, enabling low-level interoperability required by the runtime.
- NDArray and _ArrayLike: Type-level aliases used by the internal typing system to express array shapes and inputs in a statically informative way.

Typing and type variables
-------------------------
The internal type system uses Python typing to describe generic array behavior and element types:
- _DType: A TypeVar bound to dtype[Any], parameterizing generic array types by element dtype.
- _DTypeOptional: A TypeVar bound to None or dtype[Any], enabling optional dtype specialization.
- _SCT: A TypeVar bound to generic, used to describe generic container or subclass relationships in the internal API.

These type variables appear in the code to support static typing and to guide tools (such as mypy) in understanding how NumPy arrays are parameterized by element type and shape.

Key internal imports related to arrays
--------------------------------------
The code imports a broad set of symbols from NumPy’s core to model array storage and typing:
- ndarray, dtype, generic, bool_, byte, short, intc, int_, longlong, ubyte, ushort, uintc, uint, ulonglong, single, double, longdouble, void
  These provide concrete representations of element types and the base array type.
- flagsobj: Manages array flags that reflect memory layout and mutability properties.
- _ctypes: Provides plumbing for C data types used in low-level interop.

Representative internal typing and module structure
---------------------------------------------------
- NDArray and _ArrayLike: Internal aliases describing array objects and their input forms.
- _ShapeLike: Internal binding for shapes (tuples of dimensions) used across APIs.
- _DTypeLike and _VoidDTypeLike: Internal aliases representing allowable dtypes and void-like dtypes.
- _BoolCodes, _UByteCodes, _UShortCodes, _UIntCCodes, _UIntCodes, _ULongLongCodes, _ByteCodes, _ShortCodes, _IntCCodes, _IntCodes, _LongLongCodes, _SingleCodes, _DoubleCodes, _LongDoubleCodes: Internal code mappings for the various element types, facilitating compact representations and efficient internal dispatch.

Note on build-time and platform specifics
------------------------------------------
- A mypy plugin is used to ensure the correct platform-specific ctypes sub-type is imported, reflecting the close coupling between Python-level array objects and their C implementations.
- The internal typing and type aliases help the runtime dispatch algorithms select appropriate code paths for different dtypes and shapes.

Practical implications for developers
-------------------------------------
- When designing or using internal APIs, rely on NDArray and _ArrayLike to represent array inputs and outputs.
- Dtype-driven dispatch: Many internal routines specialize behavior based on the element type, leveraging the extensive set of primitive type aliases.
- Contiguity and memory layout: Access to shape and strides enables understanding of how elements are laid out in memory and how slices or views affect the underlying data.
- Type-safe interfaces: The internal _DType and related TypeVars enable more precise typing for functions that operate on arrays with specific element types.

Examples
--------
- Inspecting an array’s core metadata:
  .. code-block:: python

     import numpy as np

     a = np.arange(6, dtype=np.float64)
     print("shape:", a.shape)        # (6,)
     print("strides:", a.strides)    # (8,)
     print("dtype:", a.dtype)        # float64
     print("flags:", a.flags)        # e.g., C_CONTIGUOUS, WRITEABLE

- Typing example (conceptual, for internal tooling):
  .. code-block:: python

     from numpy import dtype
     from numpy._typing import NDArray

     _DType = TypeVar("_DType", bound=dtype[Any])
     _DTypeOptional = TypeVar("_DTypeOptional", bound=None | dtype[Any])

     def sum_float64(arr: NDArray[np.float64]) -> float:
         return float(arr.sum())
