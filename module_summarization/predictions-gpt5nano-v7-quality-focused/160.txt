Compatibility Policy
====================

Scope
-----
This policy defines the compatibility targets for the npyrandom integration used by professional developers. It covers:
- Public Python API stability for numpy.random-related modules
- Binary/ABI compatibility with the NumPy C-API
- Build, platform, and dependency requirements
- Testing, validation, and release practices
- Packaging and distribution of the compiled components

Key architectural clues from the implementation
- The npyrandom static library is built from a dedicated set of C sources (logfactorial, distributions, random_mvhg_*.c, etc.) and installed under np_dir/random/lib.
- Cython extensions for numpy.random are built and wired to a central static library (npyrandom) and optional helper libraries (npymath_lib).
- Several cython-based modules are produced, including _bounded_integers, _common, _mt19937, _philox, _pcg64, _sfc64, bit_generator, and generator wiring.
- A custom path handles a generated .pxd (_bounded_integers.pxd) via a templated input, and a temporary target is used to ensure installation of the .pxd file.
- Build-time flags include NPY_NO_DEPRECATED_API=0 to preserve compatibility with older NumPy C-API usage, and a platform-specific handling for Cygwin to export symbols.
- Legacy support is activated for mtrand via a build flag (NP_RANDOM_LEGACY=1).
- Tests and test data are shipped and installed to verify compatibility (numpy/random/tests and numpy/random/tests/data).

Public API compatibility
------------------------
- The Python API surface exposed under numpy.random remains stable across minor releases. New features may be added, but existing public functions, classes, and modules should remain callable and behavior-consistent.
- Deprecations follow the project’s standard lifecycle: announce in release notes, provide a deprecation period, and preserve behavior long enough for users to migrate.

Binary/ABI compatibility with NumPy
-----------------------------------
- Extensions are built against the NumPy C-API. The policy aims to maintain compatibility across supported NumPy versions without requiring upstream changes to downstream code.
- The build uses NPY_NO_DEPRECATED_API=0 to maintain compatibility with the older NumPy C-API surface, ensuring existing code paths continue to function.
- The presence of a static library (npyrandom) and explicit linking to np_core_dep helps maintain a consistent runtime environment across builds.

Build, Platform, and Runtime compatibility
-------------------------------------------
- Platforms: typical Linux, macOS, and Windows environments. Special handling exists for Cygwin (export-all-symbols) to preserve binary compatibility on that platform.
- Build system: Meson-based workflow, with explicit install paths (e.g., install_dir: np_dir / 'random/lib' and install of Python sources under numpy/random).
- Runtime dependencies: Python, NumPy core, and the NumPy math library as needed. The build wires into these components via py_dep, np_core_dep, and npymath_lib.
- Data and tests: A comprehensive test suite and test data are installed under numpy/random/tests to validate compatibility across platforms and versions.

Deprecations and change management
-----------------------------------
- Breaking changes to the public API are avoided where possible. When changes are required, they follow a deprecation-and-mmigration pathway documented in release notes.
- Internal changes that affect the C-API or binary compatibility are covered by the NumPy C-API policy and validated with the included tests.

Testing and verification
------------------------
- A dedicated test suite resides in numpy/random/tests, with additional data under numpy/random/tests/data to exercise numerical consistency and compatibility across implementations.
- Verification steps typically include building the project, compiling all extensions, and running the test suite to ensure no regressions.

Distribution and packaging notes
-------------------------------
- The compiled npyrandom library is installed to np_dir/random/lib.
- Python sources, stubs, tests, examples, and licenses are installed under numpy/random, including test data and example code for CFFI, Cyton, and Numba bridges.
- The build ensures generated .pxd and .pyx files are installed where required to support the public API surface and maintain compatibility with downstream usage.

Migration guidance
-------------------
- When adopting newer NumPy versions or Python runtimes, rebuild extensions to ensure binary compatibility with the current C-API surface.
- If a breaking API change is necessary, communicate in release notes, provide a migration path, and maintain deprecation support for at least one release cycle where feasible.

How to verify compatibility locally
-----------------------------------
- Build: run the project’s Meson-based build
  - meson setup build
  - meson compile -C build
- Run tests:
  - meson test -C build --section numpy/random
  - Or invoke the project’s standard Python test runner against the installed package
- Validate packaging:
  - Inspect install directories (np_dir/random/lib for the static library; numpy/random for Python modules and tests)

Appendix: Quick reference
- Static library: npyrandom (installed under numpy/random/lib)
- Core dependencies: py_dep, np_core_dep
- Optional math library: npymath_lib
- Platform nuance: cygwin builds export all symbols to maintain compatibility
- Legacy support: NP_RANDOM_LEGACY=1 enables legacy mtrand behavior

This compatibility policy aligns with the codebase’s approach to maintaining a stable public API, ensuring binary compatibility with the NumPy C-API, and validating correctness through a comprehensive test suite and data sets.
