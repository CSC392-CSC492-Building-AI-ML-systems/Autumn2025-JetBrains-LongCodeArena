Image HDU Data
==============

Overview
--------
The Data component of an Image HDU contains the actual pixel values of the image. In Astropy FITS handling, this data is exposed as a NumPy ndarray and is aligned with the header's NAXIS and NAXISn keywords describing the image geometry. The typical ImageHDU stores 2-D image data, but the underlying machinery supports higher-dimensional data if needed.

Data attribute
--------------
- hdu.data returns a NumPy ndarray representing the image pixel values.
- The array shape corresponds to the image dimensions (for a standard 2-D image, (NAXIS2, NAXIS1)).
- The dtype reflects the pixel data type used in the FITS file (commonly uint8, int16, uint16, int32, float32, float64, etc.).

Accessing and mutating data
---------------------------
- Access a pixel value:
  - value = hdu.data[y, x]
- Modify a pixel (in memory; requires write access to the FITS file if you plan to save changes):
  - hdu.data[y, x] = new_value
- For large images, you may enable memory-mapped access when opening the FITS file to avoid loading the entire array into memory.

Example
-------
- Inspect image shape and dtype:
  .. code-block:: python
     from astropy.io import fits
     with fits.open('image.fits') as hdul:
         hdu = hdul[0]
         print(hdu.data.shape)
         print(hdu.data.dtype)

- Modify a pixel (in memory):
  .. code-block:: python
     hdu.data[100, 50] = 0

- Use memory-mapped access for large files:
  .. code-block:: python
     with fits.open('image.fits', mode='readonly') as hdul:
         mmaped = hdul[0].data  # often memory-mapped internally

Data types and validation
-------------------------
- The Data array holds numeric pixel values. The FITS header (NAXIS, NAXIS1, NAXIS2, …) defines the image geometry and data type.
- When writing, ensure the data type and shape are consistent with the header to avoid corruption.

Verification and reporting
--------------------------
Astropy provides an internal verification framework used to check and report issues in the HDU contents, including the Data component.
- Verification options (examples):
  - ignore, warn, exception
  - fix, silentfix
  - fix+warn, fix+exception, silentfix+ignore, etc.
- The verification routine reports issues and can optionally fix fixable problems or raise exceptions depending on the selected option.
- See the general verify API for details and integration with the rest of the FITS tooling.

Example
-------
- Run verification with warnings:
  .. code-block:: python
     from astropy.io import fits
     with fits.open('image.fits') as hdul:
         hdu = hdul[0]
         hdu.verify("warn")

- Run verification with automatic fixes:
  .. code-block:: python
     with fits.open('image.fits', mode='update') as hdul:
         hdu = hdul[0]
         hdu.verify("fix+warn")

Notifier and change tracking
----------------------------
Image HDUs leverage a NotifierMixin-like mechanism to alert listeners when the Data changes. This enables UI updates, progress tracking, or dependent computations to respond to data modifications in real time. The mechanism uses weak references to listeners and dispatches updates on changes to the Data.

Related concepts
----------------
- Data header alignment: The Data array must correspond to the header-defined image geometry.
- WCS and coordinates: Image geometry often integrates with WCS information in the header to map pixel coordinates to world coordinates.
- Image vs. other HDUs: The Data component is specific to ImageHDU-like structures; other HDUs (e.g., BinaryTableHDU) store data in different formats.

See also
--------
- verify: General verification framework reference for how Data issues are reported and fixed.
- NotifierMixin: Change-notification mechanism used to propagate updates when Data changes.
