Security Roles
==============

Overview
--------

Security roles define what a user is allowed to do on the ftrack server. The Python API library does not implement role definitions itself; instead it authenticates as a user and relies on the server to enforce permissions. Your session uses the provided api_user and api_key to perform actions on the server, and any role-based access control is applied server-side.

Prerequisites
-------------

- A reachable ftrack server URL (server_url)
- API credentials:
  - api_key (FTRACK_API_KEY or argument)
  - api_user (FTRACK_API_USER or argument)

- Access to the server with sufficient permissions to view or modify the relevant resources (as allowed by your roles)

Authentication and identity
---------------------------

The client authenticates by attaching the following headers to every request:

- ftrack-api-key: your API key
- ftrack-user: your API user name

This is handled automatically by the SessionAuthentication object, which is used under the hood by Session.

Example: creating a session
---------------------------

This example shows the standard way to create a session with the required credentials. The headers described above are applied automatically.

.. code-block:: python

    from ftrack_api import Session

    session = Session(
        server_url='https://ftrack.example.com',
        api_key='your_api_key',
        api_user='your_username'
    )

Inspecting your own roles
-------------------------

Your current acting user is identified by api_user, and the server enforces permissions based on that user’s security roles. The client library does not expose a universal, built-in method to list security roles directly. If your server exposes user information (for example, a User object with a roles or security_roles attribute), you can fetch that information from the server.

Example (assuming the server provides a user entity with a roles attribute):

.. code-block:: python

    session = Session(
        server_url='https://ftrack.example.com',
        api_key='your_api_key',
        api_user='your_username'
    )

    # Retrieve the server-side user object for the current user
    user = session.query('User', where='name is "{}"'.format(session._api_user))

    # Access the roles attribute if exposed by the server
    roles = getattr(user, 'roles', None) or getattr(user, 'security_roles', None)
    print('Your security roles:', roles)

Note: Whether this information is available depends on server configuration and the exposed User fields. If the server does not expose roles in the client API, you must rely on server-side admin tools to view or modify roles.

Checking permissions for an operation
------------------------------------

Role-based permissions are enforced by the server. If you attempt an operation you are not permitted to perform, the server raises an access error. You can catch and handle this in client code.

.. code-block:: python

    from ftrack_api.exception import AccessDeniedError

    try:
        # Example: attempt to read or modify a resource that may be restricted
        restricted = session.query('Project', where='name is "Restricted"')
        print(restricted)
    except AccessDeniedError:
        print('Access denied: your current security roles do not permit this operation.')

Managing roles (server-side)
----------------------------

Security roles are managed on the server. The Python API library does not provide a general-purpose method to assign or revoke roles for users. To manage roles:

- Use the server admin UI to assign or revoke roles for a user.
- Use server-side APIs or admin tooling provided by your ftrack deployment if available.
- For automation, coordinate with your server administrator to modify roles or permissions as needed.

Tips and best practices
-----------------------

- Use environment variables for credentials when appropriate (FTRACK_SERVER, FTRACK_API_KEY, FTRACK_API_USER) to simplify secure deployment.
- Treat the api_user as the identity performing all operations; role-based permissions on the server determine what you can and cannot do.
- Handle AccessDeniedError gracefully to provide clear feedback when a user’s roles insufficiently permit an action.
- When auditing or debugging, leverage the session’s operation tracking (session.recorded_operations) to review actions performed under a given session.

Reference
---------

Key components related to security roles in the client context:

- SessionAuthentication: attaches ftrack-api-key and ftrack-user headers to requests.
- Session: central point for server interaction; requires server_url, api_key, and api_user; supports environment-variable fallbacks.
- Access control: enforced by the server; client-side code should anticipate AccessDeniedError when permissions are insufficient.

Code examples assume standard Python API usage for querying User-related information and handling permission errors as demonstrated above.
