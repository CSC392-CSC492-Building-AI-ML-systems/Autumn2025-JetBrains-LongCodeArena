Conventions
===========

This document outlines the conventions used across the library to ensure a consistent, maintainable, and well-documented codebase. It covers public API design, documentation style, module structure, and related practices.

Public API and module structure
-------------------------------

- Public symbols are explicitly exported via the module-level __all__.
  - Example: __all__ = ['sample_system', 'c2d'] declares the public surface of the module.
- Internal details remain private and are not part of the public API. Name internal items with a leading underscore (e.g., _optimal_defaults) to indicate private usage.
- Functions that operate on a canonical representation of an object (e.g., continuous-time LTI systems) wrap underlying functionality when appropriate (e.g., sample_system delegates to the system’s .sample method).
- Consistency in input/output types:
  - Public APIs accept/return well-defined types (e.g., LTI objects such as StateSpace or TransferFunction).
  - Return values preserve the original class where possible (e.g., sysd has the same type as the input when feasible).

Docstrings and documentation style
-----------------------------------

- All public functions include docstrings following the NumPy/SciPy style with the following sections:
  - Parameters
  - Returns
  - Notes or Description
  - Examples
- Docstrings use precise, self-contained descriptions, including:
  - Parameter types and constraints
  - Return type and meaning
  - Behavioral notes (e.g., how options are interpreted)
  - Any special cases or limitations
- Doctest-friendly examples are included in the Examples section, using the standard interactive syntax (>>>).
- Cross-references use Sphinx-style references when applicable (e.g., :class:`StateSpace`, :func:`scipy.signal.cont2discrete`).
- Licensing and copyright headers appear at the top of source files and remain unchanged in documentation examples where relevant.

Example docstring pattern (illustrative)
- The format below reflects the library’s documented style (not a required snippet to reproduce here):
  - def sample_system(sysc, Ts, method='zoh', alpha=None, prewarp_frequency=None):
    - """Convert a continuous time system to discrete time by sampling
      Parameters
      ----------
      sysc : LTI
          Continuous time system to be converted
      Ts : float > 0
          Sampling period
      method : string
          Method to use for conversion
      alpha : float or None
          Bilinear transformation weighting (see details)
      prewarp_frequency : float or None
          Prewarp frequency for bilinear method
    - Returns
      sysd : LTI
          Discrete time system
    - Notes
      See StateSpace.sample or TransferFunction.sample for details.
    - Examples
      >>> sysc = TransferFunction([1], [1, 2, 1])
      >>> sysd = sample_system(sysc, 1, method='bilinear')
      """

- The top-level module docstring should describe the module’s purpose and the primary routines it exposes.

API design and parameter handling
---------------------------------

- Optional parameters default to None when the option is not needed or when the default is intentionally unspecified by the function.
- Input validation is explicit:
  - Raise meaningful exceptions (e.g., ValueError) when inputs do not satisfy required constraints.
  - Validate ranges and compatibility with method options (e.g., prewarp_frequency only meaningful for certain methods).
- When a simple wrapper exists (e.g., c2d calling sample_system), the wrapper delegates to the underlying implementation to avoid duplication and keep behavior consistent.

Defaults and configuration
--------------------------

- Module-level defaults live in a private, centralized dictionary (e.g., _optimal_defaults) to clearly separate user-facing configuration from internal defaults.
- Public configuration keys use readable, hierarchical names (e.g., 'optimal.minimize_method', 'optimal.solve_ivp_method').
- Defaults are documented in code and, where applicable, in the library’s configuration reference (not shown here) to ensure discoverability.

Testing, examples, and doctests
------------------------------

- Doctests are encouraged in the Examples section of docstrings to verify basic usage and behavior.
- Public APIs should be covered by unit tests that exercise:
  - Typical use cases
  - Boundary conditions (e.g., invalid inputs)
  - Interaction with related components (e.g., compatibility of input/output types)
- Code blocks in documentation illustrate expected usage and edge cases.

Coding style and organization
-----------------------------

- Adhere to the project’s code style (PEP 8 for Python, project-specific conventions).
- Imports follow a conventional order: standard library, third-party libraries, local modules.
- Each module begins with a descriptive module docstring, often followed by copyright/license information.
- Where appropriate, modules expose a small, focused public API surface and keep complex logic encapsulated.

Practical guidance and examples
-------------------------------

- When documenting new features, follow the established docstring pattern (Parameters, Returns, Notes, Examples) and reference related API parts:
  - Reference internal methods via their public equivalents (e.g., use StateSpace.sample rather than duplicating sampling logic).
- For time-domain conversions (e.g., continuous-to-discrete), document the supported methods (e.g., 'zoh', 'bilinear', etc.) and any caveats (e.g., prewarping only applies to certain methods).
- When extending the optimal-control portion, keep the convention of a clear problem description, a well-defined objective, and constraints expressed in a way that mirrors optimization libraries (e.g., SciPy’s minimize).

Licensing and attribution
-------------------------

- Each source file includes the appropriate license header. Documentation should preserve references to licensing when reproducing source snippets or examples.

By following these conventions, the library remains approachable for professionals while ensuring code quality, consistency, and maintainability across modules such as dtime (discrete-time utilities) and optimal (optimization-based controllers).
