Working with entities
=====================


Overview
--------

In the ftrack API, entities represent objects stored on the ftrack server (for example, projects, tasks, shots, or custom entities). Interaction with these objects is carried out through a Session, which serves as the entry point to the API. Entities can be loaded lazily (auto-populated) and are backed by a caching layer to improve performance.

Key concepts
- Session: Central object for server communication, authentication, and configuration.
- Auto-populate: When enabled, missing entity attributes are loaded from the server on first access.
- Cache: A configurable caching layer sits on top of the server data to speed up repeated access.
- Cache key maker: Generates keys used to identify cached objects.
- Schema cache: Optional caching of server schema definitions to speed up metadata access.
- Event hub: Optional embedded publish/subscribe mechanism for non-local events.
- Plugins: Discover and load plugins that extend or modify behaviour related to entities.

Getting started
--------------

Create a session to begin working with entities. Provide the server URL and authentication details, or rely on environment variables.

Example: explicit session creation
.. code-block:: python

   import ftrack_api

   session = ftrack_api.Session(
       server_url='https://my.ftrack.server',
       api_key='your_api_key',
       api_user='your_username',
       auto_populate=True,
       schema_cache_path='/path/to/schema/cache',
       auto_connect_event_hub=True
   )

Environment-based configuration
-------------------------------

If you omit credentials and server URL, the session tries to read from environment variables:

- FTRACK_SERVER for the server URL
- FTRACK_API_KEY (or FTRACK_APIKEY for backward compatibility)
- FTRACK_API_USER

If values are not provided, a TypeError is raised.

Example: environment-based session
.. code-block:: python

   import ftrack_api

   session = ftrack_api.Session()

What you can configure
-----------------------

- server_url: URL of the ftrack server (including port if needed).
- api_key: API key for authentication.
- api_user: Username for operations against the server.
- auto_populate: Enables lazy loading of missing attributes on access.
- plugin_paths: List of filesystem paths to search for plugins.
- cache: A cache instance implementing ftrack_api.cache.Cache, or a callable that returns such an instance.
- cache_key_maker: An object implementing ftrack_api.cache.KeyMaker to generate cache keys. If not provided, a StringKeyMaker is used.
- auto_connect_event_hub: If True, automatically connects the embedded event hub to the event server; if False, only local events are available until you connect manually.
- schema_cache_path: Path to a folder to store cached server schema definitions. Set to False to disable schema caching.
- plugin_arguments: Optional mapping of keyword arguments passed to plugin registration during discovery.

Example: customize cache and key maker
.. code-block:: python

   from ftrack_api.cache import MemoryCache, StringKeyMaker
   from ftrack_api import Session

   session = Session(
       server_url='https://my.ftrack.server',
       api_key='your_api_key',
       api_user='your_username',
       cache=MemoryCache(),            # optional: provide your own memory cache
       cache_key_maker=StringKeyMaker()  # default is StringKeyMaker
   )

Lazy loading and attributes
---------------------------

With auto_populate enabled, entity attributes that are not yet loaded are fetched from the server when accessed. This allows you to work with a lightweight representation of the entity and load details on demand.

Example: accessing an attribute triggers load
.. code-block:: python

   # Assume 'entity' is an object retrieved from the server
   name = entity.name  # If not loaded yet, this triggers a server fetch

Note: The exact attribute access syntax may vary by entity type and API version. In general, entities expose attributes through standard attribute access or dictionary-like access.

Querying and navigating entities
--------------------------------

Entity data is accessed through the API’s query and factory facilities. You typically obtain entities via the query interface or via the entity factory, then navigate or filter results as needed.

- ftrack_api.query provides query-building capabilities for retrieving entities.
- ftrack_api.entity.base provides the base class and common behaviour for entities.
- ftrack_api.entity.factory helps instantiate entity objects from server data.

Example: basic query pattern (conceptual)
.. code-block:: python

   # Pseudo-example illustrating typical usage; exact syntax may vary by version
   from ftrack_api import Session

Caching and performance
-----------------------

The session integrates a layered cache to improve performance and reduce repeated server calls:
- A top-level MemoryCache sits at the front, with additional layers beneath.
- You can supply a custom cache implementing ftrack_api.cache.Cache, or pass a callable that returns such a cache.
- A cache_key_maker (default StringKeyMaker) controls how objects are identified in the cache.

Example: enabling a custom cache layer
.. code-block:: python

   from ftrack_api.cache import MemoryCache
   from ftrack_api import Session

   session = Session(
       server_url='https://my.ftrack.server',
       api_key='your_api_key',
       api_user='your_username',
       cache=MemoryCache()
   )

Schema caching
--------------

Enable server schema caching by providing a schema_cache_path. This stores metadata about entity types and fields locally to speed up initial access and discovery.

Example: enabling schema cache
.. code-block:: python

   session = ftrack_api.Session(
       server_url='https://my.ftrack.server',
       api_key='your_api_key',
       api_user='your_username',
       schema_cache_path='/path/to/schema/cache'
   )

Event hub and real-time events
------------------------------

If auto_connect_event_hub is True, the embedded event hub connects to the event server, enabling publish/subscribe to non-local events. Otherwise, you may connect later using the hub’s connect method.

Example: connecting the event hub
.. code-block:: python

   from ftrack_api.event.hub import EventHub

   session = ftrack_api.Session(auto_connect_event_hub=True)
   hub = session.event_hub
   hub.connect()

Plugins and plugin arguments
----------------------------

Plugins may extend or modify entity-related behaviour. The session can discover plugins from plugin_paths and pass plugin_arguments to registration functions. If a plugin signature does not accept the provided arguments, discovery reduces the arguments to those that are accepted and logs a warning.

Example: plugin discovery configuration
.. code-block:: python

   session = ftrack_api.Session(
       plugin_paths=['/path/to/plugins'],
       plugin_arguments={'option_a': True, 'option_b': 42}
   )

Notes
-----

- If server_url is not provided, the session attempts to read FTRACK_SERVER from the environment.
- If api_key or api_user are not provided, the session attempts to read FTRACK_API_KEY and FTRACK_API_USER (or FTRACK_APIKEY for backward compatibility), and falls back to the system user for api_user if possible.
- The session is designed to be used as a long-lived object for interacting with entities and related APIs.
