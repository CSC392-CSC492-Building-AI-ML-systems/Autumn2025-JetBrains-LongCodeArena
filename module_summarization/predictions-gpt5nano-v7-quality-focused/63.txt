Commands
==========

Overview
--------
This document describes the public API of the commands module. The module provides three commands used to manage migration templates, initialize a new script directory, and create new revision files. The implementations rely on the internal utilities and data structures from the migration system (Config, ScriptDirectory, RevisionContext, etc.).

Functions
---------

list_templates(config)
^^^^^^^^^^^^^^^^^^^^^^
List available templates that can be used with the init command.

Parameters
- config: a Config object (typically from alembic.config.Config).

Behavior
- Prints a header "Available templates:".
- Iterates over subdirectories in the template directory returned by config.get_template_directory().
- For each template, reads the first line of the template's README as a synopsis and prints "template - synopsis".
- Prints a note showing how to use templates with the init command, e.g.:
  alembic init --template generic ./scripts

Returns
- None

Notes
- This command reads template READMEs to present a short synopsis for each template.
- It uses config.print_stdout to display output and adheres to the configured messaging options.

Example
- List templates for an Alembic environment:
  commands.list_templates(config)

init(config, directory, template="generic", package=False)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
Initialize a new scripts directory from a template.

Parameters
- config: a Config object.
- directory: string path of the target directory to create.
- template: template name to use (default: "generic").
- package: when True, also write __init__.py files into the environment location and the versions directory (creates package markers).

Behavior
- If the target directory exists and is not empty, raises CommandError:
  "Directory <directory> already exists and is not empty".
- Validates that the requested template exists under config.get_template_directory().
- Creates the target directory if it does not exist (with a status message).
- Creates a versions subdirectory inside the target directory (with a status message).
- Creates a ScriptDirectory object for the target directory.
- Copies over non-template files from the chosen template:
  - If the file is alembic.ini.mako, generates the configuration file from the mako template to the configured alembic.ini path (config.config_file_name). If the config file already exists, emits a skip message; otherwise generates the file.
  - All other regular files from the template are copied into the target directory.
- If package is True, creates __init__.py files in:
  - <target>/__init__.py
  - <target>/versions/__init__.py
  (each created with a status message)
- Requires that the config file path has been determined; prints a final message prompting the user to edit configuration/connection/logging settings in the generated alembic.ini file before proceeding.

Returns
- None

Notes
- This function assumes the presence of a mako template alembic.ini.mako in the template directory and uses ScriptDirectory._generate_template for generation.
- The function respects config.messaging_opts for runtime status messages.

Example
- Initialize a new migration environment:
  commands.init(config, directory="./migrations", template="generic", package=True)

revision(config, message=None, autogenerate=False, sql=False, head="head", splice=False, branch_label=None, version_path=None, rev_id=None, depends_on=None, process_revision_directives=None)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
Create a new revision file.

Parameters
- config: a Config object.
- message: optional string, the message to apply to the revision (corresponds to -m).
- autogenerate: boolean, whether to autogenerate the script from the database (corresponds to --autogenerate).
- sql: boolean, if True, dumps the script to stdout as SQL (corresponds to --sql).
- head: head revision to build the new revision upon as a parent (default "head") (corresponds to --head).
- splice: boolean, whether the new revision should be made a new head (corresponds to --splice).
- branch_label: optional string label to apply to the branch (corresponds to --branch-label).
- version_path: optional version path identifier from the configuration (corresponds to --version-path).
- rev_id: optional revision identifier to use instead of generating one (corresponds to --rev-id).
- depends_on: optional dependency specifier(s) (corresponds to --depends-on).
- process_revision_directives: optional callable, used programmatically to alter the revision structure after generation. This mirrors the process_revision_directives hook described in the environment context.

Return value
- A Script or a list of Script objects (Union[Optional[Script], List[Optional[Script]]]).

Behavior
- Creates a ScriptDirectory from the provided config.
- Builds a command_args dictionary with the provided values.
- Creates a RevisionContext with:
  - the config
  - the script directory
  - the command_args
  - process_revision_directives (if provided)
- Reads the "revision_environment" main option from the config to determine the runtime environment handling.
- If autogenerate is requested:
  - environment is forced to True.
  - Raises a CommandError if both autogenerate and sql are requested, as this combination does not make sense.
- The code creates a helper retrieve_migrations function (incomplete in the snippet) to facilitate obtaining migrations for the new revision, and then proceeds with the revision generation workflow via the revision_context.

Notes
- The snippet provided ends before the full implementation is shown; the documented behavior reflects the visible portions and the function signature.
- This command is intended for programmatic use and for invoking the revision flow via the command interface.

Example
- Create a new revision with a message:
  commands.revision(config, message="Add users table")

- Create an autogenerate revision based on the current database state:
  commands.revision(config, autogenerate=True)

- Create a SQL-only revision based on the autogenerate script (note: this combination is not allowed and will raise a CommandError):
  commands.revision(config, autogenerate=True, sql=True)
