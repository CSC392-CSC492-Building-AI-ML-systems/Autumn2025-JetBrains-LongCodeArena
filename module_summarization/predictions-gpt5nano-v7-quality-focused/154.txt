==============

Connecting Steps Together
-------------------------

This document describes how to connect individual steps to form a
workflow graph. It explains how to obtain the proper handles from
Step objects, how to create and use Edge connections, and how to
manipulate steps to support dynamic graph construction.

Overview
--------

- A workflow is composed of discrete steps. Each step declares its
  inputs and outputs, plus a set of commands to run.
- Connections between steps are represented by Edge objects, which
  hold a source (an output handle) and a destination (an input handle).
- Steps expose handles that identify inputs and outputs in a consistent
  tuple form, enabling graph-building utilities to connect steps
  programmatically.

Key Concepts
------------

- Step: Represents a single computational unit loaded from a YAML
  configuration. The Step object exposes handles for its inputs and
  outputs and provides utilities to modify the step's interface.
- Handle: A tuple that identifies a specific input or output on a step.
  - Input handle: (step_name, 'inputs', input_name)
  - Output handle: (step_name, 'outputs', output_name)
- Edge: A simple pair (src, dst) that models a connection from an
  output handle to an input handle.

Edge: Connecting Source to Destination
--------------------------------------

- Edge objects connect a source output to a destination input.
- Creation example:

  .. code-block:: python

     from edge import Edge
     from step import Step

     A = Step('path/to/stepA')  # source step
     B = Step('path/to/stepB')  # destination step

     # Connect A's output 'out' to B's input 'in'
     edge = Edge( A.o('out'), B.i('in') )

- After creating an Edge, a graph-building component can merge multiple
  edges to form the full workflow.

Step Handles: Accessing and Using Inputs/Outputs
-----------------------------------------------

- Get a specific input handle:
  - s.get_input_handle(name)
  - s.i(name) is a shorthand for get_input_handle
- Get a specific output handle:
  - s.get_output_handle(name)
  - s.o(name) is a shorthand for get_output_handle
- Retrieve all input handles for a step:
  - s.all_input_handles()
- Retrieve all output handles for a step:
  - s.all_output_handles()
- Examples:

  .. code-block:: python

     A = Step('path/to/stepA')
     inp_handle = A.i('in1')      # e.g., ('stepA', 'inputs', 'in1')
     out_handle = A.o('out1')    # e.g., ('stepA', 'outputs', 'out1')

- Validation: The accessors assert the existence of the requested
  inputs/outputs. If a step has no inputs or no outputs, the corresponding
  accessors produce appropriate assertion messages.

Modifying Step Interfaces
-------------------------

- Extend inputs/outputs (dynamic wiring):
  - extend_inputs(list_of_names)
  - extend_outputs(list_of_names)
- Pre/post extend commands:
  - pre_extend_commands(list_of_commands)
  - extend_commands(list_of_commands)

  These helpers mutate the step’s internal configuration to support
  dynamic graph construction without editing the YAML files directly.

- Cloning:
  - Step.clone() creates a deep copy of the step’s configuration and
    allows safe reuse when building multiple graphs from a single base step.

Example: Building a Simple Graph
--------------------------------

1) Load two steps

  .. code-block:: python

     from step import Step
     A = Step('/path/to/stepA')  # source
     B = Step('/path/to/stepB')  # destination

2) Create a connection from A to B

  .. code-block:: python

     edge = Edge( A.o('out'), B.i('in') )

3) Use edges to compose a graph (typical usage in a graph builder)
   (Graph API usage shown for illustration; actual graph builder may vary)

  .. code-block:: python

     graph_edges = [ edge ]
     # graph = Graph(nodes=[A, B], edges=graph_edges)
     # graph.connect_all()

Notes on Internal Data
----------------------

- Step data is loaded from YAML via configure.yml located either in a
  step directory or in a top-level default steps directory when using
  default=True.
- Each Step stores:
  - _config: an internal dictionary containing all loaded data
  - step_dir: the relative directory of the step
  - source: the directory of the YAML source file

- The YAML data must contain a "name" field. Optional fields include
  "inputs", "outputs", and "commands". If inputs or outputs are empty or
  missing, they are handled gracefully.
- Outputs may include tagged shorthand forms that the Step class expands
  into concrete file names (e.g., mapping a shorthand to a real file path).

Best Practices
--------------

- Validate connections: Ensure the source step provides the stated output
  and the destination step declares the corresponding input before
  connecting.
- Use Step.clone() when you need multiple graph variants based on a common
  step configuration.
- When extending interfaces, prefer extend_inputs/extend_outputs to keep
  YAML changes minimal and programmatic.
