Working with entities
=====================

Overview
--------

The ftrack API uses a Session to communicate with an ftrack server and to work with server-side entities (e.g. assets, tasks, etc.). A Session encapsulates authentication, optional caching, an event hub, and a plugin system. When enabled, entity attributes can be loaded lazily from the server as they are accessed.

Getting started
---------------

Create a session by providing the server URL and credentials. Credentials can also be supplied via environment variables.

.. code-block:: python

   from ftrack_api import Session
   session = Session(
       server_url='https://my-ftrack-server',
       api_key='YOUR_API_KEY',
       api_user='YOUR_USERNAME'
   )

If you prefer, the values can be provided via environment variables:
- FTRACK_SERVER for the server URL
- FTRACK_API_KEY (or FTRACK_APIKEY) for the API key
- FTRACK_API_USER for the API user

Note: If required values are not provided, a TypeError will be raised.

Authentication helper
---------------------

For advanced usage, the session can be paired with a request-level authentication handler. The SessionAuthentication class adds the API key and user to outgoing requests.

.. code-block:: python

   from ftrack_api import SessionAuthentication
   session = Session(
       server_url='https://my-ftrack-server',
       api_key='YOUR_API_KEY',
       api_user='YOUR_USERNAME'
   )
   # Example of using the custom auth with a requests session
   # requests_session = requests.Session()
   # requests_session.auth = SessionAuthentication('YOUR_API_KEY', 'YOUR_USERNAME')

Auto-population of entity attributes
------------------------------------

By default, accessing attributes on an entity will trigger a server fetch if the value is not already loaded, provided auto_populate is True (the default).

- This lazy loading reduces the amount of data retrieved upfront.
- If you disable auto_populate, you must fetch attributes explicitly.

Caching and key management
--------------------------

Sessions use a pre-configured layered cache with a MemoryCache at the top level. You can customize or replace the cache as needed.

- cache: an instance implementing ftrack_api.cache.Cache or a callable that returns one.
- cache_key_maker: an object implementing ftrack_api.cache.KeyMaker. If not provided, a StringKeyMaker is used.

Example:

.. code-block:: python

   from ftrack_api.cache import MemoryCache, StringKeyMaker
   session = Session(
       server_url='https://my-ftrack-server',
       api_key='YOUR_API_KEY',
       api_user='YOUR_USERNAME',
       cache=MemoryCache(),                 # optional
       cache_key_maker=StringKeyMaker()     # optional
   )

Event hub and subscriptions
---------------------------

If auto_connect_event_hub is True, the embedded event hub will connect to the event server, enabling publishing and subscribing to non-local events. If False, only local events are available until you connect manually.

Plugin system
-------------

plugin_paths can be provided as a list of filesystem paths to discover plugins. plugin_arguments can contain keyword arguments to pass to plugin registration functions on discovery. If a plugin signature does not accept the provided arguments, the system will attempt to adapt and warn if needed.

Schema caching
--------------

Enable schema caching by setting schema_cache_path to a directory. If the environment variable FTRACK_API_SCHEMA_CACHE_PATH is set, it will be used automatically. If you set schema_cache_path to False, schema caching is disabled.

Operational state
-----------------

The session tracks pending operations via recorded_operations (an instance of ftrack_api.operation.Operations) and a flag (record_operations) to indicate whether to record operations. This supports batching and efficient communication with the server.

Notes
-----

- The Session initialization enforces required parameters and provides helpful error messages if values are missing.
- The API is designed to be extended with plugins and configurable caching to suit various workflows and performance requirements.

For more details, see the related modules:
- ftrack_api.entity.factory, ftrack_api.entity.base, ftrack_api.entity.location
- ftrack_api.cache, ftrack_api.query
- ftrack_api.event.hub, ftrack_api.plugin
- ftrack_api.operation, ftrack_api.accessor.server
- ftrack_api.logging and related utilities

Example workflow
--------------

1. Create a session with server URL and credentials.
2. Optionally enable auto-population and caching.
3. Retrieve and manipulate entities via the session and the entity API.
4. Use the event hub to publish/subscribe to events if enabled.
5. Rely on the operation tracker to batch changes back to the server.
