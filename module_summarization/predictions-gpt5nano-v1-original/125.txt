FITS Headers
============

Overview
--------

In the FITS (Flexible Image Transport System) format, metadata is stored in
headers composed of a sequence of 80-character ASCII cards. Each card encodes a
keyword, an optional value, and an optional comment. Headers exist in the primary
header and in any extension headers. A header ends with a card whose keyword is
END. Header blocks are padded to 2880-byte boundaries.

Card format
-----------

- Keyword: 8-character, conventionally uppercase (used as the identifier for the
  metadata item).
- Value: optional. If present, it starts after an equals sign (=) and can be a
  number (integer or float), a logical value (T/F), or a string enclosed in
  quotes.
- Comment: optional. Free-form text that describes or clarifies the value; begins
  after an optional '/' following the value.

Special cards
-------------

- COMMENT and HISTORY cards carry free-form text and do not require a value
  field. They are used for human-readable notes or a history of operations on the
  data.

Endianness, blocks, and padding
--------------------------------

- Each header card is exactly 80 characters long.
- The header block is padded with blank cards (spaces) to fill 2880 bytes.
- The END card marks the end of the header (no value for END).

Indexing and access
-------------------

- When reporting or iterating header cards in the included tooling, zero-based
  indexing is used (i.e., the first header card is index 0). This is noted in
  the documentation for consistency with common Python indexing.

Validation and error handling
-----------------------------

- A verification framework provides options to check header values and report
  issues. Available options include:
  - ignore, warn, exception
  - fix, silentfix
  - fix+ignore, fix+warn, fix+exception
  - silentfix+ignore, silentfix+warn, silentfix+exception

- Verification can report messages as warnings or raise a VerifyError with a
  structured summary of issues. Unfixable issues can be surfaced distinctly from
  fixable ones.

- Verification results are organized into a nested error-list structure, enabling
  indentation and grouping of messages according to the header’s hierarchical
  organization.

Notifying changes
-----------------

- Header objects may support a lightweight notification system. Listeners can be
  registered to be updated when the header changes, enabling other parts of the
  system to respond automatically to metadata updates.

Example header
--------------

A minimal FITS header might include:

::

   SIMPLE  =                    T / file conforms to FITS standard
   BITPIX  =                   16 / number of bits per data pixel
   NAXIS   =                    2 / number of data axes
   NAXIS1  =                 1024 / length of data axis 1
   NAXIS2  =                 1024 / length of data axis 2
   END

Notes
-----

- Strings in values are enclosed in quotes.
- The END card must be present to signify the end of the header.
- Headers and their padding allow efficient random access and compatibility
  across different software that reads FITS data.

This documentation reflects the handling of FITS headers as implemented in the
associated tooling, including the built-in verification, error reporting, and
change-notification facilities.
