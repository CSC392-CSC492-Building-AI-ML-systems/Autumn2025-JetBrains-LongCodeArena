Header Interface Transition Guide
=================================

Overview
--------
This guide documents the transition approach for the Header interface used in FITS handling. It covers the header value verification framework, how verification results are reported, how fixes are applied, and how to migrate existing code to the new interface with minimal disruption.

Header Validation and Verification Options
------------------------------------------
Validation is performed through a header-level verify mechanism. The verification option can be one of the base modes or a combination that also controls fixing behavior.

- Base modes:
  - ignore: perform no validation and return immediately
  - warn: report issues as warnings
  - exception: raise an exception for issues

- Fixing modes (with the option to combine with reporting):
  - fix: attempt to fix fixable issues and report them
  - silentfix: fixable issues are fixed silently (no messages unless there are unfixable errors)

- Combinations:
  - fix+ignore, fix+warn, fix+exception
  - silentfix+ignore, silentfix+warn, silentfix+exception
  - You may also combine fix modes with ignore/warn/exception for reporting by using the “+” form (e.g. "fix+warn").

Behavior details
- If the option ends with "warn" or "exception", fixing is disabled for those errors.
- If an error is not fixable, the message is prefixed with “Unfixable error:”.
- When fixing is performed, a standard fix message (e.g. "Fixed.") is appended to the error text.
- Messages are aggregated and then either emitted as warnings or raised as VerifyError, depending on the reporting portion of the option.

Error representation
--------------------
Verification results are represented using a nested error list structure (_ErrList). This structure supports hierarchical messages and indentation for nested verifications, making it easier to understand which header keys caused which issues.

- VerifyError: exception raised for verification failures (when reporting as exception)
- VerifyWarning: warning emitted for verification issues (when reporting as warn)
- _ErrList: internal structure for collecting and formatting error messages

Notifier and Header Changes (Overview)
--------------------------------------
Header objects may use a notifier mixin to allow listeners to react to changes in header state. Listeners register (via weak references) and are notified of changes with a standardized update method. This enables reactive UI updates or synchronization in dependent components when header values change.

Migration Guidance
-------------------
- Prefer using the standard header.verify(option) interface to perform validation.
- Use combined options (e.g. "fix+warn" or "silentfix+exception") to control both auto-fixing behavior and reporting.
- Handle VerifyError and VerifyWarning in application code as appropriate for your workflow.
- Leverage _ErrList’s structured messages to present clear, hierarchical validation output.

Examples
--------
- Basic verification with warnings:
  .. code-block:: python

     header.verify("warn")

- Automatic fixes with reporting on issues:
  .. code-block:: python

     header.verify("silentfix+warn")

- Enforcing fixes and raising on unfixable issues:
  .. code-block:: python

     header.verify("fix+exception")

Internal details (for maintainers)
----------------------------------
- VerifyError and VerifyWarning are used to signal verification outcomes.
- _Verify.run_option implements the core logic for executing a particular verify option.
- _ErrList.iter_lines provides a formatted, indented view of nested errors.
