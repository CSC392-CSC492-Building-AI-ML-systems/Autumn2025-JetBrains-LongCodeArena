Querying
========

Overview
--------

The ftrack_api.query module provides a server-side querying interface for retrieving data from an ftrack server. A query targets a specific resource type (for example, Task, Shot, AssetVersion) and supports:

- selecting which fields to return
- filtering data with constraints
- sorting and paginating results
- traversing related entities to fetch nested data

API surface
-----------

- Primary entry point: the Query class in ftrack_api.query.
- Creating a query: instantiate a Query with a root/resource type.
- Field projection: specify which fields to include in results.
- Filtering: apply constraints to restrict which records are returned.
- Sorting: order results by one or more fields.
- Pagination: limit and offset controls to manage large result sets.
- Relationship traversal: fetch related data by including fields from related entities.
- Execution: run the query to obtain results from the server.

Session interaction
-------------------

Queries are executed against an authenticated Session. The session provides server URL, credentials, and connection management, while the query defines what data to retrieve. The actual data retrieval happens on the server, reducing client-side data transfer and round-trips where possible.

Usage concepts
--------------

- Resource type: the root entity you want to query (e.g., Task, Shot).
- Fields to return: a projection of the data you need (id, name, status, related fields, etc.).
- Constraints: criteria used to filter results (e.g., status, created date, association with another entity).
- Ordering: deterministic result order to support paging and stable displays.
- Paging: control the number of results returned per request and the starting point.
- Nested data: ability to pull in related information (e.g., a task’s project data) through relationship paths.

Performance considerations
--------------------------

- Always project only the fields you need to minimize payload size.
- Use filters to narrow the dataset before retrieval.
- Prefer pagination for large result sets to manage memory and latency.

Error handling
--------------

- Query-related issues raise exceptions from ftrack_api.exception, such as QueryError, and should be handled gracefully in calling code.

Best practices
--------------

- Start with a narrow projection and add fields only as needed.
- Combine filtering with proper indexing (where supported) to improve performance.
- Use coherent sorting to ensure stable paging across large datasets.
- Validate results against expected schemas, especially when traversing relationships.

----------------------

- Apply filters to limit results.
- Sort and paginate to control data retrieval.
