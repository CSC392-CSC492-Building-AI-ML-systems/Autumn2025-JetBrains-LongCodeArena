Less Frequently Used FITS Data Structures

Introduction
Beyond the common FITS structures (images, tables, and standard HDUs), there are internal data structures and utilities that support data integrity checks and change notification within the FITS I/O ecosystem. These components are typically used by developers extending the I/O layer or building custom FITS-like containers. This document provides an overview of such lesser-used structures and how they relate to FITS data validation and observability.

Verification and error reporting structures

_ErrList: nested verification errors
- Purpose: Represents hierarchical verification messages generated by checks performed on FITS objects. The structure is a list that can contain both simple error items and nested sublists, enabling messages to reflect the class hierarchy and subcomponents of a FITS object.
- Structure:
  - Each top-level item can be either:
    - A tuple (fixable_flag, message_text), or
    - Another _ErrList instance representing a nested level.
  - The nested level can have its own unit label (default "Element"), which serves as a header when presenting the next level of messages.
- Display and iteration:
  - __str__ prints a human-readable, indented view of the messages.
  - iter_lines(filter=None, shift=0) yields lines with appropriate indentation for each level. The first element of each top-level item is used to determine indentation and header lines for nested levels.
- Usage notes:
  - The unit attribute acts as a header prefix for nested levels (e.g., "HDU 0:", "Header 1:").
  - The string representation emphasizes the hierarchical nature of failures across related components.
  - Filtering can be used to suppress or reveal specific kinds of messages when presenting results to users.

_Verify: verification framework for FITS data
- Purpose: Provides a lightweight mechanism to verify values within a FITS object and report or fix issues according to user-specified options.
- Key methods:
  - run_option(option="warn", err_text="", fix_text="Fixed.", fix=None, fixable=True)
    - Executes the verification with the selected option.
    - Behavior depends on option:
      - "warn" or "exception": issues are not fixable.
      - Other options may attempt to fix (if fix is provided) and append fix_text to the error message.
    - Returns a pair (fixable, text).
  - verify(option="warn")
    - Validates all values in the instance.
    - Recognizes a set of option strings (including combinations like "fix+warn" and "silentfix+ignore").
    - Produces a structured report of errors and handles reporting (warnings or exceptions) or fixing as configured.
- Options:
  - VERIFY_OPTIONS includes: ignore, warn, exception, fix, silentfix, fix+ignore, fix+warn, fix+exception, silentfix+ignore, silentfix+warn, silentfix+exception.
  - Options may be combined with "+ignore", "+warn", or "+exception" to control reporting alongside fixing behavior.
- Behavior highlights:
  - "ignore" skips verification.
  - "warn" emits warnings for issues.
  - "exception" raises a VerifyError for issues.
  - "fix" and "silentfix" attempt to repair fixable issues; "silentfix" suppresses output for fixed issues.
  - Special handling exists to prevent reporting when only fixed issues were addressed under certain option combinations.
- Output:
  - Verification results are assembled into messages (potentially multi-line) with a header and a note about zero-based indexing (as used by astropy.io.fits).

Notifier pattern for FITS objects

NotifierMixin: observer/listener support
- Purpose: Enables objects to register listeners that react to changes on the object. This is useful for complex FITS data containers where external components need to be informed of updates.
- Design:
  - Methods are underscored to emphasize internal use and to discourage external reliance.
  - Listeners are registered via _add_listener and are kept via weak references, so they are automatically cleaned up when no longer used.
  - _notify('change_type', *args, **kwargs) calls listener._update_change_type(*args, **kwargs) on all registered listeners.
  - If a listener does not implement the appropriate update method, it is ignored gracefully.
- Usage notes:
  - The pattern supports decoupled architectures where changes in one FITS-related object propagate to observers (e.g., UI components, caching layers, or validation dashboards).
  - Listeners should implement the corresponding update method (e.g., _update_change_type) to respond to notifications.
- Practical considerations:
  - Keeps memory usage in check via weak references.
  - Encourages a clean separation between data objects and observers.

Other internal utilities

- Internal helpers and constants (e.g., path-like type hints, basic utilities, and compatibility shims) underpin the verification and notification mechanisms but are generally not exposed as end-user APIs. They support robust verification workflows and event-driven updates in more advanced usage scenarios.

Reference and related concepts
- The verification workflow (verify, run_option) relies on the nested error structure (_ErrList) to present hierarchical diagnostics clearly.
- The option set (VERIFY_OPTIONS) enables flexible control over how issues are reported and whether they should be fixed automatically.
- The NotifierMixin provides a lightweight, non-intrusive way to observe state changes across FITS-related objects.

Note
These structures are intended for developers extending the FITS I/O layer or building complex data validation and observing mechanisms. End users generally interact with higher-level FITS features, while the components described here support advanced use cases and internal reliability.
