Working with assignments and allocations
----------------------------------------

Overview
- In ftrack, an assignment represents the linkage of a user to a particular entity (for example, a shot, task, or asset) indicating that the user is responsible for work on that entity.
- An allocation represents the allocation of a user’s or a resource’s capacity to work on an entity within a given time window. Allocations help model available work time and resource planning.

Prerequisites
- A functioning Session is required to interact with the ftrack server.
- Create a Session with server_url, api_key, and api_user (these can also be supplied via environment variables FTRACK_SERVER, FTRACK_API_KEY, and FTRACK_API_USER).
- The Session encapsulates authentication, connection management, and optional caching to speed up repeated access.
- Changes can be recorded through the session’s operation tracking mechanism for auditability.

Working with assignments
- Retrieving assignments:
  - Query assignments by assignee (user) or by the target entity to determine current responsibilities.
  - Filter results by status, start/end dates, or other domain-specific attributes.
- Creating assignments:
  - Create a new assignment by associating a user with a target entity and specifying the relevant context (e.g., role, start date, end date, due date, or status).
  - Ensure that the assignee has the necessary permissions to be assigned to the entity.
- Updating assignments:
  - Change attributes such as role, status, or the active time window of an assignment.
  - Reassign ownership by updating the assignee or shifting the assignment to a different entity.
- Removing assignments:
  - Unassign a user from an entity when the work is completed or no longer applicable.
  - Preserve history if required by project governance.

Working with allocations
- Retrieving allocations:
  - Query allocations by user, by entity, or by time window to understand capacity and commitment.
  - Use filters to view allocations within a particular date range or for a specific project.
- Creating allocations:
  - Create a new allocation to represent a user’s planned or reserved work time on an entity.
  - Specify the time window (start and end) and the allocated capacity (for example, hours or days).
- Updating allocations:
  - Adjust the allocation window or capacity to reflect changes in schedule or availability.
  - Split or merge allocations as workload changes.
- Removing allocations:
  - Delete allocations when the reserved time is cancelled or no longer needed.
- Practical considerations:
  - Align allocations with the project calendar and other scheduling constraints.
  - Validate that allocations do not exceed available capacity and that they are consistent with existing assignments.

Common workflows
- Aligning assignments with allocations:
  - Ensure that an assignee has the appropriate allocation on the target entity for the intended time period.
  - Update either the assignment or the allocation to reflect changes in scope or schedule.
- Auditing and history:
  - Use the session’s operation recording to track creation, modification, and deletion of assignments and allocations for accountability.
- Performance considerations:
  - Use schema caching where possible to speed up repetitive queries related to assignments and allocations.

Example concepts (high level)
- Assignment attributes: assignee, entity, role, start_date, end_date, status.
- Allocation attributes: assignee/resource, entity, start_date, end_date, capacity, utilization.

Tips
- Leverage the session’s environment-based configuration for consistent authentication across environments.
- Enable and utilize caching to reduce repetitive server requests when listing or filtering assignments and allocations.
- Monitor and review recorded operations to maintain an auditable history of changes to assignments and allocations.
