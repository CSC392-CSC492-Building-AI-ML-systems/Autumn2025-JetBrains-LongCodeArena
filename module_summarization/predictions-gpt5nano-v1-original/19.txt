w_crawl
=======

Overview
--------
Crawl a weighted ensemble dataset and execute a user-supplied function for each iteration. Tasks are run in parallel by iteration; there are no guarantees about the evaluation order. Optional crawler logic can coordinate initialization, per-iteration result processing, and finalization.

Usage
-----
w_crawl [OPTIONS] TASK_CALLABLE

- TASK_CALLABLE: Run TASK_CALLABLE (specified as module.function) on each iteration. Required.

- If you need initialization, finalization, or per-iteration result processing coordinated by a dedicated object, supply a crawler instance with -c/--crawler-instance (see Crawler instance below). The crawler object should be an instance of WESTPACrawler (or a subclass) and may override initialize, finalize, and process_iter_result.

Command-line options
--------------------
The following option domains are available:

- Options from WESTDataReader (data selection)
- Options from IterRangeSelection (iteration range)
- -c, --crawler-instance
  Use CRAWLER_INSTANCE (specified as module.instance) as an instance of WESTPACrawler to coordinate the calculation. Required only if initialization, finalization, or task result processing is required.

- TASK_CALLABLE
  Run TASK_CALLABLE (specified as module.function) on each iteration. Required.

- Options provided by ProgressIndicatorComponent (progress display)

Notes:
- Data selection and iteration range are provided through WESTDataReader and IterRangeSelection, respectively.
- Progress display is controlled via ProgressIndicatorComponent options.

What TASK_CALLABLE receives
---------------------------
For each iteration, the task is invoked with:
- n_iter: the iteration index
- iter_group: the data for that iteration (accessible via a data manager)

The function should return a result. The result, for each iteration, is then passed to:
- crawler.process_iter_result(n_iter, result) if a crawler instance is configured
- The per-iteration process is dispatched in parallel and may complete out of order.

Crawler integration
-------------------
If you want to coordinate extra behavior across the crawl, provide a crawler instance via -c/--crawler-instance. The crawler should implement:
- initialize(iter_start, iter_stop): called once before dispatching tasks
- finalize(): called after all tasks complete
- process_iter_result(n_iter, result): called for each completed iteration

Examples
--------
- Simple usage without a crawler (no per-iteration coordination):
  w_crawl mypackage.mymodule.my_task

- With a crawler instance (to coordinate initialization/finalization and per-iteration results):
  w_crawl -c mypackage.mypreprocessing:MyCrawler mypackage.mymodule.my_task

Implementation notes
--------------------
- Each iteration’s task is executed by _remote_task, which opens the data backing and calls the user function as taskfn(n_iter, iter_group).
- The system uses a work manager to submit tasks and collect results as they complete (order not guaranteed).
- The finalization step is always performed via crawler.finalize() if a crawler was provided; otherwise, it is a no-op.
