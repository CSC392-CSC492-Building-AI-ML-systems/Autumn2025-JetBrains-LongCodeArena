Compatibility Policy
===================

Overview
--------

This policy defines how the project preserves compatibility across releases, how deprecations are handled, and how binary/ABI stability is ensured. It applies to public Python API, extension modules, and any user-facing interfaces.

Public API compatibility
------------------------

- Public API surface is kept stable within a minor/patch release.
- Breaking changes are allowed only in major releases.
- New public features may be added in minor releases without removing existing APIs.
- Public API changes must be documented in release notes and the compatibility policy should be updated if needed.

Deprecation and removal
-----------------------

- Deprecated features are announced in release notes and code warnings (e.g., DeprecationWarning or project-specific warning).
- Deprecated features must remain accessible for a minimum deprecation window (typically at least two subsequent releases) before removal.
- When removing a deprecated feature, provide a migration path and update the documentation accordingly.

ABI and binary compatibility
----------------------------

- Binary wheels and compiled extensions should maintain the ABI within the same Python/NumPy compatibility line.
- Changes to the C ABI or to extension interfaces should be accompanied by a major version increase.
- Any required changes to NumPy/C-API usage should be clearly documented in the release notes.

Python, NumPy, and platform compatibility
-------------------------------------------

- Support is aligned with the ecosystem: Python versions, NumPy versions, and supported platforms as listed in the installation documentation.
- If a Python/NumPy upgrade introduces incompatibilities, provide a migration guide and, where possible, maintain compatibility shims for a transition period.

Documentation and release notes
-------------------------------

- All compatibility changes must be described in release notes.
- The compatibility policy document itself should be reviewed and updated as needed during releases.

Testing and verification
------------------------

- CI should cover all supported Python versions, NumPy versions, and platforms.
- Tests should exercise public APIs and extension modules to verify compatibility.

Communication of changes
------------------------

- Users should be informed about potential incompatibilities through:
  - Release notes
  - Migration guides
  - Documentation updates

End-of-life
-----------

- When a version is EOL, it is no longer supported for fixes or security updates.
- End-of-life dates should be announced well in advance in release notes and documentation.
