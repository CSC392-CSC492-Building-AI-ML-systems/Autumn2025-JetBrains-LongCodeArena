A quick dive into using the API
================================

Introduction
------------
This guide covers the core pieces of the ftrack_api you interact with most: creating a Session, authenticating, and powering basic operations with sensible defaults and useful options.

Getting started
---------------
To begin, you create a Session which represents a connection to an ftrack server. The Session handles authentication, caching, optional event hub integration, plugin discovery, and more.

Key concepts
- Server and credentials are required, either via arguments or environment variables.
- Optional features like automatic data population, event hub connection, and schema caching can be tuned at session creation.
- A simple authentication helper is provided to attach credentials to raw HTTP requests when needed.

Creating a session
------------------
Parameters (explanation):
- server_url: URL of the ftrack server (must be provided or set via FTRACK_SERVER env var).
- api_key: API key for authentication (or FTRACK_API_KEY / FTRACK_APIKEY env vars).
- api_user: User name for operations (or FTRACK_API_USER env var or the system user).
- auto_populate: If True, accessing attributes will fetch data from the server automatically.
- plugin_paths: List of paths to search for plugins (default uses FTRACK_EVENT_PLUGIN_PATH env var).
- cache: Cache instance or callable to configure caching for the session.
- cache_key_maker: Key maker for cache keys (default is StringKeyMaker).
- auto_connect_event_hub: If True, a background event hub connection is established.
- schema_cache_path: Folder path for schema caching (or FTRACK_API_SCHEMA_CACHE_PATH); False disables caching.
- plugin_arguments: Optional mapping of arguments to pass to plugin discovery.

Example
-------
Create a session with explicit credentials:

.. code-block:: python

    from ftrack_api import Session

    session = Session(
        server_url='https://my-ftrack-server.example.com',
        api_key='MY_API_KEY',
        api_user='my.username'
    )

Authentication header behavior
------------------------------
The API provides a lightweight authentication helper that attaches the credentials to HTTP requests:

.. code-block:: python

    from ftrack_api import SessionAuthentication
    import requests

    auth = SessionAuthentication('MY_API_KEY', 'my.username')
    response = requests.get('https://api.example.com/endpoint', auth=auth)

Session features in brief
-------------------------
- auto_populate
  - When enabled, entity attributes are fetched on demand if not already loaded.
- Cache
  - The session participates in a layered cache with a top-level MemoryCache by default.
  - You can provide a custom cache or a factory that receives the session.
- Cache key maker
  - Use a custom key maker or rely on the default StringKeyMaker.
- Event hub
  - If auto_connect_event_hub is True, the embedded event hub is connected in the background, enabling non-local event publishing and subscription.
  - Local event publishing/subscription is always available; a connected hub improves capabilities for remote events.
- Schema cache
  - Schema caching can speed up repeated interactions by storing schema information on disk.
  - Disable by setting schema_cache_path to False; otherwise, the path is taken from FTRACK_API_SCHEMA_CACHE_PATH or a temporary directory if not specified.
- Plugins
  - Plugins are discovered from specified paths (or the default environment lookup) and registered with optional arguments.
  - If a plugin’s signature cannot accept the provided arguments, the system reduces args to compatible ones and logs a warning.

Environment variables (defaults)
-------------------------------
- FTRACK_SERVER: server URL if not provided directly.
- FTRACK_API_KEY or FTRACK_APIKEY: API key if not provided directly.
- FTRACK_API_USER: API user if not provided directly.
- FTRACK_EVENT_PLUGIN_PATH: default plugin search paths (for plugin_paths).
- FTRACK_API_SCHEMA_CACHE_PATH: default schema cache folder.

Notes
-----
- If required values are not supplied, the Session constructor raises a TypeError with guidance on how to provide them.
- The API is designed to be ergonomic: you can start with simple credentials and later opt into advanced features like caching, plugins, and the event hub as needed.

Sample workflow summary
-----------------------
1. Create a Session with server_url, api_key, and api_user.
2. Optionally enable auto_populate to fetch data on demand.
3. Optionally configure caching and schema caching for performance.
4. Optionally enable the embedded event hub for remote events.
5. Optionally discover and load plugins.
6. Use the session to interact with ftrack data; when needed, use the raw requests with SessionAuthentication for custom HTTP calls.
