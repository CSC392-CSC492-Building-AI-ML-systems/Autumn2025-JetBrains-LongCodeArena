Understanding sessions
=====================

A session represents an isolated interaction with an ftrack server. It holds the credentials, connection settings, caching strategy, and optional embedded event hub used for publishing and subscribing to events. Sessions are created with configuration that can be supplied directly or read from environment variables.

Key concepts
- Server connection: Each session targets a specific ftrack server URL.
- Authentication: Credentials (API key and user) are attached to outgoing requests.
- Caching: Sessions integrate with a layered cache, typically starting with a MemoryCache at the top level to improve performance.
- Event hub: An optional embedded event hub can be auto-connected to enable non-local events; if disabled, only local events can be published or subscribed to until a manual connection is made.
- Plugins: Plugins can be discovered and registered with optional arguments supplied at discovery time.
- Schema caching: Optional caching of API schemas to speed up repeated API interactions.

Creating a session
- server_url
  - The URL of the ftrack server to connect to (including port if needed).
  - If not provided, the environment variable FTRACK_SERVER is used as a fallback.
  - If still not provided, a TypeError is raised.
- api_key
  - The API key used for authentication.
  - If not provided, the environment variable FTRACK_API_KEY (or legacy FTRACK_APIKEY) is used.
  - If still not provided, a TypeError is raised.
- api_user
  - The username in ftrack to perform operations against.
  - If not provided, the environment variable FTRACK_API_USER is used, or the current OS user via getpass.getuser() as a fallback.
  - If still not provided, a TypeError is raised.
- auto_populate
  - If True (default), accessing entity attributes will automatically fetch data from the server when not already loaded.
- plugin_paths
  - A list of filesystem paths to search for plugins.
- cache
  - A cache instance implementing ftrack_api.cache.Cache, or a callable that returns a cache or None. If None, normal operation continues without a dedicated cache.
  - The provided cache is integrated into a pre-configured layered cache with a MemoryCache at the top level.
- cache_key_maker
  - An instance implementing ftrack_api.cache.KeyMaker to generate keys for cached objects.
  - Defaults to ftrack_api.cache.StringKeyMaker if not specified.
- auto_connect_event_hub
  - If True, an embedded event hub is automatically connected to the event server in a background thread.
  - If False, only local events are supported until the hub is manually connected via EventHub.connect.
  - Subscribing to events does not require a connected event hub.
- schema_cache_path
  - Path to a folder for schema caching. If not set, FTRACK_API_SCHEMA_CACHE_PATH is used. If neither is set, a temporary directory may be used.
  - Set to False to disable schema caching entirely.
- plugin_arguments
  - Optional mapping of keyword arguments passed to plugin registration functions during discovery. If a plugin’s signature does not accept some keys, the discovery mechanism will drop those keys and log a warning.

Authentication
- SessionAuthentication
  - A helper class that attaches ftrack session authentication information to HTTP requests.
  - Initialized with api_key and api_user.
  - When called with a request, it updates the request headers with:
    - ftrack-api-key: <api_key>
    - ftrack-user: <api_user>

Usage notes
- Environment variables enable easy deployment without hard-coding credentials:
  - FTRACK_SERVER for server_url
  - FTRACK_API_KEY (or legacy FTRACK_APIKEY) for api_key
  - FTRACK_API_USER for api_user
  - FTRACK_API_SCHEMA_CACHE_PATH for schema caching path
- If any of server_url, api_key, or api_user is missing after considering environment variables, session creation will raise a TypeError with guidance on how to supply the required value.
- The embedded event hub is designed to improve startup times by performing its connection in a background thread. Plugins that require a connected hub should verify hub status explicitly.
- The session manages recorded operations and a flag (record_operations) indicating whether operations should be tracked.

Representative usage
- Basic session creation (credentials supplied directly):
  - s = Session(server_url='https://my.ftrack.server', api_key='my_api_key', api_user='my_user')
- Session with environment-based credentials and optional features:
  - s = Session(auto_populate=True, plugin_paths=['/path/to/plugins'], auto_connect_event_hub=True)

Notes
- The session class is designed to centralize interaction with the ftrack server, including network communication, caching strategy, and optional event-based integrations.
- The implementation relies on several ftrack_api submodules for caching, events, plugins, and data structures.
