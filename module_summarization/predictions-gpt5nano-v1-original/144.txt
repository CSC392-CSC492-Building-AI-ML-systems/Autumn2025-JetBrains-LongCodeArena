Scheduling policies
===================

This document describes the policies used by Dask's distributed scheduler to select the preference of tasks and the preference of workers.

Task preference policy
----------------------

Goal
- Determine which ready task (a task whose dependencies are satisfied) should be scheduled next and on which worker.

Key concepts
- Readiness: Only tasks with all dependencies completed are considered for scheduling.
- Priority: Ready tasks carry a priority value; higher-priority tasks are preferred when multiple tasks are ready.
- Data locality and movement: When possible, tasks are scheduled on workers that already hold the task’s input data or where the data transfer cost is minimal.
- Memory considerations: Scheduling decisions account for current memory usage and pressure to avoid over-committing or evicting useful data inadvertently.
- fairness and determinism: Scheduling aims to avoid starvation and to provide predictable behavior, with optional deterministic replay modes for debugging.

Policies in practice
- Ready task selection: Among ready tasks, the scheduler favors higher-priority tasks and those with favorable data locality.
- Data transfer costs: If two ready tasks have comparable priority, the task with lower estimated data transfer requirements may be preferred to reduce network pressure.
- Extensibility: Scheduling behavior can be extended or customized through scheduler extensions (see Extensions).

Worker preference policy
------------------------

Goal
- Decide which worker should execute a given ready task, balancing load, data locality, and resource availability.

Key concepts
- Worker capacity: Each worker’s current load (number of tasks running), cpu resources, memory usage, and capabilities determine its suitability.
- Data locality: Preference is given to workers that already hold the necessary data in memory, reducing data movement.
- Memory pressure and resources: Decisions account for memory usage and available resources to avoid overcommitment.
- Data movement cost: When assigning tasks, the cost to fetch inputs/outputs to a worker is considered.

Policies in practice
- Locality-first scheduling: Favor running tasks on workers with the data needed by the task or with minimal transfer cost.
- Load balancing: Distribute work across the cluster to prevent hotspots and idle workers.
- Work stealing: Idle workers attempt to steal tasks from busier workers to improve overall utilization (see WorkStealing extension).
- Memory-aware placement: Memory pressure triggers adjustments to avoid excessive data retention or to trigger eviction strategies, guided by memory-related extensions.

Policy extensions and components
-------------------------------

- WorkStealing (stealing): The WorkStealing extension implements a policy where idle workers actively seek tasks from other workers to balance the workload and keep workers productive.
- ReplayTaskScheduler: Provides deterministic scheduling behavior for debugging by replaying task scheduling decisions.
- ShuffleSchedulerPlugin: Affects scheduling decisions related to data shuffles, influencing where shuffle-related tasks are placed.
- ActiveMemoryManagerExtension: Manages memory pressure and can influence task placement and eviction to maintain usable memory.
- MemorySamplerExtension: Monitors memory usage to inform scheduling decisions.
- DEFAULT_EXTENSIONS: A set of built-in extensions that influence scheduling behavior, including the above.

Notes on customization
- The distributed scheduler exposes policies via extensions, allowing customization of task and worker preferences to suit workloads and cluster characteristics.
- Policies interact: data locality, memory pressure, and load balancing collectively shape scheduling decisions, with extensions providing hooks for specialized behavior.

Terminology
- Ready task: A task whose dependencies have all completed and is eligible to be scheduled.
- Priority: A numeric value indicating the relative importance of a task for scheduling.
- Locality: The proximity of the task’s input data to a worker, influencing data transfer costs.
- Memory pressure: The current memory usage level on a worker or cluster, guiding avoidance of memory-related issues.

Examples of policy-driven decisions
- A high-priority task with data already on a worker is scheduled there, if feasible.
- An idle worker with low memory usage and no local data may steal a task from a busier worker to improve utilization.
- When memory pressure is detected, scheduling may favor tasks that produce smaller or more memory-efficient results or trigger eviction strategies via the memory-management extensions.

This completes the overview of the policies that govern task and worker preferences in Dask's distributed scheduler.
