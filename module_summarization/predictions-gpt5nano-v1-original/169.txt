Contributing to Qiskit Metal
=============================

Qiskit Metal is an open-source project that welcomes contributions from the community. This guide summarizes how to contribute code, documentation, tests, and examples in a consistent, high-quality way.

Code of Conduct
-------------
All contributors are expected to follow the project’s Code of Conduct. Be respectful, constructive, and collaborative in discussions, reviews, and contributions. If you encounter behavior that violates the Code of Conduct, please report it to the project maintainers.

Getting started
-------------
Prerequisites
- Python 3.8+ (or as required by the project at the time)
- Git for version control
- A development environment (virtualenv/conda) to isolate dependencies

Repository setup
- Fork the repository and clone your fork.
- Create a feature branch for your work (e.g., feature/new-analyses or fix/issue-123).
- Install development dependencies as described in the project’s setup instructions.
- Build and run the project locally to verify changes.

Branching and naming
- Use short, descriptive branch names.
- Follow the project’s existing branching strategy (feature/bugfix/maintenance).
- Include issue or PR numbers in branch names when applicable.

What to contribute
-------------
You can contribute in several ways:
- Bug fixes and small improvements
- New features or analyses
- Documentation improvements
- Tests and test improvements
- Examples and tutorials

Understanding the codebase
-------------
Qiskit Metal organizes its code into modular packages. A typical area you may encounter is:
- qiskit_metal.analyses: contains analysis helpers and calculators.
  - Submodules such as cpw_calculations and lumped_capacitive (example from the codebase) provide specific electrical calculations used by components.
- Other components and utilities that rely on these analyses for design and verification.

Documentation guidelines
-------------
- All public functions, classes, and modules should have docstrings following the project’s documentation style (numpy/style NumPy docstrings are encouraged for consistency).
- Use clear, precise descriptions, including input/output arguments, return values, and units.
- Where appropriate, add autosummary entries so the API surface is clearly exposed in the generated docs.
- Update documentation whenever you introduce new public APIs or change existing behavior.
- If you add a new module under qiskit_metal.analyses, document its purpose, usage, and any dependencies.

Testing
-------------
- Write tests for new functionality and ensure existing tests still pass.
- Follow the project’s testing conventions (usually pytest-based).
- Run the full test suite locally before submitting a PR.
- Include tests that cover edge cases and typical usage scenarios.

Code style and quality
-------------
- Follow the project’s coding standards (PEP 8 or project-specific rules).
- Write clear, readable code with meaningful names.
- Add or update unit tests for new functionality.
- Keep changes focused and small; prefer small, well-scoped commits.

Submitting changes
-------------
- Ensure tests pass and the code builds locally.
- Provide a concise, informative pull request description:
  - What was changed and why
  - Any related issues or design considerations
  - A brief testing plan or results
- Link any related issues or discussions.
- Be prepared to respond to code reviews and iterate on feedback.

Developing with analyses modules
-------------
If you add or modify an analysis module (for example, within qiskit_metal.analyses):
- Document the module and its usage clearly.
- Keep interface stability in mind; aim for backward-compatible changes when possible.
- Update the __init__ files to export new submodules if the project’s structure requires it.
- Include unit tests that exercise the new functionality (both typical use and edge cases).

Licensing and attribution
-------------
- All contributions must be licensed under the project’s license (Apache 2.0).
- Preserve copyright notices from original files where required.
- Clearly indicate any modifications to existing files.

Getting help
-------------
- If you’re unsure how to proceed or have questions about the codebase, open an issue or seek guidance in the project’s communication channels.
- Engage with maintainers and reviewers to align on design and implementation choices.

Thank you
-------------
Thank you for contributing to Qiskit Metal. Your work helps advance quantum hardware design tooling for the community.
