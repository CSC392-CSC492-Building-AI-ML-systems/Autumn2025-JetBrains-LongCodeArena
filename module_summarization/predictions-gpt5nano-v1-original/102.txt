Autowrap: Compile and wrap codegen output into Python-callable binaries
-----------------------------------------------------------------------

Overview
--------

This module provides a common interface for wrapping code generated by a codegen
tool into executable Python callables. It supports multiple backends (e.g., f2py,
Cython, SWIG) and aims to offer a one-button workflow to obtain high-performance
binaries. The current implementation covers f2py and Cython backends, with
possibilities to extend to other backends.

The wrapped callable is a Python function that evaluates numerically and is not a
symbolic Diofant expression. If the wrapped function needs to be used inside
symbolic expressions, use binary_function() which returns a Diofant Function
object. The binary callable is attached as the _imp_ attribute and is invoked when
numerical evaluation is requested (evalf) or with lambdify.

Example
-------

The following demonstrates typical usage:

.. code-block:: python

    expr = ((x - y)**25).expand()
    binary_callable = autowrap(expr)
    binary_callable(1, 2)
    -1.0

The callable can be used within symbolic expressions via binary_function:

.. code-block:: python

    f = binary_function('f', expr)
    2*f(x, y) + y
    y + 2*f(x, y)
    (2*f(x, y) + y).evalf(2, subs={x: 1, y: 2}, strict=False)
    0.e-190

When is this useful?
--------------------

- For computations on large arrays where Python loops are too slow and NumPy
  advanced indexing can be advantageous.
- For very long expressions that are invoked repeatedly, where the compiled
  binary is significantly faster than Diofant’s evalf().
- When code is generated with the codegen utility and you want to test the
  binaries from within Diofant immediately.
- To create customized ufuncs for use with NumPy arrays (see ufuncify).

When is this not the best approach?
-----------------------------------

- If you are primarily concerned with speed or memory and can optimize directly
  with the underlying wrapper tools and low-level code.
  The generated files may still serve as a useful starting point or reference;
  temporary files are left intact if you supply tempdir=.

- If array computations can be adequately handled by NumPy without the need for
  compiled binaries in another project.

Public API and main concepts
----------------------------

Exceptions
~~~~~~~~~~

- CodeWrapError: Generic error raised during code wrapping or compilation.

Core concepts
~~~~~~~~~~~~~

- CodeWrapper: Base class for code wrappers. Key responsibilities:
  - Manage the code generation lifecycle (generation, header handling, and
    empty file handling) and the temporary build environment.
  - Provide a wrap_code() method that orchestrates code generation, file
    preparation, compilation, and import of the resulting module.
  - Maintain a per-instance state including a generator, file path, build flags,
    and verbosity.

- DummyWrapper: A test wrapper that bypasses real compilation by generating a
  simple Python module to mimic a wrapped function. Useful for testing
  independently of backends.

- CythonCodeWrapper: Wrapper that uses Cython as the backend for code wrapping.
  Provides a Cython-based path to producing binary-callable code. (The
  concrete setup template for the Cython-based build is defined as setup_template.)

Notes on behavior
~~~~~~~~~~~~~~~~~

- wrap_code() builds the code in a working directory (temporary or user-specified),
  imports the generated module, and returns the wrapped function from the module.
- _module_counter is used to generate unique module names across successive wraps.
- If a temporary directory is used, it is cleaned up automatically after wrapping
  completes.
