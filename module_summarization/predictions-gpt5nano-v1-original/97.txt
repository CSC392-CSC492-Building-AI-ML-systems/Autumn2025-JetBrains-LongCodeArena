How Mitogen Works
==================

Overview
--------

Mitogen is a high-performance framework for remote execution, module loading, and service calls across process boundaries. It uses a central broker to coordinate communication with one or more remote “slave” contexts, injecting bootstrap code into new slaves to initialize execution with minimal startup overhead.

Architecture
------------

- Broker
  - Central controller that manages routing, dispatch, and lifecycle of remote operations.
- Router
  - Maintains routes to remote endpoints and handles cleanup when endpoints disconnect or are torn down.
- Slave context
  - The remote process, thread, or environment where code is actually executed. Each new slave receives bootstrap code to initialize Mitogen functionality.
- Channel and IO
  - Communication channels and buffered IO used to send messages between the broker and slave contexts. Default buffer sizes are tuned for performance.

Message protocol
----------------

Mitogen uses a compact, opcode-based message protocol. Each message includes a type (opcode) and a payload, enabling a range of remote operations:

- GET_MODULE (100): Retrieve a module object on the remote side.
- CALL_FUNCTION (101): Invoke a function within a remote module.
- FORWARD_LOG (102): Forward log records from remote to local logger.
- ADD_ROUTE (103): Add a new route to reach a remote endpoint.
- DEL_ROUTE (104): Remove an existing route.
- ALLOCATE_ID (105): Allocate a new identifier for a remote object.
- SHUTDOWN (106): Instruct the remote side to shut down.
- LOAD_MODULE (107): Load a module in the remote process.
- FORWARD_MODULE (108): Forward a module to be loaded remotely.
- DETACHING (109): Begin detaching or tearing down a connection.
- CALL_SERVICE (110): Call a remote service.
- STUB_CALL_SERVICE (111): Stub for a remote service call.

IS_DEAD sentinel
----------------

IS_DEAD (999) is a special value signaling disconnection or inability to route a message. Receiving this typically raises a ChannelError and cleans up the corresponding route.

Bootstrapping
-------------

On creation of a new slave context, Mitogen sends a bootstrap payload. This bootstrap is a minimal, self-contained portion of the code that prepares the remote side to run Mitogen without pulling in the entire module at once. This reduces startup size and ensures the remote context can initialize the full functionality needed for routing and remote execution.

Default buffers and encoding
----------------------------

- IO buffers default to 128 KiB. This size balances the number of IO loop iterations with memory usage and socket buffer considerations.
- The code relies on a Latin-1 codec and other encodings to reliably transmit data across process boundaries and to support zero-copy memory handling where possible.

Compatibility and portability
-----------------------------

- Mitogen is designed to work with Python 2 and Python 3, with shims and compatibility layers for types like BytesType, UnicodeType, and BufferType.
- The implementation uses flexible serialization (pickle and compatible variants) to preserve object state across boundaries.

Performance considerations
--------------------------

- Buffer sizing and memory management have a direct impact on throughput and CPU usage.
- The system is built with careful consideration of memoryview, zero-copy, and efficient framing to minimize overhead in remote calls and module transfers.

This document outlines the core concepts behind Generate how Mitogen works, focusing on architecture, the message protocol, bootstrapping behavior, and performance considerations that underpin typical usage scenarios.
