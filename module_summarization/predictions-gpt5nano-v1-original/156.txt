How Mitogen Works
==================

Overview
--------

Mitogen is a lightweight framework that enables fast, pluggable execution of code in remote contexts (slaves) while keeping a central broker responsible for routing messages between local and remote endpoints. A small bootstrap implementation is sent to every new slave context to minimize initial transfer and startup costs.

Architecture
------------

- Bootstrap core
  - The primary module implements the essential package functionality while keeping non-critical code separate to reduce the size of what is shipped to remote contexts. This bootstrap is what is sent to every new slave context.
- Broker and Router
  - A central broker coordinates communication and routing between local and remote contexts.
  - The Router maintains routing information and delivers messages to the correct recipient. Routes can be added or removed as contexts join or depart.
- Channel and IO
  - Communication between endpoints happens over channels, with a default buffer size tuned for performance.
  - Side.read and Side.write manage IO with a focus on efficient buffering and minimal overhead.
- Slave/Bootstrap contexts
  - Remote contexts start with a bootstrap that loads the necessary pieces to execute code, retrieve modules, and perform function calls without transferring the entire environment.

Message Protocol
----------------

Mitogen uses a compact, opcode-based message protocol. Core operation codes (examples from the codebase) include:

- GET_MODULE
- CALL_FUNCTION
- FORWARD_LOG
- ADD_ROUTE
- DEL_ROUTE
- ALLOCATE_ID
- SHUTDOWN
- LOAD_MODULE
- FORWARD_MODULE
- DETACHING
- CALL_SERVICE
- STUB_CALL_SERVICE

Key semantics

- Messages carry a reply_to field. A special sentinel value IS_DEAD signals disconnection or an inability to route. When IS_DEAD is observed, router callbacks are notified to clean up resources and routes.
- The router can forward messages to the appropriate remote or local endpoint, or manage routes as endpoints come online or go offline.

Performance and Buffering
-------------------------

- The default size for IO operations (Side.read/Side.write) and for socketpair buffers is 128 KiB. This size represents a balance between CPU overhead from many small IO operations and memory usage for large buffers.
- Large messages benefit from larger buffers, while smaller messages avoid excessive RAM usage and copying.

Compatibility and Portability
-----------------------------

- Mitogen is designed to run on both Python 2 and Python 3, with compatibility shims for type names and built-ins (e.g., BytesType, UnicodeType, long, etc.).
- Graceful fallbacks are provided for optional modules (e.g., cProfile, thread vs threading, cPickle vs pickle) to maximize compatibility across environments.
- Warnings related to deprecated modules or import timing may be emitted or suppressed to preserve stability.

Logging and Diagnostics
-----------------------

- Logging is centralized via MITOGEN and MITOGEN.IO loggers to separate internal and IO-related messages.
- The bootstrap and routing behavior are designed to be observable through these log channels to aid debugging and performance tuning.

Notes
-----

- The bootstrap approach ensures the remote slave starts with just enough code to bootstrap the environment, loading additional functionality on demand as needed.
- The code carefully handles differences between Python versions and platform quirks (e.g., WSL detection) to maintain reliable operation across environments.
