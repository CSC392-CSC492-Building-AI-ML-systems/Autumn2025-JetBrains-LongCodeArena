Built-in kernels for the Convolution module

Overview
The Convolution module provides built-in kernel objects that represent finite impulse responses used by convolution operations. The primary kernel types are Kernel (base class), Kernel1D, and Kernel2D. These kernels are constructed from array-like data and used to perform convolutions (including FFT-based implementations) within the modeling framework.

Kernel classes
- Kernel
  - The base class for all kernels. Encapsulates the kernel data, normalization, and boundary handling.
  - Kernels are typically centered on a central element. Sizes are expected to produce a well-defined center.
- Kernel1D
  - A one-dimensional kernel. Useful for separable convolutions and 1D convolution operations.
- Kernel2D
  - A two-dimensional kernel. Used for standard 2D convolution operations.

Creation and validation
- Kernels are created from 1D or 2D array-like data. The shape and content determine the kernel’s response.
- Kernel sizes and centering
  - Kernels are intended to have a well-defined center. In practice, kernels with even axis lengths are not supported by the built-in validation, and attempting to use an even-sized axis will raise an exception.
- Exceptions
  - KernelSizeError is raised for invalid kernel sizes.
  - raise_even_kernel_exception is provided to raise a proper error when an even-sized axis is encountered.

Normalization
- MAX_NORMALIZATION
  - A constant that defines the maximum allowed normalization value for kernels. Normalization is used to scale the kernel so that its sum matches a desired value, typically ensuring the convolution preserves the signal level in appropriate circumstances.

Boundary handling
- The module supports several boundary handling strategies when performing convolutions:
  - None: Default boundary handling.
  - "fill": Pad with a fill value.
  - "wrap": Periodic boundary conditions (wrap-around).
  - "extend": Extend the boundary by repeating edge values (or via an internal extension rule).
- These options control how data outside the input array is treated during convolution.

FFT support and performance
- _next_fast_lengths(shape)
  - Utility to determine FFT-friendly padding lengths for a given shape.
  - Prefers scipy.fft.next_fast_len when SciPy is available.
  - If SciPy is not available, falls back to a precomputed list of good sizes (_good_sizes) scaled by powers of 10 as needed.
  - If no suitable length can be found, raises ValueError with an explanatory message.
- _good_sizes
  - A precomputed list of dimensions that are considered efficient for FFT-based convolution.
- _good_range
  - The log10 range used to determine appropriate scaling for sizes when SciPy is unavailable.

Implementation notes
- The actual convolution operation in the Convolution module is implemented in a performance-optimized C routine (_convolveNd_c) for multi-dimensional input.
- The module provides a compatibility layer that accepts Kernel, Kernel1D, and Kernel2D objects and applies the appropriate convolution algorithm (direct or FFT-based) depending on the kernel and data characteristics.

Usage notes
- Access the built-in kernels via:
  - from astropy.modeling.convolution import Kernel, Kernel1D, Kernel2D
- Use appropriate boundary handling options to control how edge effects are treated during convolution.
- Ensure kernel sizes are valid (odd-sized axes are generally expected) to avoid KernelSizeError or related exceptions.
- When performance is critical, be aware of FFT-related padding lengths and choose sizes that align with the _good_sizes list or leverage SciPy’s fast FFT length computation when available.
