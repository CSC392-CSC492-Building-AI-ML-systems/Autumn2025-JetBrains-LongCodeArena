Working with Text Data
======================

Overview
--------
The feature_extraction.text submodule provides utilities to convert raw text documents into numerical feature vectors suitable for machine learning. It handles tokenization, normalization, stop-word handling, and weighting (e.g., TF-IDF), enabling scalable text representations for a variety of NLP tasks.

Public API
----------
- CountVectorizer
- TfidfVectorizer
- TfidfTransformer
- HashingVectorizer
- ENGLISH_STOP_WORDS
- strip_accents_unicode
- strip_accents_ascii
- strip_tags

Notes:
- Internal helpers (prefixed with an underscore) support preprocessing and analysis pipelines, such as _preprocess, _analyze, and _VectorizerMixin.decode. They are designed for use by the public vectorizers but can be referenced for advanced customization.

Key Concepts
------------
- Preprocessor: A callable applied before tokenization, typically used to normalize text (lowercasing, removing accents, etc.).
- Tokenizer: A callable that splits preprocessed text into tokens.
- Analyzer: A high-level pipeline that can replace preprocessor/tokenizer/ngrams steps. If provided, it short-circuits the default pipeline.
- N-grams: Grouping of tokens into sequences (e.g., bigrams, trigrams) to capture local word order.
- Stop words: Common words excluded from consideration (by default or via user-defined lists).
- Accent stripping: Optional normalization of accented characters, available via helpers and vectorizer parameters.
- Decoding: Support for multiple input sources (raw strings, file paths, file-like objects) via a decode method.

Detailed API and Behavior
-------------------------
1) _preprocess(doc, accent_function=None, lower=False)
   - Purpose: Chain optional text preprocessing steps.
   - Parameters:
     - doc: str, the text to preprocess.
     - accent_function: callable or None, optional accent normalization function.
     - lower: bool, whether to convert text to lowercase.
   - Returns: preprocessed string.

2) _analyze(doc, analyzer=None, tokenizer=None, ngrams=None, preprocessor=None, decoder=None, stop_words=None)
   - Purpose: Convert a single document into a sequence of tokens (ngrams) through an optional pipeline.
   - Parameters:
     - analyzer: callable or None, if provided, replaces the rest of the pipeline.
     - tokenizer: callable or None.
     - ngrams: callable or None.
     - preprocessor: callable or None.
     - decoder: callable or None, used to decode the input document.
     - stop_words: list or set or None.
   - Returns: list of tokens (possibly n-grams).

3) strip_accents_unicode(s) and strip_accents_ascii(s)
   - Purpose: Normalize accented characters.
   - strip_accents_unicode:
     - Converts accented Unicode symbols to their simple form.
     - Slower due to Unicode normalization but handles a wide range of scripts.
   - strip_accents_ascii:
     - Normalizes to ASCII by transliterating where possible and removing non-ASCII chars.
   - Returns: normalized string.

4) strip_tags(s)
   - Purpose: Remove basic HTML/XML tags using a simple regulator.
   - Returns: string with tags replaced by spaces.

5) _check_stop_list(stop)
   - Purpose: Resolve stop word specifications into a usable iterable.
   - Behavior:
     - If stop == "english": returns ENGLISH_STOP_WORDS.
     - If stop is a string other than a built-in, raises ValueError.
     - If stop is None: returns None.
     - Otherwise: returns a frozenset of the provided collection.

6) _VectorizerMixin
   - Purpose: Shared functionality for text vectorizers (e.g., CountVectorizer, TfidfVectorizer).
   - decode(self, doc):
     - Purpose: Decode the input to a string of unicode symbols.
     - Handles:
       - input == "filename": reads from a file path
       - input == "file": reads from a file-like object
       - bytes => decode using the vectorizer’s encoding and error policy
     - Returns: decoded string
   - Notes:
     - The mixin encapsulates common decoding logic, enabling vectorizers to accept multiple input sources.

Public Vectorizers and How They Relate
--------------------------------------
- CountVectorizer: Converts a collection of text documents to a matrix of token counts. Supports customization of analyzer, preprocessor, tokenizer, stop words, and n-grams. Often used as a baseline representation.
- TfidfVectorizer: Builds on CountVectorizer by weighting counts with term frequency-inverse document frequency (TF-IDF). Suitable for text classification and information retrieval tasks.
- TfidfTransformer: Converts a term-frequency matrix to TF-IDF representation. Used when you want to transform an existing count matrix into TF-IDF features.
- HashingVectorizer: Uses the hashing trick to produce feature vectors without building an explicit vocabulary, enabling memory-efficient processing for very large corpora (with potential hash collisions to consider).

Usage Examples
--------------
Basic CountVectorizer
- Goal: Convert a small corpus to a document-term matrix.

from sklearn.feature_extraction.text import CountVectorizer

corpus = [
    "This is the first document.",
    "This document is the second document.",
    "And this is the third one.",
    "Is this the first document?"
]

vectorizer = CountVectorizer(stop_words="english")
X = vectorizer.fit_transform(corpus)

# Feature names (sorted by feature index)
try:
    feature_names = vectorizer.get_feature_names_out()
except AttributeError:
    feature_names = vectorizer.get_feature_names()

print(feature_names)
print(X.shape)

TF-IDF Vectorization
- Goal: Weight terms by TF-IDF to downscale common terms and emphasize informative ones.

from sklearn.feature_extraction.text import TfidfVectorizer

tfidf = TfidfVectorizer(stop_words="english", ngram_range=(1, 2))
X_tfidf = tfidf.fit_transform(corpus)

# Access feature names
try:
    names = tfidf.get_feature_names_out()
except AttributeError:
    names = tfidf.get_feature_names()

print(names)
print(X_tfidf.shape)

Custom Analyzer
- Goal: Demonstrate using a user-defined analyzer.

def custom_analyzer(doc):
    # Example: lowercase, then simple whitespace tokenization
    return doc.lower().split()

vectorizer = CountVectorizer(analyzer=custom_analyzer)
X_custom = vectorizer.fit_transform(corpus)

Edge Cases and Considerations
-----------------------------
- Unicode and accents: Use strip_accents_unicode or strip_accents_ascii via preprocessing to normalize input when needed. Ensure you understand the performance trade-offs of Unicode-aware normalization.
- Input sources: Vectorizers can accept raw strings, file paths, or file-like objects. The decode method handles these transparently.
- Stop words: Providing a stop word list can significantly reduce feature dimensionality. Use built-in ENGLISH_STOP_WORDS or supply a custom set.
- N-grams: Increasing ngram_range increases feature space and may improve performance on some tasks but can lead to sparser matrices.
- Memory considerations: For very large corpora, consider HashingVectorizer to avoid building an explicit vocabulary, at the cost of potential hash collisions and less interpretable features.
- Edge cases: Non-text content, extremely long documents, or mixed multilingual corpora may require custom preprocessing and careful evaluation of tokenization and normalization steps.

Practical Tips
--------------
- Start with simple defaults: CountVectorizer with default analyzer and English stop words, then move to TF-IDF weighting for text classification.
- Experiment with ngram ranges (e.g., (1, 2) for unigrams and bigrams) to capture phrases.
- Use a consistent preprocessing pipeline when comparing models to ensure features are comparable.
- Validate inputs and handle potential decoding errors gracefully when reading from files.

This documentation provides a practical overview of working with text data using the module's capabilities, focusing on how to configure preprocessing, tokenization, and weighting to obtain meaningful text features for downstream machine learning tasks.
