Working with user security roles
================================

Overview
--------

Security roles define the access level and permissions granted to users (and groups) on the ftrack server. This module provides the ability to enumerate available roles, inspect role assignments for users, and create or remove role assignments. Proper use of security roles allows fine-grained control over what users can view or modify within the system.

Key concepts
------------

- Role: A named collection of permissions that can be assigned to users. Roles are defined on the server and may include descriptions and a set of permissions.
- Role assignment: The association of a Role with a User (or potentially a Group/Team). This determines the permissions the subject has.
- Roles manager: A session-level entry point to query roles, retrieve specific roles, and manage role assignments.

API surface (high level)
------------------------

- Roles manager (accessible from a Session, e.g. `session.security_roles`):
  - list_roles() -> List[Role]
  - get_role_by_name(name: str) -> Role or None
  - list_assignments(user=None, role=None) -> List[RoleAssignment]
  - assign(user, role) -> None
  - revoke(user, role) -> None

- Role
  - name: str
  - id: str
  - permissions: List[str]
  - description: str (optional)

- RoleAssignment
  - user: User
  - role: Role
  - assigned_at: datetime (optional)

- Exceptions
  - NotFoundError, AccessError, AlreadyAssignedError, etc. (as raised by operations)

Getting started
---------------

- Create an authenticated session to the ftrack server. You will typically need: server URL, API key, and API user.

Code example:
```
from ftrack_api import Session

session = Session(
    server_url='https://your-ftrack.server',
    api_key='YOUR_API_KEY',
    api_user='your_username'
)

# Obtain the roles manager
roles = session.security_roles
```

Listing available roles
-----------------------

Code example:
```
for role in roles.list_roles():
    print('{}: {}'.format(role.name, role.description or 'No description'))
```

Retrieving a specific role
--------------------------

Code example:
```
role_name = 'Editor'
role = roles.get_role_by_name(role_name)
if role is None:
    print('Role {} not found'.format(role_name))
else:
    print('Role found: {} (id: {})'.format(role.name, role.id))
```

Inspecting a user’s roles
-------------------------

Code example:
```
# Assuming you have a User object (or fetch by query)
user = session.query("User where username is 'jdoe'").one()

assignments = roles.list_assignments(user=user)
for a in assignments:
    print('User {} has role {}'.format(a.user.username, a.role.name))
```

Assigning a role to a user
-------------------------

Code example:
```
role = roles.get_role_by_name('Editor')
user = session.query("User where username is 'jdoe'").one()

roles.assign(user, role)
print('Assigned role {} to user {}'.format(role.name, user.username))
```

Revoking a role from a user
---------------------------

Code example:
```
role = roles.get_role_by_name('Editor')
user = session.query("User where username is 'jdoe'").one()

roles.revoke(user, role)
print('Revoked role {} from user {}'.format(role.name, user.username))
```

Important considerations and edge cases
---------------------------------------

- Administrative rights: Modifying roles typically requires administrator privileges. Attempting to assign or revoke roles without sufficient rights will raise an access error.
- Locking your own access: Removing the last administrative role from your own account can lock you out of performing privileged operations. Ensure you have an alternative admin account before revoking critical roles.
- Server state and caching: Role definitions and assignments are managed by the server. Be aware of potential propagation delays or caching effects in clients.
- Consistency and idempotency: Assigning an already-assigned role should be gracefully handled (e.g., no-op with a clear result). Revoking a non-assigned role should similarly report the no-op condition.
- Error handling: Network issues, invalid user/role references, or permission violations will raise exceptions such as NotFoundError or AccessError. Implement appropriate try/except blocks to surface meaningful feedback.
- Permissions model: The actual permissions tied to roles depend on server configuration. Review role definitions on the server to understand the effective access granted by each role.
- Auditing: Role assignments are typically part of server-side auditing. Consider logging changes for traceability in your tooling.

Usage scenarios
---------------

- On-boarding: Create a standard role (e.g., Viewer) and assign it to new users to enforce minimum access.
- Role review: Periodically enumerate roles and current assignments to ensure alignment with organizational policy.
- Access changes: Quickly grant temporary permissions by assigning a short-term role, then revoke when no longer needed.

Troubleshooting
---------------

- If list_roles() returns an empty list, verify server role definitions and your permissions.
- If get_role_by_name() returns None, confirm the role name is correct and that the role exists on the target server.
- If assignment fails due to permissions, ensure you have administrative rights and that the target user exists.

Edge-case scenarios
-------------------

- Cross-organization roles: Some servers may scope roles to projects or groups; verify scope implications before assignments.
- Migration: During server upgrades or migrations, role definitions may change; review role mappings if you encounter unexpected permission behavior.

Notes
-----

- The security roles API presented here assumes a standard Roles manager exposed by the session. If your environment exposes a different interface, adapt the usage accordingly.
- Always validate operations in a staging environment before applying changes to production.
