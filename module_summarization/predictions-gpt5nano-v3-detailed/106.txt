Introduction

hyperexpand.py is a utility within the symbolic math ecosystem that builds a knowledge base for expanding hypergeometric functions and Meijer G functions into named special functions. The module is primarily used to generate documentation and to support Sphinx-based display of these expansions. It relies on a collection of pre-defined formulae and a representation of hypergeometric objects to produce LaTeX-rendered equations for documentation purposes.

Overview of purpose

- Assemble a repository of hypergeometric/Meijer G expansion rules.
- Provide an automatically generated, nicely formatted mathematical presentation of these rules for documentation.
- Offer a programmatic entry point to create a LaTeX-friendly representation of each expansion.

Key concepts and relationships

- FormulaCollection
  - A container that holds a set of formula entries (formulae) describing how certain hypergeometric objects can be rewritten in terms of known special functions.
  - Accessed via c.formulae, where each element f contains information about the hypergeometric form and its closed-form expansion.

- Hyper_Function
  - A representation of a generalized hypergeometric function with upper parameters ap and lower parameters bq.
  - Used as the functional form in expansion rules.

- HyperRep_* and meijerg
  - HyperRep_* symbols denote named representations for specific hypergeometric expansions (e.g., HyperRep_power1, HyperRep_sqrts1).
  - meijerg denotes Meijer G-function representations.

- Function and class collaboration
  - The module constructs Eq objects that equate a hypergeometric expression to its rewritten closed form and converts these to LaTeX for documentation.
  - LaTeX formatting is produced via the latex utility for inclusion in the generated docs.

- add_formulae(formulae)
  - Primary API for populating a knowledge base with expansion rules.
  - Accepts a mutable list-like container (formulae) and appends Formula entries to it.
  - Internally defines helper functions add(ap, bq, res) and addb(ap, bq, B, C, M) to build simple and block-formula entries, respectively.
  - Demonstrates the intended structure of expansion rules and their associated metadata (parameters a, b, c, z, etc.).

- _mod1(x)
  - Utility to bucket values by their fractional part modulo 1.
  - Used internally to sort or categorize indices in the Meijer G expansion machinery, with special handling for numeric inputs.

Important parameters and return values

- add_formulae(formulae)
  - Purpose: Populate the provided collection with expansion rules for hypergeometric and Meijer G functions.
  - Parameter:
    - formulae: A mutable sequence (e.g., list) to which Formula objects will be appended.
  - Return: None. The function mutates the input by appending entries.

- The top-level script logic (for documentation generation)
  - Creates a FormulaCollection (c) and iterates over c.formulae to construct a human-readable LaTeX representation of each expansion.
  - Returns a string that is assigned to __doc__ for use in documentation rendering.

Usage examples

- Generating a documentation string from the formula collection

```python
from ..hyperexpand import FormulaCollection
from ..printing import latex
from ..core import Eq
from ..functions.special.hyper import hyper

c = FormulaCollection()
doc = ''

for f in c.formulae:
    obj = Eq(hyper(f.func.ap, f.func.bq, f.z),
             f.closed_form.rewrite('nonrepsmall'))
    doc += f'.. math::\n  {latex(obj)}\n'

__doc__ = doc
```

- Populating a formula knowledge base

```python
from .hyperexpand import add_formulae

# Prepare a container to receive formula entries
formulae = []
add_formulae(formulae)

# formulae now contains Formula objects describing hypergeometric expansions
```

Edge cases and important considerations

- Order reduction and polynomial reduction
  - If b_j - a_i is a non-negative integer under certain index configurations, gamma quotients reduce to polynomials, enabling order reduction of the G-function.
  - The expansion machinery includes theoretical mechanisms (e.g., paired index theorems) described in the documentation, but practical implementation notes indicate that these reductions are complex and not fully automated in all cases.

- Inverse-shift operators and reachability
  - Four inverse-shift operators exist to move indices (A_J, B_J, C_J, D_J), but their applicability depends on the absence of cancellations (e.g., a_J != b_i for certain i).
  - Determining reachability between index quadruples is non-trivial; hence the implementation uses hand-built routines to match and instantiate formulas.

- Practical completeness
  - The code includes TODOs and partial examples (an incomplete addb invocation is visible in the snippet). The documentation reflects the intended design and the current state of the expansion knowledge base.

- Documentation-oriented purpose
  - The module’s explicit goal is to generate a documentation-friendly representation of the expansion rules (via __doc__ and LaTeX), rather than delivering a fully self-contained runtime expansion engine.

Notes for developers

- The knowledge base is intended to be extensible; adding more rules via add_formulae is straightforward but requires correct Hyper_Function and HyperRep mappings.
- The documentation generation relies on LaTeX rendering; ensure the Sphinx environment has the necessary math rendering support.
- The module documents itself by transforming formulae into LaTeX and embedding them into __doc__, making it important to keep the formatting and representations consistent with the rest of the project’s math rendering pipeline.
