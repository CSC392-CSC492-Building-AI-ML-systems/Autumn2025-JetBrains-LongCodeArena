Tensor Attributes
=================

Introduction
------------
Tensor Attributes describe metadata associated with tensor-typed values and
tensor-producing operations within the OneFlow MLIR dialect. They encode
information such as shape, element type (dtype), and layout, enabling
optimizers, code generators, and runtime components to reason about tensors
without requiring a full tensor instance. Attributes are used alongside
tensor types to annotate operations and results, guiding compilation and
execution.

Concepts and relationships
--------------------------
- Tensor Attribute concept: A lightweight metadata container attached to
  tensor-typed values or operations. Attributes convey shape, dtype, and optional
  layout or memory-related information.
- Relationship to tensor types: The tensor type encodes static information about
  shape and element type. Attributes provide additional annotations or override
  information when necessary (e.g., runtime-provided shapes, layout hints).
- Attribute containers: Attributes are typically organized as dictionaries
  (e.g., key-value pairs) or as structured attribute types (e.g., arrays, single
  values) that can be attached to operations or values in MLIR-like dialects.
- Optional vs required: Some attributes may be optional and only needed for
  certain operations or optimization passes. Others are required to preserve
  correctness or enable specific code paths.

Common attributes (conceptual)
------------------------------
- shape: Encodes tensor dimensions. Supports static shapes (e.g., [1, 3, 224, 224])
  and dynamic shapes (e.g., [-1, 3, 224, 224] or a dedicated dynamic symbol).
- dtype (element type): The scalar data type of tensor elements (e.g., f32, i64).
- layout: Memory layout hint or permutation (e.g., "NCHW", "NHWC", or custom layouts).
- strides: Optional explicit stride information for non-default layouts.
- offset: Optional byte offset for the tensor data within a larger buffer.
- tiling or blocking: Optional hints used by optimizers or runtimes for tiling.

API and usage patterns (high level)
-----------------------------------
- Attaching attributes: When defining or calling an operation, attach a tensor
  attribute dictionary to convey necessary metadata for correctness and
  optimization.
- Querying attributes: Tools and passes can query attributes to determine shapes,
  dtypes, or layouts without inspecting actual tensor data.
- Synchronization with types: Ensure that attributes are consistent with the tensor
  type carried by the operation. Inconsistent attributes and types should be treated
  as errors or trigger canonicalization passes.

Code examples
-------------
Note: The following snippets illustrate typical usage in a MLIR-like syntax. They
are representative and may differ in exact syntax in your dialect.

- Declaring a tensor with shape, dtype, and layout attributes
.. code-block:: mlir

{
  "shape" = [1, 3, 224, 224],
  "dtype" = f32,
  "layout" = "NCHW"
}

- Dynamic shape example
.. code-block:: mlir

{
  "shape" = [-1, 3, 224, 224],   // dynamic batch size
  "dtype" = f32,
  "layout" = "NCHW"
}

- Attaching attributes to an operation (conceptual)
.. code-block:: mlir

%result = "oneflow.conv2d"(%input)
  : tensor<1, 3, 224, 224>, tensor<64, 3, 7, 7>
attributes {
  "shape" = [1, 64, 112, 112],
  "dtype" = f32,
  "layout" = "NCHW"
}

Edge cases and considerations
-----------------------------
- Dynamic shapes: Represent dynamic dimensions with a special sentinel (e.g., -1)
  or a dedicated dynamic symbol. Ensure downstream components interpret dynamic
  sizes consistently.
- Inference vs explicit attributes: If the tensor type already encodes shape and
  dtype, attributes may be optional or used for overrides. Validate consistency
  during canonicalization.
- Inconsistencies: If attributes contradict the tensor type (e.g., shape or dtype
  mismatch), report an error or apply a well-defined resolution rule.
- Layout dependencies: Layout hints may affect optimization passes (e.g., layout
  conversion). Treat layout as advisory unless it is required to preserve correctness.

Best practices
--------------
- Keep attributes synchronized with the associated tensor type to avoid drift.
- Use attributes to convey information that is expensive to derive at runtime
  or that is necessary for specific optimization passes.
- Prefer explicit attributes for dynamic or non-default information, and rely on the
  type system for static, invariant information.

Usage overview
--------------
Tensor Attributes provide a lightweight mechanism to describe tensor metadata
without materializing tensor data. They are read by compiler passes, runtime
components, and code generators to optimize and validate tensor-based operations.
