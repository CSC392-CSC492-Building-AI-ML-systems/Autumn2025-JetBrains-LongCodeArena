Working with entities
=====================


Overview
--------

This topic documents how to work with ftrack entities via the core API exposed by the Session
class. Entities represent objects tracked in an ftrack server (for example projects, tasks, assets,
and more). The Session provides authentication, lazy-loading of entity attributes, and a
caching layer to optimize performance. Entities are typically retrieved and queried through the
Session and related query facilities, and can be populated on demand if auto_populate is enabled.

Key concepts include:
- Session: a dedicated interaction context with a specific server, API key and user.
- SessionAuthentication: HTTP headers required to authenticate requests against the ftrack server.
- Lazy loading: attributes are fetched from the server on first access when auto_populate is True.
- Caching: a layered cache is used to improve performance and reduce repeated server calls.
- Event hub integration: optional embedded event hub support for non-local events.

The following sections describe the main classes, their relationships, and practical usage patterns.


Session and authentication
--------------------------

The Session class is the primary entry point for interacting with an ftrack server. It encapsulates:
- server_url: the target ftrack server (including port if needed).
- api_key and api_user: credentials used for authentication.
- auto_populate: when True, missing entity attributes are fetched automatically from the server on access.
- plugin_paths, cache, cache_key_maker: extensibility and caching configuration.
- auto_connect_event_hub: controls whether an embedded event hub is connected in the background.

Key details:
- If server_url is not provided, the value is read from the FTRACK_SERVER environment variable.
- If api_key is not provided, it may be read from FTRACK_API_KEY (and FTRACK_APIKEY for backwards compatibility).
- If api_user is not provided, it can be read from FTRACK_API_USER or derived from the current user via getpass.
- The Session initializes a layered cache, with a MemoryCache at the top level for fast access.
- The Session raises clear errors if required information is missing.

SessionAuthentication attaches the necessary headers to HTTP requests:
- ftrack-api-key: the API key
- ftrack-user: the API user

Usage example
-------------

Create a session to connect to an ftrack server. This example demonstrates explicit credentials; environment-based configuration is also supported.

.. code-block:: python

    from ftrack_api import Session

    # Explicit credentials
    session = Session(
        server_url='https://your-ftrack-server',
        api_key='your_api_key',
        api_user='your_username',
        auto_populate=True
    )

    # Alternatively, rely on environment variables:
    # FTRACK_SERVER, FTRACK_API_KEY, FTRACK_API_USER

Notes:
- If required values are missing, the constructor will raise a TypeError with guidance on how to supply them.
- The session will prepare a cache and, if requested, spawn a background thread to connect an embedded event hub.


Working with entities and queries
---------------------------------

Entity access
-------------
Entities are retrieved and manipulated via the Session. With auto_populate enabled, accessing an
entity attribute will trigger a server fetch for any values not yet loaded.

.. code-block:: python

    # Example: obtain a project by id and access its attributes
    project = session.query("select id, name, full_name from Project where id = 'proj123'").first()
    print(project.name)          # may trigger lazy loading if not yet populated
    print(project.full_name)

Querying entities
-----------------
The API supports a query language (FQL/Ftrack Query Language) via ftrack_api.query. This enables
fetching multiple entities in a single operation and iterating results in a Python-friendly way.

.. code-block:: python

    # Retrieve a list of projects
    results = session.query("select id, name from Project where archived is false")
    for project in results:
        print(project['id'], project['name'])

Entity relationships and factories
----------------------------------
Entities are created and managed through the ftrack entity layer (entity.factory, entity.base, etc.).
The Session composes with these components to provide a cohesive API for creating, querying, and
manipulating server-side objects. While specific factory methods are part of the internal module
set, typical usage patterns involve obtaining a session, running a query, and operating on the
returned entity objects.

Caching considerations
----------------------
- A top-level MemoryCache is used to speed up recurrent access.
- The layered cache design helps avoid redundant server requests, especially in long-running
  sessions or batch operations.
- If you configure a custom cache, ensure it implements the ftrack_api.cache.Cache interface.

Event hub integration
---------------------
If auto_connect_event_hub is True (default in many setups), an embedded event hub will be connected
to the event server in the background. This enables publishing and subscribing to non-local events.
If you opt out of automatic hub connection, you can connect it later using the event hub API.

.. note::

    Subscribing to events does not require a connected event hub; publishing and
    subscribing to local events remains possible without a connected hub.

Edge cases and considerations
------------------------------
- Environment variable configuration: Missing server or credentials will raise a TypeError.
- Lazy population: Auto-population relies on server availability; network issues can raise exceptions when
  accessing attributes for the first time.
- Plugin support: Plugins may be discovered and registered during session setup; incompatible signatures will be adjusted with warnings.

Usage patterns
--------------

- Start with a minimal session using environment configuration, then progressively enable features
  such as auto_populate, caching strategies, and the event hub.

- Use query-based retrieval for efficient data access and to leverage server-side filtering.

- Rely on lazy loading to minimize initial data transfer; access attributes as needed.

API surface and relationships
------------------------------

- ftrack_api.Session: Core class to configure and manage a connection to the ftrack server, with
  support for lazy loading, caching, and optional event hub integration.
- ftrack_api.SessionAuthentication: Helper used by the session to attach authentication headers to requests.
- ftrack_api.cache: Provides cache implementations (MemoryCache, DiskCache, etc.) used by Session.
- ftrack_api.query: Utilities to perform server-side queries against the ftrack data model.
- ftrack_api.entity.*: Core entity model and behavior (factories, bases, and specific entity types) used
  in conjunction with the Session to represent and manipulate server-side objects.
- ftrack_api.event.hub: Event hub integration for non-local event publishing/subscription.

Practical checklist
-------------------

- Ensure server_url and credentials are correctly configured (environment variables or explicit arguments).
- Decide on auto_populate and caching strategy based on application needs.
- If using an embedded event hub, understand its impact on startup time and background threading.
- Handle network or server errors gracefully, especially around lazy attribute population.

Usage examples
--------------

Minimal session creation and a simple query:

.. code-block:: python

    from ftrack_api import Session

    session = Session(
        server_url='https://your-ftrack-server',
        api_key='your_api_key',
        api_user='your_username',
        auto_populate=True
    )

    # Simple query to fetch projects
    projects = session.query("select id, name from Project")
    for project in projects:
        print(project['id'], project['name'])

Accessing a lazily populated attribute:

.. code-block:: python

    project = session.query("select id, name, full_name from Project where id = 'proj123'").first()
    # First access may trigger a server fetch for missing data
    print(project.name)  # auto-populated on demand if needed

Note: Adapt the query language and access patterns to your specific server version and API surface, as details may evolve.
