sesans_fitting.rst
===================

Fitting SESANS Data
-------------------

Overview
========
This documentation describes the SESANS fitting workflow within sasmodels, focusing on parameter discovery across models and how to inspect which models use which parameters. The sesans_fitting module provides a small command-line utility to enumerate kernel parameters used by all models, enabling developers to understand parameter coverage and dependencies when fitting SESANS data.

Module purpose
==============
- Facilitate exploration of parameters across SESANS-related models.
- Identify which models share or require a given parameter.
- Support filtering by parameter kind (volume, orientation, sld, or none/any) to aid model selection and fitting strategies.

Key concepts
============
- Kernel parameters: The parameters that influence the kernel (scattering model) used by a model.
- Parameter kinds:
  - volume: size-related parameters (e.g., radius, length)
  - orientation: orientation-related parameters (e.g., angles)
  - sld: scattering length density related parameters
  - none/any: miscellaneous or all-parameters filters
- Model information: Each model exposes a set of kernel_parameters via model_info.parameters.kernel_parameters.

How it relates to SESANS fitting
==============================
SESANS data is modeled by the same kernel-parameterized scattering models used for SANS, adapted for the SESANS transform. Before fitting, it is often useful to know:
- Which parameters are available across models.
- Which models share a given parameter.
- Which parameter kinds are relevant for a given fitting scenario (e.g., size parameters vs. orientation parameters).

Usage and API
=============
The sesans_fitting functionality is available via a small CLI and supporting Python helpers.

- list_pars: Print all parameters across all models, optionally including the models that use each parameter.
- find_pars: Internal helper that builds a mapping {parameter_name: [model, model, ...]}.
- main: Command-line entry point that wires argparse and prints results.

Main functions
--------------
- find_pars(kind=None)
  - Purpose: Collect all parameters used by all models, optionally filtering by kind.
  - Return: A dictionary {parameter_name: [model_name, ...]}.
  - Parameters:
    - kind: one of 'volume', 'orientation', 'sld', None, or 'any' (see note below).

- list_pars(names_only=True, kind=None)
  - Purpose: Print parameter information for all models.
  - Return: None (prints to stdout).
  - Parameters:
    - names_only: If True, print only parameter names; if False, print parameter: model list mappings.
    - kind: Filter by parameter kind (as above).

- main()
  - Purpose: Command-line entry point.
  - Behavior: Parses options and calls list_pars with the appropriate arguments.
  - CLI options:
    - -v / --verbose: Print models that use each parameter.
    - kind: Optional positional argument; one of volume, orientation, sld, none, any.

Notes on parameter kind handling
--------------------------------
- The CLI supports filtering by kind with choices: volume, orientation, sld, None, any.
- The internal mapping treats:
  - 'any' -> all parameters
  - 'none' -> parameters that are not categorized as volume/orientation/sld (if such a distinction exists)
  - Other values pass through as-is
- Some models may not expose kernel_parameters; such models are effectively ignored by the listing utilities.

Usage examples
==============
List all parameters (names only)
--------------------------------
$ python -m sasmodels.list_pars

List all parameters with corresponding models
--------------------------------------------
$ python -m sasmodels.list_pars -v

List only parameters of a specific kind (e.g., volume)
-----------------------------------------------------
$ python -m sasmodels.list_pars volume

Find parameters across all models (internal helper example)
----------------------------------------------------------
from sasmodels.list_pars import find_pars
pars = find_pars()
print(len(pars), "parameters found across models")

Edge cases and considerations
=============================
- Some models may not expose any kernel_parameters. Such models do not contribute to the parameter table.
- The parameter naming is model- and implementation-specific. Ensure consistent interpretation when aggregating across models.
- When using kind filtering, ensure the provided kind aligns with the model parameter taxonomy; some models may not categorize parameters strictly by the three canonical kinds.

Code references (conceptual)
