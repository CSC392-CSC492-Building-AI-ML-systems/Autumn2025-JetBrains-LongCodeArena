Assignments and allocations in the ftrack API provide mechanisms to model who is responsible for work on a given object (assignment) and how that work is planned in terms of time/resources (allocation). This document outlines concepts, common patterns, and practical usage for working with assignments and allocations via the API.

Assignments and allocations: concepts
- Assignment
  - A link between a trackable object (e.g., an asset, task, or similar entity) and a user.
  - Captures who is responsible for work on the object, often including role, status, and time window information.
- Allocation
  - A representation of planned or allocated work time/resources for an assignment or object.
  - Typically includes start/end times, duration (hours), and status (planned, confirmed, completed).

Key concepts and relationships
- An object (task, shot, asset, etc.) can have zero or more assignments.
- An assignment ties a specific user to work on that object, potentially with a role and a time window.
- Allocations describe the time/resources allocated to an assignment or its object, enabling planning across multiple users or tasks.
- The exact attributes and relationships are governed by your server schema; attribute names may vary. Always refer to your server’s schema for the precise fields.

API usage overview
- Querying
  - Retrieve assignments for an object or a user.
  - Retrieve allocations for an assignment or object.
- Creating
  - Create a new assignment linking a user to an object with a role and timing information.
  - Create an allocation specifying the scope (assignment or object), duration, and status.
- Updating
  - Update assignment attributes (e.g., status, role, time window).
  - Update allocation attributes (e.g., hours, start/end times).
- Deleting
  - Remove an assignment or an allocation, observing any cascading effects or requirements (such as ensuring no active allocations remain when removing an assignment).

Important considerations and edge cases
- Time zones and formatting
  - Use UTC or consistent time zones; ensure start/end values are in a predictable format.
- Overlapping allocations
  - Be mindful of potential overlaps when multiple allocations concern the same user or object.
- Permissions
  - Ensure the API user has appropriate permissions to read/write assignments and allocations.
- Auditing and history
  - Consider how updates are logged and how to access historical states if needed.
- Schema variations
  - Attribute names and relations vary by server configuration; adapt code to your server’s schema.
- Performance
  - Loading large numbers of assignments or allocations can be expensive; consider paginating or filtering on the server side.

Create: create a new assignment
```python
# Pseudo-code illustrating typical pattern; adapt to your API surface
assignment_data = {
    'object': object,        # reference to the parent object
    'assignee': user,        # reference to the User
    'role': 'Engineer',
    'start': '2025-01-01T09:00:00Z',
    'end': '2025-01-15T17:00:00Z',
    'notes': 'Initial allocation'

```

Create: create an allocation
```python
allocation_data = {
    'assignment': assignment,  # or assignment.id
    'hours': 40.0,
    'start': '2025-01-01T09:00:00Z',
    'end': '2025-01-21T17:00:00Z',
    'status': 'planned'
}
allocation = session.create('Allocation', allocation_data)
session.commit()
```

Query: fetch allocations for an assignment
```python
allocations = session.query("Allocation where assignment_id = {}".format(assignment.id)).all()
for al in allocations:
    print("Hours:", al.hours, "Start:", al.start, "End:", al.end, "Status:", al.status)
```

Update: modify an assignment
```python
# Change the status of an assignment
assignment.status = 'completed'
session.commit()
```

Update: modify an allocation
```python
# Example: adjust the allocated hours
allocation.hours = 32.0
session.commit()
```

Delete: remove an assignment or allocation
```python
# Remove an allocation
session.delete(allocation)
session.commit()

# Remove an assignment (subject to server rules and cascading effects)
session.delete(assignment)

```
