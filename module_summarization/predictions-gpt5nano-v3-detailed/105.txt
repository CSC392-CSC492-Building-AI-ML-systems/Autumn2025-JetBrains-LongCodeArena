Introduction
This module auto-generates a documentation string describing Meijer G-function related formulas. When imported, it populates __doc__ with a structured, Sphinx-friendly block that lists elementary functions and functions involving specific symbols, along with their corresponding equations and optional conditions.

Overview of purpose
- Central goal: expose the mathematical definitions and relationships of elementary and derived G-functions for documentation purposes.
- The content is produced at import time by inspecting a lookup table built from the Meijer G-function rules.
- The resulting docstring is intended to be rendered by Sphinx (via the math directive) to produce readable, math-capable documentation.

Internal data structures and workflow
- t: A private lookup table
  - Type: dict[tuple[type, ...], list]
  - Initialization: collections.defaultdict(list)
  - Populated by: _create_lookup_table(t)
- Each key in t (about) is a tuple of types indicating the group of functions. The corresponding value (category) is a list of entries.
- Each entry in a category is a 4-tuple: (formula, gs, cond, hint)
  - formula: The left-hand side expression for the equation to display.
  - gs: Either a list of (factor, expression) pairs, used to construct the right-hand side g, or something else that signals a generated form.
  - cond: A condition for the equation. If True, no condition is shown; otherwise it is appended as a LaTeX-formatted phrase.
  - hint: Reserved for potential future use (not directly used in doc generation in this version).

Documentation generation logic
- The module iterates over sorted(t.items(), key=default_sort_key).
- Header construction:
  - If about == (): the header is “Elementary functions:”
  - Else: the header is “Functions involving <names>:” where <names> are the latex-rendered first occurrences of each function in the first category formula.
- For each (formula, gs, cond, hint) in a category:
  - g construction:
    - If gs is not a list: g = Symbol('\\text{generated}')
    - Else: g = Add(*[fac*f for (fac, f) in gs])
  - obj = Eq(formula, g)
  - cond rendering:
    - If cond is True: no condition text is appended
    - Else: append ',\text{ if } <latex(cond)>'
  - Appends a math directive:
    - .. math::
      <latex(obj)><cond>

Public API and public-facing behavior
- __doc__: The module-level documentation string containing the entire generated content.
- Note: There are no public functions or classes exposed by this module; its purpose is to present the generated documentation content for Sphinx.

Important considerations and edge cases
- The header generation depends on the presence and content of about.
  - about == () yields the “Elementary functions” section.
  - Otherwise, the “Functions involving …” header is created, listing functions by inspecting the first formula in the first category entry.
- The right-hand side g depends on gs:
  - A non-list gs yields a generated symbol: Symbol('\\text{generated}')
  - A list gs yields a sum Add(*[fac*f for (fac, f) in gs])
- Condition handling:
  - cond == True -> no condition text is shown
  - cond == any non-boolean value -> latex(cond) is embedded into the text as a TeX-formatted condition
- The math content relies on Sphinx with the math extension and a LaTeX-capable renderer.

Usage notes
- In Sphinx, include the generated module docstring via an automodule directive to render the content:
  - .. automodule:: yourpkg.meijerint.g_functions
     :no-members:
- At runtime, the module exposes its documentation via __doc__. For exploration:
  - from yourpkg.meijerint import g_functions
  - print(g_functions.__doc__)  # view the generated documentation block

Usage examples
- Sphinx inclusion (reST):

Representative structure of the generated documentation (illustrative)

Notes
- The exact content of the generated documentation depends on the data produced by _create_lookup_table; as such, regenerating the module (e.g., after updating the underlying rules) will update the docstring accordingly.
- This module is primarily for documentation purposes and is not intended as a runtime API for functional usage beyond exposing __doc__.
