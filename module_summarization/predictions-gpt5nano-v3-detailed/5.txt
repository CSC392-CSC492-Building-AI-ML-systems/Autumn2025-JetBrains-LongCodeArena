Publishing versions
==================

This module provides the infrastructure for publishing software versions to an ftrack-based publishing system. It encapsulates server communication, authentication, and optional caching to support robust, repeatable publish workflows.

Overview
--------

- Establish authenticated communication with an ftrack server.
- Manage session state, including optional automatic population of entity data and event hub connectivity.
- Support pluggable caching for performance and offline workflows.
- Provide a simple, consistent mechanism for attaching authentication to HTTP requests when performing publishing operations against the server.

Key components
--------------

- SessionAuthentication
  - Purpose: Attach ftrack authentication information to outgoing HTTP requests.
  - Core behavior: Injects headers ftrack-api-key and ftrack-user into each request.
  - Key members:
    - api_key: API key used for authentication.
    - api_user: User associated with the API key.
  - Typical usage: Use as a requests auth handler for HTTP requests involved in publishing.

- Session
  - Purpose: A consolidated namespace for interacting with an ftrack server.
  - Core behavior:
    - Validates required connection information (server_url, api_key, api_user).
    - Honors environment variables (FTRACK_SERVER, FTRACK_API_KEY, FTRACK_API_USER) as fallbacks.
    - Supports optional auto-population of entities, plugin loading, and event hub integration.
    - Configures a layered cache with an in-memory top layer by default.
  - Key members (illustrative):
    - _server_url: Target server URL.
    - _api_key: API key for authentication.
    - _api_user: API user identity.
    - recorded_operations / record_operations: Manage pending operations.
    - cache_key_maker: Mechanism to generate cache keys.
  - Important behavior:
    - If required values are missing, a TypeError is raised with guidance on environment variables or parameters.

Parameters and return values
----------------------------

Session constructor
- server_url (str): URL of the ftrack server to connect to (including port if needed).
  - If not provided, the environment variable FTRACK_SERVER is consulted.
- api_key (str): API key used for authentication.
  - If not provided, FTRACK_API_KEY (or FTRACK_APIKEY for backward compatibility) is consulted.
- api_user (str): Username for the API user.
  - If not provided, FTRACK_API_USER is consulted, otherwise fall back to a system user via getpass.
- auto_populate (bool): If True, accessing entity attributes may fetch data from the server on demand.
- plugin_paths (list[str]): Paths to search for plugins. If not provided, FTRACK_EVENT_PLUGIN_PATH is used.
- cache: Cache instance or callable returning a cache to be used by the session.
- cache_key_maker: Object implementing the KeyMaker interface to generate cache keys.
- auto_connect_event_hub (bool or None): If True, embed an event hub for non-local events; otherwise only local events.
- schema_cache_path (str or False or None): Path to store schema cache. False disables caching entirely.
- plugin_arguments (dict): Optional arguments passed to discovered plugin register functions.

Return values:
- Session instance: A configured session ready to perform server interactions (publishing workflows included in higher-level calls or downstream APIs).

Usage examples
--------------

Initialization (with explicit credentials)
.. code-block:: python

   from ftrack_api import Session

   session = Session(
       server_url="https://ftrack.example.com",
       api_key="your_api_key",
       api_user="your_username",
       auto_populate=True,
       cache=None,
       auto_connect_event_hub=False
   )

Publishing workflow (conceptual)
- Retrieve or create the target Version entity and associate it with the appropriate Project.
- Populate version metadata (name, description, release notes, etc.).
- Commit/publish changes to the server via the ftrack API.

Note: The exact publishing steps depend on your server's API surface and the ftrack API you use (entity operations, REST endpoints, or higher-level interfaces). The Session class provides authenticated access and a stable session for these operations.

Attaching authentication to HTTP requests (illustrative)
.. code-block:: python

   import requests
   from ftrack_api import SessionAuthentication

   # Build an authentication object
   auth = SessionAuthentication(api_key="your_api_key", api_user="your_username")

   # Use a requests.Session and attach the authentication handler
   http = requests.Session()
   http.auth = auth  # requests will invoke auth(request) to modify headers

   # Example publish request (endpoint and payload depend on your server)
   payload = {
       "name": "Release 1.2",
       "description": "Publishing version 1.2 to production"
   }
   resp = http.post("https://ftrack.example.com/api/versions", json=payload)
   resp.raise_for_status()

Edge cases and considerations
------------------------------

- Missing credentials:
  - If server_url, api_key, or api_user are not provided and cannot be inferred from environment variables, a TypeError is raised with guidance.
- Environment-based configuration:
  - The module supports fallback to environment variables for server URL and credentials, facilitating secure and automation-friendly deployments.
- Schema caching:
  - schema_cache_path enables caching of schema information to improve startup times for repeated sessions. Setting to False disables caching.
- Event hub connection:
  - auto_connect_event_hub controls whether the embedded event hub connects automatically. If your workflow depends on non-local events, consider enabling this and ensuring the hub is connected before publishing.
- Caching behavior:
  - The session places the provided cache under a pre-configured layered cache with MemoryCache at the top. Be mindful if you customize the cache layers to avoid unexpected cache invalidation or growth.
