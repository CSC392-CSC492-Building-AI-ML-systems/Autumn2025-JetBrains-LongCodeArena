Migrating to Nextcord: Widget support (WidgetChannel, WidgetMember, Widget)

Introduction
------------
This document describes Nextcord's widget data model and how to migrate to using Nextcord's widget classes. The module provides lightweight representations for guild widgets, including partial channels and members as reported by the Discord widget endpoint. These classes are designed to be convenient when handling widget data without requiring full guild state.

Overview of the widget module
-----------------------------
The widget module exposes three main classes:

- WidgetChannel
  - A lightweight representation of a channel included in a guild widget.
- WidgetMember
  - A partial representation of a guild member visible in the widget.
- Widget
  - Represents a guild widget, containing accessible channels and online members.

Key concepts and relationships
------------------------------
- WidgetChannel
  - Fields: id, name, position
  - Behaviors:
    - __str__ returns the channel name
    - __repr__ provides a debug representation
    - mention returns a string that can be used to mention the channel in messages
    - created_at uses the channel’s ID (snowflake) to derive creation time
- WidgetMember
  - Subclass of BaseUser
  - Fields include: id, name, discriminator, bot, avatar, nick, status, activity, deafened, muted, suppress, connected_channel
  - state: A ConnectionState instance (provided by the client’s internal state)
  - data: WidgetMemberPayload (the payload from the widget data)
  - connected_channel: Optional WidgetChannel indicating the member’s current channel in the widget
  - Activity is derived from the payload’s game data via create_activity
  - display_name returns nick if available, otherwise the username
- Widget
  - Represents the guild widget as a whole
  - Fields include: id, name, channels (List[WidgetChannel]), members (List[WidgetMember])
  - Widgets expose their channels and online members for convenient consumption
  - Note: Widget data is partial; offline members are not guaranteed to appear (Discord limitation)

Important parameters and return values
--------------------------------------
- WidgetChannel
  - __init__(id: int, name: str, position: int)
  - id: Channel ID (snowflake)
  - name: Channel name
  - position: Channel position within the widget’s channel list
  - created_at: datetime derived from the ID
  - mention: str in the form <#id>
  - str(widget_channel): channel name
- WidgetMember
  - __init__(state: ConnectionState, data: WidgetMemberPayload, connected_channel: Optional[WidgetChannel] = None)
  - state: client state container (needed to create user/activity); provided by Nextcord internally
  - data: payload describing the member (nick, status, etc.)
  - connected_channel: the WidgetChannel the member is connected to (optional)
  - display_name: nick if available, else username
  - status: Status enum derived from data
  - activity: BaseActivity or Spotify (if provided via the payload’s game data)
  - deafened, muted, suppress: boolean flags indicating current moderation/mute state
- Widget
  - __init__(state: ConnectionState, data: WidgetPayload)
  - id: Guild ID
  - name: Guild name
  - channels: List[WidgetChannel]
  - members: List[WidgetMember]
  - Equality/str behavior: widget objects support standard comparisons and str returns a meaningful representation (as per class docstring)

Usage examples
--------------
Example 1: Create a WidgetChannel from a payload
```python
# Payload-derived example values
channel_id = 123456789012345678
channel_name = "General"
channel_position = 0

wc = WidgetChannel(channel_id, channel_name, channel_position)
print(wc)            # General
print(wc.mention)    # <#123456789012345678>
print(wc.created_at)   # datetime in UTC
```

Example 2: Create a WidgetMember from a payload
```python
# State object would be provided by Nextcord's internal client state
state = ...  # ConnectionState instance

member_payload = {
    'nick': 'Ally',
    'status': 'online',
    'deaf': False,
    'self_deaf': False,
    'mute': False,
    'self_mute': False,
    'suppress': False,
    'game': {'name': 'Typing', 'type': 0},
    # Additional fields that may be present in a real payload
}

# connected_channel is optional
widget_channel = WidgetChannel(123456789012345678, "General", 0)

wm = WidgetMember(state=state, data=member_payload, connected_channel=widget_channel)
print(wm.display_name)  # Ally
print(wm.activity)      # Activity instance corresponding to the game payload
```

Example 3: Accessing widget data
```python
# widget_payload is the payload obtained from the widget endpoint for a guild
widget = Widget(state=state, data=widget_payload)

# List of accessible channels in the widget
for ch in widget.channels:
    print(ch.name, ch.id)

# List of online widget members
for m in widget.members:
    print(m.display_name, m.status)
```

Edge cases and considerations
------------------------------
- Activity construction:
  - If the payload lacks the game data, activity is None.
  - If game data is present, create_activity(game) is used to instantiate an Activity (or Spotify) object.
- Status handling:
  - Status is converted using try_enum; unknown statuses may result in a default or None depending on the enum behavior.
- Nicknames and display names:
  - display_name prioritizes nick; falls back to username if nick is not set.
- Widget data availability:
  - The Discord widget data is partial; offline members do not appear, as noted by Discord’s widget limitations.
- Data shape stability:
  - WidgetMemberPayload and WidgetPayload are defined in Nextcord’s types; direct dict payloads can be used to construct objects when needed, but typical usage goes through the client’s gateway/state.

Migration notes
--------------
- If you previously consumed widget data via a different path, you should now:
  - Use WidgetChannel for channel representations in the widget.
  - Use WidgetMember to access partial user data and their current activity within the widget.
  - Access the overall widget via Widget, which aggregates channels and online members.
- When parsing widget payloads, map fields to the corresponding class constructors as demonstrated in the usage examples.
- Be mindful of partial data: only online members and specified channels are guaranteed to be present in widget payloads.
