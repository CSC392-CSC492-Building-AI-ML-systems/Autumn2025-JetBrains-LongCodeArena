# Memory alignment

This documentation describes the alignment and SIMD-related diagnostics exposed by the alignment module. The module provides build-time and runtime information that influences how memory is laid out and accessed, including CPU features and SIMD extension support. It is primarily intended for diagnostic and debugging purposes, helping developers understand how NumPy may align data and select optimized paths.

## Overview

The alignment module exposes:

- A small, self-contained configuration snapshot (CONFIG) describing:
  - Compiler information (C, Cython, C++ toolchains)
  - Machine information (host/build, cross-compilation)
  - Build dependencies (BLAS/LAPACK)
  - Python information
  - SIMD extensions support (baseline features and which features are found on the current platform)
- A public API to display this information in different formats:
  - Human-readable YAML (via stdout) when PyYAML is available
  - Fallback JSON if PyYAML is not installed
  - Programmatic access to the data via a Python dictionary

Important note: many values in CONFIG are placeholders substituted during the build process (e.g., "@HOST_CPU@"). The final values reflect the environment used to build NumPy and the runtime environment.

## Public API

- DisplayModes
  - stdout: Print a human-readable representation of the configuration to stdout (preferably YAML if PyYAML is installed).
  - dicts: Return the configuration as a Python dictionary.

- show(mode)
  - Purpose: Display or return the configuration information describing the build and runtime environment related to memory alignment and SIMD features.
  - Parameters:
    - mode: One of 'stdout' or 'dicts' (use DisplayModes.stdout.value or DisplayModes.dicts.value).
  - Returns:
    - If mode is 'dicts', returns a dict representing CONFIG.
    - If mode is 'stdout', prints the configuration (YAML if PyYAML is available, otherwise JSON) and returns None.
  - Errors:
    - Raises AttributeError if an invalid mode is provided.

- _cleanup(d)
  - Internal helper used to recursively remove empty values from dictionaries so that the configuration is concise.

- CONFIG
  - A nested dictionary containing:
    - Compilers: c, cython, c++
      - name, linker, version, commands, args, linker args
    - Machine Information: host, build, cross-compiled
      - cpu, family, endian, system
    - Build Dependencies: blas, lapack
      - name, found, version, detection method, include directory, lib directory, openblas configuration, pc file directory
    - Python Information: path, version
    - SIMD Extensions: baseline, found, not found
      - baseline: built-in baseline features
      - found: features supported by the current CPU
      - not found: features not supported by the current CPU

- _check_pyyaml()
  - Attempts to import PyYAML; used by show(mode=stdout) to print YAML output when available.

## How memory alignment and SIMD information are exposed

- SIMD Extensions
  - The CONFIG entry under "SIMD Extensions" includes:
    - baseline: The set of SIMD features that NumPy assumes as baseline for alignment and vectorized paths.
    - found: Features that NumPy detects as available on the current CPU (e.g., SIMD instruction sets that can be used for aligned memory operations).
    - not found: Features not available on the current CPU.
  - This information helps diagnose which alignment-optimized code paths may be active.

- Build and host information
  - The "Machine Information" section captures host and build CPU characteristics, including endianess and CPU family. This affects how memory alignment is determined and how data is laid out in memory for optimal access patterns.

- Build dependencies and compilers
  - The "Build Dependencies" and "Compilers" sections document the toolchain used to build NumPy (C, C++, cython) and their configuration. This can influence how memory alignment is negotiated during compilation and which intrinsics are enabled.

## Usage examples

- Example 1: Get the configuration as a Python dictionary
```python
from alignment import show, DisplayModes

config = show(DisplayModes.dicts.value)
print(config["SIMD Extensions"]["found"])  # features supported by the current CPU
```

- Example 2: Print a human-readable configuration (YAML if PyYAML is installed)
```python
from alignment import show, DisplayModes

show(DisplayModes.stdout.value)
```

- Example 3: Handle the absence of PyYAML (fallback to JSON)
```python
from alignment import show, DisplayModes

# If PyYAML is not installed, this will print JSON to stdout
show(DisplayModes.stdout.value)
```

## Important considerations and edge cases

- Build-time placeholders:
  - Many fields in CONFIG use placeholders (e.g., "@HOST_CPU@"). These are substituted at build time. The actual values depend on the system and build configuration.
- PyYAML availability:
  - If PyYAML is installed, stdout mode prints YAML for readability. If not, a JSON representation is printed.
- Invalid mode handling:
  - Passing an invalid mode to show() raises an AttributeError with a message listing valid options.
- Edge cases in alignment data:
  - On some builds or environments, certain SIMD features may be reported as found or not found based on runtime detection, CPU capabilities, or cross-compilation results. The accuracy of the “found” list depends on how the detection was performed during the build and at runtime.
