Internals: Code Explanations – Internal organization of NumPy arrays
===============================================================

Introduction
------------
This document explains the core internal concepts that back NumPy arrays, focusing on the ndarray container, its memory layout, data types, and related internal utilities. It also notes ancillary internals present in the same module, such as build-time configuration introspection, which are not directly part of array data structures but are relevant for developers exploring NumPy’s internals.

Overview of key concepts
------------------------
- ndarray: The central array container used to store homogeneous data in a multi-dimensional, contiguous or non-contiguous memory block.
- dtype: Describes the data type of each array element, including kind, itemsize, and metadata.
- strides: The number of bytes to skip to move to the next element along each axis; crucial for mapping between multi-dimensional indices and the underlying 1-D memory buffer.
- flags: Properties that indicate contiguity, ownership of memory, writability, alignment, and other invariants of the array’s memory layout.
- _ctypes and typing: Internal typing helpers and low-level ctypes integration used for efficient interfacing with C-level code and platform-specific type representations.
- Internal type codes: Mappings used by NumPy’s internals to encode dtypes and kinds (e.g., boolean, integers, floating types) for performance-critical paths.
- Build-time introspection: Utilities that reflect how NumPy was built (e.g., compilers and environment) that are present in the same module but are not part of the array data model.

Core types and their relationships
----------------------------------
- ndarray
  - Purpose: A multi-dimensional, homogeneous data container with a shape, dtype, and memory layout defined by strides.
  - Key attributes:
    - shape: Tuple[int, ...] describing the size along each axis.
    - dtype: A dtype object describing the element type.
    - strides: Tuple[int, ...] giving the byte-step between elements along each axis.
    - data buffer: The underlying memory block that stores the elements.
    - flags: A flags object describing contiguity, ownership, writability, and alignment.
    - base: Optional reference to the base object when the array is a view.
  - Important concepts:
    - Contiguity: Whether memory is laid out contiguously in C order (row-major) or Fortran order (column-major).
    - Views vs copies: Slicing and transposing typically create views; .copy() creates an independent array.
    - Alignment and itemsize: Proper alignment and the per-element size affect performance and interoperability with C extensions.

- dtype
  - Purpose: Encapsulates the type information for array elements (e.g., int32, float64, complex128).
  - Key properties:
    - kind: Character code indicating the general category (e.g., 'i' for signed int, 'f' for float, 'b' for boolean).
    - itemsize: Number of bytes per element.
    - metadata: Optional information (endianness, subarray structure, etc.).

- flagsobj
  - Purpose: Internal representation of an array’s memory-related flags.
  - Common flags accessed via arr.flags:
    - C_CONTIGUOUS / F_CONTIGUOUS: Memory layout in C-order (row-major) or Fortran-order (column-major).
    - OWNDATA: Whether the array owns its memory.
    - WRITEABLE: Whether the array’s data can be modified.
    - ALIGNED: Whether the data is properly aligned for efficient access.
  - Relationship: The ndarray exposes a flags object that provides a convenient API for inspecting and reasoning about memory layout.

Internal typing and low-level utilities
---------------------------------------
- _ctypes and platform types:
  - The internals import and use platform-appropriate ctypes to interact with C-level data and to represent pointers or integer indices in a way that matches the underlying platform’s conventions.
  - Example: A note in the internals indicates the use of a platform-specific ctypes subtype for int pointers to satisfy static type checking (mypy) while remaining compatible with the runtime behavior.
- Type aliases and TypeVars:
  - The code defines TypeVars bound to numpy.dtype[Any], generic, and other internal typing constructs to express constraints on functions and templates that operate on arrays.
  - These typing helpers enable more accurate static typing in the internal codepath while maintaining Python’s dynamic behavior at runtime.

Internal type codes for dtypes
------------------------------
- NumPy maintains internal lists of codes corresponding to different dtype kinds (e.g., boolean, unsigned integers, signed integers, floating types, complex types, etc.).
- Internal type codes help route performance-critical operations through specialized implementations depending on the dtype category.

Build-time introspection utilities
----------------------------------
- The module contains a small, generated CONFIG structure and a show(mode=...) function that can display the environment and build information (e.g., compilers, host/build machine details, dependencies).
- Note: This introspection is for developer debugging and does not affect array semantics or runtime behavior for typical usage.

Key functions and their roles
-----------------------------
- show(mode)
  - Purpose: Display NumPy build/configuration information.
  - Parameters:
    - mode: 'stdout' or 'dicts'. Determines how the configuration is outputted.
  - Returns:
    - If mode is 'dicts', returns a dictionary with build information.
    - Otherwise prints to stdout (optionally formatted if PyYAML is available).
  - Usage example:
    - show(DisplayModes.stdout.value)
    - cfg = show(DisplayModes.dicts.value)

- _cleanup(d)
  - Purpose: Recursively remove empty values from a dictionary (used when constructing CONFIG).
  - Usage: Internal utility; not intended for public use.

Usage examples: inspecting ndarray internals
--------------------------------------------
- Basic array introspection
  ```python
  import numpy as np

  a = np.arange(6).reshape(2, 3)
  print("shape:", a.shape)
  print("dtype:", a.dtype)
  print("strides:", a.strides)
  print("nbytes:", a.nbytes)
  print("flags:", a.flags)
  ```

- Non-contiguous view (slicing)
  ```python
  b = a[:, ::2]
  print("b.shape:", b.shape)
  print("b.strides:", b.strides)
  print("C-contiguous:", b.flags.c_contiguous)
  print("F-contiguous:", b.flags.f_contiguous)
  ```

- Fortran-order array
  ```python
  f = np.asfortranarray(a)
  print("Fortran order strides:", f.strides)
  print("F-contiguous:", f.flags.f_contiguous)
  ```

- Raw data size and element size
  ```python
  print("itemsize:", a.dtype.itemsize)
  print("nbytes:", a.nbytes)
  ```

Important considerations and edge cases
---------------------------------------
- Contiguity versus views:
  - Slicing and transposing often produce views with non-trivial strides. Such arrays may be non-contiguous in either C or Fortran order, affecting performance for certain operations.
- Multidimensional strides:
  - Strides define how many bytes to skip to move to the next element along each axis, which is essential when computing memory offsets for indexing.
- Alignment and platform differences:
  - Internal ctypes and pointer representations depend on the platform; static typing hints in the codebase accommodate these differences.
- Zero-sized and 1-D edge cases:
  - Arrays with zero size or 0-th dimension have special handling in terms of shape, strides, and memory layout; ensure code handles these gracefully.
- Build-time introspection:
  - The CONFIG and show() components are intended for debugging and build verification, not for daily array computation. They can help diagnose environment-related issues when developing NumPy or embedding it in other projects.

Notes on the module's scope
---------------------------
- While this documentation centers on internal array organization, the source file also includes utilities for build-info introspection (CONFIG and show) that are part of the same internals module. Those utilities are ancillary to the ndarray data model and are provided for debugging and transparency about the build environment.

Usage context
-------------
- This information is intended for developers contributing to NumPy’s core or building advanced tooling that interacts with NumPy’s internal array representation. For typical numerical tasks, the public ndarray API (shape, dtype, strides, flags, etc.) is sufficient.
