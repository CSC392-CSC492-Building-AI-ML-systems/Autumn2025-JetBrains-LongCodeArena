The Metadata Routing API
========================

Overview
--------

The Metadata Routing API provides lightweight utilities to route and adapt metadata and method calls through wrappers around estimators. This enables nested or composite estimators to expose a clean external API while delegating actual work to internal sub-estimators. The routing layer handles the translation of external method calls or metadata keys to their corresponding internal targets, and can validate or adjust parameters accordingly.

Key components
--------------

- MetadataRouter
  - A central object that describes how external metadata or method calls should be mapped to internal targets. It aggregates one or more MethodMapping entries and can be consulted to determine how routing should be applied for a given estimator graph.

- MethodMapping
  - Represents a single mapping rule from an external symbol (e.g., a method or metadata key) to an internal target path understood by the routing system. A mapping typically pairs an external name with an internal destination (for example, mapping an external "fit" to an internal estimator’s "fit" method).

- process_routing
  - A utility that takes a MetadataRouter and a dictionary of parameters or metadata and applies the routing rules to produce routed, internal-ready parameter structures. It is the primary entry point for converting external calls/metadata into their internal equivalents.

- _routing_enabled
  - An internal helper to determine whether routing should be applied for a given estimator or wrapper. Used to guard routing behavior.

- _raise_for_params
  - An internal validation helper that ensures provided parameters conform to what routing expects. It raises informative errors when invalid parameters are encountered.

Public API reference
--------------------

- MetadataRouter
  - Purpose: Configure and store routing rules between external metadata/method calls and internal targets.
  - Typical usage: Instantiate with a collection of MethodMapping items that describe how to map external symbols to internal destinations.

- MethodMapping
  - Purpose: Describe a single routing rule between an external symbol and an internal target.
  - Typical usage: Create mappings such as external_name -> internal_path to guide process_routing.

- process_routing(router, params)
  - Purpose: Apply the routing rules described by a MetadataRouter to a given parameter/metadata dictionary.
  - Return: A new dictionary containing the routed parameters suitable for consumption by internal estimators.

Internal utilities (not part of the public API)
----------------------------------------------

- _routing_enabled(estimator)
  - Purpose: Check whether routing should be enabled for the given estimator.
  - Note: Internal to the routing system; behavior may depend on estimator configuration.

- _raise_for_params(params)
  - Purpose: Validate parameters against routing expectations and raise descriptive errors on mismatches.
  - Note: Internal to the routing system.

Usage patterns
--------------

Example 1: Basic routing setup
- Objective: Map an external method to an internal target so a wrapper can delegate work to a sub-estimator.

.. code-block:: python

   from sklearn.utils.metadata_routing import MetadataRouter, MethodMapping, process_routing

   # Define a simple mapping: external 'fit' maps to internal 'estimator.fit'
   router = MetadataRouter([MethodMapping("fit", "estimator.fit")])

   # External parameter dictionary (illustrative)
   external_params = {"fit": {"X": X, "y": y}}

   # Apply routing to obtain internal-ready parameters
   routed_params = process_routing(router, external_params)

Example 2: Routing with multiple operations
- Objective: Route multiple external symbols to their internal targets (e.g., predict and score).

.. code-block:: python

   router = MetadataRouter([
       MethodMapping("predict", "sub_estimator.predict"),
       MethodMapping("score", "sub_estimator.score"),
   ])

   external_params = {"predict": {"X": X}, "score": {"y": y}}

   routed_params = process_routing(router, external_params)

Example 3: Validation and guarded routing
- Objective: Validate parameters and ensure routing is only applied when enabled.

.. code-block:: python

   # Suppose we have an estimator wrapped in routing-enabled context
   router = MetadataRouter([...])

   params = {"unexpected_key": 1}

   # _raise_for_params will raise an error for unexpected keys (internal usage)
   try:
       _raise_for_params(params)
   except ValueError as e:
       print("Routing validation error:", e)

Edge cases and considerations
------------------------------

- Fallback behavior: If a symbol has no routing rule defined, its parameter metadata may pass through unchanged or be ignored, depending on the routing configuration.

- Conflicts: If multiple mappings could apply to the same external symbol, define a clear precedence or raise a configuration error to avoid ambiguous routing.

- Disabled routing: Routing can be conditionally disabled for certain estimators; ensure _routing_enabled returns True when routing should be active.

- Nested estimators: When wrapping multi-estimator graphs, carefully define internal paths to avoid conflicts and ensure consistent routing across the graph.

- Validation: Always validate routed parameters with _raise_for_params to catch misconfigurations early.
