# Calculator Interface

Introduction
The Calculator Interface provides a uniform way to explore the parameters used by all models in the SASmodels calculator ecosystem. It exposes both programmatic helpers and a command-line interface to discover which parameters exist across models and which models use each parameter.

Overview
- Core functionality is exposed through a small set of helpers that inspect model metadata.
- A CLI entry point allows listing parameters across all models, with optional filtering by parameter kind and an option to show the models that use each parameter.
- Parameters are retrieved from each model’s metadata, specifically from the kernel parameters associated with a model.

API Reference

1) find_pars(kind=None)
- Purpose: Gather all parameters used by all models, optionally filtering by parameter kind.
- Parameters:
  - kind: Optional string. Filters parameters by their type. If None, no filtering is applied.
    - Typical values (as used by the code): 'volume', 'orientation', 'sld'.
    - Any other value that matches a parameter type in models may also work depending on model metadata.
- Returns: A dictionary mapping parameter names to a list of model names that use that parameter.
  - Type: Dict[str, List[str]]
  - Example: {'scale': ['ModelA', 'ModelB'], 'radius': ['ModelC']}
- Notes:
  - The function iterates over all models returned by list_models(), loads model information with load_model_info(name), and inspects model_info.parameters.kernel_parameters.
  - If kind is None, all parameters are included. If kind is provided, only parameters whose p.type equals kind are included.

2) list_pars(names_only=True, kind=None)
- Purpose: Print parameters (and optionally the models that use them) to stdout.
- Parameters:
  - names_only: If True (default), print only the parameter names. If False, print each parameter with the models that use it.
  - kind: Optional string to filter by parameter type, as described above. If None, no filtering is applied.
- Returns: None. Side-effect is stdout printing.
- Behavior:
  - Retrieves the parameter table via find_pars(kind).
  - If names_only is True: prints a sorted, columnized list of parameter names.
  - If names_only is False: prints lines of the form "parameter: model1, model2, ...".
- Notes:
  - The presentation uses a column-friendly format via the columnize helper from sasmodels.compare.

3) main()
- Purpose: Command-line interface to list parameters across all models.
- Behavior:
  - Parses arguments with argparse.
  - Options:
    - -v / --verbose: If set, show, for each parameter, the models that use it (names_only becomes False).
    - kind (positional, optional): Filter by parameter kind. Accepted values include volume, orientation, sld, none, any. Internally, 'any' and 'none' map to None, effectively disabling filtering.
  - Calls list_pars(names_only=not args.verbose, kind=args.kind) to produce the output.
- Return: Exit status 0 on success; prints to stdout.

Usage and Examples

- List all parameter names across all models:
  - python -m sasmodels.list_pars

- List parameter names in a column (default behavior):
  - python -m sasmodels.list_pars

- List parameters with models that use them (verbose mode):
  - python -m sasmodels.list_pars -v

- List parameters of a specific kind (e.g., volume):
  - python -m sasmodels.list_pars volume

- List parameters of a specific kind with models:
  - python -m sasmodels.list_pars -v volume

Implementation Details and Considerations

- Parameter metadata
  - Each model exposes a set of kernel parameters via model_info.parameters.kernel_parameters.
  - Each parameter object provides:
    - name: The parameter’s name (string).
    - type: The parameter kind (e.g., 'volume', 'orientation', 'sld', etc.).
  - The API relies on these fields to filter and categorize parameters.

- Performance considerations
  - The listing operations iterate over all available models. For large model sets, the command may take noticeable time on first run as model metadata is loaded and cached by the underlying loader.

- Edge cases
  - No models or models without parameters: the output is empty.
  - Unknown parameter kinds: filtering by an unknown kind results in an empty listing.
  - Duplicate parameter names across models: handled by aggregating models per parameter name in the output when not in names-only mode.

- Extensibility
  - New parameter kinds can be introduced in model metadata; the CLI will pick them up as long as they populate p.type.
  - The code structure cleanly separates data collection (find_pars) from presentation (list_pars), making it straightforward to adapt the interface for alternate outputs (e.g., JSON, XML, or a GUI).

Code Snippets (for developers)

- Programmatic listing of all parameters with their models:
  - from sasmodels.list_pars import find_pars
  - pars = find_pars()
  - for name, models in sorted(pars.items()):
  -     print(f"{name}: {', '.join(models)}")

- Programmatic listing of parameters (names only):
  - from sasmodels.list_pars import list_pars
  - list_pars(names_only=True, kind=None)

Notes
- The calculator interface described here is implemented in the sasmodels.list_pars module, which provides both programmatic helpers and a CLI entry point (python -m sasmodels.list_pars).
- The columnize helper is used to present results in a readable, multi-column format when only parameter names are requested. It is imported from sasmodels.compare.

End-user guidance
- Use the default listing to survey what parameters exist across all models.
- Use the verbose option to quickly identify parameter usage across models, which is helpful when diagnosing model compatibility or parameter sharing.
- Use the kind filter to focus on specific parameter categories (e.g., volume-related parameters) during analysis or debugging.
