Array API support
===================

Overview
--------
This module documents how the project interacts with the Array API standard to enable interoperability of array-like inputs across libraries. The goal is to allow estimators to accept array objects that conform to the Array API while preserving NumPy-based internal computations.

Scope and status
----------------
- Status: Experimental/introductory support. Core computations rely on NumPy arrays, with conversion utilities to and from Array API arrays when available.
- Intent: Provide guidance for developers on how Array API inputs are recognized, converted, validated, and used within estimators. Establish expectations for behavior, edge cases, and testing.

Key concepts
------------
- Array API backend: Any implementation that provides array-like objects and the Array API surface (e.g., asarray, zeros, ones, etc.). If present, inputs may be passed through a conversion layer to NumPy for processing.
- Canonical representation: Internally, computations are performed on NumPy arrays. Array API inputs are converted to NumPy arrays before core algorithms run.
- Duck-typing interoperability: The system relies on duck-typing and optional protocol support to detect Array API arrays and apply conversions when necessary.
- Backward compatibility: When Array API support is not available or not applicable to a given input, the code gracefully falls back to NumPy-based processing.

How it works (high-level)
--------------------------
- Input handling: Before computation, inputs are inspected. If an object implements the Array API interface (or is recognized as an Array API array), it is converted to a NumPy array.
- Validation and casting: After conversion, standard validation and casting rules apply (dtype, shape checks, etc.).
- Computation: All algorithms receive NumPy arrays and operate using the existing code paths.
- Output: Results are returned as NumPy arrays. If desired, downstream code can convert back to an Array API array.

Supported backends and compatibility
-----------------------------------
- Primary backend: NumPy arrays are the canonical representation for internal computations.
- Array API backends: If an Array API implementation is available in the environment, inputs may be converted automatically. The project does not require a specific backend and aims to remain backend-agnostic.
- Sparse, sparse-like, and mixed inputs: Behavior depends on whether a dedicated conversion path exists. When not supported, inputs should be converted to dense NumPy arrays if feasible.

Important considerations and edge cases
---------------------------------------
- Type and dtype handling: Array API arrays may have dtypes that differ from NumPy defaults. Conversions should cast to a compatible NumPy dtype, with explicit attention to precision and range.
- Contiguity and memory layout: Post-conversion arrays should adhere to NumPy expectations (usually C-contiguous). If non-contiguous, a copy may be performed to ensure compatibility.
- Performance: Conversion incurs overhead. For large datasets, consider buffering strategies or performing preprocessing to minimize conversions.
- Partial support: Some Array API operations (beyond simple conversions) may not be fully supported. In such cases, inputs should be converted early to NumPy to ensure deterministic behavior.
- edge case inputs: Nested structures, masked arrays, or unusual shapes should follow the same validation rules as NumPy inputs after conversion.

Usage examples
--------------

Example 1: Accepting an Array API array and fitting a model
----------------------------------------------------------
This example demonstrates how an Array API array can be used as input, with automatic conversion to NumPy for computation.

```python
# Pseudo-example: Array API array input
try:
    import array_api as ar  # hypothetical Array API backend
    have_array_api = True
except Exception:
    have_array_api = False

import numpy as np
from sklearn.linear_model import LinearRegression

# Create input features (Array API array if available, otherwise NumPy array)
X = ar.asarray([[1.0, 2.0], [3.0, 4.0]]) if have_array_api else np.array([[1.0, 2.0], [3.0, 4.0]])
y = ar.asarray([1.0, 2.0]) if have_array_api else np.array([1.0, 2.0])

# Fit the model (internal conversion to NumPy occurs if needed)
model = LinearRegression()
model.fit(X, y)
```

Example 2: Converting to NumPy explicitly (when needed)
------------------------------------------------------
If automatic detection is not available, explicitly convert input data before passing to estimators.

```python
import numpy as np
try:
    import array_api as ar
    X_api = ar.asarray([[5.0, 6.0], [7.0, 8.0]])
    X = np.asarray(X_api)
except Exception:
    X = np.asarray([[5.0, 6.0], [7.0, 8.0]])

# Use with a scikit-learn estimator
from sklearn.ensemble import RandomForestRegressor
model = RandomForestRegressor(n_estimators=100, random_state=0)
model.fit(X, np.asarray([1.0, 2.0]))
```

Example 3: Back-and-forth with downstream Array API consumers
-------------------------------------------------------------
If you produce results intended for a downstream Array API consumer, you may convert back:

```python
import numpy as np
from sklearn.metrics import mean_squared_error

# Suppose y_true and y_pred are NumPy arrays
y_true = np.array([1.0, 2.0, 3.0])
y_pred = np.array([1.2, 1.9, 3.1])

mse = mean_squared_error(y_true, y_pred)

# Optional: convert to Array API array if needed by downstream
try:
    import array_api as ar
    mse_api = ar.asarray(mse)
except Exception:
    mse_api = mse
```

Developer guidance
------------------
- Detection and conversion: Implement or rely on a lightweight conversion utility that checks for an Array API-like interface and performs a safe conversion to NumPy arrays.
- Validation: After conversion, continue to apply existing input validation and shape checks as if the inputs were NumPy arrays.
- Testing: Add tests that cover:
  - Pure NumPy inputs (no conversion path taken)
  - Array API inputs that convert to NumPy
  - Edge cases with dtypes and shapes
  - Behavior when an Array API backend is unavailable
- Documentation touchpoints: Document any new utilities or helper functions introduced for Array API support, including expected input types, return types, and error messages.
