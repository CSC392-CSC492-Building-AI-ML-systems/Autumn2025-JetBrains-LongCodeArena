# timer.rst — Using timers

Introduction
- Timers provide lightweight mechanisms to measure elapsed time and to schedule functions to run after a delay or at regular intervals. They help you implement time-based logic without blocking the main thread and with clear, testable semantics.

Overview
- Key concepts:
  - One-shot timers: schedule a callable to run once after a specified delay.
  - Stopwatches: measure the duration of a code block or operation.
  - Repeating timers (pattern): execute a callable at regular intervals until explicitly canceled.
  - Lightweight timing utilities: decorators and context managers to easily capture timing information.

API and concepts
- Timer (one-shot)
  - Purpose: execute a callable after a specified delay.
  - Typical methods:
    - start(): begin the countdown and execute the function when the delay elapses.
    - cancel(): prevent the scheduled function from running.
    - is_alive(): indicate whether the timer is still waiting to fire.
    - join(): wait for the timer thread to finish (if implemented with a thread).
  - Common usage:
    - Schedule a function to run after a delay without blocking the current thread.

- Stopwatch (block timing)
  - Purpose: measure how long a block of code takes to execute.
  - Typical pattern:
    - Context manager for automatic start/stop.
    - Property elapsed (or similar) to read the duration in seconds.
  - Common usage:
    - Wrap a well-defined block and obtain the elapsed time for reporting or optimization.

- Timeit utilities (decorators and context managers)
  - Purpose: capture execution time for functions or blocks in a concise way.
  - Common patterns:
    - Decorator: timeit on a function to log or return timing information.
    - Context manager: time a code block with a with-statement for quick timing.

Important considerations
- Clock sources and accuracy
  - Use monotonic clocks (where available) to avoid issues with system clock changes.
  - Be aware that thread scheduling and OS behavior can affect scheduling accuracy for short delays.
- Threading and resources
  - Timers often run in background threads; ensure callbacks are thread-safe.
  - Daemon threads may prevent program termination if not managed carefully.
- Exceptions in callbacks
  - Exceptions raised in timer callbacks should be logged and handled to avoid silent failures.
- Integration with async code
  - For asyncio-based apps, prefer asyncio-compatible timers (call_later, call_at) or an event loop integration.
  - The timer module described here can be used standalone or as a simple bridge in synchronous code paths.

Usage examples
- One-shot timer
```python
# Schedule a function to run after 5 seconds
from timer import Timer

def greet(name):
    print("Hello, {}".format(name))

t = Timer(5.0, greet, args=("Alice",))
t.start()
# Optionally cancel before it fires:
# t.cancel()
```

- Stopwatch (block timing)
```python
# Timing a block of code
from timer import Stopwatch

with Stopwatch() as sw:
    # code to measure
    total = 0
    for i in range(1000000):
        total += i

print("Elapsed: {:.6f} seconds".format(sw.elapsed))
```

- Recurring timer (pattern)
```python
# Simple recurring timer pattern (reschedules itself)
from timer import Timer

def tick():
    print("tick")
    global t
    t = Timer(1.0, tick)
    t.start()

t = Timer(1.0, tick)
t.start()
# To stop:
# t.cancel()
```

- Timing a function with a decorator (timeit)
```python
# Decorator-based timing
from timer import timeit

@timeit
def compute():
    sum(i * i for i in range(100000))

compute()
# The decorator can log or return timing information depending on implementation.
```

- Timing with a context manager (timeit context)
```python
# Context-manager timing for a code block
from timer import timeit as timeit_ctx

with timeit_ctx() as t:
    # code to time
    for _ in range(100000):
        pass

print("Elapsed: {:.6f} seconds".format(t.elapsed))
```

Edge cases and best practices
- Use long-running timers with care; cancel timers when no longer needed to avoid unnecessary callbacks.
- For high-precision timing, prefer monotonic clocks if your timer implementation exposes them (e.g., time.perf_counter or time.monotonic-equivalent).
- When timing very fast operations, be mindful of overhead introduced by the timer mechanism itself.
- In multi-threaded scenarios, ensure the callback logic is thread-safe and does not introduce race conditions.
