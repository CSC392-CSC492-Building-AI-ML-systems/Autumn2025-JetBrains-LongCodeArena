# Using custom attributes

Custom attributes provide a flexible mechanism to store user-defined metadata on ftrack entities. They are defined on the server and attached to entity types (such as Asset, Shot, Task, etc.). The Python API allows reading, updating, and persisting values for these attributes, while the server handles validation, permissions, and storage.

## Overview

- Custom attributes are defined via attribute definitions and attached to specific entity types.
- Each entity instance can hold values for its defined custom attributes.
- The session object can be configured to automatically populate attribute values on access (auto_populate).
- Changes to custom attribute values are persisted through the session (typically via commit).

Key concepts
- Attribute definition: Metadata describing a custom attribute (name, type, allowed values, description, target entity type, etc.).
- Custom attribute value: The actual data stored on an entity for a given attribute definition.
- Entity: ftrack objects (e.g., Asset, Shot, Task) that can host custom attributes.

## Accessing and mutating custom attribute values

This section covers common patterns for reading and writing custom attribute values on entities.

### Reading values

- If auto-population is enabled on the session, accessing a custom attribute value may trigger a server fetch.
- Values are retrieved from the entity using a key corresponding to the attribute name.

Code example:
```python
# Assume `entity` is a fetched ftrack entity (e.g., a Shot, Asset, or Task)
# Read a custom attribute named 'estimated_hours'
estimated_hours = entity.get('estimated_hours')  # returns None if not set
# Alternatively, if the attribute is guaranteed to exist, direct access may be used
# estimated_hours = entity['estimated_hours']
```

### Updating values

- Assign a Python value to the attribute key on the entity.
- The value must conform to the attribute definition (type, allowed values, etc.).

Code example:
```python
# Set a custom attribute on the entity
entity['estimated_hours'] = 6.5
```

### Persisting changes

- To persist changes to the server, commit the session after modifying attribute values.

Code example:
```python
# Persist changes to the server
session.commit()
```

## Working with attribute definitions

Custom attributes are defined on the server. The Python API can be used to inspect and, where supported, create or modify attribute definitions.

Notes:
- Attribute definitions control the type (e.g., string, integer, float, boolean), constraints, and the set of entity types that can host the attribute.
- Creating new definitions and attaching them to an entity type is typically performed via server-side tooling or API calls that create an AttributeDefinition and associate it with a target entity type. Refer to the server/API documentation for the exact workflow and permissions.

Code sketch (conceptual; exact API names may vary):
```python
# Conceptual example: defining a new custom attribute (pseudo-API)
from ftrack_api.attribute import AttributeDefinition

attr_def = AttributeDefinition(
    name='estimated_hours',
    attribute_type='float',
    description='Estimated effort in hours',
    entity_type='Shot'  # or the appropriate target entity type
)

session.create(attr_def)
session.commit()
```

## Querying and listing attributes

You can inspect available custom attributes and their definitions to understand what attributes exist for a given entity type.

Code example (conceptual):
```python
# List attribute definitions for a given entity type
definitions = session.query_attribute_definitions(entity_type='Shot')
for d in definitions:
    print(d.name, d.attribute_type, d.description)
```

Or, once you have an entity, you can enumerate its defined attributes and their current values:
```python
for name in entity.get_attribute_names():  # hypothetical helper
    value = entity.get(name)
    print(f'{name}: {value}')
```

## Important considerations and edge cases

- Type safety: Ensure values assigned to custom attributes conform to the attribute definition (type, range, allowed values).
- Missing attributes: Accessing an attribute that is not defined for the entity type may raise an error or return None, depending on the API version.
- Permissions: Write access to custom attributes requires appropriate permissions on the entity and the attribute definition.
- Auto-population: Enabling auto_populate on the session can simplify development but may increase initial fetch time and server load.
- Schema caching: If schema caching is enabled, attribute definitions may be cached locally. Be mindful of schema changes and cache invalidation during development or server upgrades.
- Persistence: Changes are not sent to the server until you commit the session. Plan batching of updates when performing mass attribute edits.

## Usage scenarios

- Tracking custom metadata (e.g., estimated hours, risk level, review status) on production entities.
- Extending built-in entity types with project-specific fields without modifying the core schema.
- Building UI tools that present custom attributes in dashboards or reports.

## Examples

- Example 1: Read a custom attribute value from a Shot
```python
# Retrieve a Shot entity (implementation dependent)
shot = session.query('Shot').filter_by(id='123').one()

# Read the custom attribute 'estimated_hours'
estimated_hours = shot.get('estimated_hours')
print('Estimated hours:', estimated_hours)
```

- Example 2: Update and persist a custom attribute value
```python
# Update a custom attribute on an entity
shot['estimated_hours'] = 8.0

# Persist changes to the server
session.commit()
```

- Example 3: Inspect available custom attributes for an entity type (conceptual)
```python
# Retrieve attribute definitions for Shots
definitions = session.query_attribute_definitions(entity_type='Shot')
for d in definitions:
    print(f'{d.name} ({d.attribute_type}): {d.description}')
```
